## freee での経験（5 分・深掘りされたら）

### 構成

1. プロダクトの概要 + Ken の役割
2. 印象に残るプロジェクトを 1 つ（STAR で）
3. そこで学んだ設計・運用の教訓
4. 反省点（弱みを正直に）

### 草案（STAR で印象に残るプロジェクトを話す）

**Situation:** freee で会計の中の特定機能を扱うチームにいました。
[STAR ストーリー本体は `aspects/job/search/interviews/technical-interview-cheatsheet.md` または `resilire/prep.md` に既存版があれば流用]

**Task:** [具体的なタスク]

**Action:** [取った行動]

**Result:** [成果 + 学び]

### kickflow に活きるポイント

> freee の経験で特に活きそうなのは 3 つ。
>
> 1. **Rails の大規模コードベースでの設計判断** — 巨大な ActiveRecord モデルが既にあるところで、新機能をどう乗せるか・どう切り分けるかをずっと考えてきた
> 2. **エンタープライズ顧客の要望と PdM の調整** — 個別要望をプロダクトに取り込むかどうかの判断、何度もやった
> 3. **本番障害の経験** — N+1、ロック、Sidekiq の詰まりは一通り踏んできて、再発防止の仕組みも入れてきた

### 弱み（聞かれたら）

> 自分でも自覚があるのは、テーブル設計の最初の解像度が甘くなりがちなところです。正規化や将来の拡張を後から「あ、こう持たせておけばよかった」となるパターン。
>
> 対策として、Design Doc を書く習慣を維持して、変更コスト・異常系・スケール時の挙動を 3 点セットでレビュー観点に入れるようにしています。最近は AI に「この設計で 5 年後の負債は何？」を聞いてセルフレビューする形も組み合わせています。

### ポイント

- 弱みは「自覚 + 対策 + 進行中」のセットで答える
- kickflow が「spec 駆動開発」を掲げているので、Design Doc 文化との親和性をさりげなく示す

## life OS の 2 年（深掘りされたら）

### 構成

1. なぜやったか（freee の苦労 + AI の波）
2. 何をやったか（具体的な仕組み）
3. 何を学んだか（kickflow に持ち込めるもの）

### 草案

> freee を辞めたあと、2 年弱「life OS」という個人プロジェクトに集中していました。きっかけは 2 つで、1 つは freee で「自分の働き方を自分で設計する」感覚を持てなかった反省、もう 1 つは Claude Code が出てきて「AI と一緒に開発する」のが現実になってきたタイミングだったからです。
>
> 内容は、自分の生活全体（食事・運動・学習・教会・恋愛まで）を Git リポジトリで管理して、Claude Code をエージェントとして走らせる仕組みです。Skill という単位で手順を切り出して、Hook で自動化して、ルールファイルで Claude の挙動を制御する。普通の SaaS を作っている感覚と本質は変わらないです。
>
> やっていて学んだのは、**AI と協働するときに一番大事なのは「ルールを書ききること」と「失敗パターンを覚えさせる仕組み」**だということ。これは kickflow さんが Claude Max や Devin を入れて生産性を上げようとしている文脈と、まったく同じ問題意識だと思います。

### ポイント

- 「2 年ブランク」を「AI-assisted SE の craft を 2 年積んだ」とリフレーミング
- kickflow の AI ツールの個別名（Claude Max / Devin / CodeRabbit / Dify）を覚えておく
- 「ルールを書ききる」「失敗パターン」を強調 → kickflow のエンジニアが実際に困っているはずの領域

## 逆質問（5 個用意、3 個に絞って聞く）

カジュアル面談で MUST 4 個は既に聞いたので、**EM 面接向け = 技術寄りに振った逆質問** を用意する。
重複を避けて、EM だからこそ聞ける内容に。

### 強い質問（kickflow 特有・EM 向け）

1. **EM ご自身は今、チームのどんな課題に一番時間を使っていますか？**
   - 直属の上司の関心領域を知る → 入社後すり合わせの起点
2. **AI ツールを全社で導入されていますが、エンジニアの設計判断やコードレビューの粒度はどう変わりましたか？逆に AI に任せない部分はどこに線を引いていますか？**
   - kickflow の AI 1st 文化への深い関心を示す + EM が一番考えている領域
3. **稟議ドメインで、過去 1 年で一番難しかった設計判断は何でしたか？（フローの複雑化、テナント分離、検索など）**
   - 技術への興味 + EM 自身の経験を引き出す
4. **エンタープライズ顧客の個別要望が来たとき、プロダクトに取り込むかどうかの判断プロセスはどうなっていますか？**
   - freee で苦労したポイント、kickflow の決め方を知りたい
5. **「機能数が足りていない」とカジュアル面談で伺いましたが、EM の立場から「今期一番ここを伸ばしたい」という機能領域はどこですか？**
   - カジュアル面談の発言と接続 → 記憶力 + 関心の継続を示す

### ポイント

- カジュアル面談で聞いた話を引用する逆質問は強い印象を残す（質問 5）
- EM だからこそ答えられる質問を選ぶ（質問 1, 2, 3）
- 1 個は「自分が入った後の話」を聞く（質問 4）
