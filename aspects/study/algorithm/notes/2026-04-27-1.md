# 3. Longest Substring Without Repeating Characters

## 🧒 一言で言うと

Sliding Window 可変長の最も典型な問題。`map[byte]int` で出現回数を持ち、重複が出たら left を進めて回数を減らす。Window が valid になった後で長さを更新する。

## 🎯 今日の目標・疑問

- Sliding Window 可変長の型を Go で書けるか
- left/right ポインタの動かし方と「いつ best を更新するか」を理解できているか

## 📝 ノート

### 自分の解（count 版）

```go
func lengthOfLongestSubstring(s string) int {
    count := make(map[byte]int)
    l, longest := 0, 0
    for r := 0; r < len(s); r++ {
        count[s[r]]++
        for count[s[r]] > 1 {
            count[s[l]]--
            l++
        }
        longest = max(longest, r-l+1)
    }
    return longest
}
```

### 詰まったポイントと修正

1. **`l = r = 0` は Go では書けない** — Python/C 風の連鎖代入は不可。`l, r := 0, 0` を使う
2. **`map[string]int` ではなく `map[byte]int`** — `s[r]` は byte 型（string アクセスの戻り値）。string に変換すると無駄
3. **`while` キーワードは Go にない** — `for` が全部兼ねる。条件のみの形 `for cond { ... }` で書く
4. **`r++` の位置** — `longest` 更新の**前**だと、ウィンドウ長さの式 `r-l+1` がズレる（r が既に進んでるので実体は `r-l`）。**inner for を抜けた直後・r++ の前**に best を更新するのが正しい順序

### Go の落とし穴と覚えどころ

| トピック | 要点 |
|---|---|
| string の index | `s[i]` は byte（ASCII OK・Unicode は `[]rune(s)` 経由） |
| `'a'` リテラル | rune 型（int32）。byte 比較なら `byte('a')` 明示 or 型推論で OK |
| map の zero value | 存在しないキーは zero value を返す → `count[k] > 1` で直接判定可 |
| 空 map 作成 | `make(map[K]V)` が慣習。`map[K]V{}` も同等だがリテラル用 |
| `Math.max` | Go 1.21+ の **built-in `max(a, b)`** が使える（`math.Max` は float64 専用） |

### 計算量

- **時間: O(n)** — r も l も最大 n 回しか進まない（amortized）。各文字は count に 1 回入って 1 回出る
- **空間: O(min(n, Σ))** — Σ は文字種数。ASCII なら O(128) = O(1)

### Sliding Window 可変長の型

```go
for r := 0; r < n; r++ {
    /* window に s[r] を追加 */
    for /* 制約違反の間 */ {
        /* window から s[l] を取り除く */
        l++
    }
    /* best を更新（window が valid な状態） */
}
```

「外で右を伸ばし、内で左を縮め、抜けたら best を更新」の3拍子。

### 別解1: last-seen-index 版

重複を見つけたら left を**ジャンプ**させる別バリアント:

```go
func lengthOfLongestSubstring(s string) int {
    lastSeen := make(map[byte]int)
    l, longest := 0, 0
    for r := 0; r < len(s); r++ {
        if idx, ok := lastSeen[s[r]]; ok && idx >= l {
            l = idx + 1
        }
        lastSeen[s[r]] = r
        longest = max(longest, r-l+1)
    }
    return longest
}
```

- 利点: inner for がなく、各 r で O(1) 処理 → 確実に O(n)
- 欠点: 「at most K distinct characters」など発展形に**拡張しにくい**

count 版は型が綺麗で拡張性が高い。面接では count 版から書き始め、最適化として last-seen 版を提示するのが王道。

### 別解2: ASCII 特化（配列版）

問題制約が ASCII のみなら map → 配列に変えると定数倍が速い:

```go
func lengthOfLongestSubstring(s string) int {
    var count [128]int
    l, longest := 0, 0
    for r := 0; r < len(s); r++ {
        count[s[r]]++
        for count[s[r]] > 1 {
            count[s[l]]--
            l++
        }
        longest = max(longest, r-l+1)
    }
    return longest
}
```

| | map 版 | 配列版 |
|---|---|---|
| 書きやすさ | △ delete 必要なケースあり | ◎ zero に戻すだけ |
| 速度 | ハッシュ計算オーバーヘッド | メモリ直アクセス・キャッシュ効率◎ |
| 汎用性 | Unicode OK | ASCII 限定 |

## 🔑 キーワード

- Sliding Window（可変長）
- 2 ポインタ（left / right）
- byte vs rune
- map vs 配列（`[128]int`）
- map の zero value
- built-in `max` / `min`（Go 1.21+）
- amortized O(n)

## 💡 まとめ

1. **可変長 Sliding Window の型は3拍子**: 右で入れる → 内で出す → 抜けて更新
2. **best 更新は inner for を抜けた後**（window が valid な状態でのみ）
3. **`map[byte]int` の zero value を活用**して存在判定を省略
4. **文字種が固定なら配列版が定数倍速い**（`[128]int` が ASCII の正解）
