# 出力前チェックリスト

`aspects/job/` 配下のファイルを書き終わったら、出力前に以下 7 項目を確認する。漏れがあれば書き直す。

```
□ 1. 「担当」「実現」「貢献」「主導」「最大化」「組み込み」「一貫して」「横断的に」が同段落に 2 回以上出ていない
□ 2. 「〜することができます」→「〜できます」に短縮済み
□ 3. 全角ダッシュ（——）→ カッコ or 読点に置換済み
□ 4. 「**ラベル:** 内容」形式の箇条書きが残っていない
□ 5. AI 臭フレーズを削った箇所に「具体事実」が埋まっている（削るだけで終わっていない）
□ 6. 数字は実測 or 根拠あり（盛っていない、面接で深掘りされて答えられる）
□ 7. ですます調は保てている（カジュアルに振りすぎていない、媒体は履歴書）
```

---

## 各項目の補足

### 1. 空疎動詞の連発

`patterns.md` C1 参照。同段落で 2 回以上出たら 1 つは具体動作に書き換える。

### 2. 「〜することができます」短縮

`patterns.md` A14 参照。grep ですべての出現を検出して短縮できる。

### 3. 全角ダッシュ

`patterns.md` A7 参照。`——`（U+2014 連続）が NG。`—`（単独 em dash、`**bold** — 補足` 形式）は OK。

### 4. 太字 + コロン箇条書き

`patterns.md` A5 参照。`- **〇〇:** 〇〇` の形式を全削除。ラベルなしで内容だけ書く。

### 5. 削るだけで終わっていない

これが最重要。「削る + 埋める」のペアで完結しているか。**削っただけだと履歴書としての規模感・難易度が消える**。

確認方法: 削除した AI 臭フレーズに対して「これで業務の規模感が伝わるか?」を自問する。伝わらないなら、その場所に具体事実（数字・固有名詞・状況描写）を埋める。

### 6. 数字の根拠

**面接で「これどうやって測りました?」と聞かれて 5 秒で答えられない数字は使わない**。

| 数字 | 根拠の例 |
|---|---|
| レスポンスタイム | Datadog / New Relic / 自社モニタリング |
| ビルド時間短縮 | GitHub Actions ログの before/after |
| churn 率改善 | Redash / Mixpanel の数値 |
| 離脱削減 | Google Analytics の funnel |

確認できない場合は数字を削る。「向上」「改善」で逃げず、別の角度の具体事実で埋める。

### 7. ですます調保持（媒体性ガード）

履歴書・職務経歴書では「思ってます」「感じてます」は NG。「考えています」「目指しています」程度の温度感に留める。

媒体別の目安は `SKILL.md` の「媒体別の注意」表参照。

## post-check モード（`/humanize <file>`）での適用

既存ファイルに対しては、上記 7 項目を **検出ロジック** として使う:

1. 項目 1〜4 は機械的に grep / pattern match で検出可能
2. 項目 5〜7 は文脈判断が必要、Claude が読んで判定する
3. ヒットした箇所を「Before / 削った理由（項目番号） / After + 具体事実」の 3 列で diff 提示
4. ユーザー承認後に Edit 適用
