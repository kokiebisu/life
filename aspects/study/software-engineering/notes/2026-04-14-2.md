# 勉強 - 2026-04-14

## 🎯 今日の目標・疑問

ユニークID生成のスループット計算・64bitの設計上の意義・BASE62 vs 数値IDのトレードオフを理解する

## 📝 ノート

### マルチマスターレプリケーションによるID生成

複数のDBマスターがそれぞれ異なるステップで auto_increment を発行する方式。

```
マスター3台の場合:
  1号機: 1, 4, 7, 10 ...  （+3ずつ）
  2号機: 2, 5, 8, 11 ...
  3号機: 3, 6, 9, 12 ...
```

既存の auto_increment の仕組みをそのまま使えるため追加実装がほぼ不要。

**非推奨の理由:**
- スケールが硬直する：マスター台数を変えるたびに全台の step 設定を変更する必要がある
- 時系列ソート不保証：タイムスタンプを含まないのでIDから生成順がわからない
- 複数DC展開が難しい：ネットワーク遅延で step 管理が複雑になる

| | マルチマスターレプリケーション | Snowflake |
|--|------|------|
| 実装コスト | 低（既存DBの設定のみ） | 専用サービスが必要 |
| スケールの柔軟性 | ❌ 台数変更が困難 | ✅ マシンIDを増やすだけ |
| 時系列ソート | ❌ 保証なし | ✅ タイムスタンプ入り |

> 面接では「この方式も検討したが〇〇の理由で採用しなかった」と棄却理由を述べると設計の幅広さが伝わる

### ソート可能性がDB設計に影響する理由

DBインデックス（B-tree）はソートされた順で管理されているため、単調増加IDは常に末尾追記のみ→高速。
ランダムID（UUID v4）は挿入のたびにB-treeのページ分割が発生して書き込みが遅くなる。

```
連番/Snowflake: [1][2][3][4][5] ← 末尾追記のみ
UUID v4:        挿入位置がバラバラ → ページ分割多発
```

### 連番 vs タイムスタンプ系のトレードオフ

| 方式 | ソート可 | 分散対応 | 弱点 |
|------|---------|----------|------|
| auto increment | ✅ 完全単調増加 | ❌ 単一DB必須 | スケールしない |
| Snowflake | ✅ 概ね順序保証 | ✅ | クロックスキュー |
| UUID v4 | ❌ | ✅ | DB性能劣化・長い |

### 数字のみ vs 英数字（BASE62）のトレードオフ

| | 数字のみ（BIGINT） | BASE62（英数字） |
|--|-------|---------|
| DBでの型 | `BIGINT`（8バイト固定） | `VARCHAR`（可変） |
| 変換コスト | なし | 生成→変換→保存→比較 の手順が増える |
| インデックス効率 | ✅ 整数型は速い | 文字列型は遅い |
| 向いているシステム | システム内部ID | URL等で外部露出するID |

**BASE62 vs UUID の比較（どちらも英数字文字列）**

BASE62 はエンコード方式、UUID v4 はID生成アルゴリズム。比較するなら「Snowflake（BASE62表現）vs UUID v4」が正確。

| | BASE62（Snowflake IDをエンコード） | UUID v4 |
|--|------|------|
| 元のサイズ | 64bit | 128bit |
| 文字数 | 約11文字 | 36文字（ハイフン込み） |
| 中身 | タイムスタンプ+マシンID+連番 | 完全ランダム |
| ソート可 | ✅ 時系列順 | ❌ |
| 一意性の保証 | 設計で保証（衝突ゼロ） | 確率的（128bitの乱数） |

UUID に BASE62 エンコードを適用すれば約22文字に短縮できるが、ソートできない・長い欠点は変わらない。

**BASE62変換のコスト（Chapter 7 で採用しない理由）**
- 生成（数値）→ BASE62変換 → 保存（文字列）→ 比較・ソートも文字列 → 必要なら数値に戻す
- 毎秒数百万ID生成するシステムでは変換コストが積み重なる
- URLに露出しないなら `BIGINT` のままが最適

→ **Chapter 7（内部ID）は BIGINT、Chapter 8（URL Shortener）は BASE62** と設計方針が分かれる理由がここにある

### IDの長さと64bitの意義

**スループット計算（Snowflake）**
- 12bit sequence → 1msに 2^12 = 4,096個/マシン
- 1秒 = 1,000ms → 約400万個/秒/マシン
- 最大1,024台（10bit マシンID）→ 全体で約42億個/秒

**64bit以内に収まる意義**

| 観点 | 内容 |
|------|------|
| DB | BIGINTと対応。追加設定ゼロ、固定8バイト |
| JS | Number型は53bit精度 → 64bitをそのまま渡すと末尾が変わる |
| CPU | レジスタサイズと一致 → ビット演算が1命令で完結 |
| 寿命 | 41bitタイムスタンプで約69年分（エポックをシステム開始年に設定） |

```go
// Snowflake IDの組み立てはビットシフトとORだけ
id := (timestamp << 22) | (machineID << 12) | sequence
```

**JavaScriptの53bit問題**
- TwitterがAPIで `id`（数値）と `id_str`（文字列）を両方返していた理由
- JSON経由でIDを渡す設計では文字列として扱うか53bit以内に収める必要がある

### IDの長さの考慮観点（抱えられるID数以外）

1. **DBインデックスサイズ**：長いIDはインデックスが膨らみメモリに乗らなくなる
2. **ネットワーク転送**：大量のJSONレスポンスに含まれると塵積も
3. **CPU効率**：64bit = レジスタサイズ一致で演算が高速
4. **人間の視認性**：ログでIDを目視するとき（障害調査・サポート）
5. **衝突確率**：設計で管理できれば冗長な長さは避ける

### マシンID変更が発生するシチュエーションと対処

**発生するケース:**
- オートスケーリング（AWS EC2・Kubernetes でインスタンスが動的に増減）
- マシン障害・交換
- データセンター移行

**なぜ大変か:** IDの再利用が危険。廃止したマシンIDを新マシンに割り当てると、クロックスキューと組み合わさって過去のIDと衝突するリスクがある。

**対処法:**

| 方法 | 概要 | 向いている環境 |
|------|------|------|
| ZooKeeper動的割り当て | 起動時にIDをリクエスト・停止時に返却 | オートスケーリング環境 |
| 静的設定ファイル | 各マシンに固定IDを設定 | 台数固定の小規模環境 |
| IPアドレスから導出 | IPの末尾をIDとして使用 | ❌ クラウドでは再起動でIP変化のリスク |

Snowflakeの10bitマシンIDは「固定した少数台で動かす」前提の設計。クラウドネイティブ環境ではZooKeeperなどの調整サービスとセットで使うのが現実解。

**ZooKeeper は SPOF にならない？**
ZooKeeper 自体がクラスター構成（アンサンブル）で動くため SPOF にはならない。奇数台で構成し、過半数（Quorum）が生きていれば継続動作する。

```
3台構成 → 1台落ちても動く（2/3生存）
5台構成 → 2台落ちても動く（3/5生存）
```

ZooKeeperクラスターごと落ちた場合は、SnowflakeサーバーがマシンIDをメモリにキャッシュしているため、再起動しない限り継続動作できる。

現代では etcd（Kubernetes）・Consul（HashiCorp）が代替として使われることも多いが、同じ「分散合意（Quorum）」の考え方で SPOF を避けている。

### StatefulSet と ULID（Kubernetes 環境での代替）

**StatefulSet**
Kubernetes の Pod 管理方式。Deployment と違い、各 Pod に固定の識別子がつく。

```
Deployment:   pod-a8f3x, pod-b2q9k ... ランダム名・使い捨て
StatefulSet:  pod-0, pod-1, pod-2  ... 固定連番・再起動しても同じ名前
```

Pod名から数字を取り出してマシンIDとして使えるため、ZooKeeper不要でSnowflakeを運用できる。DB（MySQL・Cassandra）・Kafka・Elasticsearchなど「ノードに個性があるサービス」全般に使われる。

**ULID（Universally Unique Lexicographically Sortable Identifier）**
128bit を BASE32 で26文字にエンコード。マシンID不要でソート可能。

```
構造: [48bit タイムスタンプ ms精度][80bit ランダム]
例:    01ARZ3NDEKTSV4RRFFQ69G5FAV
```

| | Snowflake | UUID v4 | ULID |
|--|------|------|------|
| ビット数 | 64bit | 128bit | 128bit |
| マシンID | 必要 | 不要 | 不要 |
| ソート可 | ✅ | ❌ | ✅ |
| K8s対応 | 要工夫 | ✅ | ✅ |
| 文字数 | 約11文字(BASE62) | 36文字 | 26文字 |
| 衝突保証 | 設計で保証 | 確率的 | 確率的（80bitランダムで実質不可能） |

**ULID の 80bit がランダムである理由**
increment にすると複数マシンでカウンターを共有する調整が必要になる → ZooKeeper → Snowflake と同じ問題に戻る。
ランダムにすることで「調整なし」を実現している。

80bit ランダムの衝突確率（誕生日問題）:
- 同一ミリ秒に 50% 衝突が起きるには約 2^40 ≈ 1兆個 必要 → 現実には起きない

**UUID v7 はハイブリッド（increment + ランダム）**

```
[48bit タイムスタンプ][12bit 単調増加カウンター][62bit ランダム]
```

同一ミリ秒内では 12bit のカウンターを increment し、オーバーフローしたらランダム部で吸収する。
1マシン内の単調増加を保証しつつ、マシン間の調整は不要。

**選び分け:**
- Snowflake → 超高スループット・衝突ゼロ保証・台数固定環境
- ULID → K8s上のオートスケーリング環境・実装シンプル・ソートも必要
- UUID v7 → ULID より同一ms内の順序が重要な場合・標準化済み（RFC 9562）

### Snowflake の先頭1bitが常に0の理由

`BIGINT` は符号付き64bit整数。先頭1bitが符号ビット（0=正、1=負）。

```
[符号 1bit][タイムスタンプ 41bit][マシンID 10bit][連番 12bit]
     0     |        41bit        |     10bit     |   12bit
```

1にするとIDが負の数になり、ソート崩れ・アプリの `id > 0` チェック崩れ等が起きる。
0に固定することでIDが必ず正の整数になる。実質63bit分しか使えないが、41bitのタイムスタンプで約69年分あるので問題なし。

### Snowflake はハッシュ関数ではない

ハッシュ関数 = 外部の入力データを受け取って固定長の出力に変換する。衝突が原理上ありうる。

Snowflake = 現在の「状態」（タイムスタンプ・マシンID・連番）をビット結合するだけ。入力データなし。

```go
id := (timestamp << 22) | (machineID << 12) | sequence
```

| | Snowflake | UUID v4 | ハッシュ関数 |
|--|------|------|------|
| 入力データ | なし（状態を使う） | なし（乱数） | あり |
| 衝突 | ゼロ保証 | 確率的に無視できる | 原理上ありうる |
| 一方向性 | なし（タイムスタンプ復元可） | なし | あり |
| 順序性 | ✅ 時系列 | ❌ | ❌ |

「ハッシュっぽい」のは Chapter 8 の URL Shortener 側：long URL（入力）を MD5/SHA-1 にかけて short URL を生成する、あちらが本来のハッシュ利用。

## 🔑 キーワード

- B-tree: ソートされた順でインデックスを管理するデータ構造。単調増加IDと相性が良い
- Snowflake: 64bit = 41bit(タイムスタンプ) + 10bit(マシンID) + 12bit(連番)
- クロックスキュー: サーバー間の時刻ズレでID順序が狂う可能性
- BASE62: 0-9・a-z・A-Zの62文字でIDをエンコード。URLに露出するIDに有効
- JavaScript 53bit問題: Number型の精度限界。64bit整数をそのまま渡すと末尾が変わる

## 💡 まとめ

1. **IDの設計はトレードオフの連鎖** — ソート可能性・分散対応・変換コスト・衝突保証はすべて連動している。BIGINT は DB 効率、BASE62 は URL 露出、ULID は K8s 対応と「どこで使うか」で選択が変わる
2. **Snowflake の 64bit 設計は意図的な制約** — 符号ビット固定・BIGINT 一致・ビット演算の効率・約 69 年の寿命、全てが 64bit に収まるように逆算されている
3. **分散環境では「調整」のコストをどこで払うか** — Snowflake は ZooKeeper で払う、ULID は 80bit ランダムの広大な空間で代替、UUID v7 は 12bit カウンターで 1 マシン内のみ払う
