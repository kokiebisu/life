# 勉強 - 2026-03-18

## 🎯 今日の目標・疑問

Rate Limiter（Ch.4）の内容を理解する

## 📝 ノート

### サーバーサイド vs クライアントサイド
- サーバーサイド: API提供者がサーバーリソースを守る。改ざん不可
- クライアントサイド: API利用者が外部APIの制限を守る自衛手段。バイパス可能（リバースエンジニアリング・DevTools）

### 分散型レートリミッター
- 複数サーバーがローカルカウントすると合計が把握できない
- Redisの共有カウンターで一元管理が定石
- 競合状態（Race Condition）はLua スクリプトのアトミック操作で防ぐ
- RedisはLuaインタープリターを内蔵。EVAL コマンドでスクリプトを送るとシングルスレッドで実行（割り込み不可）

### 制限の粒度
- ユーザーID: ログイン済み向け。厳しく設定できる
- IP: 未ログイン向け。企業・VPNで複数人が同一IPになるため緩めに設定
- 実務では組み合わせが多い

### アルゴリズム5種（優先度順）
1. **トークンバケット ★★★** - バーストOK。一般的なAPI向け
2. **固定ウィンドウ ★★★** - シンプル。境界バースト問題あり
3. **スライディングウィンドウカウンタ ★★** - 境界問題を近似で解決
4. **スライディングウィンドウログ ★** - 精度最高・メモリ大
5. **リーキーバケット ★** - 流量を均一化。決済・メール向け

### アーキテクチャ
- ミドルウェアが主流（漏れがない・関心の分離）
- 大規模ではAPIゲートウェイ（Kong、AWS API Gateway）でアプリ手前でブロック
- ルール: S3に設定ファイル → 起動時にメモリキャッシュ
- カウンター: Redis（共有）
- SPOF対策: Redisレプリケーション or Redis Cluster or ElastiCache（実務最多）

### OSIモデルとの関係
- L7（アプリ層）: ソフトウェアレベル。HTTPの中身を見て制限
- L4（トランスポート）: TCPコネクション数・SYN flood対策
- L3（ネットワーク）: IP単位のパケット制限・帯域制限

### クライアントのベストプラクティス
- キャッシュ ＋ Exponential Backoff をセットで使う
- キャッシュで429自体を防ぎ、429時はstaleキャッシュでUX維持

## 🔑 キーワード

- Race Condition（競合状態）
- Lua スクリプト / EVAL コマンド
- Token Bucket / Leaky Bucket / Fixed Window / Sliding Window
- Sticky Session
- Redis Cluster / ElastiCache
- Consistent Hashing（→ Ch.5）
- OSIモデル L3/L4/L7

## 💡 まとめ

Rate Limiterは「誰が・どこで・何を守るか」で設計が変わる。分散環境ではRedisの共有カウンター＋Luaのアトミック操作が定石。アルゴリズムはトークンバケットと固定ウィンドウを押さえておけば面接では十分。

## 🔁 復習で詰まったところ

### 2026-04-28 - 🔺 あいまい
- **Q: OSI L3/L4/L7 でのレート制限の違いは？**
  - 詰まった内容: L3（IP）と L7（ユーザーID・HTTPエンドポイント）は答えたが、L4 を「リクエストの上限」と説明してしまった（L4 は TCP コネクション数）
  - 正解ポイント: L4（トランスポート層）は **TCP コネクション数の制限・SYN flood 対策**（接続単位）。「リクエスト」は HTTP の概念なので L7 の領域。階層が上がるほど解像度が上がる: パケット(L3) → 接続(L4) → アプリ意味論(L7)

### 2026-04-24 - 🔺 あいまい
- **Q: OSI L3/L4/L7 でのレート制限の違いは？**
  - 詰まった内容: L3（IP・パケット）と L7（アプリ層・ユーザーID）は答えられたが、L4 の役割が説明できなかった
  - 正解ポイント: L4（トランスポート層）は **TCP コネクション数の制限・SYN flood 対策**。L3 がパケット単位、L4 が TCP 接続単位、L7 が HTTP の中身（パス・ユーザー）と階層ごとに解像度が上がる
