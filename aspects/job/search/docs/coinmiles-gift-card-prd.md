# PRD: ギフトカード仮想通貨キャッシュバック — Coinmiles

**著者:** Ken Oki  
**ステータス:** リリース済み（2022年6月）  
**ステークホルダー:** CEO、CTO、決済チーム、CSチーム  
**最終更新:** 2022年5月

---

## 背景・課題

### ビジネス背景

Coinmiles は、提携店舗での購入に対して Bitcoin キャッシュバックを提供するモバイルアプリ（カナダ市場）。2021年末時点で MAU 約8万人。収益モデルはアフィリエイト手数料で、ユーザーが提携店舗で購入するほど Coinmiles の収益も増加する。

**問題:**  
ギフトカードは北米における小売市場で年間 $1,600億ドル規模の市場（Mercator Advisory Group, 2021）。提携小売店の多くはギフトカードを販売しているが、ユーザーがアプリ外でギフトカードを購入しても Coinmiles はキャッシュバックを付与できず、アフィリエイト収益が発生しない。

**定量的な課題:**
- ユーザーインタビュー（n=30）において、72%が「過去3ヶ月以内にギフトカードを購入した」と回答
- そのうち 90%以上がアプリ外（提携店舗の公式サイトや店頭）で購入しており、Coinmiles にはゼロ収益
- 既存の物理購入連携（手動突き合わせ）はキャッシュバック付与まで平均 36時間かかっており、CSへの問い合わせの約 28%が「キャッシュバックが届かない」

### 技術的背景

当時の Coinmiles バックエンドは全て AWS Lambda（Serverless Framework）で構成されており、Node.js / TypeScript で実装されていた。決済機能は Stripe を既に使用していたが、Payment Intent 等の高度な機能は使っておらず、Checkout による単純なサブスクリプション課金のみだった。

バックエンドチームは CTO と リードエンジニアの同時退職（2021年6月）後、筆者を含む3名体制で運用していた。実装工数・保守性が設計の大きな制約となっていた。

## 目標

| 優先度 | ゴール | 計測方法 |
|--------|--------|---------|
| P0 | アプリ内でギフトカードを購入し、決済完了後 60秒以内に BTC キャッシュバックが付与される | SQS ワーカーの処理完了ログ |
| P0 | Coinmiles サーバーにカード情報が一切保存されない（PCI DSS 非準拠環境での運用継続） | Stripe Dashboard + コードレビュー |
| P1 | サードパーティ・ギフトカードプロバイダーの API キーが 30日ごとに自動ローテーション | EventBridge 実行ログ |
| P1 | 決済フロー中のネットワーク切断・Lambda タイムアウト等のエッジケースで二重請求・キャッシュバック不整合が発生しない | Stripe Payment Intent ステータスと DynamoDB の突き合わせ監視 |
| P2 | 将来的に他の決済・ギフトカードプロバイダーへ切り替え可能な設計 | コードレビューでインターフェース分離を確認 |

## スコープ外（V1）

| 項目 | 理由 |
|------|------|
| 返金フロー | リリース初期は CS 対応で吸収。発生率が低い（推定 < 1%）ため V2 で自動化 |
| BTC 換算レートのリアルタイムロック | 為替変動リスクは短時間（< 60秒）なら許容範囲。V2でヘッジ検討 |
| 複数ギフトカードプロバイダー対応 | 提携先は 1社のみ。V2でプロバイダー選択肢を拡大 |
| ギフトカードの残高管理・リロード | プロバイダー側の機能。Coinmiles はキャッシュバック付与にフォーカス |

## ユーザーストーリー

**新規ユーザーとして、**  
既存の仮想通貨ウォレットがなくてもクレジットカードでギフトカードを購入し BTC キャッシュバックを得たい。ウォレット開設フローがあっても 5分以内で完結してほしい。

**既存ユーザーとして、**  
ギフトカード購入後すぐにウォレットにキャッシュバックが反映されるのを確認したい。「いつ付与されるのか」が不透明な状態が一番ストレスになる。

**オペレーションチームとして、**  
ギフトカード API の認証情報が自動でローテーションされ、「期限切れに気付かなかった」という理由による本番障害を撲滅したい（過去に 2回発生）。

**CSチームとして、**  
「キャッシュバックが届かない」問い合わせの根本原因を DLQ + DynamoDB で追跡できるようにしたい。現在は問い合わせを受けるたびにエンジニアへのエスカレーションが必要で、週次で 5〜8件発生している。

## 制約条件

### セキュリティ・コンプライアンス

- **PCI DSS 非準拠環境:** Coinmiles はカード会員情報環境（CDE）の認定を受けていない。カード番号・CVV・有効期限が自社サーバーに触れる設計は不可。Stripe のクライアントサイド SDK によるトークン化が必須。
- **HTTPS 必須:** API Gateway エンドポイントは全て TLS 1.2 以上。

### 技術的制約

- **AWS Lambda ランタイム:** 全バックエンドは Serverless Framework による Lambda（Node.js 16）。最大実行時間 15分。同期 API は 29秒のレスポンスタイムアウト制約がある。
- **チーム規模:** バックエンド実質 1名（筆者）。過度に複雑な設計は保守不能になるリスクがある。
- **Stripe 依存:** 既存インフラが Stripe を使用。第二の決済プロバイダー導入は避ける。

### ビジネス制約

- **リリース期限:** 2022年6月末（Q2 末）。CEO からの優先プロジェクト指定。
- **ギフトカードプロバイダー制約:** API キーの有効期限は 30日間。キーはクライアント証明書を使って取得する独自フローがあり、Stripe のような標準的な OAuth ではない。

## リスクと対策

| リスク | 発生可能性 | ビジネスインパクト | 対策 |
|--------|----------|-----------------|------|
| 決済成功後にキャッシュバックワーカーが失敗し付与漏れが発生 | 中（ネットワーク障害時） | 高（ユーザー信頼の毀損） | SQS + DLQ による冪等リトライ。DLQ 深度 > 0 でアラーム |
| ギフトカードプロバイダーの API キー有効期限切れ | 高（過去 2回発生） | 高（機能停止） | Secrets Manager + EventBridge による自動ローテーション |
| Webhook 重複配信による二重キャッシュバック | 低（Stripe 保証レベル） | 高（資金損失） | `payment_intent.id` による DynamoDB での冪等チェック |
| ネットワーク切断でトランザクション状態が曖昧になる | 中 | 中（CS 問い合わせ増加） | Stripe Payment Intent ステータスのポーリングによるバックアップ確認 |
| プロバイダー API の仕様変更 | 中（API ドキュメントの整備が不十分） | 中（緊急対応が必要） | Clean Architecture でプロバイダーをインターフェースの裏側に隔離。変更箇所をインフラ層に限定 |
