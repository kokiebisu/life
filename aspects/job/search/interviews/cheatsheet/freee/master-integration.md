## 📌 freee 福利厚生 従業員マスタ統合 - 面接かんぺ(セリフ版)

### 🎤 状況設定(30〜40 秒)

「freee 福利厚生は、freee 人事労務のユーザーに対して福利厚生サービスを提供するプロダクトで、中身は **『社宅』と『ベネフィットサービス』の 2 つ**で構成されています。社宅は自社開発、ベネフィットサービスは OEM 提供。freee としての差別化は、**人事労務との連携をどれだけ深く出せるか**に集約される設計で、これは社宅・ベネフィットサービスの両方に共通する軸です。

ただ当時、福利厚生は独自 DB で従業員データを持っていて、人事労務側の共通従業員マスタとは分かれていました。そのため、人事労務から福利厚生にアップセルするときに従業員をもう一度登録する必要があって **両プロダクトとも導入体験が毀損** するうえ、登録後も両側のデータが連携されないので **従業員情報の不整合も起こりうる** 状態でした。結果として **freee としての差別化動線も成立していない** という問題が起きていて、バックエンド側で本件の設計とリードを担いました。

一番難しかったのは、**福利厚生 DB を残したまま基本情報の管理だけを共通マスタへ寄せる動線**を、**既存ユーザー影響ゼロ**と**人事労務側のスキーマ無改修**という 2 つの縛りの中で成立させることでした。」

---

### 🎤 制約(45〜60 秒)

「制約は 4 つあります。

まず **福利厚生 DB は廃止できない**。福利厚生 DB には社宅情報や OEM へのログイン ID など、共通マスタには含まれない**福利厚生固有のデータ**が乗っていて、これは人事労務側の関心事ではないので、共通マスタには寄せられません。なので『全員を共通マスタへ寄せて福利厚生 DB を潰す』という一括移行は最初から取れない選択肢でした。できるのは**基本情報の管理方式だけ**を寄せるところまで、という前提です。

次に **既存ユーザー影響ゼロ**。福利厚生をすでに使っている企業が一定数いて、CSV 登録の動線を強制的に切り替えると不具合時の戻しが重くなります。既存ユーザーは『現状維持』も選べる形にしておく必要がありました。

さらに **人事労務側のスキーマには手を入れない**。共通マスタは人事労務側の管理で、福利厚生のために構造変更を依頼するのはコストも合意難度も高い。**吸収責任は福利厚生側で取る**という前提です。

最後に **共通マスタの属性変更 pubsub が未整備**でした。共通マスタ自体が新しいマイクロサービスへの移行途中で、**削除イベントの pubsub は存在していましたが、属性変更を通知する pubsub イベントはまだ作られていません**。新規に整備するなら**通知基盤チームとの合意形成と実装工数が重い**ので、属性変更を購読してローカルコピーを同期する CDC・イベント駆動の構成は、この時期には現実的に取れる選択肢ではない、という前提で設計を始めました。

加えてスキーマレベルの食い違いも 2 種類あって、氏名の表記ゆれ、共通マスタは『ユーザー名』だけで登録できるが福利厚生は社宅サービス等で住所・電話番号などを必須化していた、この 2 つも吸収する必要がありました。」

---

### 🎤 設計判断(60 秒)

「採用した設計の核は **UI レベルで経路を分けて、入口でレコードの起源を一意に決める**ことです。

ありがちな設計は、登録ボタンを 1 つにして API 内部で『新規? 既存?』を判定して**起源を分岐**するというものですが、これは**起源が文脈依存**になります。ロジックにバグがあった瞬間に取り違えるし、後から『なぜこのレコードがこの起源で作られたか』を追うのもログ依存になる。

なので、既存ユーザー向けには『CSV 経由で登録』と『従業員マスタから取り込む』の 2 ボタン、新規ユーザー向けには『従業員マスタから取り込む』の 1 ボタンだけ、という形で UI を分けました。**ユーザーがどのボタンを押したかでレコードの起源が入口で決まる**ので、API 側に分岐ロジックを書く必要がなくなります。

スキーマ側では、福利厚生従業員テーブルに `master_employee_id` という **NULL 許容のカラム**を 1 本追加して、**NULL なら CSV 起源、非 NULL なら共通マスタ起源**、と起源を一意に表現しました。既存カラムを流用すると 1 つの値が文脈で違う意味を持つダブルミーニングになるので、新カラム追加で『値の有無』で意味を切り分ける形にしました。

紐付けキーはメールアドレスではなく `master_employee_id` です。メールアドレスは変更される可能性があって、人事労務側で変えられると福利厚生から『別ユーザー』に見える瞬間が出る。統合キーは表示情報ではなく、変更されない内部 ID に寄せるべきという判断です。」

### 🎤 スキーマ食い違いの吸収(45 秒、強い武器)

「スキーマレベルの食い違いは 2 種類あって、両方とも**吸収責任を福利厚生側に倒す**方針で揃えました。

1 つは表記ゆれです。氏名で言うと、福利厚生は『姓』『名』の 2 カラム、共通マスタは『ユーザー名』単一カラム。しかも共通マスタ側では『姓』『名』が任意項目なので、取り込みレコードに姓・名が空のものが普通に来ます。これは福利厚生側で勝手に補完せず、**取り込み一覧 UI でエラー表示にして、新規タブで人事労務側の編集画面を開く補完導線**を引きました。モーダルにリフレッシュボタンを置いて、補完したらモーダルを閉じずに再取得できる形です。福利厚生 UI で代替表示で繕うと、共通マスタ側の不備が放置されて他サービスでも再発するので、**源泉を直す動線**にしたところがポイントです。

もう 1 つは必須項目の食い違いです。福利厚生 CSV は社宅サービス等のために住所・電話番号などを必須化していましたが、共通マスタは『ユーザー名』だけで登録できる仕様。これは **『保存時の必須』と『サービス利用時の必須』を概念的に分離**して吸収しました。福利厚生 DB の保存時必須項目は最小化して、住所などはサービスを実際に申請するときに補完入力させる形です。」

### 🎤 整合性担保(45 秒)

「整合性については、**そもそも跨り書き込みが構造的に発生しない構造**にしました。

前提として、**福利厚生は人事労務とは別マイクロサービス**です。共通マスタ自体は当時、論理的には人事労務から独立した位置付けでしたが、物理的にはまだ人事労務側に同居している状態で、人事労務内のモジュール(例: freee E ラーニング)は共通マスタのリポジトリを DB レベルで直接使っていました。一方、**福利厚生は外側のサービスなので共通マスタの DB に直接触れる経路がなく、HTTP リクエスト経由でのみ読みに行く**形になります。

なので書き込み先は常に福利厚生側のユーザーテーブルだけで、共通マスタ起源のレコードでも `master_employee_id` を保存して『どの従業員マスタから来たか』を示すだけ。共通マスタ側には書きません。2PC や Saga のような分散トランザクションは**そもそも検討対象に入らない**構造で、**整合性の難しさは跨り書き込みから生まれる**以上、跨らない構造にすれば議論自体が不要になります。

データ取得の考え方だけが起源で変わって、master 起源なら従業員マスタから基本情報を取って、CSV 起源なら福利厚生 DB から取る。この分岐を画面・サービスに散らさないために**リポジトリパターン**を入れて、呼び出し側は『従業員を取得する』だけに見えるようにしました。後から『この画面は master 起源のときだけ古い値を見ている』という事故が起きないようにしています。」

## 💬 想定 Q&A(セリフ版)

### Q1. なぜ段階移行にしたのか?

「そもそも福利厚生 DB は廃止できないからです。社宅情報や OEM へのログイン ID など、共通マスタには含まれない**福利厚生固有のデータ**が福利厚生 DB に乗っているので、『全員を共通マスタへ寄せて福利厚生 DB を潰す』という一括移行はそもそも構造的に取れない選択肢でした。

できるのは『基本情報の管理方式だけを切り替えられるようにする』ところまで。新規ユーザーは強制的に新動線、既存ユーザーは現状維持か新動線かを選べる、という段階移行に倒しました。」

### Q2. 整合性をどう担保したのか?

「**書き込み先は常に福利厚生 DB だけ**で、共通マスタには HTTP で読みに行くだけ。一切書かないので跨り書き込みが構造的に発生しません。整合性の難しさは跨り書き込みから生まれる以上、**跨らない構造にすれば議論自体が不要**になります。

レコードの起源(CSV 起源 / master 起源)も UI で経路を分けて入口で決めるので、API 内部で起源を判定する必要がなく、取り違え事故も構造的に防げます。」

### Q3. ダブルミーニング回避の具体的な設計は?

「福利厚生従業員テーブルに `master_employee_id` という NULL 許容カラムを追加しました。**NULL なら CSV 起源、非 NULL なら共通マスタ起源**、と起源を一意に表現しています。既存カラムを流用すると同じ値が文脈で異なる意味を持つダブルミーニングになりますが、**NULL/非 NULL で切り分ける**ことで起源の判別が NULL 判定一発になり、運用での誤判定を防げます。」

### Q4. 入口で起源を分けた発想はどこから?

「API 内部で分岐ロジックを書くと、判定ロジックにバグがあった瞬間に**起源を取り違えます**(master 起源として作るべきものが CSV 起源として作られる、など)。後から『なぜこのレコードがこの起源で作られたか』を追うのもログ依存です。

逆に、ユーザーの操作経路で起源が決まるようにすれば、**ボタン名がそのまま起源を表現する**ので運用でも追いやすい。判定ロジック自体を書かなくて済むので、バグの混入リスクが構造的に減ります。**『起源 = 状態判定』ではなく『起源 = ユーザー操作の経路』と読み替えた**形です。」

### Q5. メールアドレスではなく `master_employee_id` で紐付けた理由は?

「メールアドレスは変更される可能性があるからです。人事労務側でログインメールアドレスを変更したとき、福利厚生側がすぐ察知できないと、福利厚生には古いメール、人事労務には新しいメール、という状態になります。このときメールアドレスをキーにしていると、同じ人なのに別ユーザー扱いになる。

統合キーには、ユーザーが変更できる属性ではなく、**従業員マスタ上の変更されない内部 ID**を使うべきです。なので `master_employee_id` を追加して、共通マスタ起源のユーザーはそのキーで紐付ける形にしました。」

### Q6. master と CSV の両方に登録していたらどうする?

「1 レコードの起源は `master_employee_id` の NULL/非 NULL で一意に決める方針なので、**同じ福利厚生ユーザーが両方の起源を持つ状態は作りません**。

もし既存の CSV ユーザーと取り込もうとしているユーザーが同じ人に見える場合でも、**メールアドレスだけで自動マージはしません**。候補として警告表示はできますが、最終的には管理者に確認させるか、明示的な紐付け操作で既存ユーザーに `master_employee_id` を付与する形を取ります。自動で『同じ人だからマージ』はやらない、というのが安全側の判断です。」

### Q7. リポジトリパターンを入れた理由は?

「master 起源と CSV 起源で**基本情報の取り方が変わる**からです。master 起源なら従業員マスタから、CSV 起源なら福利厚生 DB から基本情報を取ります。

この分岐を画面やサービス層に散らすと、どこかで片方の起源だけ古い見方をしてしまうリスクがあります。なのでリポジトリパターンで**呼び出し側からは『従業員を取得する』だけに見える**ようにして、`master_employee_id` の有無による分岐をリポジトリ層 1 箇所に閉じ込めました。」

### Q8. 一括招待で重複したらどうなる?

「`master_employee_id` にユニーク制約を設けて、同じ共通マスタ従業員を二重に取り込めないようにしています。UI 側でも取り込み済みの従業員はチェックできない表示にして、UI と DB の双方で重複を防いでいます。**部分失敗**が起きたら成功分はコミットして、失敗分はエラー一覧で再試行できる形です。」

### Q9. スキーマの食い違いはどう吸収したのか?

「2 種類ありました。1 つは表記ゆれで、福利厚生は『姓』『名』、共通マスタは『ユーザー名』。共通マスタ側では『姓』『名』が任意項目なので、取り込みレコードに姓・名が空のものが来ます。これは**取り込み一覧 UI でエラー表示にして、新規タブで人事労務側を開く補完導線**を引きました。モーダルのリフレッシュで再取得できます。

もう 1 つは必須項目の食い違いで、福利厚生 CSV は社宅サービス等で住所・電話番号などを必須化していましたが、共通マスタは『ユーザー名』だけで登録できる仕様。これは**『保存時の必須』と『サービス利用時の必須』を概念分離**して、住所などはサービス申請時に補完入力させるフローにしました。

両方とも吸収責任は**常に福利厚生側に倒す**方針で揃えています。共通マスタ依頼は制約 3 に反するし、強引な変換(ユーザー名を空白で分割するなど)はデータ品質が落ちるので避けました。」

### Q10. 氏名の表記ゆれに対して、他にどんな案があり得たか?

「3 つあります。

1 つ目は **強引な自動分割**。共通マスタの『ユーザー名』を空白・カナ漢字境界などで姓・名にパースする案です。これは『山田太郎』のような曖昧ケースで誤分割が出やすく、**氏名フィールドのデータ品質が落ちる**ので不採用にしました。社宅契約や税務で氏名の正確性が要るケースに耐えません。

2 つ目は **代替表示で繕う**。福利厚生 UI 上で姓・名が空でも『ユーザー名』をそのまま表示してスルーする案です。これは**共通マスタ側の不備が放置**されて、他サービスでも同じ問題が再発します。福利厚生だけ凌いでも源泉が直らない、というのが致命的。『取り込みは通すが警告マークを付ける』バリエーションも検討しましたが、警告は実質放置されるので同じ問題に着地します。

3 つ目は **共通マスタ側のスキーマで姓・名を必須化する**。これは制約 3『**人事労務側のスキーマには手を入れない**』に正面から違反します。共通マスタは人事労務側の管理で、福利厚生のために必須化を依頼するのは合意難度・他ユースケースへの波及の両方で重い判断でした。

採用案の『**取り込み UI でエラー → 共通マスタ側を新規タブで開く → モーダルリフレッシュ**』は、この 3 つのトレードオフのすき間を通すルートです。**福利厚生で繕わず、共通マスタのスキーマも触らず、人間に源泉を直してもらう導線だけ引く**。直せば他サービスも恩恵を受ける、というポジティブな副作用も含めての判断でした。」

### Q11. `master_employee_id` に DB の外部キー制約を貼らなかった理由は?

「2 つあります。1 つ目は**サービス境界**の問題で、福利厚生は人事労務とは別マイクロサービスなので、共通マスタの DB には直接アクセスできません。アクセスは HTTP リクエスト経由のみで、DB レベルの FK 制約はそもそもかけようがない、という前提です。共通マスタ自体は当時、物理的にはまだ人事労務側に同居していて、人事労務内のモジュール(freee E ラーニングなど)は共通マスタのリポジトリを DB レベルで使っていましたが、福利厚生は外側のサービスなのでその経路は使えません。

2 つ目は**疎結合化の方針**で、freee 全体で人事労務の周辺機能を疎結合化していく流れがあったので、仮に DB 直結できる構成だったとしても DB 層で強く繋ぐのは避ける判断でした。DB で繋ぐと人事労務側のスキーマ変更が福利厚生のエラー要因になりサービス分離が進みません。

引き換えに DB の自動保護は失うので、`master_employee_id` のユニーク制約、リポジトリ層の存在チェック、孤児レコードの定期検知、共通マスタ側の削除イベントの伝達経路、この 4 つで実質的に担保する形にしました。

JOIN したい時は **HTTP 越しなので SQL JOIN はそもそも引けない**ため、**リポジトリ層で別クエリで取って、アプリ側で hash join 相当**(ID → オブジェクトの map で引き直す)を自前でやる、という形です。」

### Q12. CDC やイベント駆動で同期する案は検討したか?

「検討して見送りました。当時、共通マスタ側で **属性変更を通知する pubsub イベントがまだ作られていない**状況で(削除イベントの pubsub は存在)、属性変更を購読してローカルコピーを同期する構成は、まず **属性変更イベントの新規整備**から始める必要がありました。通知基盤チームとの合意形成と実装工数を引き受けるには、本プロジェクトのスコープに対して重すぎる判断です。

実際の立ち回りは、**福利厚生側のレコードは `master_employee_id` だけ保存して、氏名・メアドなどの基本情報はローカルにコピーしない**形にしました。画面表示や処理で基本情報が必要になった瞬間に共通マスタを叩いて取得するので、共通マスタ側で属性が変わっても **次の読み取りで自動的に最新値になる**。pubsub で変更通知を受け取って自前テーブルを更新する、という仕組みを **そもそも不要にした** のがポイントです。一覧画面で N+1 が出る箇所はリポジトリ層が `master_employee_id` の集合を bulk fetch でまとめます。

ローカルコピーを持つ構成は **整合性の窓・イベント遅延・順序逆転・重複配信**などの考慮が新規に発生しますが、今回の構成はローカルコピー自体を持たないので不整合の窓が原理的に存在しません。**整備コストも整合性の難しさも、まとめて回避できる**という判断でした。削除だけは別で、共通マスタから『居なくなった』を読み取り側からは気付けないので、**削除 pubsub を購読して孤児レコードを片付ける**形にしています。

将来、属性変更 pubsub が整備されたら、頻繁にアクセスされる項目だけローカルキャッシュ + イベント無効化、という最適化はあり得ます。今は『読み取り時に取りに行く』で十分というスタンスです。」

### Q13. 既存ユーザーの最終的な出口戦略は?

「**『福利厚生 DB の廃止』は出口にはなりません**。社宅情報や OEM ログイン ID など固有データが乗っているので、福利厚生 DB は永続的に残ります。

なので出口戦略は『既存ユーザーが基本情報の管理を共通マスタ参照側に切り替えるかどうか』に絞られて、強制切り替えはせず、各社の判断で取り込みに移ってもらう運用です。最終的に CSV 登録ユーザーがゼロになるかは事業判断で、技術側からは強制しないという立て付けにしています。」

### Q14. なぜこのプロジェクトが事業的に重要だったのか?

「freee 福利厚生は『社宅』と『ベネフィットサービス』の 2 プロダクトで構成されていて、特に **ベネフィットサービス側は OEM 提供**です。OEM 部分は中身が freee 独自開発ではないので、**freee として売り出す意味が『人事労務との連携』に集約**されます。連携が弱いと、市場の他社福利厚生アプリと差別化要素がなくなって単純な価格勝負になる。価格勝負に持ち込まれると freee の強みが消えるので、人事労務エコシステムの土俵で戦えるようにすることが必須でした。

従業員マスタ統合は、UX 毀損の解消であると同時に、**freee 福利厚生の事業ポジショニングそのものを成立させる手段**だった、という位置付けです。」

### Q15. 1 社が 1000 人規模のとき、一括取り込みの UX は破綻しないか?

「一括招待モーダルで全件を一度に選ばせる形にしているので、件数自体は破綻しません。ポイントは**エラー解消のためにモーダルを閉じさせない**ことで、不備のある行は新規タブで人事労務側を開いて補完したら、モーダルのリフレッシュで最新状態を取り直せます。

『行ったり来たりでやり直し』ではなく『**エラー行だけ直して再読み込み**』に近い操作感になるので、企業規模が大きくても定数コスト寄りに収まる設計です。」

### Q16. 今やり直すなら何を変える?

「2 つあります。

1 つ目は、福利厚生 DB のスキーマを**『基本情報領域』と『福利厚生固有データ領域』で初期から明示的に分ける**こと。今は同じテーブルにどちらも乗っていて、共通マスタ参照に切り替えたとき『どこまでが上書き対象か』が暗黙になります。最初から領域を切っておくと、共通マスタ起源データと福利厚生固有データの責任範囲が読みやすくなります。

2 つ目は、`master_employee_id` のユニーク制約や一括招待の冪等性を**テストで明示的に保護する**こと。NULL/非 NULL での起源判別ロジックは単純ですが、契約として残すならテストで縛っておくと、後の改修で意味がブレません。」

### Q17. これは expand / contract(parallel change) の考え方ですよね?

「はい、フレームとしてはまさにそれです。

- **Expand**: 既存の CSV 経路は壊さず、`master_employee_id` カラムと master 経由のユーザー作成パスを**追加するだけ**で広げました。既存スキーマは互換のまま残しています。
- **Parallel**: 入口で『master 側に該当者が居れば master、居なければ CSV』と振り分けて、両経路を一定期間**並行運用**しました。
- **Contract**: 既存ユーザーが master 側に出揃った段階で CSV 経路を畳んでいく、というのが出口戦略です(Q13)。

DB の外部キーをアプリ層で見る判断は厳密には別軸(別マイクロサービス境界で物理的に FK を貼れない)ですが、参照先が揃わない時期を許容する設計、という意味で expand/parallel フェーズの発想とは噛み合っています。」
