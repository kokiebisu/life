# 勉強 - 2026-03-17

## 🎯 今日の目標・疑問

ユーザーが増えるにつれてシステムをどう段階的にスケールさせるか、各フェーズの設計判断と理由を理解する。シャーディングのジョイン問題と対処法まで押さえる。

## 📝 ノート

### ゼロからスケールへ

**一言で言うと（ファインマン式）**
ユーザーが増えるにつれて、一台のサーバーから始まり、ロードバランサー・DBレプリケーション・キャッシュ・CDN・シャーディングと段階的に設計を進化させていく話。

**スケールの進化フロー**

- Phase 1: 単一サーバー — アプリサーバーとDBが同一マシンに共存
- Phase 2: スケールアップ vs スケールアウト — スケールアウト（水平）でロードバランサーを導入
- Phase 3: DBレプリケーション（マスター/スレーブ）— マスター：書き込み専用 / スレーブ：読み取り専用。フェイルセーフ・読み書き分離
- Phase 4: キャッシュ — 頻繁な読み取りの結果をメモリに保存。DB負荷↓・レイテンシ↓
- Phase 5: CDN — 静的コンテンツを地理的に近いサーバーから配信
- Phase 6: ステートレス化 — セッションデータを専用データストアで管理
- Phase 7: データセンター（マルチリージョン）— 地理的冗長性・レイテンシ改善
- Phase 8: メッセージキュー — コンポーネント間を疎結合に。重い処理を非同期化
- Phase 9: DBシャーディング — テーブルの行を複数シャードに分散

**キャッシュ戦略まとめ**

| 戦略 | 誰が制御 | 特徴 |
|------|----------|------|
| Cache-aside | アプリ | 最も柔軟・実務で最多 |
| Read-through | キャッシュ層 | アプリがDB直接叩かない |
| Write-through | キャッシュ層 | 同期でDBに書く → 整合性高い・書き込み遅い |
| Write-back | キャッシュ層 | 非同期でDBに書く → 速い・障害時データ損失リスク |

**SQL vs NoSQL の選択基準**
- NoSQL: Eventual Consistency許容 / 超低レイテンシ / 1台を超えるデータ量 / SQLレプリの限界を超えるQPS
- SQL: ACID保証 / 複雑なJOIN / スキーマが安定

「読み取りが多い = NoSQL」は誤り。**判断の核心は「データ量がマシン1台を超えるか」「ACIDが必要か」。**

### DBスケーリング詳細（Phase 9 深掘り）

**シャードとは**
テーブルの**行のグループ**。1つの行は必ず1つのシャードに属する。

```
usersテーブル全体:
  シャードA（サーバA）→ user_id 1〜1000万 の行
  シャードB（サーバB）→ user_id 1000万〜2000万 の行
```

**ジョイン問題**
シャーディングするとデータが別サーバに分散するため、**クロスシャードJOINが難しくなる**。

**対処法まとめ**

| 状況 | 対処法 |
|------|--------|
| 関連データの更新が少ない | 非正規化（user_nameをordersに持たせるなど） |
| 関連テーブルを同じキーで分割できる | シャードキー設計（user_idで揃える） |
| 大規模システムでアプリ側を単純にしたい | ミドルウェア（Vitess/Citus） |
| 複雑な分析クエリが必要 | OLAP / データウェアハウス |

- **非正規化**: 更新が少ないデータ向け（更新頻度が高いと更新コストが爆発）
- **ミドルウェア**: 代表例 **Vitess**（MySQL）、**Citus**（PostgreSQL）
- **OLAP / データウェアハウス**: 更新系（OLTP）とは別に全データを集めた読み取り専用DB

## 🔑 キーワード

- **スケールアウト（水平スケール）** — サーバーを複数台に増やす
- **ロードバランサー** — 複数Webサーバーにトラフィックを分散
- **DBレプリケーション** — マスター（書き込み）/ スレーブ（読み取り）
- **Cache-aside** — アプリが制御する最も一般的なキャッシュ戦略
- **Write-back** — 非同期でDBに書く。速いが障害時にデータ損失リスク
- **ステートレス** — セッションをWebサーバーに持たない
- **メッセージキュー** — 疎結合・非同期処理
- **シャーディング** — テーブルの行を複数サーバーに分散
- **クロスシャードJOIN** — 別サーバーにまたがるJOIN。パフォーマンス問題が発生
- **非正規化** — JOINをなくすためにデータを冗長に持つ。更新頻度が低いデータ向け
- **Vitess / Citus** — シャードルーターの代表例
- **OLTP vs OLAP** — OLTP: 更新系 / OLAP: 分析系

## 💡 まとめ

1. スケールは段階的に進化させる。最初から全部入れない
2. 読み取りはキャッシュ＋レプリカで捌ける。書き込みが増えたらシャーディング
3. NoSQLはデータ量がマシン1台を超えるか・ACID不要な場合に選ぶ
4. シャーディングするとクロスシャードJOINが難しくなる。設計段階でキーを揃えて回避するのが基本
5. 非正規化は更新が少ないデータに有効。分析クエリはデータウェアハウスを別に持つ
