## 0. 面接中に見るカンペ

### 30秒版

「freee Eラーニングでは、OEM 側との OIDC 認証フロー統合を担当しました。一番難しかったのは、freee は 1 ユーザーが複数事業所に所属できる一方で、OEM 側は 1 ユーザー 1 会社に近い前提だったことです。OEM 側はリアーキ中でスキーマ変更もできなかったので、既存のログイン ID フィールドに `freee_user_id` と `company_id` の組み合わせを入れ、OIDC トークンと照合する設計にしました。PSIRT からのなりすまし懸念には、OEM 側のユニーク制約とトークン照合で説明し、リリースまで持っていきました。」

### 2分版

「freee Eラーニングでは、freee 人事労務から OEM 提供の Eラーニング基盤へ SSO するために、OIDC 認証フローを設計しました。難しかったのは、freee と OEM のユーザーモデルが合っていなかったことです。freee では 1 人のユーザーが複数事業所に所属できますが、OEM 側は 1 ユーザー 1 会社に近い構造でした。

理想だけで言えば、OEM 側に `(freee_user_id, company_id)` を表す中間テーブルや external_id を追加するのが自然です。ただ、OEM 側はリアーキ中で、スキーマ変更や API 変更を依頼できない状況でした。そこで、OEM 側に既に存在していたログイン ID フィールドに、`freee_user_id` と `company_id` の組み合わせを埋め込む設計にしました。

セキュリティ面では、PSIRT から『ログイン ID を書き換えられるなら、他人になりすませるのでは』という懸念が出ました。ここは、OEM 側のユニーク制約で他人が使っているログイン ID には変更できないこと、OIDC トークン内の値とログイン ID から取り出した値を照合することを説明しました。つまり、既存カラムを使いながらも、DB 制約とトークン検証を組み合わせて本人性を担保しました。

この話のポイントは、理想設計を理解した上で、変更できない外部システムの制約の中で、既存のデータ構造を安全に読み替えたことです。」

### 深掘り版

- 判断軸: freee の 1 user : N companies と OEM の 1 user : 1 company の不一致
- 検討した選択肢: 中間テーブル追加、external_id 追加、OEM 主キー指定、既存ログイン ID への合成 ID 埋め込み
- 採用理由: OEM 側のスキーマ/API を変えずに、事業所コンテキストを OIDC に載せられる
- トレードオフ: 既存ログイン ID に意味を持たせるため、検証ロジックと説明責任が増える
- 一番のリスク: ログイン ID 書き換えによるなりすまし懸念
- 今なら改善: 外部 ID カラムや中間テーブルを OEM 側に正式に持たせる。監査ログと異常検知もより明示的に設計する

## 1. 状況設定

freee Eラーニングは、freee 人事労務上で OEM 提供元の Eラーニング基盤を呼び出すプロダクト。OEM 側に従業員アカウントを作成し、OIDC で SSO 認証を行うことで、freee からシームレスに学習画面に遷移する設計。

- 2025 年 7 月〜2026 年 2 月、3 名チームでゼロから構築
- 2026 年 2 月にリリース達成、本番障害ゼロ、認証成功率 99.9% 以上
- **一番難しかったのは、freee と OEM の構造的な不一致を、OEM 側に変更を一切依頼せずに解決したこと**

---

## 2. 制約の整理

### 制約 1: freee の構造(1 ユーザー : N 事業所)
- freee は 1 ユーザーが**複数の事業所**に所属可能
- 副業、グループ会社の親子関係などで日常的に発生する構造
- ユーザー側からは「今どの事業所のセッションか」を切り替えながら使う

### 制約 2: OEM の構造(1 ユーザー : 1 会社)
- OEM の OIDC は **1 ユーザー = 1 会社**という前提で設計されている
- OEM 側のユーザーテーブルにも会社 FK が単一カラムで持たれる
- freee の 1:N 構造を素直に表現する手段がない

### 制約 3: OEM がリアーキ中で仕様変更不可
- OEM 自体が大規模リファクタ中
- 追加のスキーマ変更・API 仕様変更には**一切応じてもらえない**
- 「OEM 側にテーブルを足す」「外部 ID を持たせる」系の正攻法はすべて NG

### 制約 4: チーム規模
- 3 名(自分 + ジュニア 2 名)
- 保守容易性が強い制約
- 複雑な認証ロジックは将来の運用で事故になりやすい

### 制約 5: リリース日固定
- ビジネス側のローンチ計画が動いていてリリース日は動かせない
- 「設計し直して半年遅延」は不可

### 制約 6: PSIRT(社内セキュリティチーム)通過必須
- freee は金融・労務系プロダクトを抱える会社、セキュリティ審査が厳しい
- 認証フローは PSIRT のレビュー通過がリリース条件
- なりすまし・権限混線リスクは特にシビアに見られる

## 3. 課題の本質

### 構造的不一致

```
freee 側                              OEM 側
┌───────────────┐                    ┌──────────────────┐
│ User A        │                    │ User in Company1 │
│  ├─ 事業所1   │  ← OIDC →          │ User in Company2 │
│  ├─ 事業所2   │                    │ User in Company3 │
│  └─ 事業所3   │                    │ ...              │
└───────────────┘                    └──────────────────┘
   1 user : N companies               1 user-row = 1 company
```

### 認証時に発生する曖昧性

OIDC のフローでは「誰がログインするか」を ID として OEM に渡す。
ところが freee の User A は事業所 1, 2, 3 のどれとしてもログインしうる。

- OEM 側に渡す `sub`(認証主体)が `User A` のままだと、OEM は「どの会社のユーザー?」が決められない
- 仮に OEM が `User A` を Company1 と紐付けて作っていたら、Company2 として研修を受けたい時にログインできない
- **「freee の事業所コンテキストを OIDC に乗せて運ぶ手段が必要」**

### 普通の解 → ぜんぶ封じられている

| 普通の解 | なぜ NG か |
|---|---|
| OEM 側に `(user_id, company_id)` 中間テーブル追加 | OEM がリアーキ中で仕様変更不可 |
| OEM のユーザーテーブルに `external_id` カラム追加 | 同上 |
| OEM の API 仕様を拡張して認証時に company_id を別途送る | 同上 |
| freee 側で会社別に別ユーザーを作って OEM と紐付け | OEM の課金単位が「ユーザー数」、コスト爆発 |

## 4. 設計判断 1: 中間テーブル(理想設計)を捨てた理由

### 課題
- 1:N の関係を素直に表現するなら、間に `(freee_user_id, freee_company_id, oem_user_id)` の中間テーブルを作るのが王道

### 検討した選択肢

#### 案 A: 中間テーブルで多対多を表現(理想設計)
- freee 側 + OEM 側の両方に紐付けテーブルを作り、OIDC の `sub` は中間テーブルの ID を使う
- **OEM 側のスキーマ変更が必要**
- 当時の OEM 状況では実現不可能

#### 案 B: 既存ログイン ID フィールドに合成 ID を埋め込む ⭐ **採用**
- OEM 側の任意文字列フィールド(ログイン ID)に `{user_id}_{company_id}` を埋め込む
- OEM のスキーマには 1 行も触らない

#### 案 C: OEM の主キー(企業アカウント PK)に freee 側の company_id を使ってもらう
- OEM が企業アカウントを作成するときに、自社採番ではなく freee 側の `company_id` を PRIMARY KEY として登録してもらう
- これなら freee の `(user_id, company_id)` をそのまま OEM 上のキーで突合できる
- **却下: OEM 側が単加採番(auto-increment)で PK を発番している実装で、外部から ID 値を指定する API も用意されていなかった**
- スキーマ・採番ロジックの両方に手を入れる必要があり、リアーキ中の OEM では実現不能

### 比較表

| 観点 | 案 A: 中間テーブル | 案 B: ログイン ID 埋め込み | 案 C: OEM 主キーに freee company_id |
|---|---|---|---|
| OEM 仕様変更 | 必要(スキーマ) | **不要** | 必要(採番ロジック + API) |
| 実現可能性 | × OEM リアーキ中 | ⭐ | × OEM が単加採番、外部指定 API なし |
| 実装スピード | × | ⭐ | × |
| 副作用範囲 | 小 | 小(専用フィールド) | 大(OEM 全体の PK 採番に影響) |
| PSIRT 通過 | ○ | △(後述) | ○ |
| 将来の巻き直し | n/a | やりやすい | n/a(実現不能) |

### 受け入れたトレードオフ
- 「合成 ID を文字列に埋める」は理想設計ではない
- ただし**制約条件下では他に手がない**(OEM 仕様変更不可 × リリース日固定)
- 巻き直しは「OEM がリアーキ完了後にスキーマ拡張に応じてくれた段階」で別途プロジェクト化できる
- ロードマップ上、3 社・4 社所属など複雑な拡張は計画されていない

## 5. 設計判断 2: 既存ログイン ID への埋め込み採用

### 課題
- OEM のどのフィールドに合成 ID を入れるか

### 判断軸 3 つ

1. **任意文字列であること** — フォーマット制約が緩く、合成 ID を入れても OEM のバリデーションに引っかからない
2. **専用フィールドであること** — 名前・メールなど他機能と意味が衝突するフィールドは避ける
3. **ユニーク性を担保できること** — 認証 ID なので一意性は外せない

### なぜ「ログイン ID」フィールドだったか

| フィールド候補 | 任意文字列? | 専用? | ユニーク? | 採用? |
|---|---|---|---|---|
| ログイン ID | ○ | ○(認証用) | ○(後述ユニーク制約) | ⭐ |
| 表示名 | ○ | × 表示用途 | × | × |
| メールアドレス | △ フォーマット制約 | × 通知用途 | ○ | × |
| メモ・備考 | ○ | ○ | × | × |

ログイン ID は元々「認証主体を表す任意文字列」として設計されているフィールド。**意味的にも一致**しているのが決め手。

## 6. 設計判断 3: フォーマット選定

### 課題
- 合成 ID のフォーマットをどう決めるか

### 採用フォーマット
```
{user_id}_{company_id}
```

例: `12345_678` → freee の user_id=12345 が事業所 678 としてログイン

### 判断軸

#### 区切り文字に `_` を選んだ理由
- `user_id` と `company_id` はどちらも整数
- 数字に出てこない、URL セーフ、ログでも目立つ
- 候補比較:

| 区切り | 数字に含まれない? | URL safe | OEM のログイン ID 制約に違反しない? | 評価 |
|---|---|---|---|---|
| `_` | ○ | ○ | ○ | ⭐ |
| `-` | ○ | ○ | ○ | ○(`_` と並んで第二候補) |
| `:` | ○ | △ | △ OEM 側の許容文字次第 | × |
| `.` | ○ | ○ | △ 浮動小数と誤認の可能性 | × |
| `\|` | ○ | × | × | × |

#### エスケープ問題
- 両方が整数なので**区切り文字の衝突は発生しない**
- 仮に将来 user_id・company_id が文字列化したら、エスケープ or URL-safe Base64 化が必要
- 今は整数なので素朴な split で十分

#### 文字数制限
- OEM のログイン ID フィールドは 64 文字程度の上限
- `user_id`(最大 10 桁) + `_` + `company_id`(最大 10 桁) = 21 文字
- 余裕は十分。将来 prefix(`v1_` など)を付ける拡張余地もある

## 7. 設計判断 4: なりすまし対策(DB 制約をセキュリティ境界として再解釈)

### 課題と PSIRT からの指摘

- ログイン ID は OEM 側で**後から書き換え可能**なフィールド
- 攻撃シナリオ: User A が自分のログイン ID を `99999_678`(他人の合成 ID)に書き換えて、他人の事業所として認証成立
- PSIRT「これなりすましじゃないか?」

### 解決: ユニーク制約を OEM 側にお願いした

- OEM 側でログイン ID にユニーク制約を設けてもらう
- → 既に `99999_678` が存在するので、別ユーザーが同じ値に書き換えようとすると DB レベルで弾かれる
- **物理的に他人の値への書き換えが不可能になる**

### なぜ「DB 制約」でセキュリティ境界として通せたか

通常、セキュリティ境界というと「アプリ層の認可チェック」「ミドルウェアの検証」をイメージする。今回はそれを **DB 制約に読み替えた**:

| レイヤー | 通常のセキュリティ境界 | 今回の設計 |
|---|---|---|
| アプリ層 | パラメータバリデーション、認可チェック | フォーマット検証(後述) |
| ミドルウェア | OIDC ライブラリの token 検証 | 標準のまま |
| **DB 層** | 通常はデータ整合性のため | **ユニーク制約 = なりすまし防止の境界** |

PSIRT の論理:
- 「アプリ層のチェックは抜け道があるかもしれないが、**DB ユニーク制約が effective なら、書き込み時点で物理的に弾かれる**」
- 「攻撃者がアプリを通さず DB に直接書ける状況は、もはや OEM 全体の侵害なので別問題」
- → 承認

### この判断の核となる洞察
- **DB 制約は「データ整合性」だけのものではなく、攻撃者の取れる行動を物理的に制限する装置として読める**
- アプリ層 + DB 層の二重で安全側に倒す、ではなく、**今回はアプリ層では弾けないので DB 層に押し込んだ**

### Migration イメージ

```ruby
# OEM 側でやってもらった migration(イメージ)
class AddUniqueIndexToOemUsersLoginId < ActiveRecord::Migration[7.0]
  def change
    add_index :oem_users, :login_id, unique: true
  end
end
```

freee 側でも、合成 ID を生成する直前に**重複チェックの API**(OEM が提供している `findByLoginId`)を呼んで、二重作成を防ぐ:

```ruby
def ensure_unique_oem_login_id!(user_id, company_id)
  login_id = "#{user_id}_#{company_id}"
  if OemClient.find_user_by_login_id(login_id).present?
    raise OemUserAlreadyExistsError, login_id
  end
  login_id
end
```

## 8. 設計判断 5: 暗黙の契約と silent failure 検知

### 課題
- `{user_id}_{company_id}` というフォーマット自体が **どこにも明文化されていない暗黙の契約**
- 壊れると「認証は通るが間違った事業所として通る」silent failure が発生する
- silent failure は本番で気付かないのが最悪

### 壊れるシナリオ

| シナリオ | 何が起きるか |
|---|---|
| OEM 側がログイン ID のフォーマット制限を変更(例: 数字のみに制限) | 合成 ID が作れず認証 API でエラー |
| freee 側で同じフィールドを別用途で使い始める | `{user_id}_{company_id}` ではない値が混入し、パースで誤判定 |
| `_` を使う別 prefix 機能が後から入る(例: `v2_12345_678`) | 区切り解析が崩壊して company_id が誤判定 |
| user_id / company_id がいずれ string 化(UUID 化等) | パース時にエスケープが必要になる |

### 採用した検知設計

#### Layer 1: 入口バリデーション(同期失敗)

OEM ユーザー作成 API を叩く側(自分たち)で、API レスポンスの validation エラーをハードフェイルにする。

```ruby
class OemUserCreator
  def create!(user_id, company_id)
    login_id = build_login_id(user_id, company_id)
    response = OemClient.create_user(login_id: login_id, ...)
    raise OemValidationError, response.errors if response.errors.present?
    response
  rescue OemValidationError => e
    DeadLetterQueue.push(
      kind: :oem_user_create_validation_failed,
      payload: { user_id:, company_id:, login_id:, errors: e.message }
    )
    PagerDuty.notify("OEM user create validation failed", e)
    raise
  end

  private

  def build_login_id(user_id, company_id)
    "#{user_id}_#{company_id}"
  end
end
```

OEM 側のフォーマット仕様変更があった場合、validation エラーで **必ず DLQ に積まれる + PagerDuty 通知**が飛ぶ。リリース後の沈黙障害を避ける。

#### Layer 2: パース時のフォーマット検証

OIDC 認証時に OEM から渡ってくるログイン ID を、必ずパーサ経由で扱う。

```ruby
class OemLoginIdParser
  FORMAT = /\A(\d+)_(\d+)\z/.freeze

  Result = Struct.new(:user_id, :company_id)

  def self.parse(login_id)
    m = FORMAT.match(login_id)
    raise InvalidFormatError, login_id unless m
    Result.new(m[1].to_i, m[2].to_i)
  end
end

# OIDC コールバック内
class OidcController
  def callback
    oem_login_id = oidc_userinfo.sub
    parsed = OemLoginIdParser.parse(oem_login_id)
    sign_in_with_company(parsed.user_id, parsed.company_id)
  rescue OemLoginIdParser::InvalidFormatError => e
    DeadLetterQueue.push(
      kind: :oidc_login_id_parse_failed,
      payload: { login_id: oem_login_id, error: e.message }
    )
    PagerDuty.notify("OIDC login id parse failed", e)
    render :error, status: :unprocessable_entity
  end
end
```

#### Layer 3: 暗黙の契約をテストで保護

```ruby
# spec/oem_login_id_format_spec.rb
RSpec.describe OemLoginIdParser do
  it "is the contract: '{user_id}_{company_id}'" do
    # この test が落ちたら、フォーマット契約が壊れたサイン
    expect(OemLoginIdParser.parse("12345_678").user_id).to eq(12345)
    expect(OemLoginIdParser.parse("12345_678").company_id).to eq(678)
  end

  it "rejects non-conforming formats" do
    expect { OemLoginIdParser.parse("12345-678") }.to raise_error(InvalidFormatError)
    expect { OemLoginIdParser.parse("v2_12345_678") }.to raise_error(InvalidFormatError)
    expect { OemLoginIdParser.parse("abc_def") }.to raise_error(InvalidFormatError)
  end
end
```

#### Layer 4: ドキュメント化
- ADR(Architecture Decision Record)に「OEM ログイン ID = `{user_id}_{company_id}` 契約」を明文化
- パーサとビルダを単一ファイルに集約 (`OemLoginIdParser` / `OemLoginIdBuilder`) し、grep 一発で全使用箇所が見つかるようにする

## 9. 設計判断 6: 将来の拡張性

### Q: 1 ユーザーが 3 社、4 社所属になった場合、この設計は破綻するか?

**結論: 破綻しない**

- 合成 ID は `{user_id}_{company_id}` の組み合わせ単位で OEM 側にユーザーレコードができる
- 3 社所属になれば OEM に 3 行できる(`12345_678`, `12345_679`, `12345_680`)
- 認証時はユーザーがどの事業所として入るかで `sub` が変わるだけ
- アプリロジックは `(user_id, company_id)` の組み合わせを引くだけなので変更不要

### Q: フォーマット変更が必要になった場合の影響範囲は?

- すべて `OemLoginIdParser` / `OemLoginIdBuilder` に集約
- バージョニングを足すなら `v2_{user_id}_{company_id}` のように prefix で識別可能(64 文字制約に余裕あり)
- パーサ側で `v2_` の場合の分岐を追加するだけで段階移行可能

### Q: 中間テーブル巻き直し計画は?

- ロードマップ上、近い将来の大きな拡張はなし
- 巻き直しコスト > 維持コストの状態が当面続く見込み
- OEM のリアーキ完了後、スキーマ拡張が解禁されたタイミングで再評価する

## 11. 面接ストーリー(完成版・ノーカット)

### 状況設定(30 秒)

「freee Eラーニングは、OEM 提供元の Eラーニング基盤を OIDC 連携で呼び出すプロダクトでした。3 名チーム 7 ヶ月でゼロから作って、2026 年 2 月にリリースしました。

一番難しかったのは、freee と OEM の構造的な不一致を、OEM 側に変更を一切依頼せずに解決したことです。」

### 制約整理(45 秒)

「具体的には、freee は 1 ユーザーが複数の事業所に所属できる構造で、副業やグループ会社でよくあるケースなんですが、OEM 側の OIDC は 1 ユーザー = 1 会社という前提で設計されていました。

普通に解くなら OEM 側に中間テーブルを足すか、外部 ID カラムを足すかなんですが、OEM 自体がリアーキ中で、追加のスキーマ変更には一切応じられないという状況でした。リリース日は固定で、PSIRT の審査も通す必要がある。

つまり、OEM 側は 1 行も触れない、けど 1:N の関係を OIDC に乗せて運ぶ必要がある、という制約条件でした。」

### 設計判断(1 分)

「採用したのは、OEM 側の既存のログイン ID フィールドに `{user_id}_{company_id}` という合成 ID を埋め込む方式です。

中間テーブルが理想なのは分かっていましたが、OEM 仕様変更不可という制約で実現不可能。別カラムを流用する案もあったんですが、専用フィールドではないので副作用が読めず却下しました。

ログイン ID フィールドを選んだ理由は 3 つで、任意文字列でフォーマット制約が緩いこと、認証主体を表す専用フィールドで意味が一致していること、ユニーク性を担保できる前提だったこと。区切り文字は `_` にしました。user_id・company_id がどちらも整数なので、数字と衝突しない、URL セーフ、ログで目立つ、という基準です。」

### セキュリティ判断(45 秒)

「PSIRT のレビューで指摘されたのが、ログイン ID は後から書き換えできるフィールドだから、他人の合成 ID に書き換えてなりすまし可能じゃないか、というものでした。

これに対して、OEM 側でログイン ID にユニーク制約を設けてもらうことで解決しました。物理的に既存の値への書き換えができなくなるので、攻撃者の取れる行動が DB レベルで制限される。

これは、いつもはデータ整合性のために使う DB 制約を、セキュリティ境界として再解釈した判断でした。アプリ層では弾けないチェックを、DB 層に押し込む形です。私としては「攻撃者が DB に直接書ける状況はもう OEM 全体の侵害なので認証フローのスコープ外」という整理でレビューに出し、結果として承認をもらえました。」

### 検知設計(30 秒)

「もう一つ意識したのが、`{user_id}_{company_id}` というフォーマットがどこにも明文化されていない暗黙の契約だったことです。これが壊れると、認証は通るけど間違った事業所として通る silent failure になる。

なので、OEM ユーザー作成 API のバリデーション失敗を DLQ で受けてログ(Datadog / Bugsnag)から追える形にする、OIDC コールバックでも parse 失敗を同様に DLQ に逃がす、フォーマット契約自体を rspec で保護する、ADR にドキュメント化する、という多層の検知を入れました。能動的な PagerDuty アラートは入れていなかったので、今やり直すなら DLQ → PagerDuty まで繋ぎたいです。

結果、認証成功率 99.9% 以上、本番障害ゼロでリリースできました。スピード重視のスタートアップ的な状況でしたが、唯一の懸念だったセキュリティは DB 制約で担保し、silent failure リスクは検知レイヤーで担保する、という整理ができたのが良かったと思っています。」

## 12. 想定 Q&A

### Q1: なぜ中間テーブルにしなかったのか?(理想設計を聞かれる)
「理想は中間テーブルでした。ただ OEM 側のスキーマ変更が必要で、OEM がリアーキ中で仕様変更不可という制約があった。リリース日固定の中で、OEM 仕様変更を待つ選択肢はなかったので、`{user_id}_{company_id}` 埋め込みに倒しました。OEM のリアーキ完了後にスキーマ拡張が解禁されたら、巻き直しを再評価する想定です。」

### Q2: なぜ任意文字列フィールドへの埋め込みか?
「ログイン ID は元々『認証主体を表す任意文字列』として設計されているフィールドで、意味的にも合成 ID の用途と一致していたからです。表示名やメールなど、他機能と意味が衝突するフィールドは副作用が読めないので避けました。」

### Q3: 区切り文字に `_` を選んだ理由は?
「user_id と company_id がどちらも整数なので、数字と衝突しない、URL セーフ、ログで目立つ、という基準です。`-` も同条件ですが、OEM のログイン ID 制約に違反しないかを確認した上で `_` の方が他のフィールドで先例があったので採用しました。」

### Q4: ユニーク制約で本当にセキュリティ要件を満たすか?
「アプリ層のチェックは抜け道がありえますが、DB ユニーク制約は書き込み時点で物理的に弾かれるので、なりすましの実行手段がなくなります。攻撃者が DB に直接書ける状況はもはや OEM 全体の侵害なので認証フローのスコープ外、という整理でレビューに出し、結果として承認をもらえました。」

### Q5: 暗黙の契約が壊れる silent failure をどう検知する?
「3 層あります。OEM の create_user API バリデーション失敗を DLQ で受けてログから追える、OIDC コールバックの parse 失敗を同様に拾う、フォーマット自体を rspec の契約テストで保護する。仕様変更があれば必ずどれかで検知できる構造にしました。リリース時点では能動的な PagerDuty アラートまでは入れていなかったので、今やり直すなら DLQ → PagerDuty まで繋ぎたいです。」

### Q6: パース処理にバグがあった時のフェイルセーフは?
「parse 失敗は raise して認証を中断します。間違った事業所として通すよりログインさせない方が安全側、という判断です。同時に DLQ に積まれて Datadog / Bugsnag のログから追えるので、長時間の沈黙障害にはなりません。能動的な通知層は今やり直すなら PagerDuty で繋ぎたいです。」

### Q7: 1 ユーザーが 3 社、4 社所属の拡張時に破綻するか?
「破綻しません。OEM 側に `{user_id}_{company_id}` の組み合わせ単位で行ができるので、3 社所属なら 3 行できるだけです。アプリは組み合わせを引くだけなので変更不要です。」

### Q8: 技術負債としての扱いは?将来巻き直す計画は?
「明確に技術負債として ADR に書いています。ただロードマップ上の近い将来に大きな拡張はないので、巻き直しコスト > 維持コストの状態が続く見込み。OEM のリアーキ完了後にスキーマ拡張が解禁されたタイミングで再評価します。」

### Q9: PSIRT との議論で一番争点になったのは?
「『書き換え可能なフィールドにセキュリティ依存していいのか』という点です。最終的には『DB ユニーク制約が effective である限り書き換え経路が物理的にない』という整理で通りました。アプリ層のチェックを増やすより、書き換え自体を不可能にする方が監査しやすいという合意でした。」

### Q10: 今やり直すなら?
「2 つあります。1 つは、OEM 側のスキーマ拡張が解禁されたら中間テーブルに巻き直す。これは負債解消です。もう 1 つは、合成 ID にバージョン prefix(`v1_` 等)を最初から入れておくこと。今のフォーマットだと将来の段階移行で `v2_` を導入するときに分岐が必要になりますが、最初から `v1_` を付けておけば parser の分岐が自然に書けます。文字数制約上も余裕があるので、最初からやるべきだったと思っています。」

## 15. 議論の経緯メモ

このドキュメントは、freee Eラーニング案件の OIDC 統合パートを、面接で深掘りされた時の引き出しとして整理したもの。元のカンペ(technical-interview-cheatsheet.md L55-L86)は要点だけを箇条書きで書いており、設計判断の流れまでは展開されていなかった。

設計の進化:

1. **第 1 案: 中間テーブル(理想設計)**
   - freee + OEM 両方に紐付けテーブル
   - OEM 仕様変更不可で実現不能 → 却下

2. **第 2 案: OEM 主キーに freee の company_id を使ってもらう**
   - OEM 企業アカウント作成時に PRIMARY KEY として freee 側の `company_id` を採用してもらう
   - 却下: OEM が単加採番(auto-increment)で PK を発番する実装で、外部から ID を指定する API もなし
   - スキーマと採番ロジック両方への変更が必要 → リアーキ中の OEM では実現不能

3. **第 3 案: ログイン ID フィールドへの `{user_id}_{company_id}` 埋め込み**(採用)
   - 任意文字列 + 専用フィールド + ユニーク性担保可能、の 3 条件を満たす唯一の選択肢

4. **セキュリティ懸念の発見と対処**
   - PSIRT「書き換えてなりすましできるのでは?」
   - → OEM 側ログイン ID にユニーク制約 → DB レイヤーをセキュリティ境界として再解釈
   - 「アプリ層では弾けない、DB 層で物理的に弾く」というのが核となる洞察

5. **検知設計の追加**
   - 「フォーマットが暗黙の契約」リスクへの対処
   - 4 層構造: API バリデーション失敗 / parse 失敗 / 契約テスト / ADR ドキュメント
   - silent failure を防ぐ多層検知

各設計判断は「課題 → 選択肢 → 採用 → 理由 → トレードオフ」の流れで整理。面接で深掘りされた時に、設計プロセスの質と、制約条件下での合理的判断を語れる構成にしている。
