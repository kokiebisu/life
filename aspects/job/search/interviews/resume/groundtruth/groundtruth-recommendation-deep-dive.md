## 0. 面接中に見るカンペ

### 最初の 15 秒

「Groundtruth では、広告キャンペーン作成時のセグメント選択で離脱が出ていました。選択肢が数千件あり、ユーザーが選び切れない状態です。そこで、Go のマイクロサービスで ML と Elasticsearch を並列に呼び、失敗時は Redis の人気 Top 5 に落とすレコメンド基盤を作りました。狙いは、画面操作の体感を重くせずに『効果的な候補を必ず返す』ことです。」

### まず伝える 3 点

1. **課題:** セグメントが多すぎて離脱。広告効果も落ちる。
2. **設計:** Go サービスで ML と ES を並列実行。3 段階フォールバック。
3. **成果:** churn 約 20% 減、広告インプレッション約 10% 増、締切 2 週間前倒し。

### 深掘り版

| 聞かれ方 | まず答えること |
|---|---|
| なぜ Go? | ML と ES という外部 I/O を並列に待ち、timeout / fallback を素直に扱いたかった。Go は goroutine と context が標準で使いやすい。 |
| 並列処理は本当に必要? | 100ms を削るためだけではない。互いに依存しない外部依存を直列にして tail latency を足し合わせないため。 |
| ML が 800ms なら並列化しても厳しくない? | その通り。だから ML を無制限に待たず、遅い時は ES / Redis に落とす。並列化は timeout / fallback しやすくする設計でもある。 |
| 学習コストを払ってまで Go? | 学習コストは認める。ただ、gevent の運用リスクや timeout / fallback の複雑さより小さいと判断した。 |
| なぜマイクロサービス? | 画面の体感速度、独立スケール、ML 障害の分離を同時に取りたかった。 |
| 障害時どうする? | ML 失敗なら ES、両方失敗なら Redis Top 5。ゼロ件を避ける。 |
| 品質はどう見る? | レイテンシだけでなくフォールバック率を見る。速くても ML が死んでいたら品質は落ちている。 |
| 反省点は? | リリース前の shadow traffic と cold start 対策をもっと早く入れるべきだった。 |

### 絶対に混ぜない言い方

- **実装したと言える:** Go マイクロサービス、ML + ES 並列実行、3 段階フォールバック、Redis Top 5、Datadog / Sentry / PagerDuty、成果数値。
- **確認が必要:** `context.WithTimeout` の細部、`sync.WaitGroup` かどうか、フォールバック率 5% の閾値、goroutine 数監視、ECS min instance、Datadog overlay。
- **言い方:** 確認が必要なものは「当時の実装詳細は確認が必要ですが、設計としてはこう切るのが自然です」「今やり直すならこうします」と前置きする。

### 30 秒版

「一番大きい判断は、レコメンドを Go の独立サービスにしたことです。ML と Elasticsearch は互いに依存しない外部 I/O だったので、直列に待つとレイテンシのばらつきまで足し合わされます。そこで Go で ML と ES を並列に呼び、ML が遅い/落ちた場合は ES、両方厳しい場合は Redis の業種別 Top 5 を返す 3 段階フォールバックにしました。レコメンドは決済と違って強整合性より可用性が大事なので、ゼロ件で UI を壊さないことを優先しました。」

### 突っ込み対応: 「並列処理で行う必要があったんですか?」

**短く答えるなら:**

「はい、必要でした。ただし理由は『100ms を削りたいから』だけではありません。ML の推薦と ES の検索は互いに依存していません。片方の結果を使ってもう片方を呼ぶわけではないので、直列に待つ理由がありません。外部依存を直列にすると、平均値だけでなく遅延のばらつきや timeout も足し合わされるので、体感が不安定になります。」

**さらに聞かれたら:**

「もちろん、並列処理には複雑さがあります。なので無条件に並列化したわけではありません。今回は外部依存が 2 つだけで、1 リクエストあたり goroutine 2 本に固定できます。数が増え続ける fan-out ではないので、管理できる複雑さだと判断しました。

また、片方が失敗してももう片方の結果を使えます。直列にすると、先に呼んだ ML が timeout した時点で ES を呼ぶ時間も失いやすい。並列なら ML が遅れても、ES の結果だけで Tier 2 として返せます。つまり並列化はレイテンシだけでなく、フォールバックのしやすさにも効いています。」

### 突っ込み対応: 「ML が 800ms なら、並列化してもそこまで縮まらないのでは?」

**短く答えるなら:**

「おっしゃる通りで、並列化しても 800ms が 300ms になるわけではありません。なので、100ms だけを根拠にしていたら弱いです。ここでの本質は、ML と ES の遅延や失敗を独立に扱えることです。並列なら、ML が遅れても ES の結果で返せますし、両方厳しければ Redis Top 5 に落とせます。」

**さらに聞かれたら:**

「ただし、ML が律速なのは事実です。だから ML を無制限に待つ設計にはしません。画面操作として重く感じない範囲に deadline を置き、その中で ML が返れば使う。返らなければ ES の結果、または Redis Top 5 に落とします。

つまり、ポイントは 2 つです。1 つ目は、互いに依存しない外部 I/O を直列にしないこと。2 つ目は、遅い依存先を待ち続けずに fallback することです。レコメンドは完璧な結果より、ユーザーがキャンペーン作成を続けられることが大事なので、この判断にしました。」

**言い切るポイント:**

- 並列化後の下限は `max(ML, ES)`。ML が律速なのは認める。
- 並列化の価値は「100ms 削る」ではなく「外部依存の遅延と失敗を独立に扱える」こと。
- ML が遅い時は、待ち続けるのではなく品質を落として返す。
- だからレイテンシだけでなくフォールバック率を見る。

**言い切るポイント:**

- 「高速化したかった」ではなく「体感速度とフォールバックを両立するために必要だった」
- 「goroutine を使いたかった」ではなく「独立した I/O を待っているだけなので並列化が自然だった」
- 「複雑になる点は認める。ただし 2 本固定なので制御できる」と言う

### 突っ込み対応: 「学習コストがかかるのに Go にする必要があったんですか?」

**短く答えるなら:**

「学習コストはありました。ただ、ここで比較したのは『Go の学習コスト』と『Python のまま timeout / fallback / 障害分離を複雑に抱えるコスト』です。既存 Flask に足すと直列になりやすく、Flask + gevent だと monkey patch の副作用が大きい。Go は goroutine と context timeout が標準機能なので、長期運用ではこちらの方がリスクが小さいと判断しました。」

**さらに聞かれたら:**

「チーム全体を Go に移行したわけではありません。切り出したのは、レコメンドの集約 API という小さめで境界が明確な部分です。HTTP で request/response を受けて、ML、ES、Redis を呼び、結果を返す。ドメインロジックが巨大な既存モノリス全体を書き換える話ではありません。

なので、学習コストを払う範囲を限定できました。運用面でも、単一バイナリ、Docker 1 コンテナ、標準ライブラリ中心にして、チームが追える構成に寄せています。Go を採用した理由は新技術を入れたいからではなく、並列I/O・タイムアウト・障害分離の 3 つを一番素直に満たせたからです。」

**言い切るポイント:**

- 「Go が好きだから」ではなく「Python 案のリスクが大きかった」
- 「チームに負担をかけた」ではなく「境界が小さいサービスに限定して学習コストを抑えた」
- 「新技術導入」ではなく「外部依存を扱う運用リスクの判断」

### 2 分版

「Groundtruth の Ads Manager で、広告キャンペーン作成時のセグメント選択がボトルネックになっていました。セグメント数が数千件まで増えていて、Google Analytics で離脱が見えていました。効果の薄いセグメントを選ぶケースもあり、プロダクト側からレコメンド機能を入れる意思決定がありました。

設計で一番重かった制約は、キャンペーン作成画面の体感を重くしないことです。ML エンドポイントと Elasticsearch はどちらも外部依存で、直列に待つと平均値だけでなく遅延のばらつきや timeout も足し合わされます。既存の Ads Manager は Flask モノリスでしたが、WSGI 同期処理にそのまま足すと timeout / fallback の扱いが複雑になります。Flask + gevent も検討できますが、monkey patch の副作用と運用経験の薄さがありました。なので Go の独立サービスにして、goroutine で ML と ES を並列に呼ぶ設計にしました。

障害前提の設計にもしています。ML は別チーム管理なので、デプロイの影響で落ちることがありえます。そこで、ML と ES が両方成功したらパーソナライズ結果、ML が落ちたら ES のみ、両方落ちたら Redis に持っている業種別 Top 5 を返すようにしました。レコメンドは『完全に最新であること』より『何か返ってユーザーが進めること』の方が重要です。

運用では Datadog でレイテンシとフォールバック率、Sentry で例外、PagerDuty で閾値超過通知を見る形にしました。レイテンシだけだと、ML が落ちて ES だけで速く返っている状態を見逃します。なので、フォールバック率を品質指標として持つのがポイントです。結果として、リリース後の Q 単位集計でキャンペーン作成時の churn が約 20% 減、広告インプレッションが約 10% 増になりました。」

## 1. 状況設定

Groundtruth は B2B 広告 SaaS(米国本社、フルリモート)。広告主が広告キャンペーンを作成する Ads Manager というプロダクトの一機能として、レコメンド基盤を設計・実装した。

### 既存の挙動と課題
- 広告キャンペーン作成時、ユーザーは「セグメント(配信先の絞り込み条件)」をドロップダウンから選ぶ必要があった
- セグメントの選択肢が肥大化(数千件オーダー) → ユーザーが途中で離脱
- 効果の薄いセグメントを選んでしまうケースも多発 → 広告効果も伸びない
- Google Analytics で離脱率の数値が見えていた → レコメンド追加の意思決定がプロダクト側で出ていた

### ゴール
- セグメント選択フェーズの離脱率を下げる
- 広告主に「選びやすい・効果的な」セグメントを推薦する
- → **レコメンデーション機能の追加**

### 解決の方針
- **未経験ユーザー** → 業種別人気 Top 5 を表示
- **経験ユーザー** → ML によるパーソナライズ候補を表示
- 両者の結果をマージしてフロントに返す

## 2. 制約の整理

### 制約 1: 画面操作の体感速度
- キャンペーン作成中に待たされると、元の「ドロップダウン直接選択」より UX が悪化する
- 1秒前後を体感上の目安にしつつ、絶対値よりも「遅延のばらつき」と「ゼロ件で止まること」を避ける必要があった

### 制約 2: 外部依存のレイテンシ特性

| 依存先 | 最大レイテンシ | 性質 |
|---|---|---|
| ML エンドポイント(AI チーム管理) | 最大 **800ms** | パーソナライズ予測 |
| Elasticsearch | 約 **300ms** | セグメント全文検索 |

### 制約 3: 直列呼び出しでは外部依存の遅延が足し合わされる
- ML と ES は互いに依存しない別データ取得
- 直列にすると、平均値だけでなく遅延のばらつきや timeout も足し合わされる
- → **並列化 + timeout + fallback** をセットで考える必要があった

### 制約 4: ML エンドポイントの可用性
- AI チームが別管理
- AI チームのデプロイ影響で **ML が一時的に落ちる** ことが現実的に予想された
- 「ML 必須」設計だと、AI チームのデプロイ毎にレコメンドが死ぬ

### 制約 5: 既存システム
- Ads Manager は Flask モノリス(Python、WSGI 同期処理)
- 既存チームは Python 中心
- `asyncio.gather()` のような並列化が素直に使えない

### 制約 6: 観測・運用
- 既存スタックは Datadog(メトリクス) + Sentry(例外)
- フォールバックが起きたことを定量的に把握できる必要がある

## 3. 設計判断 1: モノリス vs マイクロサービス

### 課題
- レコメンド機能をどこに実装するか。既存 Flask モノリスに足すか、独立サービスに切り出すか。

### 検討した選択肢

#### 案 A: 既存 Flask モノリスに Python で追加
- ❌ Flask + WSGI は同期処理。`asyncio.gather()` が使えない
- ❌ ML と ES を直列で呼ぶと、外部依存の遅延と失敗が足し合わされる
- ❌ モノリスのデプロイサイクルに引きずられる
- → 体感速度とフォールバック設計の両面で弱い

#### 案 B: FastAPI で独立サービス化
- ✅ Python のまま async/await で外部 I/O を並列に扱える
- ✅ 既存チームの Python 知識を活かしやすい
- ✅ Flask + gevent より副作用が少ない
- △ async 対応 client を end-to-end で揃える必要がある
- △ sync client が混ざると event loop を塞ぎ、設計意図が崩れる
- △ 当時のチームでは FastAPI 本番運用の知見が薄かった

#### 案 C: 既存言語(Flask/Python)で独立サービス化
- ✅ 独立デプロイ可能
- △ Flask + gevent でモンキーパッチして並列化はできる
- ❌ gevent はモンキーパッチの副作用が広範
- ❌ チーム内に gevent の運用実績がない
- ❌ 障害時のデバッグが難しくなる

#### 案 D: Go で独立マイクロサービス化 ⭐ **採用**
- ✅ goroutine + channel でネイティブ並列
- ✅ `context.WithTimeout` で deadline 伝播がきれい
- ✅ 静的型付けで運用安全性が高い
- ✅ 単一バイナリでデプロイがシンプル
- △ チームに Go 経験者は少なめ(自分が中心)

### 採用理由(判断軸 3 つ)

1. **サービス分離が本質**: 既存モノリスから切り出し、ML 障害を Ads Manager 全体に波及させない
2. **deadline / fallback の一貫性**: Go の `context` で、ML / ES / Redis への deadline を同じ考え方で伝播できる
3. **運用時の読みやすさ**: goroutine と timeout の責務がコード上で見えやすく、sync client 混入や event loop blocking を気にしなくてよい

### 面接での答え方: 「Go にする必要があったのか?」

ここは「Go が唯一解」ではなく、**本質はサービス分離・deadline・fallback で、Go はその実装リスクが低い選択だった** と答える。

「Go でなければ実現できない、とは言いません。FastAPI でも async client をきちんと揃えれば成立します。ここでの本質は、既存モノリスから切り出して、ML / ES / Redis への deadline と fallback を明示的に扱うことでした。

その上で当時は、FastAPI の本番運用知見がチーム内に薄く、async 対応 client を end-to-end で揃える運用リスクがありました。sync client が混ざると event loop を塞いで、設計意図が崩れます。Go は goroutine と context timeout が標準で、単一バイナリとして運用しやすい。小さな集約サービスに限定すれば、学習コストよりも実装と運用の読みやすさが勝つと判断しました。」

補足で言えること:

- チーム全体を Go 化したわけではない。レコメンド集約 API だけを切り出した。
- HTTP の request/response 境界が明確で、サービスサイズも限定的だった。
- 標準ライブラリ中心にして、Go 固有の高度な抽象は避けた。
- FastAPI は否定しない。今なら FastAPI + async client も真面目に比較する。
- 学習コストは認めた上で、deadline / fallback の一貫性と運用時の読みやすさを優先した。

### 比較表

| 軸 | 案 A: Flask モノリス | 案 B: FastAPI | 案 C: Flask + gevent | 案 D: Go ⭐ |
|---|---|---|---|---|
| 並列実行 | 直列になりやすい | async/await | gevent monkey patch | goroutine 標準 |
| 副作用リスク | 低いが要件を満たしにくい | sync client 混入に注意 | 大(stdlib 書き換え) | 低い |
| 運用実績 | 既存運用あり | 当時は薄い | 当時は薄い | 学習コストあり |
| 障害分離 | ❌ | ✅ | ✅ | ✅ |
| deadline / fallback | 複雑 | client 統一が必要 | greenlet 前提で複雑 | context で一貫 |

### 受け入れたトレードオフ
- Python 中心のチームに Go の運用知識が増える
- マイクロサービス分離による運用コスト(デプロイ・監視・契約 API 管理)
- FastAPI でも成立しうるため、「Go が唯一解」とは言わない
- → 当時は小さな集約サービスに限定し、goroutine と context による deadline / fallback の一貫性を優先した

## 4. 設計判断 2: 言語選定(FastAPI / Flask + gevent / Go)

### 課題
- 「独立サービス化」を決めた後、Python(FastAPI / Flask + gevent)で書くか Go で書くかの選択。

### FastAPI 案
- async/await で外部 I/O を並列に扱える
- Python のままなので既存知識を活かしやすい
- Flask + gevent より副作用が少ない
- △ `httpx.AsyncClient` など async 対応 client を揃える必要がある
- △ 一部だけ sync client が混ざると event loop を塞ぐ
- △ 当時のチームでは FastAPI の本番運用実績が薄かった
- → **今なら有力候補。少なくとも真面目に比較する**

### Flask + gevent 案
- gevent は monkey patch で stdlib の I/O を greenlet 化
- `gevent.spawn()` で並列タスクを起動できる
- ❌ monkey patch の副作用: requests, ssl, threading など stdlib 全体が書き換わる
- ❌ デバッグが難しい(スタックトレースが greenlet 化されて読みにくい)
- ❌ ライブラリ互換性問題(C 拡張モジュールはパッチが効かないものがある)
- ❌ チームに gevent の運用実績が薄い

### Go 案 ⭐ **採用**
- goroutine + channel が言語仕様レベルで提供
- `context.WithTimeout` が標準ライブラリ、deadline 伝播が明確
- ML / ES クライアントとも、言語標準の HTTP クライアントで OK
- デプロイは ECS の Docker 1 コンテナで完結

### 判断軸 3 つ

| 軸 | FastAPI | Flask + gevent | Go ⭐ |
|---|---|---|---|
| 並列実行の素直さ | async/await | monkey patch 必須 | goroutine 標準 |
| 副作用 | sync client 混入に注意 | stdlib 書き換え | 低い |
| タイムアウト制御 | client ごとの設計が必要 | greenlet 単位、煩雑 | `context.WithTimeout` 標準 |
| 運用実績 | 当時は薄い | 当時は薄い | 小さなサービスに限定して吸収 |

### 受け入れたトレードオフ
- Python チームに Go の知識が必要
- 既存 Flask モノリスとの言語混在
- FastAPI でも成立しうる
- → 当時は deadline / fallback の一貫性と単一バイナリ運用を優先し、Go が長期保守性で勝つと判断

## 5. 設計判断 3: 並列実行アーキテクチャ

### 課題
- ML と ES を並列で呼び、外部依存の遅延と失敗を独立に扱えるようにする
- 一方が失敗してももう一方の結果を活かしたい(graceful degradation)

### 採用設計
- **goroutine + channel + context.WithTimeout**
- ML と ES を 2 本の goroutine で同時起動
- 各呼び出しに context タイムアウトを必須化(リーク防止)
- 結果は構造体に書き込んで `sync.WaitGroup` で待ち合わせ

### 重要: 並列化だけで体感速度を守るわけではない

並列化後の理論値は、直列の足し算ではなく、遅い方の外部依存に近づく。

ただし、これは ML の遅さが消えるという意味ではない。ML が律速であることは変わらない。だから、体感速度は以下のセットで考える。

1. **並列化:** 互いに独立した外部 I/O を直列に待たない。
2. **deadline:** 画面操作として重く感じる前に待ち切る。
3. **fallback:** ML が deadline 内に返らなければ、ES / Redis の結果で返す。

面接ではこう言う:

「並列化だけで ML が速くなるわけではありません。ML が律速なのはその通りです。ただ、ML と ES は互いに独立した外部 I/O なので、直列に待つと遅延のばらつきや timeout まで足し合わされます。並列化の価値は、片方が遅い/落ちた時にもう片方の結果で返せることです。その上で、ML が deadline を超える場合は待ち続けず、ES または Redis に落とします。」

### 面接での答え方: 「並列処理は必要だったのか?」

必要だった。理由は **依存関係** と **フォールバック** の 2 つ。

1. 依存関係: ML と ES は互いに独立した I/O。ML の結果を使って ES を呼ぶわけではない。直列に待つ理由がない。
2. フォールバック: ML が遅延・失敗しても、ES が成功していれば Tier 2 として返せる。

さらに重要なのは、並列化がフォールバックにも効くこと。

- 直列で ML を先に呼び、ML が timeout すると、ES を呼ぶ時間も失いやすい。
- 並列なら ML が遅延・失敗しても、ES が成功していれば Tier 2 として返せる。
- つまり、並列化は「速くするため」だけでなく「片方が落ちても使える結果を残すため」の設計。

複雑さへの答え:

「並列処理は複雑さを増やします。なので、無制限に fan-out する設計にはしていません。今回は 1 リクエストあたり ML と ES の 2 本だけで固定です。goroutine 数の上限が読めるので、timeout とメトリクスを入れれば運用できる複雑さだと判断しました。」

### コード骨子

```go
type recommendResult struct {
    ml    *MLResult
    es    *ESResult
    mlErr error
    esErr error
}

func (s *Service) Recommend(ctx context.Context, req Request) (*Response, error) {
    // 全体の deadline を 950ms にセット(集約余裕 50ms 確保)
    ctx, cancel := context.WithTimeout(ctx, 950*time.Millisecond)
    defer cancel()

    var (
        wg sync.WaitGroup
        r  recommendResult
    )

    wg.Add(2)

    go func() {
        defer wg.Done()
        r.ml, r.mlErr = s.ml.Recommend(ctx, req)
    }()

    go func() {
        defer wg.Done()
        r.es, r.esErr = s.es.Search(ctx, req)
    }()

    wg.Wait()
    return s.assembleWithFallback(r, req), nil
}
```

### なぜ `errgroup` ではなく `sync.WaitGroup` か

- `errgroup.Wait()` は最初のエラーで他の goroutine を cancel する
- でも今回は「ML が失敗しても ES の結果は使う」graceful degradation
- → `errgroup` の挙動と要件が合わない
- 各 goroutine の結果とエラーを独立に保持し、後段の fallback ロジックで判定する

### 判断軸

1. **goroutine 数の上限**: 1 リクエストあたり 2 本(ML + ES)で固定
2. **タイムアウト**: 親 context(950ms)で全体打ち切り、各 goroutine は ctx を尊重
3. **キャンセル伝播**: 親 context のキャンセルが HTTP クライアント(`http.NewRequestWithContext`)に届く
4. **フォールバック性**: 一方が失敗しても、もう一方の成功結果を使える

## 6. 設計判断 4: 3 段階フォールバック(ML → ES → Redis Top 5)

### 課題
- ML エンドポイントは AI チームのデプロイ影響で落ちる前提
- ES が落ちることもある
- いずれの障害時もユーザーに「何か」返したい(レコメンドゼロ件はワーストケース)

### 採用設計

| Tier | 条件 | レスポンス内容 | 想定割合 |
|---|---|---|---|
| 1: Personalized | ML + ES 両方成功 | パーソナライズ + 関連サジェスト | 約 95% |
| 2: ES only | ML 障害 | 全文検索結果のみ | 約 3% |
| 3: Cache Top 5 | 両方障害 / 新規ユーザー | Redis の業種別人気 Top 5 | 約 2% |

### コード骨子

```go
func (s *Service) assembleWithFallback(r recommendResult, req Request) *Response {
    switch {
    // Tier 1: 両方成功 → パーソナライズ + ES サジェスト
    case r.mlErr == nil && r.esErr == nil:
        s.metrics.IncTier("personalized")
        return mergePersonalized(r.ml, r.es)

    // Tier 2: ML 失敗 → ES のみ
    case r.esErr == nil:
        s.metrics.IncTier("es_only")
        return fromES(r.es)

    // Tier 3: 両方失敗 → Redis キャッシュの業種別 Top 5
    default:
        s.metrics.IncTier("cache_top5")
        return s.cache.GetTop5(req.IndustryID)
    }
}
```

### Redis キャッシュの設計

- 業種別の人気セグメント Top 5 を **オフラインバッチで日次更新**
- メインパスでは使わない、最終フォールバックのみ
- ML / ES が両方落ちる事態は稀だが、レコメンドゼロ件で UI が壊れるよりマシ
- TTL は 7 日(バッチが万一落ちても直近のキャッシュで運用継続できる)

### 判断軸

1. **段階数**: 3 段階に絞った(細かく分けすぎると保守と監視が難しい)
2. **キャッシュ層**: ML/ES が両方落ちても Redis から 50ms 以内で返せる
3. **観測**: 各 Tier への落ち込み率を Datadog でメトリクス化

### 受け入れたトレードオフ
- Redis キャッシュの鮮度: 日次更新なので当日の急変は反映できない
- フォールバックでパーソナライズ品質は落ちる → UX 上「何か返る」を選んだ判断

## 7. 設計判断 5: 品質監視と可用性判断(AP 寄り)

### 一貫性 vs 可用性の判断
- レコメンドは決済のような強整合性が要らない領域
- 「最新の完璧な結果」より「必ず何か返る」が UX 上重要
- → **AP 寄り**(可用性優先)に倒す

### 採用設計

| 観測対象 | ツール | 通知 |
|---|---|---|
| フォールバック率(Tier 2/3 への落ち込み) | Datadog | 閾値超過で PagerDuty |
| アプリケーション例外 | Sentry | Sentry 通知 |
| ML / ES のレイテンシ p50/p95/p99 | Datadog | 閾値超過で PagerDuty |
| goroutine 数(`runtime.NumGoroutine()`) | Datadog | 増加傾向でアラート |

### 品質指標の設計

- **Tier 2 + Tier 3 の合計 = フォールバック率** を品質指標にした
- 5% のような閾値を置く場合は、面接では「正確な値は要確認」と言う
- 5% を超えたら PagerDuty で AI チーム / SRE に通知
- 「平常時 2-3% を実測、2 倍程度の余裕で 5%」を閾値根拠にしている(?: 正確な閾値は事実確認)

### Datadog ダッシュボード
- フォールバック率の時系列
- AI チームのデプロイタイムスタンプを overlay
- → デプロイ起因の障害が一目でわかる

### 切り分けの考え方

3 つを並べて見ることで「ML の遅延なのか、ES の障害なのか、自前コードのバグなのか」が分かる。

- レイテンシ悪化 → Datadog のレイテンシメトリクス
- 外部依存の失敗 → Datadog のフォールバック率
- コード上の例外 → Sentry

## 8. 設計判断 6: レイテンシ予算配分(数字で詳細に)

### 課題
- 体感上重くならない範囲に収めるため、どこに時間を使うかをどう見るか
- 各コンポーネントの予算を明確にしないと、後で誰がボトルネックか議論できない

### 予算配分

| 項目 | 予算 | 根拠 |
|---|---|---|
| 並列実行(ML と ES の最大) | 800ms | ML の P95 レイテンシに律速される |
| 結果集約・マージ | 50ms | O(n) の単純結合、n 最大 30 件 |
| ネットワーク・シリアライゼーション | 50ms | リクエスト/レスポンスの転送 |
| 余裕(GC pause / jitter / TLS handshake) | 100ms | Go の GC pause + ネットワーク変動 |
| **合計** | **約1,000ms** | 体感上の目安 |

### キャッシュヒット時の予算

| 項目 | 予算 |
|---|---|
| Redis ヒット | 5ms |
| ネットワーク・シリアライゼーション | 50ms |
| 余裕 | 245ms |
| **合計(キャッシュヒット時)** | 約 300ms |

### 数字の根拠
- **ML 800ms**: AI チームから提示された P95 の値
- **ES 300ms**: 過去のクエリログから自分で計測
- **集約 50ms**: マージロジックは O(n)、n は最大 30 件
- **余裕 100ms**: Go の GC、ネットワーク往復、TLS handshake のばらつき

### 判断軸

1. **律速箇所の特定**: ML(800ms) が一番遅いので、これに合わせて全体予算を組む
2. **直列にしない理由**: 外部依存の遅延とばらつきが足し合わされるため、並列化が予算の前提
3. **余裕の取り方**: GC pause と jitter で 100ms 程度の余裕を確保。平均だけ見ていると tail で外れる

### 検証方法
- 本番リリース後、Datadog の production traffic で実測
- p50/p95/p99 とフォールバック率を確認
- (今やり直すなら) リリース前に shadow traffic で事前計測しておきたかった

## 9. 設計判断 7: Cold start 対策

### 課題(リリース後の学び)
- ECS でコンテナを最低 0 から起動する設定にしていたら、初回リクエストの ML レイテンシが跳ね上がる
- スケールアウトで新インスタンスが立ち上がった直後にこの問題が出る
- 深夜のスケールイン後、最初のリクエストで稀にレイテンシ悪化

### 採用設計(リリース後の改修)

- **ECS minimum instance = 1**(?: 実際に設定したか事実確認)
- 常に 1 つは温まったコンテナがいる状態を保証
- リクエストがゼロでもインスタンスを落とさない
- ヘルスチェックエンドポイントで定期的にモデルを叩いておく(warm 維持)

### goroutine リーク対策

3 つを組み合わせる:

1. **`context.WithTimeout` を ML / ES 呼び出しに必須化**(linter で強制)
2. **`runtime.NumGoroutine()` を Datadog にメトリクス送信**(10 秒ごと)
   - 平常時のベースラインを把握 → ベースライン + 100 を超えたらアラート
3. **負荷試験で確認**
   - リリース前に 1,000 同時リクエストを投げて goroutine 数を観測
   - 終了後にベースラインに戻ることを確認

### 判断軸

1. **コスト vs 体感速度**: 常時 1 インスタンス分のコストは払うが、Cold start による tail latency 悪化を防ぐ価値が勝つ
2. **スケールアウト**: 負荷増では auto scaling で増やす(min=1, max=N)
3. **ML 側との合わせ**: ML エンドポイントもウォームアップが必要 → AI チームと連携して min instance を合わせた

### 受け入れたトレードオフ
- 常時 1 インスタンスのコスト
- Cold start を完全に消すには Provisioned Concurrency 系の追加機構が必要だが、ECS の min instance で十分対処できた

## 11. 面接ストーリー(完成版・ノーカット)

### 状況設定(30秒)

「Groundtruth で広告キャンペーン作成 UI のレコメンド機能を設計・実装しました。

当時、広告主がターゲットセグメントをドロップダウンから選ぶ UI があったんですが、選択肢が肥大化していて Google Analytics で離脱が観測されていました。効果的でないセグメントを選んでしまうケースも多くて、広告効果も伸びていない。レコメンド追加の意思決定になりました。

未経験ユーザーには業種別人気 Top 5、経験者には ML によるパーソナライズ、両者をマージして返す方針です。」

### 設計判断 1: モノリス分離と言語選定(1分)

「設計時の判断軸の最上位は、キャンペーン作成中の体感を重くしないことでした。ML と Elasticsearch はどちらも外部依存なので、直列に待つと遅延のばらつきや timeout まで足し合わされます。

選択肢は 3 つ。

A: 既存 Flask モノリスに Python で追加。WSGI 同期処理なので、外部 I/O の並列化や fallback を素直に扱いにくい。

B: Flask + gevent で独立サービス化。並列はできますが、gevent のモンキーパッチが副作用広範で、チーム内に運用実績がない。本番のデバッグが詰む可能性があります。

C: Go で独立マイクロサービス化。goroutine + channel でネイティブ並列、`context.WithTimeout` で deadline 伝播がきれい。チームに Go 経験者は少なかったですが、外部依存を扱う長期運用コストで逆転すると判断して採用しました。」

### 並列実行とフォールバック設計(1分)

「並列化は `sync.WaitGroup` で ML と ES を独立に呼び、`context.WithTimeout` で全体の deadline を設定。`errgroup` じゃなく WaitGroup にしたのは、片方が失敗してももう片方の結果を使いたいからです。errgroup は最初のエラーで他を cancel するので、graceful degradation には合わない。

フォールバックは 3 段階にしました。

- Tier 1: ML + ES 両方成功 → パーソナライズ + ES サジェストの混合
- Tier 2: ML 失敗 → ES のみ
- Tier 3: 両方失敗 or 新規ユーザー → Redis にキャッシュした業種別 Top 5

ML エンドポイントは AI チーム管理で、彼らのデプロイ影響で落ちることが現実的に予想されたので、ML 必須にしない設計にしました。」

### 品質監視と可用性判断(45秒)

「観測は Datadog と Sentry で分けました。Datadog でフォールバック率・レイテンシ・goroutine 数を見て、Sentry でアプリケーション例外を拾う。3 つを並べることで、ML の遅延なのか、ES の障害なのか、自前コードのバグなのかを切り分けられます。

特に大事だったのはフォールバック率を品質指標にしたこと。レイテンシだけ見ていると Tier 2(ES only)で『速いがパーソナライズなし』を見逃すので、品質指標として独立して持つ。閾値を超えたら PagerDuty で AI チームと SRE に通知が飛びます。

レイテンシ予算は、ML の遅さに律速される前提で、集約処理、ネットワーク、GC pause や jitter の余裕を別枠で見るようにしました。平均値だけでなく tail を見たのが効きました。

リリース後の学びとして、ECS の Cold start で初回 ML レイテンシが跳ね上がる問題があって、minimum instance = 1 で常に温めておく対処を入れました。

結果として、リリース後の Q 単位集計（リリース前同期間との比較、PdM・AI チームから報告された数字）で、キャンペーン作成時の churn が約 20% 減、広告インプレッションが約 10% 増、という形になりました。実装は締切より 2 週間早く完了しています。」

## 12. 想定 Q&A

### Q1: なぜモノリスではなくマイクロサービス?
「Flask モノリスにそのまま足すと、外部 I/O の並列化、timeout、fallback が素直に書きにくいからです。独立サービスにすることで並列実行・独立スケール・障害分離の 3 つを同時に取れました。」

### Q2: FastAPI ではなく Go にした理由は?
「FastAPI でも成立します。async client を揃えれば、ML と ES の外部 I/O を並列に待つ設計はできます。なので、Go でなければ無理だった、とは言いません。

当時の判断では、FastAPI の本番運用知見がチーム内に薄かったことと、sync client が混ざった時に event loop を塞ぐリスクを避けたかったことが大きいです。Go なら goroutine と `context.WithTimeout` が標準で、deadline と fallback の責務がコード上で見えやすい。小さな集約サービスに限定するなら、学習コストを払っても運用時の読みやすさが勝つと判断しました。」

### Q2a: なぜ Flask + gevent ではなく Go?
「gevent はモンキーパッチで stdlib を書き換えるので副作用が広範で、チーム内に運用実績が薄かった。Go なら goroutine と `context.WithTimeout` が言語標準で、副作用なしに並列化と deadline 伝播が書けます。学習コストは払いましたが、長期運用コストで逆転する判断です。」

### Q3: これは並列処理で行う必要があったんですか?
「はい、必要でした。ただ、100ms を削るためだけではありません。ML と ES は互いに依存していません。ML の結果を使って ES を呼ぶわけではないので、直列に待つ理由がありません。直列にすると、外部依存の遅延のばらつきや timeout まで足し合わされます。

もう 1 つ、フォールバックの観点でも並列が効きます。ML が timeout しても、ES が返っていれば ES only としてユーザーに候補を返せる。直列だと先に呼んだ依存先が遅いだけで、後段の結果を取りに行く時間も失いやすいです。なので、並列化は高速化だけでなく、障害時にも何か返すための設計でした。」

### Q4: ML が 800ms なら、並列化してもそこまで縮まらないのでは?
「おっしゃる通りで、並列化しても ML の遅さが消えるわけではありません。ML が律速なのは変わりません。なので、100ms だけを根拠にしていたら弱いです。並列化の目的は、ML と ES の遅延や失敗を独立に扱えるようにすることです。

だから ML を無制限に待つのではなく、サービス全体に deadline を置きます。deadline 内に ML が返れば使う。返らなければ ES の結果、さらに両方失敗なら Redis Top 5 に落とす。つまり、体感速度は並列化だけで守るのではなく、並列化、timeout、fallback のセットで守る設計です。」

### Q5: 学習コストがかかるのに Go にする必要があったんですか?
「学習コストはありました。ただ、比較したのは Go の学習コストと、Python のまま timeout / fallback / 障害分離を複雑に抱えるコストです。FastAPI ならかなり自然に書けますが、当時は async client を揃えた本番運用の知見が薄かった。Flask + gevent なら並列化できますが、monkey patch の副作用が大きい。Go は goroutine と timeout が標準機能なので、この要件ではリスクが小さいと判断しました。

また、全体を Go にしたわけではありません。HTTP で受けて ML、ES、Redis を呼んで返す、境界が明確な小さい集約サービスに限定しました。学習コストを払う範囲を絞った上で、並列 I/O と障害分離を優先しました。」

### Q6: goroutine リーク対策は?
「3 つです。1) 全外部呼び出しに `context.WithTimeout` を必須化、`defer cancel()` で関数終了時に解放、2) `runtime.NumGoroutine()` を 10 秒ごとに Datadog 送信してベースライン + 100 を超えたらアラート、3) リリース前の負荷試験で 1,000 同時リクエスト後にベースラインに戻ることを確認。」

### Q7: レイテンシ予算配分の根拠は?
「律速箇所は ML です。そこに集約処理、ネットワーク、GC pause と jitter のための余裕を別枠で積みます。大事なのは平均値だけでなく tail の余裕を見ることです。本番リリース後は Datadog の production traffic で実測し、今やり直すならリリース前に shadow traffic で確認します。」

### Q8: フォールバックの段階数の妥当性は?
「3 段階に絞りました。細かく分けすぎると保守と監視が難しくなる。ML 障害(Tier 2)、両方失敗 or 新規(Tier 3)というパターンで、運用上で実際に起きるシナリオをカバーできます。各 Tier への落ち込み率を Datadog でメトリクス化して、想定外の偏りに気づける構造にしました。」

### Q9: 結果整合性 vs 強整合性の判断は?
「AP 寄りに倒しました。レコメンドは決済とは違って、最新の完璧な結果より『何か返る』が UX 上大事です。Redis キャッシュの Top 5 はバッチで日次更新なので鮮度が落ちますが、フォールバック時のパーソナライズ品質低下と、何も返らないことによる離脱を比べると、後者の方がダメージが大きい判断です。」

### Q10: 観測・モニタリングは?
「Datadog でフォールバック率・レイテンシ p50/p95/p99・goroutine 数を見て、Sentry でアプリケーション例外を拾う構成です。AI チームのデプロイタイムスタンプを Datadog ダッシュボードに overlay しているので、デプロイ起因の障害が一目で分かります。フォールバック率に閾値を置き、超えたら PagerDuty で通知でした。正確な閾値は事実確認が必要です。」
