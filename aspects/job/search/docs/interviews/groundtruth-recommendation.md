# 面接対応: 広告セグメントレコメンデーション（Groundtruth）

**期間:** 〜2024年6月リリース  
**役割:** バックエンドエンジニア（5名チーム）

---

## 30秒イントロ

当時、広告主はキャンペーン作成時にセグメントをアルファベット順のドロップダウンから選ぶしかなく、数百件のリストから何を選べばいいかわからない状態でした自然とリスト上位のセグメントを選びがちになってたんですが、アルファベット前半のセグメントが必ずしもそのユーザーのキャンペーンにとって費用対効果の高いものではなかった。その結果、キャンペーン作成フローのセグメント選択ステップでの離脱が多く、CTR も業界平均を下回っていました。

そこで、ユーザーごとに費用対効果の高いセグメントをレコメンドする機能と、絞り込み検索機能をリリースすることになりました。結果として CTR 向上と離脱率削減を達成しました。

---

## 核心となる設計判断

### 1. 既存の Ads Manager モノリスにエンドポイントを追加するのではなく、独立マイクロサービスとして構築した

判断の根拠は3点：検索・推薦ロジックを Ads Manager 本体から切り離して責務を明確にできること、独立サービスの方がスケーリングしやすいこと、機械学習やElasticsearchの障害が Ads Manager 全体に波及しないこと。

特に今回は、Elasticsearch、機械学習エンドポイント、Redis、S3 といった外部依存に加えて、並列実行とキャッシュ制御が多い。接続プール枯渇、メモリ使用量増加、タイムアウト増加のようなリソース問題が起きた場合でも、Ads Manager モノリス全体ではなく推薦サービス単体で隔離・再起動・スケールできる構成にした。

**反論への回答:**
「マイクロサービスは運用コストが増える」は正しいが、Fargate により OS パッチ・ノード管理が不要。チームに ECS の運用経験があり、実質的な増加は「デプロイ設定の追加」程度だった。

---

### 2. 機械学習のフォールバック戦略（可用性を AI チームの稼働状況に依存させない）

機械学習エンドポイントが一定時間内に返答しない場合や障害の場合、文字入力ありならElasticsearch検索のみで継続し、文字入力なしの経験ユーザーなら人気上位5件へフォールバックする。機械学習が完全に落ちても Ads Manager の機能に影響しない設計。

Elasticsearch は文字入力時の主経路として扱う。つまり、経験ユーザー・未経験ユーザーのどちらでも、ユーザーが検索ワードを入力したら Elasticsearch を叩く。その上で経験ユーザーの場合は機械学習によるパーソナライズも並列に呼び、検索結果の並び替え・補完に使う。機械学習が失敗した場合はElasticsearch検索結果のみを返す。未入力時は、経験ユーザーなら機械学習によるパーソナライズ推薦、未経験ユーザーなら Redis にキャッシュした「業種別・費用対効果上位5件」を表示する。

機械学習はElasticsearchの検索結果を受け取って再ランキングするだけではなく、検索クエリとユーザー・キャンペーン文脈から独立して推薦候補を返せる経路だった。そのため、Elasticsearch検索と機械学習推薦を並列に実行し、最後にサービス側で重複排除・スコア統合・選択済みセグメント除外を行った。

ただし、文字入力のたびに毎回機械学習エンドポイントを叩くのは無駄が大きい。キャンペーン作成中は業種・地域・目的・予算などの文脈がほぼ固定なので、その文脈をキーにして機械学習前段にRedisキャッシュを置いた。キャッシュヒット時はElasticsearch検索とキャッシュ済みパーソナライズ候補をマージするだけなので、通常は約300msで返せる。キャッシュミス時だけ機械学習エンドポイントを呼び、初回読み込みもP99 1秒以内に収めた。

フォールバック率5%を運用品質目標として定義し、Datadog で監視。AI チームのデプロイが影響するたびに検知できるようにした。これにより「AI チームのデプロイに自分たちのサービスが引きずられる」リスクを定量的に管理できた。

---

### 3. 入力時は並列、未入力時は分岐で応答時間を管理した

入力ありの場合は、Elasticsearch検索を必ず使う。経験ユーザーでは同時に機械学習によるパーソナライズ推薦も呼び、検索結果を並び替え・補完する。機械学習が失敗した場合はElasticsearch検索のみで継続する。入力なしの場合は、経験ユーザーには機械学習によるパーソナライズ推薦、未経験ユーザーには人気上位5件を返す。

背景: レコメンドサービスの99パーセンタイル目標は1,000ms。文字入力ありの経験ユーザーでは、キャッシュミス時にElasticsearch検索が約300ms、機械学習推論が最大800ms程度かかる見積もりだった。直列にすると1,100ms程度になり目標を超えるため、Goで並列実行して初回でも最大800ms程度で返す設計にした。キャッシュヒット時はElasticsearch検索が律速になり、約300msで返せる。入力なしの経験ユーザー経路は、機械学習推論が最大800ms程度を超えたら人気上位5件へフォールバックする。入力なしの未経験ユーザー経路はRedisキャッシュから返す。経路ごとに最悪時の振る舞いを決めた。

これにより、検索とパーソナライズを組み合わせても1秒以内に収めやすくした。入力時の機械学習遅延や障害時はElasticsearch検索のみで継続し、未入力時の機械学習遅延や障害時は人気上位5件へ倒す。Flaskモノリスに直接追加するより、独立サービスとして応答時間・フォールバック率・エラー率を個別に監視しやすくなった。

## よく聞かれる質問と回答

**Q: AI チームとどう連携したか？**

OpenAPI仕様を事前に合意し、実装の独立性を確保した。機械学習エンドポイントが「どういうリクエストを受けてどういうレスポンスを返すか」を文書化してから実装を開始したので、AI チームのモデル実装と並行して進められた。また「99パーセンタイル目標から逆算して機械学習に割り当てられる上限は800ms」という計算を根拠に目標値を提示し、説得力ある交渉ができた。

**Q: 新規ユーザーと経験ユーザーの分岐をなぜ設けたか？**

過去のキャンペーン作成が5件未満のユーザーに機械学習でパーソナライズを出しても意味がない（学習データがない）。この場合は Redis から業種別・費用対効果上位5件を返すだけで十分で、機械学習呼び出しのコストを払わなくていい。ユーザーの状態によって必要な処理を変えることで、コストと応答時間を最適化した。

**Q: そもそもなぜアルファベット順が問題になったのか？**

当初はセグメント数が少なく、アルファベット順でも問題なかった。しかし年々セグメントが増え続け、数百件を超えた時点でスクロールが現実的でなくなった。プロジェクト開始時点では約 3,500件に達していた。ユーザーは「とりあえず上から選ぶ」という行動をとるようになり、これは認知バイアスの一種で、アルファベット前半のセグメントが必ずしも高 CTR ではなかったため、広告インプレッションの機会損失に繋がっていた。Google Analytics のファネルデータで離脱項目として確認し、「表示順に問題がある」と結論づけた。レコメンド機能で「費用対効果の高いセグメント」を上位に入れることで、このバイアスを構造的に解消することが目的の一つだった。

**Q: 99パーセンタイルの目標値を実際に達成できたか？**

正確な実測値は手元に残っていないが、応答時間目標は達成できたと認識している。Datadog で監視していて、運用品質目標のアラートが継続的に鳴らなかったのが根拠。入力時はElasticsearch単体へのフォールバック、未入力時は人気上位5件へのフォールバックが目標内で推移していた。

**Q: Redis キャッシュの設計で気をつけたことは？**

月次 cron で費用対効果上位5件を計算し S3 に書き込む。キャッシュはcache-aside方式にした。つまり、アプリ側でまずRedisを見て、なければS3から取得してRedisに載せる構成。人気上位5件は頻繁に変わるものではなく、即時性も求められないため、イベント駆動で即時更新するよりTTLで自然に更新する方が制御しやすいと判断した。未経験ユーザーの未入力時と、入力なしの経験ユーザーで機械学習が失敗した時にこの人気上位5件を使えるため、機械学習障害がサービス全体の障害にならない設計にした。

## 深掘り質問への準備

### リリース後、想定外のことはありましたか？

リリース直後、毎朝最初のリクエストで機械学習エンドポイントの応答時間が跳ね上がる現象が発生した。原因は ECS がアイドル時間にコンテナをスケールダウンしていて、朝一番のリクエストでコンテナの起動待ちが生じるコールドスタートだった。Datadog のフォールバック率アラートで気づき、AI チームに調査を依頼した。最終的に機械学習サービス側で最小起動数を1に設定することで解消した。フォールバック率監視を入れていなければ気づくのが遅れていたと思う。

人気上位5件で、食品・日用品などのセグメントが常に上位を占める問題が出た。これらのカテゴリはクリック率の絶対値が高い傾向があり、不動産のような高単価・低クリック率のカテゴリと同じ母集団で正規化しても、業界間のバイアスを消しきれなかったのが原因だった。設計時から「将来は業界別の人気上位5件に分割する」とドキュメントに残していたが、その問題がリリース直後に顕在化した。

---

### 今やり直すなら何を変えますか？

1つ目は **人気上位5件を業界別にすること**。V1では全業界共通のスコアを出したが、食品・日用品と不動産ではクリック率の絶対値が5〜10倍違う。標準化して補正してもバイアスが残る。最初から業界タグを持ってセグメントを管理し、「同じ業界内での人気上位5件」を出す設計にすれば精度が上がった。当時は「まず動くものを出す」を優先したが、このバイアスはリリース前に想定できた問題だった。

2つ目は **フォールバック戦略をアーキテクチャ決定と同時に固めること**。今回はマイクロサービスの設計を決めてから「機械学習が遅かったらどうするか」を考えた順序だった。機械学習の実装前に正確な99パーセンタイルはわからないが、「目標値を守れないリスクがある前提でフォールバックを先に設計しておく」ことは最初からできた。依存する外部エンドポイントがある場合は、そのエンドポイントが機能しない前提でシステムを設計してから統合するのが正しい順序だと思う。
