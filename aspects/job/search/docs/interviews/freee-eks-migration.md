# 面接対応: EKS クラスタ統合マイグレーション（freee）

**期間:** 2025年初〜5月リリース  
**役割:** マイグレーションリード（エンジニア 4名チーム）

---

## 30秒イントロ

freee 福利厚生の独自 EKS クラスタは、3〜4ヶ月ごとにバージョンアップグレードが必要で、毎回 1〜2週間の作業が発生していた。4名チームでこれが年 3〜4回あり、年間で 4〜8週間が保守に消えていた。

それを解決するために、人事労務の既存クラスタに namespace を追加して統合し、ロードマップからアップグレード作業を完全に除外した。ダウンタイムゼロ、Cron 二重起動ゼロで完了した。特に社宅利用者の家賃など会社負担情報を CSV 出力する Cron の二重実行防止が最大のリスクだった。

---

## 核心となる設計判断

### 1. Cron 二重起動を「実装で防ぐ」のではなく「設計で不可能にした」

**状況:** Blue/Green で移行する以上、新旧クラスタが同時稼働する時間帯がある。その時間に Cron が実行されると、同じジョブが両方で走る。特に問題になるのが、社宅利用者の家賃など会社負担情報を CSV で出力するジョブで、二重実行されると経理データが重複して出力される。

**選択肢:**
- 分散ロック（Redis）で重複を防ぐ → 実装が正しければ防げるが、実装の信頼性に依存する
- タイムラインで物理的に重ならない構造にする → **採用**

**判断:**
全 Cron を 04:00 以降に集約し、03:00〜04:00 を「Cron ゼロ時間帯」にした。切り替え後は即座に旧クラスタを停止（replicas を 0 に）。Blue/Green の同時稼働時間をこの窓に収めることで、ロックの正しさに賭けず、物理的に二重起動が起きない設計にした。

**結果:** 移行当日の Cron 二重起動 0件。

---

### 2. メール冪等実装を「やらない」と決めた

**状況:** 移行ウィンドウ直前に起動していた非同期ワーカーが途中停止→再試行でメール重複送信するシナリオをレビュー時に指摘された。

**技術的解決策のコスト:** 全メール送信ジョブ（8本）に冪等キーを実装。推定 2週間。

**判断の構造:**
- 深夜 03:00 にユーザー操作でトリガーされたジョブが動いている確率 → 実績上ほぼゼロ
- 発生しても自動回復する問題（重複メールは翌日から正常）
- CS チームへの事前通知で十分カバーできる

「技術で解決できる問題はすべき」は正しいが、発生確率と影響の定量評価をした上で「2週間のコストは正当化できない」と判断した。

**結果:** 当日の重複メール 0件（深夜の稼働ジョブがゼロだったため）。

---

### 3. 旧 Node Group を当夜は残した（ロールバックの RTO 設計）

**理由:** 旧 Node が起動済みであれば、ロールバックは Pod の再起動だけで 2〜3分で完了。Node から起動する場合は 5〜10分の追加遅延が発生する。

旧 Node Group を移行と同時に削除してしまうと、問題発生時にロールバックで Node 起動待ちが必要になり、RTO 目標（15分）を超える可能性があった。1夜 $10 のコストで保険として残し、翌朝問題がなければ削除した。

---

### 4. 3フェーズの段階的移行（判断基準を事前定義）

Integration → Staging → Production の 3段階。各フェーズで「ここで問題が出たら本番に進まない」という Go/NoGo 基準を事前に文書化した。

Staging の通過基準: エラー率 < 0.1%（24時間）+ 全 Cron 正常終了（7日間）。これを満たさなければ本番移行日を延期するルールにした。「Staging で確認したから大丈夫」の曖昧さを排除した。

## よく聞かれる質問と回答

**Q: ダウンタイムゼロをどう実現したか？**

ALB のターゲットグループ切り替えが中心。新クラスタの Pod 起動確認 → ヘルスチェック通過 → ALB 切り替えの順で、旧クラスタは切り替え後も数分間待機させてから停止した。

**Q: IRSA の移行が難しかった理由は？**

Kubernetes の Pod が AWS リソースにアクセスする IRSA は、クラスタごとに異なる OIDC プロバイダー URL と紐付く。クラスタを変えると IAM ロールの AssumeRolePolicy も変える必要があり、Terraform の参照先を全て変更した。Integration 環境で S3・RDS・Secrets Manager への接続を全て確認してから Staging に進んだ。

**Q: 一番のリスクはどこだと思っていたか？**

Cron の二重起動。設計段階でのリスクとして最も影響が大きかった。分散ロックで解決する案もあったが、「実装が正しければ防げる」という依存を嫌って、タイムライン上で二重起動が物理的に起きない設計を選んだ。

**Q: このプロジェクトで最も重要な判断は？**

メール冪等実装を「やらない」と決めたこと。技術的には解決できる問題を、確率・影響・コストで定量評価して「2週間は払えない」と判断した。移行プロジェクトのゴールは「インフラ保守コストの削減」であり、新規機能開発を止めてまで対応する必要はなかった。

## 深掘り質問への準備

### 移行当日、想定外のことはありましたか？

当日は計画通りに進みました。ただ、それ自体が想定外でした。IRSA の移行が一番リスクが高いと思っていたので、Integration と Staging で十分テストしてから本番に臨んだ結果だと思っています。Staging で一度 IRSA 周りの問題が出て本番前に直せたのが大きかった。

---

### 今やり直すなら何を変えますか？

2点ある。

1つ目は **Cron スケジュール変更の自動化**。移行前に全 Cron を 04:00 以降に集約したが、これは手作業で変更した。Cron の数が多くなると漏れのリスクがある。`cronjob.yaml` の時刻を一括変換するスクリプトを作って CI でバリデーションする仕組みがあればよかった。

2つ目は **ロールバック判断基準の自動化**。「エラー率 > 0.5% が 5分以上」をロールバック基準にしたが、移行当夜は自分が Datadog を目視で監視していた。Datadog のコンポジットアラームで「この条件になったら Slack に自動通知」を事前に仕込んでおけば、深夜の監視負担を下げられた。

---

### 答えられないときの動き

深掘りされて詰まった場合:

> 「正直なところ、そこまでは検討していませんでした。ただ今考えると、〜という観点が重要になりそうですね。当時は〔何を優先していたか〕を最優先にしていたので、そこのトレードオフを意識できていなかったと思います。」

知ったかぶりより、「考え方を見せる」方が評価される。
