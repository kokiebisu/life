## 📊 Groundtruth — Recommendation Microservice（Go版）

### 状況

ユーザーがキャンペーン作成時にセグメントを選ぶ画面があったのですが、選択肢が肥大化して「何を選んでいいかわからない」という状態になっていました。Google Analyticsでchurnが確認されていたので、ユーザーがセグメントを迷わず選べるUXを提供することになりました。

まず「なぜ選べないのか」を分解しました。ユーザーには3つの状態があると判断しました。「自分のビジネスに合うセグメントがある程度わかっている人」「何があるか探したい人」「そもそも何を軸に選べばいいかわからない人」。それぞれに対応して、Personalizedレコメンデーション・Elasticsearch全文検索・Top 5 Popularの3モードを設計しました。1つの機能でカバーしようとするより、状態ごとに最適な入口を用意した方がchurnの根本原因に刺さると判断しました。

### 設計判断

設計時の最初の問いは「既存のFlaskモノリスに追加するか、新しいサービスとして切り出すか」でした。判断軸は2つです。まずデプロイ頻度。レコメンドロジックはMLチームのAPIスキーマ変更への追従・ESクエリのチューニング・フォールバック閾値の調整など、モノリスの他機能とは独立した変更サイクルを持ちます。この変更のたびにモノリス全体をデプロイするのは影響範囲が大きすぎます。次に影響範囲。新機能のトラフィックがどれだけ伸びるか読めない状態でモノリスに同居させると、ESへのクエリ負荷が上がったときにキャンペーン作成や他機能のスレッドプールを奪い合うことになります。切り出すことで、レコメンデーションのトラフィックに応じて独立してスケールアウトでき、モノリス側への影響をゼロにできます。トラフィックが読めないのに影響範囲がコントロールできない構成は怖いと判断して、サービスを切り出す決断をしました。

次に「切り出すとしてどの言語か」。当時社内でGoを全社的に採用していく方針があり、他チームにすでに運用知見が蓄積されていました。新規サービスを立ち上げるタイミングでGoに揃えるのは自然な判断でした。技術的にも、MLとESの両方から候補を並列取得してマージする設計に対して、Goのgoroutineとcontextでファンアウト・ファンインのパターンが自然に書けて、タイムアウト管理も明示的にできる点が合っていました。

フォールバック設計も重要でした。MLエンドポイントはAIチームが別管理で落ちることが予想された。リクエストのパターンは3つで、入力ありでML成功→Personalized、入力ありでML障害→Elasticsearchのみ、入力なし またはML/ES両方障害→RedisにキャッシュしたTop 5 Popular。DatadogでML障害によるESフォールバック率が5%以下であることをSLOとして監視しました。

### スコープ管理

プロジェクト途中でPMから「セグメント選択時にリーチ数（何人にリーチできるか）を表示したい」というスコープ追加要求がありました。広告文脈では自然な要求ですが、別途集計パイプラインが必要でGoサービスの外側に新たな依存が増えるため、現スプリントには入らないと判断しました。リリース日を守る方が事業的な優先度が高いとPMに数字で示して合意してもらい、次フェーズに切り出しました。コアの3モード設計を守ったことで、当初スケジュールより2週間早くリリースできました。

### 結果

churnを20%減、広告インプレッションを10%向上。churn削減はセグメント選択の改善による直接の結果で、インプレッション向上はレコメンドが正しく機能して費用対効果の高いセグメントを選べるようになったことで広告配信数が増えた結果です。学んだのは、デプロイ頻度・影響範囲・全社の技術方針を軸に技術選択をすること。同じことが御社のモノリス採択にも作用していると思います。


**核トレードオフ**

- モノリス追加vsマイクロサービス：デプロイ頻度・影響範囲の独立性の天秤
- Go採用：全社方針への整合 + goroutineによる並列処理の書きやすさ
- フォールバックの段階数：3段階がそのトラフィックで適切、過剰にしない
- スコープ調整：早期リリースvs機能充実のトレードオフ

**想定ツッコミ**

- **Q: テスト戦略は？** → MLエンドポイントはAIチームからONNXファイルをもらってローカルで推論サーバーを立て、本物の推論結果でテストした。フォールバックのシナリオはステータスコード・タイムアウトをモックして検証。ESはLocalStack OpenSearchを使ったAPIテストで実際のクエリ挙動まで確認した
- **Q: goroutineの漏れ対策は？** → context.WithTimeoutでML/ES周辺にタイムアウト。runtime.NumGoroutine() をDatadogに送信し続けて、増加傾向が見えたらアラート
- **Q: ML と ES の並列取得はどう実装したか？** → errgroup は1つ失敗した時点で context をキャンセルして他も止める。フォールバックしたいので使えない。WaitGroup + mutex で結果を独立して収集し、Wait() 後に何が成功したかで判断した。channel は先着順処理に向いているが、今回は「全部終わってから判断」なので不要と判断した
- **Q: タイムアウトはどう設定したか？** → ML は AI チームと p99 ベースの SLO を設けて 1000ms。ES は自分たちの p99 実測値（fuzzy matching + 大きい index サイズで 600ms）をそのまま採用。p99 をタイムアウトにすることで 99% の正常リクエストを通しつつ、外れ値は切ってフォールバックに流す設計にした。p50 だと切りすぎ、バッファを持たせると遅いリクエストを待ち続ける → フォールバックがある設計なので p99 タイトな設定が UX として正しいと判断した
- **Q: マイクロサービスにしたことで増えた複雑さは？** → ネットワーク遅延が増えた。モノリスなら関数呼び出しで済む処理がHTTP越しになるので、タイムアウト設計と分散トレーシング（Datadog APM）が必要になった。それでも独立デプロイ・独立スケールのメリットが上回ると判断した
- **Q: churn 20%の計測方法は？** → Google Analyticsでキャンペーン作成フローの離脱率をリリース前後で比較。A/Bテストはインフラが必要でこの規模感では過剰と判断した
- **Q: Top 5 PopularのRedisキャッシュはどう更新してた？** → バッチで1時間おきに集計してRedisに書き込む設計。Top 5 Popularはフォールバック用途なので多少古くても問題ない。リアルタイム集計（Redis Sorted Set）も検討したが、セグメント選択のたびに書き込みが発生して負荷が増える割に得られるものが小さいと判断した
- **Q: 業種別にしなかったのか？** → まずグローバルTop 5からシンプルに始めて効果を測ることにした。業種別は改善余地として次フェーズに残した
- **Q: Redisによる推論コスト最適化とは？** → ユーザーが何も入力していない状態でML推論を走らせないためにTop 5 PopularをRedisにキャッシュ。ML呼び出しゼロで返せるのでコスト削減になった
- **Q: リリース戦略は？** → Feature flagによる段階的ロールアウト（10% → 50% → 100%）。LaunchDarklyを使ってユーザー単位で制御した。リクエスト単位だと同じユーザーが機能のON/OFFを交互に経験してUXが壊れるため、ユーザー単位にして体験を一貫させた。フラグONユーザーのchurn率もきれいに計測できる。新規マイクロサービスの初回本番デプロイはgoroutineリーク・メモリ増加・レイテンシスパイクが負荷が増えないと見えないため、blast radiusを最小化した。GAでchurnの改善を確認しながら拡大。Shadow modeも検討したがフォールバック設計が堅牢なのでオーバーキルと判断した
- **Q: ESのインデックス設計は？** → セグメント名・説明文は `text` のみ。新機能でユーザーが何を入力するか読めないため、まずリコールを広める方向で keyword の完全一致は外した。fuzzy の fuzziness は `AUTO`。最初から数値固定で絞るより、文字数に応じて自動調整される AUTO でバランスを見て、実測後にチューニングする方針にした
- **Q: PMのスコープ追加をどう断ったか？** → 「断った」ではなく「次フェーズに移した」。リーチ数プレビューは別途集計パイプラインが必要で現スプリントに入らないことを工数で示し、リリース日を守る方が事業的優先度が高いと合意してもらった
- **Q: MLとESが両方落ちたらどうなる？** → RedisのTop 5 Popularだけ返す。ユーザーには常に何かを返せるのでキャンペーン作成フローが完全に止まらない設計になっている。エラーにしない判断は、セグメント選択が必須ステップなので空白にするとフロー自体が詰まるため
- **Q: なぜ3モードと決めたのか？PMとどんな議論をした？** → PMが顧客会話から「何を選べばいいかわからない」という声を持ってきた。「なぜ選べないか」をエンジニア側で分解すると、セグメントを減らすと機能が失われるので選びやすくするしかない → どんなユーザーが選べないのかをGAとPMの知見で整理 → ユーザーの状態が3種類に分かれた → 状態ごとに入口を用意する3モード設計に落ち着いた。ユーザーインタビューはしていないが、リリース後にchurn 20%改善という結果が出たので仮説は正しかったと判断した
- **Q: 今やり直したら？** → 同じ構成。フォールバックの段階設計とスコープを早めに切り出した判断は正しかった
