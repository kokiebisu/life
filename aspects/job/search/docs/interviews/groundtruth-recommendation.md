# 面接対応: 広告セグメントレコメンデーション（Groundtruth）

**期間:** 〜2024年6月リリース  
**役割:** バックエンドエンジニア（5名チーム）

---

## 30秒イントロ

当時、広告主はキャンペーン作成時にセグメントをアルファベット順のドロップダウンから選ぶしかなく、数百件のリストから何を選べばいいかわからない状態でした自然とリスト上位のセグメントを選びがちになってたんですが、アルファベット前半のセグメントが必ずしもそのユーザーのキャンペーンにとって費用対効果の高いものではなかった。その結果、キャンペーン作成フローのセグメント選択ステップでの離脱が多く、CTR も業界平均を下回っていました。

そこで、ユーザーごとに費用対効果の高いセグメントをレコメンドする機能と、絞り込み検索機能をリリースすることになりました。結果として CTR 向上と離脱率削減を達成しました。

---

## 核心となる設計判断

### 1. 既存の Ads Manager モノリスにエンドポイントを追加するのではなく、独立マイクロサービスとして構築した

判断の根拠は3点：Ads Manager のモノリス（Flask は同期ブロッキングで、I/O の待機中に他の処理ができない）に追加すると ML と Elasticsearch を順番に呼び出すことになるが、FastAPI の独立サービスなら並列実行できること（これによりレイテンシ目標を達成できた）、独立サービスの方がスケーリングしやすいこと、ML 障害が Ads Manager 全体に波及しないこと。

**反論への回答:**
「マイクロサービスは運用コストが増える」は正しいが、Fargate により OS パッチ・ノード管理が不要。チームに ECS の運用経験があり、実質的な増加は「デプロイ設定の追加」程度だった。

---

### 2. ML フォールバック戦略（可用性を AI チームの SLA に依存させない）

ML エンドポイントが一定時間内に返答しない場合や障害の場合、Elasticsearch の結果のみを返す（フォールバック）。実装は `asyncio.wait_for()` でタイムアウトを設定し、超過または例外時は ML の結果を使わず ES の結果だけ返す。ML が完全に落ちても Ads Manager の機能に影響しない設計。

さらに「ML 失敗 × ユーザーがまだ検索ワードを入力していない」状態では ES の結果も存在しないため、Redis にキャッシュした「業種別・費用対効果上位 5件」を最終フォールバックとして表示する。これにより ML・ES の両方が機能しない最悪ケースでも画面が空欄にならない。

フォールバック率 5% を SLO として定義し、Datadog で監視。AI チームのデプロイが影響するたびに検知できるようにした。これにより「AI チームのデプロイに自分たちのサービスが引きずられる」リスクを定量的に管理できた。

---

### 3. ML レコメンドと Elasticsearch 検索を `asyncio.gather()` で並列実行した

経験ユーザーへの応答では ML エンドポイントと Elasticsearch を同時に呼び出す設計を採用した。

背景: レコメンドサービスの P99 目標は 1,000ms。ML 推論が最大 800ms、ES 検索が加わると順番に実行すれば合計が目標を超える。`asyncio.gather()` で並列化すれば律速は ML の 800ms のみになり、サービス内オーバーヘッドを加味しても目標内に収まる計算だった。

これが Flask モノリスへの追加を選ばなかった直接の理由でもある。WSGI 同期アーキテクチャでは `asyncio.gather()` が使えず、並列実行自体ができない。FastAPI（ASGI）の独立サービスにしたことで初めてこの設計が成立した。

## よく聞かれる質問と回答

**Q: AI チームとどう連携したか？**

OpenAPI spec を事前に合意し、実装の独立性を確保した。ML エンドポイントが「どういうリクエストを受けてどういうレスポンスを返すか」を文書化してから実装を開始したので、AI チームのモデル実装と並行して進められた。また「P99 目標から逆算して ML に割り当てられる上限は 800ms」という計算を根拠に SLA を提示し、説得力ある交渉ができた。

**Q: 新規ユーザーと経験ユーザーの分岐をなぜ設けたか？**

過去のキャンペーン作成が 5件未満のユーザーに ML でパーソナライズを出しても意味がない（学習データがない）。この場合は Redis から業種別・費用対効果上位5件を返すだけで十分で、ML 呼び出しのコストを払わなくていい。ユーザーの状態によって必要な処理を変えることで、コストとレイテンシを最適化した。

**Q: そもそもなぜアルファベット順が問題になったのか？**

当初はセグメント数が少なく、アルファベット順でも問題なかった。しかし年々セグメントが増え続け、数百件を超えた時点でスクロールが現実的でなくなった。プロジェクト開始時点では約 3,500件に達していた。ユーザーは「とりあえず上から選ぶ」という行動をとるようになり、これは認知バイアスの一種で、アルファベット前半のセグメントが必ずしも高 CTR ではなかったため、広告インプレッションの機会損失に繋がっていた。Google Analytics のファネルデータで離脱項目として確認し、「表示順に問題がある」と結論づけた。レコメンド機能で「費用対効果の高いセグメント」を上位に入れることで、このバイアスを構造的に解消することが目的の一つだった。

**Q: P99 の目標値を実際に達成できたか？**

正確な実測値は手元に残っていないが、レイテンシ目標は達成できたと認識している。Datadog でモニタリングしていて、SLO アラートが継続的に鳴らなかったのが根拠。ML フォールバック率も目標内で推移していた。

**Q: Redis キャッシュの設計で気をつけたことは？**

月次 cron で費用対効果上位 5件を計算し S3 に書き込む。Redis は read-through で、MISS 時に S3 から読んでキャッシュに載せる。TTL は月次更新サイクルに合わせて設定し、古い結果を返し続けないようにした。最終フォールバック（ML 失敗 × 未入力）時もこの流れで対応でき、キャッシュ障害がサービス全体の障害にならない設計にした。

## 深掘り質問への準備

### リリース後、想定外のことはありましたか？

リリース直後、毎朝最初のリクエストで ML エンドポイントのレイテンシが跳ね上がる現象が発生した。原因は ECS がアイドル時間にコンテナをスケールダウンしていて、朝一番のリクエストでコンテナの起動待ちが生じる cold start だった。Datadog のフォールバック率アラートで気づき、AI チームに調査を依頼した。最終的に ML サービス側で minimum task count を 1 に設定することで解消した。フォールバック監視を入れていなければ気づくのが遅れていたと思う。

Top 5 Popular で、食品・日用品などのセグメントが常に上位を占める問題が出た。これらのカテゴリはクリック率の絶対値が高い傾向があり、不動産のような高単価・低クリック率のカテゴリと同じ母集団で正規化しても、業界間のバイアスを消しきれなかったのが原因だった。設計時から「将来は業界別 Top 5 に分割する」とドキュメントに残していたが、その問題がリリース直後に顕在化した。

---

### 今やり直すなら何を変えますか？

1つ目は **Top 5 Popular を業界別にすること**。V1 では全業界共通のスコアを出したが、食品・日用品と不動産では CTR の絶対値が 5〜10倍違う。z-score で補正してもバイアスが残る。最初から業界タグを持ってセグメントを管理し、「同じ業界内での Top 5」を出す設計にすれば精度が上がった。当時は「まず動くものを出す」を優先したが、このバイアスはリリース前に想定できた問題だった。

2つ目は **フォールバック戦略をアーキテクチャ決定と同時に固めること**。今回はマイクロサービスの設計を決めてから「ML が遅かったらどうするか」を考えた順序だった。ML の実装前に正確な P99 はわからないが、「SLA を守れないリスクがある前提でフォールバックを先に設計しておく」ことは最初からできた。依存する外部エンドポイントがある場合は、そのエンドポイントが機能しない前提でシステムを設計してから統合するのが正しい順序だと思う。

---
