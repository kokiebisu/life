**ステークホルダー:** PM、Eラーニングエンジニアチーム（3名）、OEM提供元、freee PSIRT、基盤チーム、CSチーム

## 背景・課題

### ビジネス背景

freee は人事労務 SaaS において給与計算・勤怠管理などのコア機能を提供しているが、企業の HR 担当者が必要とする「従業員研修管理」領域にはプロダクトを持っていなかった。競合他社（SmartHR 等）も研修管理機能を持ち始めており、HR 領域のフルスタック化が戦略的課題となっていた。

### 顧客ヒアリングによる発見（2025年4〜6月）

筆者が 5社の顧客商談に同席し、研修管理ツールに関するヒアリングを実施。当初の想定と大きく異なる発見があった。

**当初の想定（仮説）:**  
「社員が自由に研修を受けられるプラットフォーム（自由受講型）が欲しい」

**実際の顧客ニーズ（ヒアリング結果）:**
- 5社中 5社が「部署別・役職別の**必修研修の管理**」を最優先課題として挙げた
- 「誰がどの研修を完了したか追跡できない」という管理側の問題が本質
- 研修コンテンツそのものより「**受講状況の可視化と未受講者への自動リマインド**」を求めていた
- 既存の研修ツール（外部 LMS）はコンテンツが人事労務データと断絶しており、従業員異動・入退社のたびに手動でアカウントを同期する必要があった

**この発見が製品戦略を変えた:**  
自由受講型プラットフォームを freee 福利厚生の一機能として組み込む計画を全面見直し、強制型研修管理に特化した「freee Eラーニング」として独立プロダクト化する方針に転換。

## 目標

| 優先度 | ゴール | 計測方法 |
|--------|--------|---------|
| P0 | HR 管理者が部署・役職単位で従業員に研修を割り当て、受講状況を一元管理できる | 機能リリース後の利用率 |
| P0 | freee 人事労務の従業員データ（入退社・部署異動）がリアルタイムで Eラーニングに反映される | 従業員データ同期成功率 ≥ 99.5% |
| P0 | freee PSIRT のセキュリティ要件（なりすまし防止・OIDC 標準準拠）を満たす | PSIRT レビュー通過 |
| P1 | 未受講者への自動リマインドメールが設定した期日・条件で配信される | メール配信成功率 ≥ 99% |
| P1 | OEM 提供元の Rate Limit（100 req/分）の制約下で、1,000名規模の従業員データ連携が完了する | 連携完了時間 ≤ 20分（1,000名） |
| P2 | 企業独自の研修コンテンツをアップロードできる | 対応ファイル形式: PDF, MP4, SCORM |

## スコープ外（V1）

| 項目 | 理由 |
|------|------|
| 研修コンテンツの制作・編集機能 | OEM 提供元のコンテンツライブラリ（1,000コース以上）を利用。制作は V2 以降 |
| 修了証の自動発行 | 法的要件がある業種向けに V2 で対応 |
| freee 給与・会計との連携 | 研修管理と給与連携のユースケースが現時点では見えていない。V3 以降 |
| スマートフォンアプリ | Web 優先。受講は PC が主流という顧客フィードバック |

## ユーザー

### HR 管理者（主要ユーザー）

**特徴:** 100〜2,000名規模企業の人事担当者。複数の SaaS を管理している。  
**ゴール:** 年次必修研修の受講状況を一目で把握し、未受講者に個別連絡しなくて済むようにしたい。  
**現状の痛み:** 外部 LMS とのアカウント同期が手動。入退社のたびに二重入力が発生。

### 一般従業員（受講者）

**特徴:** 研修を「業務の一部」として受講する。研修自体に積極的なモチベーションがない場合が多い。  
**ゴール:** 割り当てられた研修をできるだけ早く完了させたい。複数のログイン情報を管理したくない。  
**現状の痛み:** 研修ツール専用のアカウントとパスワードを覚えなければならない。

## ユーザーストーリー

**HR 管理者として、**  
新入社員が入社した際に freee 人事労務側で登録すれば、Eラーニングのアカウントが自動で作成されてほしい。今は毎月 5〜10件の新入社員のアカウントを手動で二重登録しており、1件あたり 10〜15分かかっている。

**HR 管理者として、**  
「法定研修（ハラスメント防止）」を全従業員に割り当て、部署別の受講率をダッシュボードで確認したい。未受講者に期日 3日前と当日に自動リマインドが送られてほしい。

**従業員として、**  
freee のアカウントでそのまま研修プラットフォームにログインできてほしい。別途パスワードを設定したくない。

**OEM 提供元との関係として、**  
Eラーニングプラットフォーム本体（コンテンツ配信・受講記録）は OEM 提供元のシステムを使い、freee は従業員管理・認証・受講状況の連携に特化する。

## 技術的制約

### OEM 提供元からの制約

- **Rate Limit:** 100 req/分。1,000名の従業員を一括登録する場合、単純実装だと 10分かかるがユーザー体験上は「処理中」表示が必要
- **認証フロー:** OIDC フローで認証するが、OEM 側は 1事業所 = 1ユーザー（1:1）設計。freee 側は 1ユーザーが複数事業所に所属できる（1:n）設計。構造的不一致がある
- **リアーキ中の制約:** OEM 提供元は自社システムのリアーキ真っ最中であり、大きな仕様変更に応じられない制約があった

### freee セキュリティ要件（PSIRT）

- なりすまし防止（external_company_id と user_id の不変性保証）
- OIDC 標準準拠（nonce/state 検証によるリプレイアタック・CSRF 対策）
- セッションのセキュアな管理

### freee 基盤チームとの調整

- Eラーニングを「freee 人事労務のモジュール」として統合するか「スタンドアローン（別サービス）」として構築するかの方針決定が必要
- Design Doc を作成し基盤チームとの合意形成を主導（詳細は Design Doc 参照）

## リスクと対策

| リスク | 発生可能性 | 影響 | 対策 |
|--------|----------|------|------|
| PSIRT とOEM の要件が衝突し、どちらも譲れない状況になる | 高（実際に発生） | 高（リリース不可） | 「プロトコルレベルで担保できる部分」と「アプリレベルで補う部分」を切り分けた代替設計を提案。3者合意を形成 |
| OEM の Rate Limit で大量データ連携が UX を損なう | 高（設計時点で確定） | 中（ユーザー体験の悪化） | 非同期ワーカー + 進捗表示 UI で対応。ユーザーには「バックグラウンド処理中」として許容可能な体験を提供 |
| 人事労務本体のモジュール統合でデプロイが複雑になる | 中 | 中（開発速度の低下） | 統合のメリット（認証・DB 共有）とデメリット（デプロイ複雑化）を比較し、モジュール統合を選択（Design Doc に詳細） |
| OEM 提供元のリアーキで API が変わる | 中 | 高（緊急対応が必要） | インターフェース層でアダプターパターンを採用。OEM 固有のロジックを isolate |
