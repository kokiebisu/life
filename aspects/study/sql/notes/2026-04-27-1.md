# SQL: JOIN

## 🎯 今日の目標・疑問

- INNER / LEFT 以外の JOIN（RIGHT / FULL OUTER / CROSS / SELF）を押さえる
- 「INNER は NULL を弾く」より精緻な定義に直す
- LEFT JOIN の `WHERE` vs `ON` の罠を理解する

## 📝 ノート

### サンプルデータ（このノート全体で使う）

```
employees                departments
id | name | dept_id      id | name
1  | A    | 10           10 | Sales
2  | B    | 20           20 | Eng
3  | C    | NULL         30 | HR
```

### INNER JOIN — マッチしない行は除く

> ❌ 「INNER は NULL を弾く」
> ✅ 「INNER は JOIN 条件が TRUE になる行だけを返す」

```sql
SELECT e.name, d.name
FROM employees e
INNER JOIN departments d ON e.dept_id = d.id;
-- 結果: A | Sales, B | Eng （2行）
```

- C が消えた理由 → `NULL = 10/20/30` は SQL では `UNKNOWN` になる（`NULL = NULL` も `UNKNOWN`）。TRUE じゃないので弾かれる
- HR が消えた理由 → どの employees の `dept_id` とも一致しないから

「NULLを弾く」と言うと dept_id に値があってもマッチしなければ消える点を説明できない。**マッチしない行を弾く**が本質。

### LEFT JOIN — 左テーブル全部 + マッチした右

```sql
SELECT e.name, d.name
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.id;
-- 結果:
-- A | Sales
-- B | Eng
-- C | NULL    ← C は employees にいるので残る、d.name は NULL
```

**HR は出てこない**。LEFT JOIN は左 (employees) を残すだけで、右側の孤立行 (HR) は対象外。

```
employees             departments
┌─────┐            ┌─────┐
│ A   │ ────match──│ Sales│
│ B   │ ────match──│ Eng  │
│ C   │     ✗      │ HR   │ ← 誰ともマッチせず孤立 → 消える
└─────┘            └─────┘
```

### FULL OUTER JOIN — 両方全部

HR も含めて4行欲しいなら FULL OUTER JOIN。

```sql
SELECT e.name, d.name
FROM employees e
FULL OUTER JOIN departments d ON e.dept_id = d.id;
-- 結果:
-- A    | Sales
-- B    | Eng
-- C    | NULL
-- NULL | HR      ← departments にあって employees にマッチしない
```

### JOIN 5種を1枚で整理

| JOIN | 残す行 |
|---|---|
| INNER | 両方マッチする行のみ |
| LEFT | 左 全部 + 右マッチ |
| RIGHT | 右 全部 + 左マッチ（実務ではあまり使わない、LEFT で書き直す方が読みやすい） |
| FULL OUTER | 両方全部 |
| CROSS | 全行 × 全行（直積、JOIN条件なし） |

**面接 tip:** RIGHT JOIN を使う人は少ない。「LEFT JOIN を逆にすれば読みやすい」と言える。

### SELF JOIN — 同じテーブルを自分自身と JOIN

階層構造（社員→上司、カテゴリ→親カテゴリ、サプライヤー→親会社など）に使う。

```
employees
id | name | manager_id
1  | A    | NULL       ← CEO
2  | B    | 1
3  | C    | 1
4  | D    | 2
```

```sql
SELECT
  e.name AS 社員名,
  m.name AS 上司名
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;
-- 結果:
-- A | NULL    ← CEO
-- B | A
-- C | A
-- D | B
```

**ポイント:**
- `e` = 社員役、`m` = 上司役。中身は同じ employees テーブル
- `LEFT JOIN` にして、`manager_id = NULL` の CEO も `上司名 = NULL` で残す
- **SELF JOIN は1階層しか辿れない**。任意の深さは再帰 CTE (`WITH RECURSIVE`) が必要 → CTE セクションへの伏線

### LEFT JOIN の罠: `WHERE` vs `ON`

「Sales 部署の社員 + 他部署 / 未配属の社員も全部 NULL で残したい」要件の場合:

```sql
-- ❌ NG: LEFT JOIN なのに INNER JOIN になる
SELECT e.name, d.name
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.id
WHERE d.name = 'Sales';
-- 結果: A | Sales のみ（C, B が消える）

-- ✅ OK: 条件を ON に入れる
SELECT e.name, d.name
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.id AND d.name = 'Sales';
-- 結果:
-- A | Sales
-- B | NULL
-- C | NULL
```

**なぜ `WHERE` だと壊れるか:**
1. LEFT JOIN は先に走る → A, B, C が残る（C は d.name = NULL）
2. `WHERE d.name = 'Sales'` のフィルタで、`'Eng' = 'Sales'` は FALSE、`NULL = 'Sales'` は **UNKNOWN** → 両方弾かれる
3. 残るのは A だけ → INNER JOIN と同じ挙動

**例外（`WHERE` が正しい場合）:** アンチJOINパターン。「マッチしなかった行だけ取る」とき。

```sql
-- 部署が割り当てられていない社員
SELECT e.name
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.id
WHERE d.id IS NULL;
```

## 🔑 キーワード

- INNER JOIN / LEFT JOIN / RIGHT JOIN / FULL OUTER JOIN / CROSS JOIN / SELF JOIN
- `NULL = X` → `UNKNOWN`（TRUE じゃない）
- `ON` vs `WHERE`（LEFT JOIN で外側の表を巻き込まないため `ON` に条件を書く）
- アンチJOIN（`WHERE ... IS NULL`）
- 階層構造 → SELF JOIN（1階層）/ 再帰 CTE（任意の深さ）

## 💡 まとめ

1. **INNER の本質**は「マッチしない行を弾く」。NULL が消えるのは副作用
2. **LEFT JOIN は左を残す**だけ。右側の孤立行は出てこない
3. **両方残したいなら FULL OUTER**、左右を入れ替えたいなら LEFT を逆に書く
4. **SELF JOIN は階層 1段**まで。深さ任意は再帰 CTE
5. **LEFT JOIN の条件は `ON` に書く**。`WHERE` に書くと INNER JOIN になる（IS NULL のアンチJOIN は例外）

### 2026-04-28 - ❌ 忘れた
- **Q: INNER JOIN の結果予測と消えた行の理由（実践: クエリ予測）**
  - 詰まった内容: 「INNER JOIN は NULL を弾く」と回答（ノートが警告している誤解）。HR が消える理由を言えなかった
  - 正解ポイント: 「ON 条件が TRUE になる行だけ返す」。C は `NULL=10` が UNKNOWN で TRUE じゃない → 弾かれる。HR は dept_id 側にマッチがない → 弾かれる
- **Q: LEFT JOIN クエリ作成（全社員 + 部署、部署無しは NULL）**
  - 詰まった内容: `SELECT name, departments FROM ...` と書き、`name` 曖昧 + `departments` がテーブル名なのにカラム位置に置いた
  - 正解ポイント: 同名カラムは `e.name` `d.name` で修飾、`AS` で別名。`SELECT e.name AS social_name, d.name AS dept_name FROM employees e LEFT JOIN departments d ON e.dept_id = d.id;`
- **Q: アンチJOIN クエリ（部署無しの社員だけ）**
  - 詰まった内容: `ON e.dept_id != d.id` と書いた。`!=` だと組合せが膨張して同じ社員が複数行出る
  - 正解ポイント: 普通の `=` JOIN + `WHERE d.id IS NULL`。LEFT JOIN で全員残し、マッチしなかった目印（d.id IS NULL）でフィルター
- **Q: FULL OUTER JOIN の実務ユースケース**
  - 詰まった内容: 「わからん」と即答
  - 正解ポイント: ① データ移行の整合性チェック ② ファイナンシャル照合 ③ 監査・ドリフト検知 ④ コホート比較。「マッチしないものを両方探したい時」が合言葉
