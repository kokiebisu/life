# 勉強 - 2026-04-17

## 🎯 今日の目標・疑問

- Chapter 2: 論理設計・物理設計の全体像とレプリケーションを理解する

## 📝 ノート

### 論理設計（論理スキーマ）

**定義:** 物理制約にとらわれない設計（CPUパワー・ストレージ場所・データ型・SQL文などを考慮しない）

**ステップ:**

1. **エンティティの抽出** — 物理実態あり/なし問わず、概念も含む
2. **エンティティの定義** — 属性（2次元表の「列」）を定める。「キー」= 特定の列の値を決めるための列
3. **正規化** — データの登録・変更・削除が整合的に行えるようフォーマットを整理。エンティティをさらに細かく分割
4. **ER図の作成** — 大量のエンティティ同士の関係を表現する見取り図

### 物理設計

**定義:** データを格納するための領域や格納方法を決める

**ステップ:**

1. テーブル定義
2. インデックス定義
3. ハードウェアのサイジング（キャパシティ観点・パフォーマンス観点：応答時間・QPS）
4. ストレージの冗長機構
5. ファイルの物理配置

### レプリケーション

**構成:** マスタ → スレイブ（トランザクションログの更新データを送った後に同期）

**メリット:** ①可用性向上（冗長性・災害対策）  ②負荷分散（リードレプリカ）

| 方式   | 仕組み                                  | 特徴                                   |
| ------ | --------------------------------------- | -------------------------------------- |
| 同期   | 全レプリカ書き込み完了後にコミット通知    | 遅延のボトルネックになりやすい         |
| 非同期 | スレイブの書き込みを待たずにコミット完了 | 高速だが障害時にデータロスの可能性あり |

**障害時の指標:**

- **RPO（Recovery Point Objective）**: どの時点まで復旧できるか（データロス許容量）
- **RTO（Recovery Time Objective）**: どれだけ早く復旧できるか（ダウンタイム許容量）

レプリケーション = リアルタイム同期・直近のみ保持
バックアップ = 過去の任意時点に戻れる

### クラウド（AWS RDS）

**マルチAZ構成:** 複数AZにレプリカを配置。リードレプリカは非同期レプリケーション

**SLA:**

- 99.95%（スリーナイン）= 約4時間/年のダウンタイム
- 99.999%（ファイブナイン）= 約5分/年（金融・インフラ系で要求）

| 構成           | 方式   | コスト |
| -------------- | ------ | ------ |
| マルチAZ       | 非同期 | 中     |
| クロスリージョン | 非同期 | 最高   |

**バックアップ（S3 Snapshot）:** スレイブDBから取得。取得時に一時的なI/O中断の可能性あり

**クラウドのメリット:** ①初期費用安い ②スケールが容易 ③運用保守要員不要 ④可用性向上が容易

**クラウドのデメリット:** ①カスタマイズ不可 ②コスト増の可能性 ③オンプレ連携困難 ④GDPR等のセキュリティ制約 ⑤障害時にエンドユーザーは無力

### バックアップ戦略

**増分バックアップ:** データ量・時間が最小 ↔ リカバリが最も複雑・RTOが最長

**選択観点:** ①復旧すべき時点 ②バックアップウィンドウ ③RTO ④保持世代数（RPO）

**一般的な構成:** フルバックアップ＋差分 or フルバックアップ＋増分

**補足: 差分と増分の違い（トランザクションログの範囲）**

```
月曜: フルバックアップ
火曜: 変更A
水曜: 変更B
木曜: 変更C

差分の場合:
  火曜のバックアップ → A
  水曜のバックアップ → A + B
  木曜のバックアップ → A + B + C  ← 毎回フル以降の全部

増分の場合:
  火曜のバックアップ → A
  水曜のバックアップ → B
  木曜のバックアップ → C  ← 前回からの差分だけ
```

増分はデータ量が小さい反面、リカバリ時は全ての増分を順番に適用する必要があるためRTOが最長になる

### リストア → リカバリ → ロールフォワード

1. フルバックアップをリストア
2. 差分/増分のトランザクションログを適用（リカバリ）
3. DBサーバーに残存するログを適用（ロールフォワード）

**補足: 各ステップの詳細**

- **リストア** = バックアップ用ストレージ（S3等、本番とは物理的に別の場所）に保管されたフルバックアップのスナップショットをダウンロード＋DBとして使える状態に展開すること。この時点でDBは起動できるが、データはフルバックアップ取得時点の古い状態
- **リカバリ** = リストアで戻ったDB（フルバックアップ時点）に対して、差分/増分バックアップに含まれるトランザクションログを適用し、より新しい時点まで進めること
- **ロールフォワード** = バックアップには含まれないが、DBサーバー上に残存しているトランザクションログを適用して障害直前まで追いつくこと

**リストアは常にフルバックアップから始まる。** 差分/増分だけではDBを復元できない。フルバックアップという「土台」があって初めて、その上にログを適用できる

```
S3 (スナップショット保管)
  ↓ ダウンロード＋展開 = リストア
DBサーバー (フルバックアップ時点のデータで起動)
  ↓ 差分/増分ログ適用 = リカバリ
DBサーバー (最新バックアップ時点まで復旧)
  ↓ 残存ログ適用 = ロールフォワード
DBサーバー (障害直前まで復旧)
```

## 💡 まとめ

1. **論理設計**: 物理制約に縛られず「何のデータをどう関係させるか」を決める。エンティティ抽出 → 定義 → 正規化 → ER図の順で進める
2. **物理設計**: テーブル・インデックス・ハードウェアサイジング・冗長構成・ファイル配置を決める。レプリケーションとバックアップが可用性の両輪
3. **レプリケーション**: 同期（確実だが遅い）vs 非同期（速いがデータロスリスク）。マルチAZ→クロスリージョンの順でコスト・可用性が上がる
4. **バックアップ復旧**: リストア → リカバリ → ロールフォワードの3ステップ。RPO/RTOで方式を選ぶ

### 2026-04-24 - 🔺 あいまい
- **Q: 論理設計と物理設計の違いは？それぞれのステップを順に挙げよ**
  - 詰まった内容: 論理設計のステップで「正規化」が漏れた（抽出 → 定義 → ER図 と答えた）。物理設計でもハードウェアサイジング・冗長機構・ファイル物理配置の3項目が抜けた
  - 正解ポイント: 論理設計 = 物理制約にとらわれない設計（抽出 → 定義 → **正規化** → ER図）。物理設計 = 格納方法を決める（テーブル → インデックス → **ハードウェアサイジング** → **冗長機構** → **ファイル物理配置**）
- **Q: RPO と RTO の違いは？**
  - 詰まった内容: RTO を「どれくらいの時間でのバックアップを目指したいか」と説明（バックアップ時間と混同）
  - 正解ポイント: RTO = **復旧までのダウンタイム許容量**（"どれだけ早く戻せるか"）。RPO = "どこまで戻せるか"（データロス許容量）
- **補足: 物理設計はクラウドでも適用される（マネージドDBでは選択肢に置き換わる）**
  - ハードウェアサイジング → インスタンスクラス選択（db.r5.large 等）。Aurora Serverless なら自動
  - 冗長機構（RAID）→ EBS が裏で自動冗長化（意識不要）
  - 冗長機構（レプリ）→ Multi-AZ / Read Replica をボタン1つで設定
  - 冗長機構（バックアップ）→ 自動スナップショット
  - ファイル物理配置 → マネージドDBでは隠蔽。EBS タイプ（gp3 vs io2）で IOPS 調整は可能
  - **面接フレーズ:** 「論理/物理設計の概念はクラウドでも有効。マネージドDBなら細部は意識せず、サイジングと冗長性の選択に集中できる」
- **補足: AWS RDS のレプリケーション方式**
  - Multi-AZ Deployment（Primary/Standby、同一リージョン別AZ）→ **同期**
  - Read Replica（同一リージョン）→ **非同期**
  - Cross-Region Replica → **非同期**
- **補足: 物理設計の3項目を簡潔に**
  - ハードウェアサイジング = 必要スペック計算（キャパシティ + パフォーマンス）
  - 冗長機構 = RAID / レプリケーション / バックアップでデータ守る
  - ファイル物理配置 = データとログを別ディスクで I/O 分散
