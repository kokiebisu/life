# Day 4: 復習で言えなかったこと

## 2026-04-13

### Go: Error Handling

- **errors.Is vs errors.As の区別が最初曖昧だった**
  - structかどうかで切り分ける: sentinel error → Is、カスタム型（struct） → As
  - errors.As の第2引数は `&myErr`（ポインタのポインタ）

- **typed nil の説明（問題・理由・対策）を最初から言えなかった**
  - interface は (型情報, 値) のペア → 型があるとnil判定がfalse
  - 対策: nilを返すときは必ず `return nil` と明示する

### システム設計: 災害アラート

- **テナント・サプライヤー・施設の概念整理が必要だった**
  - テナント = Resilireの顧客企業
  - サプライヤー = テナントの取引先
  - 施設 = サプライヤーが持つ工場・倉庫（被災判定に使う）
  - 通知先はテナントのユーザー（担当者）

- **設計の選択肢を自分から全部出す習慣がなかった**
  - DLQを提案した後に「他の方法は？」と聞かれて止まった
  - 面接では「他にはX・Y・Zがあり、DLQを選んだ理由は〜」と自分から言い切る必要がある
