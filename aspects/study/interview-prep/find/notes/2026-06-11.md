# 勉強（面接対策） - 2026-06-11

## 🎯 今日の目標・疑問

注文処理システムのコードレビュー課題を通じて、保守性・設計（責務）・信頼性の観点から指摘・リファクタリングを実践する。

## 📝 ノート

### 課題コード

```typescript
async function processOrder(user: User, items: OrderItem[]): Promise<number> {
  let total = 0;
  for (let i = 0; i < items.length; i++) {
    total += items[i].price * items[i].quantity;
  }
  if (user.isPremium) {
    total = total * 0.9;
  }
  await fetch(`https://api.example.com/notify?email=${user.email}&msg=OrderProcessed`);
  return total;
}
```

### 指摘ポイント

**良い点**
- グローバル変数を使っていない（メモリリークなし）
- JS の camelCase 命名規則を守っている
- 引数の mutation が起きていない

**改善点①: readonly 引数**
引数は不変であるべき。将来的に関数内で変更されないよう `Readonly<User>` / `readonly OrderItem[]` を付ける。

**改善点②: マジックナンバー**
`0.9` を定数化。さらに浮動小数点数の演算誤差のリスクがあるため、本番では `BigInt` や `BigDecimal.js` などのライブラリ使用が理想。

**改善点③: fetch の直接使用 → 依存性の注入（DI）**
- 上位レイヤーが下位レイヤー（fetch）に直接依存 → テストでモックできない
- fetch を axios などに差し替えたい時に変更漏れのリスク
- 解決策: `HttpClient` インターフェースを定義し、constructor に注入する

**改善点④: ハードコードされた URL**
`https://api.example.com` を環境変数（`API_BASE_URL`）に切り出す。ドメイン変更時のメンテコスト削減。

**改善点⑤: エラーハンドリング不在**
fetch は 4xx・5xx では `catch` に入らない（重要！）。`response.ok` のチェックが必須。
- 4xx → リトライせず `UserError` を throw してモニタリングへ
- 5xx → exponential backoff でリトライ。失敗したら `SystemError` を throw
- ネット切断・CORS など → catch に入る。re-throw して上位に透過させる

**改善点⑥: 通知の非同期化**
メール送信を同期処理に含めると：
- シングルスレッドで他の処理をブロック
- バースト時にサーバーがパンク
- 解決策: キューに積んでワーカーで処理。rate limit / throttling も考慮。

**改善点⑦: 冪等性**
ジョブ化した場合、`job_id` をキーにキャッシュし、完了済みのジョブを再実行しない設計が必要。未送信は DLQ（Dead Letter Queue）で管理。

### 最終的なリファクタリング構造

```typescript
const API_BASE_URL = process.env.API_BASE_URL;

interface HttpClient {
  fetch(url: string): Promise<Response>;
}

interface INotificationService {
  send(): Promise<void>;
}

class Order {
  constructor(
    private notificationService: INotificationService,
    private user: User
  ) {}

  public async process(items: OrderItem[]): Promise<number> {
    const total = this.calculateOrderItemTotal(items);
    const discounted = new Discount(this.user).apply(total);
    await this.notificationService.send(); // try/catch は上位に任せる
    return discounted;
  }

  private calculateOrderItemTotal(items: OrderItem[]): number {
    return items.reduce((sum, item) => sum + item.price * item.quantity, 0);
  }
}

class NotificationService implements INotificationService {
  constructor(private client: HttpClient, private user: User) {}

  public async send(): Promise<void> {
    try {
      const response = await this.client.fetch(
        `${API_BASE_URL}/notify?email=${this.user.email}&msg=OrderProcessed`
      );
      if (!response.ok) {
        if (response.status >= 400 && response.status < 500) {
          throw new UserError(`通知送信失敗: ${response.status}`);
        } else if (response.status >= 500) {
          await this.retryWithBackoff(); // 3回試みて失敗したら↓
          throw new SystemError(`通知サービス停止: ${response.status}`);
        }
      }
    } catch (err) {
      this.logLocalFailure(err);
      throw new SystemError(`通知サービスが完全に停止: ${(err as Error).message}`);
    }
  }
}

class Discount {
  private readonly PREMIUM_DISCOUNT = 0.9;
  constructor(private user: User) {}

  public apply(amount: number): number {
    return this.user.isPremium ? amount * this.PREMIUM_DISCOUNT : amount;
  }
}
```

### re-throw パターン（面接キラーフレーズ）

> 「下位レイヤーでエラーをキャッチした際は、**揉み消さずに上に透過させる（re-throw）**設計を意識しています。各レイヤーでは、リトライや例外の翻訳など『そのレイヤーの責務』だけを行い、最終的なログ監視やユーザーへのレスポンス整形は最上位のミドルウェアに集中させることで、関心の分離がされた堅牢なアーキテクチャになると考えています。」

re-throw する 2 つの理由：
1. **エラーの文脈（コンテキスト）を追加・翻訳する** — `Network Error` を `SystemError: 通知サービスが停止` に翻訳してミドルウェアに届ける
2. **局所的な回復と全体への報告を両立する** — リトライまで頑張ったが失敗したことを呼び出し元に知らせる。silent fail は NG

### エラー分類の設計

```typescript
class UserError extends Error {}   // 4xx: リトライ不要、モニタリングに通知
class SystemError extends Error {} // 5xx/ネット障害: アラート、Datadog へ

// ミドルウェアで型を見てトリアージ
if (err instanceof SystemError) {
  datadog.alert(err);
} else if (err instanceof UserError) {
  logger.warn(err);
}
```

メリット：
- アラートのノイズ（オオカミ少年化）防止
- Datadog の課金コスト削減（システムエラーのみ送信）

## 🔑 キーワード

- `依存性の注入（DI）` `HttpClient インターフェース` `INotificationService`
- `依存性逆転の原則（DIP）`
- `response.ok` — fetch は 4xx/5xx で catch に入らない
- `exponential backoff` — 5xx 時のリトライ戦略
- `re-throw` — エラーを揉み消さず上位に透過させる
- `UserError / SystemError` — カスタムエラーによるトリアージ
- `DLQ（Dead Letter Queue）` — 送信失敗ジョブの管理
- `冪等性` — job_id をキーにした重複実行防止
- `throttling` — 外部 API の rate limit 対応
- `BigDecimal / BigInt` — 浮動小数点演算誤差の回避

## 💡 まとめ

1. **fetch は 4xx/5xx で catch に入らない** — `response.ok` チェックは必須。面接で突っ込まれやすい罠
2. **re-throw パターン** — 各レイヤーは自分の責務（リトライ・翻訳）だけをやって、最終判断はミドルウェアに任せる
3. **エラーを型で分類する** — `UserError` vs `SystemError` でミドルウェアがトリアージできる設計が本番レベル
4. **DI はテスタビリティのため** — `INotificationService` を constructor で受け取ることで、テスト時にモックに差し替えられる
