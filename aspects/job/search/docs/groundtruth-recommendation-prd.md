# PRD: 広告セグメントレコメンデーション機能 — Groundtruth

**著者:** Ken Oki  
**ステータス:** リリース済み（2024年6月）  
**ステークホルダー:** プロダクトマネージャー、バックエンドチーム（5名）、AI基盤チーム、データチーム  
**最終更新:** 2024年5月

---

## 背景・課題

### ビジネス背景

Groundtruth は位置情報を活用した広告プラットフォーム（ニューヨーク本社）。広告主は Ads Manager（Web UI）でキャンペーンを作成し、ターゲットとなる消費者セグメント（年齢層、位置情報、興味関心など）を選択して広告を配信する。

2024年初頭時点で利用可能なセグメント数が約 3,500 件に達し、以下の問題が表面化した：

**Google Analytics で確認された定量データ（プロジェクト開始時点）:**
- キャンペーン作成フロー全体のコンバージョン率: 62%
- そのうちセグメント選択ステップでの離脱率: 全離脱の **38%**（離脱の最多ステップ）
- セグメント選択に費やす平均時間: 4分12秒（社内ベンチマークの倍以上）
- 直近 6ヶ月のキャンペーン平均 CTR: 0.48%（業界平均 0.71%）

**根本的な課題:**
1. ドロップダウンに 3,500 件のセグメントが並んでおり、どれを選べばいいかわからない
2. 選択のガイダンスがないため、効果の低いセグメントを選びがちになり ROI が出ない
3. ROI が出ないと広告主が離れる → Groundtruth の収益減

### 技術的背景

既存の Ads Manager は Python / Flask（WSGI 同期）で構築された Rails 風のモノリシックアプリケーション。データは PostgreSQL に格納されており、セグメント検索は SQL LIKE クエリで実装されていた（インデックスなし）。

- セグメント検索クエリの P99 レスポンスタイム: **3.2秒**（ユーザーから遅いという苦情）
- AI基盤チームが既にユーザーの行動履歴を学習した LLM エンドポイントを保有しているが、Ads Manager への統合は未着手

## 目標

| 優先度 | ゴール | 計測方法 |
|--------|--------|---------|
| P0 | セグメント選択ステップでの離脱率を現状比 20%削減する | Google Analytics（キャンペーン作成ファネル） |
| P0 | 新規ユーザーが迷わない「デフォルト推薦」をドロップダウンの上部に表示する | UX テスト + 定量ファネル計測 |
| P1 | 経験ユーザーへのパーソナライズ推薦により、キャンペーン CTR を業界平均（0.71%）に近づける | キャンペーン CTR（Ads Manager 内レポート） |
| P1 | ML エンドポイント障害時に機能が完全停止しない（フォールバックで継続） | フォールバック発動率の監視 |
| P1 | セグメント検索レスポンスタイムを現状 P99 3.2秒 → **1秒以内**に改善 | APM メトリクス（Datadog） |
| P2 | レコメンド機能を独立したサービスとして Ads Manager のデプロイサイクルから切り離す | デプロイ頻度（週次目標） |

## スコープ外（V1）

| 項目 | 理由 |
|------|------|
| レコメンデーションモデルの学習パイプライン | AI基盤チームが担当。Groundtruth のバックエンドチームはエンドポイントの消費者に徹する |
| ユーザーフィードバック収集（「役に立った/立たなかった」） | PM から追加提案があったが、技術工数を分析し期限内完了が困難と判断。V2 候補 |
| セグメントの新規作成・編集 | 別チームが管理。スコープ外 |
| モバイルアプリ向け対応 | Ads Manager は Web のみ。モバイルアプリは使用率が低く後回し |

## ユーザーセグメント

### 新規ユーザー（選択履歴 0〜2件）

**特徴:** 初めて、または数回しかキャンペーンを作ったことがない。選択肢の意味がわからず「なんとなく」選ぶか、離脱する。  
**ニーズ:** 「まずこれを試せ」という信頼できるデフォルト推薦。迷わせない。

### 経験ユーザー（選択履歴 3件以上）

**特徴:** 複数のキャンペーンを継続運用している広告主。過去のパターンがあり、似たターゲティングを繰り返すことが多い。  
**ニーズ:** 「前回うまくいったセグメント」に近い提案。毎回ゼロから探す無駄を省きたい。

---

## ユーザーストーリー

**新規ユーザーとして、**  
キャンペーンを作る際に「人気のセグメントトップ5」が一番上に表示されてほしい。3,500件のリストを全部見たくない。この推薦を使えば最低限の効果が出るという安心感がほしい。

**経験ユーザーとして、**  
自分の過去のキャンペーンパターンを踏まえた「あなたにおすすめ」が表示されてほしい。毎回同じような設定を手動でやり直すのは時間の無駄。

**プロダクトマネージャーとして、**  
「セグメント選択ステップの離脱を減らした」ことを Google Analytics のファネルデータで定量的に報告したい。

**AI基盤チームとして、**  
LLM エンドポイントの呼び出しインターフェース（リクエスト/レスポンス形式）を明確に定義し、Ads Manager 側の実装詳細に関与せずに進めたい。

## 制約条件

### チーム間依存

- **AI基盤チーム:** パーソナライズ推薦は AI基盤チームの LLM エンドポイントに依存。このチームとはスプリントスケジュールが独立しており、API 仕様変更・エンドポイント障害も考慮が必要。
  - **対策:** インターフェースを事前に書面で合意（OpenAPI spec）。エンドポイント障害時のフォールバックを必須設計とする。
- **DevOps チーム:** Elasticsearch / ElastiCache クラスター構築に DevOps との協力が必要。
  - **対策:** インフラ要件を事前に文書化し、スプリント 1週目にキックオフミーティングを設定。

### 技術的制約

- **既存 Ads Manager はモノリス（Flask）:** 大規模なリファクタなしに既存アプリへの統合は困難。新機能は独立したマイクロサービスとして構築する。
- **タイムゾーン差による AI基盤チームとのコミュニケーション:** AI基盤チームは西海岸（PST）、Groundtruth バックエンドチームは東海岸（EST）。非同期コミュニケーションが基本。
  - **対策:** API 仕様を OpenAPI spec で事前合意し、互いの実装依存を最小化。

### ビジネス制約

- **スプリント途中でのスコープ追加:** PM から「ユーザーフィードバック収集機能」の追加提案があった。技術工数を見積もり（約 3スプリント分）、Q2 末の期限内完了が困難と判断。PM と合意の上 V2 に移動。

## リスクと対策

| リスク | 発生可能性 | ビジネスインパクト | 対策 |
|--------|----------|-----------------|------|
| ML エンドポイントが SLA を守れない（推論時間 > 2秒） | 中（AI基盤チームの SLA が未確定） | 中（レコメンド全体がタイムアウト扱いになる） | タイムアウト 2秒 + Top 5 Popular へのフォールバック。フォールバック率 > 10%でアラーム |
| Elasticsearch インデックス設計ミスによる検索精度低下 | 中 | 高（UX が改善しない） | Analyzer・Boost 設定の A/B テストをリリース前に社内で実施 |
| スコープ追加による期限超過 | 高（実際に発生） | 高（Q2 コミットメント違反） | 追加スコープの技術工数を即座に見積もり。期限内不可能なものは次スプリントへ。この判断で 2週間の余裕が生まれた |
| AI基盤チームの API 仕様変更 | 低〜中 | 中（レコメンドサービスの修正が必要） | OpenAPI spec でインターフェースを合意・バージョニング。変更時はマイクロサービス内のアダプター層を修正するだけ |
