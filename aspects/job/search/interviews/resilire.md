## ⚡ Rate Limit 深掘り Q&A（メイントピック）

**Q: 一括招待の rate limiting 設計は？**
→ OEM のユーザー作成 API に rate limit がある。worker を urgent と bulk に分け、それぞれに独立した Redis rate limiter を用意した。

- **bulk worker**: 一括招待と組織同期を担当。保守的な閾値でスロットリング
- **urgent worker**: ロール変更・従業員削除など即時性が必要な操作を担当。bulk が quota を使い切っても urgent の quota は独立して確保

Lua スクリプトで「残り確認→インクリメント」をアトミックに実行。2コマンドに分けると複数ジョブが同時に「残り1」を読む race condition が起きる。OEM のレスポンスヘッダー（`X-RateLimit-Remaining`系）は制御には使わず、アラート指標として使用。exponential backoff をリトライに組んだが本番では一度も発火しなかった。

**Q: なぜ OEM ヘッダーを制御に使わなかったか？**
→ OEM ヘッダーはリクエストを送った後にしか読めない。「送る前の判断（制御）」には使えない。事前にスロットリングするには自前カウンタが必要で、ヘッダーは事後モニタリングにしか使えない。

**Q: 自前カウンタと OEM ヘッダーでズレは生じないか？**
→ 窓の実装が違う場合（自前 fixed window、OEM rolling window）は境界付近でズレが起きうる。対策は2層。①閾値を OEM の limit より保守的に設定して安全マージンを確保。②万が一 429 が返ってきても exponential backoff で復帰できる設計。OEM ヘッダーで実際の残量を事後モニタリングし、ズレが常態化していないか監視する運用も入れた。

**Q: テーブルロックが urgent 操作と競合するリスクは？**
→ UI レベルで制約を設けた。100件ずつの chunk でしか招待できないページネーションにした。1チャンクのトランザクション保持時間が短くなり、urgent とのロック競合の確率と影響範囲を下げた。管理者がページをめくる操作の間隔が urgent の入れる窓になる。

**Q: ジョブ失敗時の再実行で OEM API を二重呼び出しするリスクは？**
→ `provisioning_status` カラムで冪等性を担保した。`pending` / `success` / `failed` の3ステータスを持ち、OEM API 呼び出し前に `success` でないことを確認してから実行。

**Q: ユーザー削除後のセッション無効化は？**
→ Webhook が打てない（基盤側実装待ち）という制約があった。OEM のセッション TTL が元々 30 分で freee のセッションと同じ長さだったので変更不要と判断した。webhook 実装待ちの間の許容できるトレードオフ。

## 📊 Groundtruth — Recommendation Microservice（Go版）

**なぜ Resilire に刺さるか**

Go + goroutine 並列処理・マイクロサービス vs モノリスの判断軸・3段階フォールバック、Datadog SLO 監視。御社の SCRM 3.0 で「モジュラモノリス採択」という同系の判断をしているので、**判断軸で会話ができる**。

**Resilire 向けフック**

> 「御社が SCRM 3.0 でマイクロサービス vs モジュラモノリスを議論して、チーム規模と試行錯誤フェーズを理由にモノリス採択された記事を読みました。私も Groundtruth でレコメンデーション機能を作るとき、逆の判断をしたことがあります。そのときの判断軸をお話しさせてください」

**話すストーリー本文（3-4分）**

「ユーザーがキャンペーン作成時にセグメントを選ぶ画面があったのですが、選択肢が肥大化して「何を選んでいいかわからない」という状態になっていました。Google Analytics で churn が確認されていたので、レコメンデーション機能を追加することになりました。

設計時に選択肢が3つありました。パターン A: 既存モノリスに追加。パターン B: 既存言語（Flask/Python）で独立サービス。C: 別言語（Go）で独立サービス。

判断軸は3つでした。最上位は P99 1,000ms。ML エンドポイント（最大 800ms）+ Elasticsearch 検索が直列で走るので並列化が必須。パターン A は消えました。Flask は WSGI 同期で asyncio.gather() が使えない。gevent で無理矢理退けることもできたが、モンキーパッチの副作用・社内に運用実績がないことを懸念して Go にしました。goroutine で ML と ES を並行呼び出しして集約する設計です。

フォールバック設計も重要でした。ML エンドポイントは AI チームが別管理で落ちることが予想された。3段階のフォールバック: ML 成功→Personalized、ML 障害→Elasticsearch のみ、ユーザー未入力→Redis にキャッシュした Top 5 Popular。Datadog でフォールバック率 5% を SLO として監視しました。

結果、churn を 20% 減、広告インプレッションを 10% 向上。学んだのは、具体的な SLO とチームの運用能力を軸に技術選択をすること。同じことが御社のモノリス採択にも作用していると思います」

**核トレードオフ**

- モノリス追加 vs マイクロサービス：SLO 達成と独立性の天秤
- Flask/gevent vs Go：チームの運用経験の有無で判断
- フォールバックの段階数：3段階がそのトラフィックで適切、過剰にしない

**想定ツッコミ**

- **Q: goroutine の漏れ対策は？** → context.WithTimeout で ML/ES 周辺にタイムアウト。runtime.NumGoroutine() を Datadog に送信し続けて、増加傾向が見えたらアラート
- **Q: 今やり直したら？** → 同じ構成。ただ Pub/Sub 代わりに gRPC を検討する（御社も gRPC + SQLBoiler を使っているので話が合いやすい）

## 📝 PRD・仕組み化

### Q: PRD を書く上で意識していることは？

3つ意識している。

**① 「決めていないこと」を先に書く**

決めたことの羅列より、まだ判断が固まっていないポイントを早めに可視化してステークホルダーと合意を取る素材にすることを優先する。PRD は書いて完成ではなく合意を取るための道具。

**② 読む人の層を分けて書く**

E ラーニングの PRD は PSIRT・OEM・Eng の三者が読む必要があった。層ごとに知りたいことが違うので、要約→詳細の構造にするか、セクションを明示的に分ける。

**③ 状態遷移で網羅する**

「新規追加だけ」書くと実装段階で詰まる。「更新・削除・権限変更・異動・休職」まで洗い出してから書き始める。ハッピーパスを肉付けするのではなく、状態遷移を起点に要件を炙り出す順序が大事。

### Q: PRD を書いた経験は？

「freee E ラーニングで PdM と組みながら PRD / Design Doc の壁打ちを常にしていました。組織同期機能の場合、従業員データの状態遷移パターン（新規・更新・削除・権限変更など）を網羅的に洗い出して PRD に落としました。OIDC 認証フローも PSIRT・OEM・Eng の三者が読める形で Design Doc を作り、施行前に合意を取りました。PRD・Design Doc ともに「決めたこと」より「まだ決めていないこと」を明記するスタイルです」

### Q: Design Doc はどう使っていたか？

技術的な詳細は Design Doc に書きすぎないようにしていた。実装の詳細・コードレベルの設計は PR の説明に書く方が適切で、Design Doc に詰め込むと読む人が多くて合意が取りにくくなる。

Design Doc は「まだふわっとしている事柄についての合意を取る場」として使った。「なぜこのアプローチを選んだか」「何と何を比較して、どの軸で判断したか」という決定の背景と根拠を残すもの。OIDC の認証設計で PSIRT・OEM・Eng 三者が読む Design Doc を作ったときも、実装詳細より「なぜ sub ではなく email を突合キーにしたか」「なぜ freee 側でセッション管理するか」という判断軸を中心に書いた。

結果として技術詳細は PR に、判断の WHY は Design Doc にという分離が生まれ、設計変更時に「あの判断どこに書いてある？」という検索コストが下がった。

### 開発プロセスの標準化（仕組み化の具体例）

3名チームで AI を「個人の工夫」でなく「チームの仕組み」に組み込んだ。

- **`.claude/skills`** にチーム共通の開発フローをコード化。「PR を single responsibility で出す」スキルを全員で共有し、ジュニアの PR サイズが平均 500 行→150 行に改善
- **`CLAUDE.md`** で開発ルール・コーディング規約を共有。Claude Code が自動参照するので、口頭で毎回説明しなくて済む標準が生まれた
- **CI に Claude GitHub Actions + CodeRabbit を組み込み**、全 PR に AI レビューが入る体制。人間のレビューに入る前にパターン的な指摘が潰れる構造にした。レビュー TTA が 2日→4時間に
- **マルチエージェントレビュー**：1つのエージェントに全観点を見させると見落としが出る。脆弱性・アクセシビリティ・リンター・文法・ロジックと観点ごとにエージェントを分けて、single responsibility を課した。エージェント同士が干渉しないので並列実行できてレビュー時間が短い。観点ごとに分かれているので「なぜその指摘が出たか」のトレーサビリティも明確になる
- **Claude Code hooks**：`PostToolUse` フックでファイル変更を検知し、変更対象の RSpec を自動実行した。背景として、PR の CI でテストが落ちることが多く、approve を取った後に修正するとまた approve が外れてレビューし直しになるブロッカーが繰り返し起きていた。ローカルで変更するたびに即座にテストが回るので「CI に上げて初めて気づく」サイクルを断ち切れた。approve のやり直しコストが実質ゼロになった
- 結果として「ジュニアが AI 生成コードを読まずにマージしかける」事故を仕組みで防けた

ポイントは**個人の工夫を仕組みに変えたこと**。同じツールを使っても属人化したままでは再現性がない。御社の「仕組みで守る」文化と同じ発想で AI 活用を設計した。

### Freee Eラーニング プロダクトリード立ち上げ @ Freee（概要）

24卒・25卒を含む 3名、7ヶ月、OEM 上場企業・リリース日契約固定の制約で 2026年 2 月に 1,000コース・6,000本動画の受講管理 SaaS をリリース。

**核となる判断ポイント**

- **モジュール統合 vs standalone** → モジュール統合（セキュリティ + 開発速度）
- **分散並行 vs ペアプロ小単位インクリメント** → 後者（総合速度で勝つ）
- **個人の努力 vs 構造的打ち手** → 後者（スケールするリーダーシップ）
- **全部拾う vs 捨てる判断** → 中盤で「何を捨てるか」をステークホルダーと再合意

**想定ツッコミ**

- **Q: プロダクトリードと EM の違いは？** → EM はチーム運営にフォーカス。プロダクトリードは What/Why から How まで横断。PdM と協働で PRD を書き、技術選定を主導し、リリース判断までコミット
- **Q: 3名で完遂できた要因は？** → ①技術選定（モジュール統合）で複雑性を削った、②スクラム再設計で「小さいインクリメント」に転換、③構造的打ち手で信頼関係を立て直した
- **Q: 一番の失敗は？** → 前半、自分が個人の努力で全てを解決しようとしてチーム停滞を招いた。構造の問題に気づいて打ち手を切り替えた
- **Q: 次に同じ状況なら何を変える？** → 「引き受ける」判断と「何を捨てるか」の合意をセットで初日にやる
- **Q: 運用・CS まで関わった経験は？** → リリース後の監視設計から担った。Sidekiq queue latency・429 率・acquire 待機時間をキーメトリクスに設定し、OEM 側 vs 自社 worker の問題を切り分けられる体制を整えた。CS 寄りの役割として、OEM 先・PSIRT・PdM の3者コミュニケーションハブを担い、契約上の制約を「なぜ」だけ翻訳してエンジニアに流す運用を作った

### キャリア深掘り：技術追及Q&A

#### E-Learning — OIDC 認証設計

**Q: freee 1:n と OEM 1:1 の不一致は具体的にどう解いたか？**
→ OEM のユーザー作成 API に `{user_id}_{company_id}` を login_id として渡して provisioning。認証時は OIDC 後に userinfo の email + company_id で突合する設計。sub は freee 側で暗号化済みで復号アルゴリズムを OEM に渡せないため使えない。

振り返ると突合キーに email を使ったのは改善余地がある。①email はドメイン移行・改姓で変わりうる。②派遣社員・バイトは freee に email を登録していないケースが多い。`freee_user_id` を userinfo のカスタムクレームに追加すれば両方解決できた。プロジェクト序盤で認証基盤チームに早めに打診すべきだったという反省点。

**Q: `{user_id}_{company_id}` 設計で将来 user_id が変わったら？**
→ freee の user_id は内部連番（変わらない）。変わる場合は migration スクリプト + OEM 側ログイン URL リダイレクトで対応する設計を Design Doc に明記した。

**Q: 組織同期の状態遷移、どのパターンを洗い出したか？**
→ 新規追加 / 属性更新 / 会社間異動 / 休職 / 退職（論理削除）/ 権限変更（昇格/降格）の 6 パターン。過渡期の UX（同期中に別タブでアクセスしたら？）は楽観的ロック + リトライで設計。

**Q: PSIRT 承認で何が懸念されたか？**
→ パース失敗時の挙動（不正アクセスにならないか）→ エラー時は 401 で弾く設計を明示。

#### E-Learning — チーム運営・構造的打ち手

**Q: 意思決定権限の「明文化」はどんな形式で？**
→ 作るきっかけは「先週決めたはずの議論が今週また始まる」という繰り返しだった。口頭合意は記録が残らないので、次の会議で「そんな決定はしていない」と言えてしまう。

Confluence に「意思決定マトリクス」を1ページ作り、3分類を表にした。

| 決定カテゴリ | 誰が決める |
|------------|----------|
| 技術選定・実装方式 | Eng 単独可 |
| スコープ変更・仕様変更 | PdM + リード合意必須 |
| リリース判断 | リード + PM 双方サイン |

決定が出たら Design Doc にリンクを貼り「この判断は済み」と明示した。境界が曖昧な案件（「これは技術選定か仕様変更か」）が出たときもマトリクスを参照点にして「どちらに近いか」を話し合えた。OEM 側の固定制約は「Eng 判断不可・OEM 固定」として別枠に入れて、チームが無駄に検討しない構造にした。

ADR がある御社なら、このマトリクス自体を ADR に昇格させるのが自然だと思っている。「なぜこの分担にしたか」の WHY が残るので再議論のコストも下がる。

**Q: スケジュール管理はどうやっていたか？**
→ リリース日契約固定という制約があったので、クリティカルパス分析をスケジュール管理の軸にした。全タスクを洗い出してから依存関係を整理し、「これが遅れるとリリースが遅れる」パスを特定した。

クリティカルパス上のタスクには不確実要素（OEM との調整・PSIRT 承認・外部 API の仕様確定）が乗りやすいので、そこだけ優先的にリスクを可視化してステークホルダーに共有した。「何が終わらなければ他が進まないか」を全員が見える状態にしておくと、チームが自分で優先度を判断できるようになる。クリティカルパス外のタスクは後回しにすると明示することで、何を捨てるかの判断もスムーズになった。

**Q: 「小さいインクリメント」転換後、スプリント構成はどう変わったか？**
→ 最初は 2 週スプリントで大機能を並行担当。転換後は 3-4 日マイクロスプリントで「1機能を 2 人でペア、その日中にデプロイ可能な状態にする」。PR サイズが平均 500 行→150 行、レビュー TTA が 2 日→4 時間に。

**Q: 「仕組みで守る」効果を定量的に測ったか？**
→ 完全な定量化は難しいが、proxy 指標として3つ追った。

- 意思決定会議の回数：週 3 回→週 1 回
- PR サイズ：平均 500 行→150 行
- レビュー TTA：2 日→4 時間

「待ち時間」が具体的に何だったかというと、仕様確認のために PdM を待つ時間・誰かの承認待ちでブロックされる時間・「あの決定どうなった？」を確認するための会話コスト。これが減ったことで、エンジニアが集中できる時間が増えた体感があった。

ただ proxy なので正直に言うと限界もある。「構造が改善されたから会議が減った」のか「全員疲れてきて会議をサボり始めた」のかは数字だけでは判断できない。だからこそ定量指標と並行して「ジュニアが自分の判断で動けているか」という定性的なシグナルも見ていた。メンバーが「この判断は私がやっていい範囲ですか？」と確認しなくなったタイミングで、マトリクスが機能していると判断した。

#### テスト哲学（Resilire は testcontainers / mock 禁止派）

**Q: テストでモックを使っていたか？**
→ 最初はモックを使っていた。ただ Rails/ActiveRecord は「モデルが DB と一体」なので、テスト用に DB だけ差し替える仕組みがない。ActiveRecord のメソッドを個別に stub するしかなく、実装が少し変わるたびにモックが壊れる。外部 API は WebMock で HTTP 層をスタブする形に落ち着いた。

DB については transactional fixtures（本物の test DB を使い、テスト後にロールバック）に切り替えた。これは Rails コミュニティが収束した標準的な手法。

ただし CI 環境との DB バージョン差異が原因でテストが不安定になることがあった。ローカルは PostgreSQL 14、CI は apt の PostgreSQL 12 という状態になりがちで、`ORDER BY` を省略したクエリの結果順序や JSONB 演算子のインデックス利用可否がバージョンで変わる。ローカルでは通るのに CI だけ落ちる場合、「たまたまローカルが通っていただけ」なので根本原因の特定が難しい。Docker Compose で管理していても、`docker-compose.yml` と CI の `services:` ブロックが別ファイルに分かれているので更新漏れによる drift が起きる。

今やり直すなら testcontainers を使う。コンテナの定義がテストコード内に入るので git で管理され、誰がどこで実行しても同じイメージが起動する。Docker Compose のように「テスト前に別コマンドでコンテナを立てる」手順も要らない。「ローカルで通ったから CI も通るはず」が本当に保証される構造で、御社の方針と同じ発想だと思っている。

**Q: Testing Trophy（統合テスト重視）に対してどう思うか？**
→ 賛成。ユニットテストだけだと、統合境界（DB・API）のバグを見逃す。一方 E2E だけだと遅くてフィードバックループが長い。統合テストを厚くして、ユニットは純粋な計算ロジックに絞る方針が再現性と速度のバランスで一番良いと実感している。

Rails との相性もいい。request spec（コントローラー〜モデル〜DB を通す）がちょうど統合テスト層に対応していて、transactional fixtures で速度も確保できるのでフレームワークが自然に Trophy の戦略に乗せてくれる。

**Q: どのようにテストを書いていったか？**
→ カバレッジの閾値を設定して正常系は必ず網羅する方針にした。ジュニアメンバーへの進め方として TDD を取り入れた。実装の前にテストケースのタイトルだけを列挙してもらい、そこをレビューする手順を踏んだ。タイトルの列挙はその人が仕様をどう理解しているかが透けて見えるので、「何を確認するか」の認識を実装前に合わせられる。実装後に「思ってたのと違う」という手戻りが減った。

**Q: 非同期のテストを書く上で苦労したことは？**
→ 非同期処理の正しさは単体テストで部品レベル（冪等性・rate limiter のロジック）を確認できても、並列実行時の挙動や大量データでのメモリ挙動は単体テストでは再現できない問題が残る。どれだけテストを書いても例外は起きうると考えて、負荷テストで検証する判断をした。

実際、負荷テストで 1,000 名超の一括招待でワーカーが OOM で落ちることが初めてわかった。単体テストでは絶対に発見できなかった問題で、chunk 分割で対応して 2,000 名で安定確認してからリリースした。単体テストが「部品が正しく動くか」を確認するもの、負荷テストが「並列・大量処理で何が壊れるか」を確認するもの、と役割が違うと整理している。

**深掘り: 負荷テストで OEM API を実際に叩いたら rate limit に引っかかるのでは？**
→ 負荷テストの目的はワーカーのメモリ挙動・並列処理の検証であって、OEM API 自体のテストではない。OEM API は WebMock でスタブして、ワーカーが OOM しないかだけを確認した。rate limit のロジックは Redis を使った統合テストで別途確認した。役割が違うので分けて検証した。
