## 1. 面接中に見るカンペ

### 30秒版

- freee Eラーニングで、OEM の Eラーニング基盤と API 連携していた
- 従業員アカウント作成、招待、削除、ロール変更、組織同期が対象
- OEM 側に **1 IP あたり 300 req/sec** の rate limit があった
- freee 側は全顧客のリクエストが同じ出口 IP から出る構造だった
- そのため、複数社が同時に一括招待すると OEM の上限に当たりやすかった
- 対応として以下を入れた
  - Sidekiq による非同期化
  - N+1 クエリの解消
  - 2つの worker による処理分散
  - worker 負荷試験による一括招待上限の把握
  - lock を見て、片方が詰まってももう片方が取得できる構成
  - Redis rate limiter
  - 同一企業の組織同期を並列実行しない mutex
  - retry と冪等性の担保
- UX は理想形までは作れていない
- 今なら、進捗表示や既存ジョブへの合流 UX を追加したい

### 2分版

「freee Eラーニングでは、OEM の Eラーニング基盤と API 連携していました。従業員アカウント作成、招待、削除、ロール変更、組織同期などが対象です。OEM 側には 1 IP あたり 300 req/sec の rate limit があり、freee 側は全顧客のリクエストが同じ出口 IP から出る構造でした。つまり、複数社が同時に一括招待すると、全社で 300 req/sec を奪い合う状態になります。

対応として、重い処理は Sidekiq で非同期化し、N+1 クエリを解消しました。さらに worker を 2 つに分け、負荷試験で一括招待上限を把握しました。API 呼び出し側には Redis rate limiter を入れ、同一企業の組織同期が並列で走らないよう mutex も入れました。リトライは単純に叩き続けるのではなく、冪等性を保った上で失敗時に再実行できるようにしました。

当時の割り切りとして、UX は最低限でした。管理者が待っている操作と、一括招待のようなバッチ処理を完全には分け切れていません。今なら urgent / bulk queue を分け、管理者操作を優先し、一括招待は soft cap をかけます。さらに進捗表示や、後から押した同期ボタンを既存ジョブに合流させる UX を作ります。

この話で伝えたいのは、rate limit 対策は単にリトライを入れる話ではなく、外部 API、worker、DB、UX をまとめて詰まらせない設計だという点です。」

### 深掘り版

- 判断軸: OEM API の上限、全顧客で出口 IP を共有する構造、worker の詰まり、管理者 UX
- 検討した選択肢: 同期処理のまま待つ、非同期化だけする、rate limiter / mutex / retry / 冪等性まで入れる
- 採用理由: OEM 側を守りつつ、freee 側の worker も詰まらせない必要があった
- トレードオフ: UX は最低限。進捗表示や既存ジョブへの合流 UX は後回し
- 一番のリスク: 一括招待が増えた時に、管理者操作まで巻き込んで遅くなること
- 今なら改善: urgent / bulk queue 分離、bulk soft cap、進捗表示、失敗分だけ再試行、既存ジョブへの合流 UX

## 2. 状況

- プロダクト:
  - freee Eラーニング
  - 人事労務から OEM 提供元の Eラーニング基盤を呼び出す

- OEM API で行っていた操作:
  - ロール変更
  - ユーザー削除
  - 招待
  - 組織同期

- 前提:
  - 研修を割り当てる前に、OEM 側に従業員アカウントと組織ツリーが必要
  - 招待や組織同期は、1操作に複数 API コールが発生する

---

## 3. 制約

- OEM 側の制約:
  - 1 IP あたり 300 req/sec
  - 交渉して引き上げ済み
  - これ以上は上げられない

- freee 側の制約:
  - 全顧客のリクエストが同一 IP から OEM に送られる
  - 顧客ごとに出口 IP が分散しない
  - 全社で 300 req/sec を共有する必要があった

- 操作の重さ:
  - 1 従業員招待 = アカウント作成 + 組織ツリー同期などで複数コール
  - 1 社 100 名一括招待 = 数百 API コールになり得る
  - 1,000 名以上の一括招待では、OEM API だけでなく worker 側の CPU もボトルネックになった
  - 複数社が同時に一括招待するとすぐ上限に近づく

- チーム/環境:
  - Rails + Sidekiq
  - Redis 利用可能
  - AWS スタックあり
  - WebSocket は人事労務で前例がなく、当時のスコープでは避けた
  - チーム規模も大きくなく、保守しやすい設計が必要だった

---

## 4. 操作ごとの性質

| 操作 | 性質 | 面接での整理 |
|---|---|---|
| ロール変更 | 管理者が UI で待つ | 即時性が高い |
| ユーザー削除 | 管理者が UI で待つ | 即時性が高い |
| 組織同期ボタン | 管理者が明示的に押す | 即時性が高い |
| 招待前の組織同期 | 招待ジョブの前処理 | バッチ寄り |
| 一括招待 | 時間がかかってもよい | バッチ寄り |

話すポイント:

- 同じ OEM API でも、呼び出し文脈で重要度が変わった
- 特に組織同期は dual-trigger だった
  - 管理者ボタンならユーザーが待っている
  - 招待前処理ならバッチの一部
- ただし当時の実装は、`urgent` / `bulk` という綺麗な queue 分離ではなかった
- `urgent` / `bulk` は、**今ならこう分ける**という改善案として話す

## 5. 実装したこと

### 5.1 非同期化

- 一括招待は Sidekiq job に逃がした
- UI リクエスト内で大量の OEM API を叩かないようにした
- job 側で OEM API を順次呼び出す
- 429 や一時的な失敗は Sidekiq retry に乗せた

話し方:

- 「まず同期処理のままだと UI も OEM も詰まるので、重い処理は Sidekiq に逃がしました」
- 「ただし、1本の worker だけだと lock 待ちで詰まりやすいので、worker を2つ用意しました」

### 5.2 2 worker + lock 取得

- worker は2つ用意していた
- 当時の実装は、`urgent` / `bulk` という綺麗な分け方ではなかった
- それぞれの worker が、好きなタイミングで取得できる仕事を取得する
- 片方の worker が lock 対象の処理で待っている場合、もう片方の worker が別の取得可能な処理を進められる
- 目的は、片方の処理待ちで全体が止まるのを避けること

補足:

- これは「厳密な優先度制御」というより、詰まりを減らすための並列度確保
- lock がある処理では、片方が取れなくてももう片方が進める余地を作れる
- 管理者操作を完全に優先保証する設計ではない

深掘りされたら:

- 「当時は `urgent` / `bulk` queue で綺麗に分けたわけではなく、worker を2つ用意して lock を見ながら取得する形でした」
- 「片方が lock で進めない時に、もう片方が別の処理を拾えるようにしていました」
- 「今なら、管理者操作をより守るために urgent 専用 queue / worker を分けます」

### 5.3 N+1 クエリの解消

- 一括招待では、従業員ごとに関連データを参照する処理があり、N+1 が起きやすかった
- API 呼び出し前の DB 参照が重くなると、OEM の rate limit 以前に worker 側が詰まる
- 対応として、招待対象のユーザー・所属・権限など、必要な関連データを事前にまとめて取得する形に寄せた
- ループ内で都度 DB を引くのではなく、先に必要なデータを preload / eager load してから job 内で処理した

話し方:

- 「rate limit 対策だけではなく、worker 内の DB アクセスも見ました」
- 「一括招待は人数分ループするので、ループ内に DB クエリが残ると N+1 で worker 側が詰まります」
- 「OEM API の 300 req/sec だけを見ても不十分で、その手前の DB 取得と worker CPU もボトルネックになります」

深掘りされたら:

- 「N+1 は、1人処理するたびに関連データを追加で取りに行く状態です」
- 「100人なら100回、1,000人なら1,000回に増えるので、一括招待では効いてきます」
- 「なので、招待対象を先にまとめて読み、job 内ではメモリ上のデータを使う形にしました」

### 5.4 worker 負荷試験と一括招待上限

- worker の負荷試験を行った
- その結果、1,000 名以上の一括招待は難しいことが分かった
- 原因は OEM の rate limit だけではなかった
- worker 側の CPU が足りず、プロセスが `SIGKILL` / exit 9 で落ちるケースがあった
- つまり、ボトルネックは以下の複合だった
  - OEM API rate limit
  - worker CPU
  - DB 読み込み
  - job の処理時間

判断:

- 「rate limit を守れば無限に流せる」わけではない
- 一括招待の上限を、外部 API 制約だけでなく worker capacity からも見る必要があった
- 大量招待は chunk に分ける、worker を増やす、CPU サイズを上げる、処理をさらに分割する余地があった

話し方:

- 「負荷試験で、1,000名以上を一括で流すと worker CPU が足りず、SIGKILL で落ちることが分かりました」
- 「なので、rate limit 対策だけでは不十分で、worker 自体の処理能力も見ました」
- 「この結果から、一括処理は chunking や worker sizing も含めて設計する必要があると分かりました」

今なら改善:

- 招待対象を 100〜200 名単位の child jobs に分割する
- parent job は進捗管理だけを持つ
- child job は短時間で終わる粒度にする
- worker の CPU / memory を負荷試験結果に合わせて sizing する
- job timeout / kill を前提に、途中から再開できる checkpoint を持つ

### 5.5 Redis rate limiter

- OEM の 300 req/sec を超えないように Redis で制御した
- 全社共通で見る必要があった
- 顧客単位ではなく、OEM API 全体で共有する limiter にした

実装イメージ:

- 秒単位のカウンタを Redis に置く
- OEM API を呼ぶ前に `acquire`
- 取れなければ少し待ってから再試行
- 429 を受けた場合は例外にして Sidekiq retry に乗せる

面接での言い方:

- 「token bucket というより、Redis の秒単位カウンタで上限を見ていました」
- 「重要なのは全社共通の出口 IP なので、会社ごとではなく全体で見る必要があった点です」

### 5.6 今なら: urgent / bulk の rate limit 分離

- 当時の実装は、`urgent` / `bulk` という綺麗な分け方ではなかった
- ただ、今振り返ると、管理者操作と一括処理は分けた方がよい
- urgent と bulk を同じ扱いにすると、bulk が詰まっている時に管理者操作も待たされる
- 今なら class を分ける
  - urgent: 管理者操作
  - bulk: 一括招待など
- bulk 側に上限を持たせ、urgent 用の余地を残す設計にする

話す時の注意:

- 「urgent が必ず即時に通る」とは言わない
- 「bulk が全枠を食い尽くさないようにした」と言う
- 「管理者操作の待ち時間を減らすための設計」と言う

深掘りされたら:

- きれいな言い方:
  - 「全体上限は OEM の 300 req/sec」
  - 「今なら bulk が全枠を使わないように soft cap を置きます」
  - 「urgent は bulk より優先されるようにします」
- さらに深掘りされたら:
  - 「本当に厳密にやるなら、global counter と bulk counter を別に持つ必要があります」
  - 「単純な1カウンタだけだと、説明上の soft cap と実際の挙動がズレます」

### 5.7 組織同期の per-company mutex

- 問題:
  - 同一企業で組織同期が同時に走る可能性があった
  - 管理者ボタンと招待前処理が重なることがある
  - OEM 側の race condition が怖い
  - rate limit 枠も無駄になる

- 対応:
  - 企業単位で mutex を入れた
  - 同じ company の組織同期は同時に1本だけ
  - 後発はブロックする

- DynamoDB を使った理由:
  - conditional write で atomic にロック取得できる
  - 既存 AWS スタックに乗っていた
  - TTL で掃除しやすい

話す時の注意:

- 「TTL で正しさを担保した」とは言わない
- DynamoDB TTL は即時削除ではない
- 正しさは conditional write と期限切れ判定で担保する
- TTL は掃除用と説明する

深掘りされたら:

- 「ロックは `attribute_not_exists` または期限切れなら取得、という形です」
- 「TTL は物理削除のための掃除で、ロックの正しさは condition expression 側で見ます」
- 「組織同期が長くなるなら heartbeat で期限延長する設計にします」

### 5.8 retry と冪等性

- 429 や一時的な失敗は Sidekiq retry に乗せた
- Sidekiq retry は指数バックオフ + jitter がある
- 自前で retry ロジックを増やしすぎないようにした

- 二重作成対策:
  - DB の unique 制約
  - request_id / company_id / user_id などを使った冪等性

話し方:

- 「retry は Sidekiq の仕組みに寄せました」
- 「retry で同じ処理が再実行されても、DB 側の unique 制約で二重作成しないようにしました」

## 6. 当時の割り切り

### 6.1 同時クリック UX

実装したこと:

- 同じ企業で組織同期が実行中なら、後発クリックはブロック
- 「他のユーザーが組織同期を実行中です」のような表示
- 合流 UX までは作っていない

当時の判断:

- まず並列実行を止めることを優先
- UX は最低限
- リリーススコープ内では、進捗共有や合流まではやらなかった

今なら改善:

- 後発クリックをエラーにしない
- 既存ジョブの状態を返す
- 同じ進捗バナーを見せる
- 「自分のクリックで新規起動」か「既存ジョブに合流」かを UI で自然に扱う

話し方:

- 「ここは正直、当時は最低限でした」
- 「今やり直すなら、後発クリックを既存ジョブに合流させます」
- 「サポート問い合わせを減らすなら、その方が良いです」

### 6.2 組織同期ボタンの進捗表示

実装したこと:

- 組織同期は async で job を投げる
- UI はリロードなどで完了確認する形
- 1秒ポーリングやリアルタイム進捗は未実装

当時の判断:

- WebSocket は前例がなく、導入コストが重かった
- ポーリングもリリーススコープ外だった
- まず rate limit と排他を優先した

今なら改善:

- 1〜2秒ポーリング
- もしくは SSE
- 進捗バー
- rate limit 待ちと失敗を UI 上で分ける

話し方:

- 「rate limit class の urgent と、UI を同期にするかは別問題でした」
- 「組織同期は管理者操作なので urgent ですが、処理自体は数秒から数十秒かかり得ます」
- 「今なら async + 進捗ポーリングにします」

### 6.3 監視

実装記憶が曖昧なこと:

- company_id タグ付きの Datadog metrics
- class 別の acquire 待機時間
- 429 率の詳細メトリクス

話す時:

- 記憶が曖昧なら盛らない
- 「見たい指標」と「実際に入れた指標」を分ける

安全な言い方:

- 「実装の記憶が曖昧なので断言は避けますが、見るべき指標は acquire 待機時間、429 率、job 完了時間です」
- 「今なら class 別、company 別に分解して見ます」

## 7. 深掘り Q&A

### Q1. なぜ一括招待を非同期にした？

- 同期リクエスト内で数百 API コールを叩くと UI が待ちすぎる
- OEM 側の rate limit にも当たりやすい
- 失敗時の retry も HTTP リクエスト内では扱いづらい
- Sidekiq job にすることで以下を扱いやすくした
  - retry
  - rate limit 待ち
  - 途中失敗
  - 再実行

### Q2. 当時の2 worker方式と、urgent / bulk 分離はどう違う？

- 当時:
  - worker を2つ用意した
  - lock で片方が詰まっても、もう片方が別の処理を取れるようにした
  - 目的は、全体停止を避けること
- 今なら:
  - 管理者が UI で待つ操作を urgent にする
  - 一括招待など待てる処理を bulk にする
  - bulk が管理者操作を巻き込まないようにする
- まとめ:
  - 当時の方式 = 詰まりを減らす設計
  - urgent / bulk = 大事な操作を守る設計

### Q3. soft cap の考え方は？

- これは **今なら入れる改善案**
- OEM の全体上限は 300 req/sec
- bulk が全枠を使うと、ロール変更などが詰まる
- bulk には上限を置き、urgent の余地を残す
- ただし、urgent の完全即時性を保証するものではない
- あくまで混雑時の優先度制御

### Q4. strict partition ではダメ？

- 例:
  - urgent 100
  - bulk 200
- 問題:
  - bulk が idle でも urgent は 100 までしか使えない
  - OEM の 300 req/sec に余裕があるのに詰まる
- だから、完全分離より soft cap の方が現実的

### Q5. Redis rate limiter の実装で注意することは？

- 1カウンタだけだと表現しきれない場合がある
- global 上限と bulk cap を両方見るなら、少なくとも以下が必要
  - global counter
  - bulk counter
- acquire 時に atomic に両方を見る
- Redis Lua などでまとめて判定する

面接での言い方:

- 「実装の細部は、global と class 別のカウンタを atomic に見る必要があります」
- 「そこを雑にすると、説明上の soft cap と実際の挙動がズレます」

### Q6. DynamoDB mutex の TTL は安全？

- TTL の物理削除は遅れる
- なので TTL 削除そのものには依存しない
- ロック取得時に `expires_at < now` を condition に含める
- TTL は掃除用
- 長い job なら heartbeat で延長する

### Q7. 同時クリック時はどうした？

実装:

- 後発クリックはブロック
- 進行中メッセージを表示

今なら:

- 既存 job の `sync_id` を返す
- 後発も同じ進捗を見せる
- エラーではなく「同期中」に合流させる

### Q8. WebSocket を使わなかった理由は？

- 当時の人事労務で前例がなかった
- 運用・実装コストが増える
- リリース優先だった
- まず rate limit と排他を優先した

今なら:

- WebSocket よりまず 1〜2秒ポーリング
- さらに良くするなら SSE
- 双方向通信が不要なら WebSocket まではいらない

### Q9. 429 を受けたらどうする？

- 429 は一時的な混雑として扱う
- job 内で例外にして Sidekiq retry
- Sidekiq の指数バックオフ + jitter を使う
- 同じ処理が再実行されても冪等性で二重作成を防ぐ

### Q10. 監視するなら何を見る？

- class 別の acquire 待機時間
- 429 率
- job 成功/失敗数
- job 完了時間
- queue latency
- company 単位の偏り

話し方:

- 「実際に全部入っていたかは記憶が曖昧ですが、今ならこのあたりを見ます」

### Q11. N+1 問題は rate limit 対策とどう関係する？

- rate limit は外部 API 側の上限
- N+1 は自社 DB / worker 側の詰まり
- 一括招待では人数分のループがあるので、ループ内 DB クエリが残ると人数に比例して重くなる
- OEM API を呼ぶ前に worker が詰まると、rate limiter 以前の問題になる
- だから、関連データを先にまとめて取得してから job を進めた

話し方:

- 「外部 API の 300 req/sec だけを守っても、worker 側が N+1 で詰まったら意味がありません」
- 「なので、DB アクセスを先にまとめることで、worker 内の処理を軽くしました」

### Q12. 1,000名以上の一括招待が難しかったなら、どう判断した？

- 負荷試験で、1,000名以上を一括処理すると worker CPU が足りないことが分かった
- `SIGKILL` / exit 9 が出て、プロセスが落ちるケースがあった
- これは OEM rate limit だけでは説明できない
- worker CPU、DB 読み込み、job 処理時間も含めた capacity 問題

話し方:

- 「負荷試験で、一括招待の上限は外部 API だけでなく worker capacity にも制約されると分かりました」
- 「なので、今なら 100〜200名単位の child jobs に分け、parent job で進捗を管理します」
- 「worker が kill されても途中から再開できるように checkpoint を持たせます」

## 8. シニア面接で突っ込まれた時

### Q1. 2 worker で本当に十分だった？

答え方:

- 十分と言い切るより、当時の制約下での最小構成として話す
- 当時はまず、1本の処理待ちで全体が止まることを避けたかった
- worker を2つにすることで、片方が lock 待ちでももう片方が進める余地を作った
- ただし、管理者操作の優先保証としては弱い
- 今なら urgent / bulk queue を分ける
- さらに必要なら urgent 専用 worker process を用意する

短く言う:

- 「2 worker は全体停止を避けるための最小構成でした」
- 「管理者操作を強く守る設計としては不十分なので、今なら urgent / bulk に分けます」

### Q2. lock 取得に失敗した job はどうなる？

答え方:

- 同一企業の組織同期が進行中なら、後発は実行しない
- 目的は、同じ company の組織同期を二重に走らせないこと
- 後発リクエストには「進行中」を返す
- バッチ側なら、少し待って retry するか、既存同期完了後に続行する
- ここは実装詳細を曖昧に盛らない

今なら:

- lock 取得失敗時に、既存の `sync_id` を返す
- UI は既存 job の進捗に合流する
- job 側は「既存同期が完了していれば後続の招待に進む」
- 完了待ちを worker sleep で長時間塞がないように、再enqueue / scheduled retry にする

短く言う:

- 「後発は二重実行せず、進行中として扱います」
- 「今なら既存 job に合流させ、worker を長時間 sleep で塞がない形にします」

### Q3. starvation は起きない？

starvation:

- 片方の種類の仕事が、ずっと後回しになって進まない状態

答え方:

- 当時の2 worker方式では、厳密には starvation を完全には防げない
- 特に大量の一括処理が流れ続けると、管理者操作が遅れる可能性はある
- ただし、当時はまず rate limit 超過と同一企業の二重同期を止めることを優先した

今なら:

- urgent / bulk queue を分ける
- urgent 専用 worker または高優先度 queue を用意する
- bulk に soft cap を置く
- queue latency を監視する
- urgent の待ち時間が閾値を超えたら alert を出す

短く言う:

- 「当時の構成では starvation の完全防止まではできていません」
- 「今なら queue 分離、bulk cap、queue latency 監視で管理します」

### Q4. rate limit 待ち中に worker を sleep で塞いでいない？

答え方:

- 長時間 sleep で worker を塞ぐのは良くない
- 短い待機なら許容できる場合もある
- ただし、rate limit 待ちが長くなるなら worker 資源を無駄にする
- 当時の実装詳細は盛らず、「今なら」を明確に話す

今なら:

- acquire できなければ scheduled retry / reenqueue にする
- 例えば「数秒後に再実行」として worker を解放する
- Sidekiq retry だけに任せるのではなく、rate limit 待ち用の再enqueue を設計する
- 429 は指数バックオフ + jitter
- limiter で事前に詰まった場合は短い delayed job

短く言う:

- 「worker を長時間 sleep で塞ぐ設計は避けたいです」
- 「今なら acquire できない時は再enqueue して worker を解放します」

### Q5. 429 が増えた時、誰がどう気づく？

答え方:

- 見るべき指標は明確にある
- 429 率
- rate limiter の acquire 待機時間
- queue latency
- job 失敗率
- job 完了時間
- company 単位の偏り

alert 例:

- 429 率が一定時間で閾値超え
- urgent 相当の処理の queue latency が閾値超え
- job retry 数が急増
- 特定 company の同期失敗が連続

今なら:

- Datadog で class / company / endpoint 別に見る
- PagerDuty 通知
- ダッシュボードで「OEM 側が詰まっているのか」「自社 worker が詰まっているのか」を分けて見られるようにする

短く言う:

- 「429 だけでなく、429 の手前の acquire 待機時間と queue latency を見ます」
- 「外部 API が詰まっているのか、自社 worker が詰まっているのかを分けて判断したいです」

### Q6. ロック TTL が切れた時に二重実行されない？

答え方:

- TTL だけに依存すると危ない
- DynamoDB TTL の物理削除は遅れる
- 逆に TTL が短すぎると、処理中に別 worker がロックを取り直す可能性がある
- 正しさは condition expression と lock owner の確認で担保する

最低限必要:

- `lock_id` を持つ
- release 時に `lock_id` が一致する時だけ削除
- lock 取得時に期限切れ判定を入れる
- job が長いなら heartbeat で期限延長する

今なら:

- 組織同期の最大時間に合わせて TTL を長めにする
- heartbeat で `expires_at` を延長する
- heartbeat が止まった lock だけ期限切れとして再取得する
- 二重実行されても壊れないように、処理自体も冪等にする

短く言う:

- 「TTL は掃除用で、正しさは lock_id と条件付き更新で守ります」
- 「長い処理なら heartbeat を入れます」

### Q7. どこまでが実装で、どこからが改善案？

実装した:

- Sidekiq 非同期化
- 2 worker + lock 取得
- Redis rate limiter
- company 単位 mutex
- retry
- 冪等性
- 一括招待の進捗画面
- 完了メール

改善案:

- urgent / bulk queue 分離
- bulk soft cap
- urgent 専用 worker
- 同時クリックの合流 UX
- 組織同期ボタンのポーリング
- SSE
- 詳細 metrics
- 失敗ユーザーのみ再試行 UI

短く言う:

- 「当時やったのは、事故を止める最小構成です」
- 「今なら、優先度制御と UX と監視を足します」

### Q8. シニアとしての設計判断をどう説明する？

答え方:

- 最初から理想構成を全部入れない
- まず外部 API の制約で事故る箇所を止める
- 次に観測する
- 実測を見て、優先度制御や UX を足す
- チーム規模とリリース期限に合わせて、複雑さを抑える

短く言う:

- 「最初から理想構成にするより、まず事故を止める最小構成を入れます」
- 「その上で metrics を見て、queue 分離や UX 改善を追加します」
- 「このケースでは、当時は 2 worker + lock + rate limiter を優先し、今なら urgent / bulk と合流 UX を足します」

## 9. 面接での話し方テンプレ

### 9.1 最初の回答

- OEM API に 1 IP あたり 300 req/sec の制限があった
- freee 側は全顧客が同じ出口 IP だった
- 一括招待や組織同期で数百 API コールが発生する
- そのため、全社共通の rate limit と worker の詰まり対策が必要だった
- 実装では以下を入れた
  - Sidekiq 非同期化
  - N+1 クエリ解消
  - worker 負荷試験
  - 2 worker + lock 取得
  - Redis rate limiter
  - 組織同期の company 単位 mutex
  - retry と冪等性
- UX は最低限で、今なら urgent / bulk 分離、進捗表示、合流 UX を改善したい

### 9.2 深掘りされた時

- 2 worker にした理由:
  - lock で片方が詰まっても、もう片方が別の処理を取れるようにするため

- 今なら urgent / bulk に分ける理由:
  - 管理者操作が一括処理に巻き込まれないようにするため

- N+1 を解消した理由:
  - 一括招待は人数分ループするので、ループ内 DB クエリが worker CPU と DB の負荷になるため

- 負荷試験で分かったこと:
  - 1,000名以上では worker CPU が不足し、SIGKILL / exit 9 で落ちるケースがあった
  - rate limit だけでなく worker capacity も制約だった

- mutex を入れた理由:
  - 同じ企業の組織同期が同時に走ると、OEM 側の race condition と rate limit の無駄があるため

- DynamoDB を使った理由:
  - conditional write で atomic に取れる
  - AWS スタックに乗っていた
  - TTL で掃除しやすい

- UX の割り切り:
  - 当時は後発クリックをブロック
  - 合流 UX は未実装
  - 今なら進捗 job に合流させる

### 9.3 最後に添える改善案

- 今なら以下を入れる
  - urgent / bulk queue 分離
  - bulk soft cap
  - 既存ジョブへの合流 UX
  - 1〜2秒ポーリングまたは SSE
  - class 別 / company 別の metrics
  - urgent 専用 worker 分離
  - bulk cap の動的調整
