# /kondate コマンド設計書

**Date:** 2026-03-17
**Status:** Approved

---

## 概要

`/kondate` は、冷蔵庫の在庫ベースで作り置きを考慮した複数食分の献立を AI が自動提案し、ユーザー承認後に daily ファイルと Notion meals DB を一括登録するコマンド。

### `/meal` との棲み分け

| コマンド | 用途 |
|---------|------|
| `/kondate` | **計画**: これから食べる献立を提案・登録 |
| `/meal` | **記録**: 食べた食事を記録（未登録 or 差し替え） |

## フロー

### Step 1: 状況把握

```bash
TZ=Asia/Tokyo date "+%Y-%m-%d %H:%M %a"
```

- `aspects/diet/fridge.md` を読んで在庫把握
- 直近の `aspects/diet/daily/YYYY-MM-DD.md` を確認（登録済み食事の有無）
- Notion events DB で今後のイベント（外食・飲み会）を確認し、その食事枠は献立から除外

### Step 2: 食数を確認してレシピ検索

「何食分計画しますか？」と聞く（例: 朝3・昼3・夕3 = 9食など）。

回答を受けたら、以下の優先順位で献立を構成する:

1. **作り置き向き食材を特定** — fridge.md の中で量が多いもの（肉・キャベツ等）を複数食にまたがって使い切る計画を立てる
2. **実在レシピをプラットフォームから検索** — 以下の優先順で使用するプラットフォーム:
   - [クラシル](https://www.kurashiru.com/) — メイン（普段の献立）
   - [白ごはん.com](https://www.sirogohan.com/) — 和食基本
   - [Nadia](https://oceans-nadia.com/) — アレンジ
   - [DELISH KITCHEN](https://delishkitchen.tv/) — 簡単メニュー
3. **NG食材を除外** — トマト、マヨネーズ、ケチャップ、マスタード
4. **材料の完全リストを提示** — レシピの全材料を1人前換算で表示

### Step 2 の提案フォーマット

```
【月 朝食】オートミール + 蜂蜜
  材料: オートミール 40g、蜂蜜 大さじ1
  推定: ~300kcal
  在庫: ✅ 全材料あり

【月 夕食】豚キャベツ蒸し（クラシル）
  材料: 豚こま 200g、キャベツ 1/4個、生姜 1片、
        酒 大さじ1、醤油 大さじ1、ごま油 小さじ1
  推定: ~480kcal
  在庫: ✅ 全材料あり

【火 夕食】豚の生姜焼き + 玄米（白ごはん.com）※作り置き豚こま使用
  材料: 豚こま 150g（作り置き）、玄米パック 1食、
        醤油 大さじ1.5、みりん 大さじ1、酒 大さじ1、砂糖 小さじ1
  推定: ~550kcal
  在庫: ✅ 全材料あり
```

在庫不足の食材がある場合:
```
  在庫: ⚠️ 不足: 長ねぎ（買い出し必要）
```

### Step 3: ユーザー承認 or 修正

- 「OK」「LGTM」→ Step 4 へ
- 「〇〇を変えて」「火曜の夕食なしにして」→ 該当部分のみ差し替えて再提案
- 修正は何度でも対応する

### Step 4: 一括登録

承認後、全食事を以下の手順で登録する:

#### 4a. Notion 重複チェック

登録前に `validate-entry.ts` で重複確認:
```bash
bun run scripts/validate-entry.ts --date YYYY-MM-DD --title "メニュー名" --start HH:MM --end HH:MM
```
終了コード 1 → 類似エントリあり。ユーザーに確認。

#### 4b. Notion meals DB に登録

```bash
# 自炊（レシピ自動生成あり）
bun run scripts/notion-add.ts --db meals --title "メニュー名" --date YYYY-MM-DD --start HH:MM --end HH:MM
```

`notion-add.ts` が内部で `notion-recipe-gen.ts` を呼び出し、レシピを自動生成して Notion ページ本文に書き込む。手動での `notion-update-page` は不要。

食事時間のデフォルト:
- 朝食: 08:00〜09:00
- 昼食: 12:00〜13:00
- 夕食: 19:00〜20:00

#### 4c. daily ファイル作成・更新

`aspects/diet/daily/YYYY-MM-DD.md` を作成または更新。フォーマットは既存ファイルに準拠:

```markdown
# YYYY-MM-DD（曜日）

## 朝食 08:00-09:00
メニュー名
- 材料1
- 材料2
- ~XXX kcal

## 昼食 12:00-13:00
...

**合計: ~XXXX kcal**
```

#### 4d. fridge.md は変更しない

消費時にイベント駆動で減算する既存ルール通り。`/kondate` では触らない。

### Step 5: 報告

```
9食分の献立を登録した。
月: 朝 オートミール(300) / 昼 — / 夕 豚キャベツ蒸し(480)
火: 朝 オートミール(300) / 昼 — / 夕 豚生姜焼き+玄米(550)
水: 朝 オートミール(300) / 昼 — / 夕 豚キャベツ炒め(460)
合計推定: ~2,390kcal/日平均
⚠️ 不足食材: 長ねぎ（火曜夕食用）→ 買い出しリストに追加推奨
```

## 制約・ルール

- **NG食材チェック必須**: `profile/health.md` 参照（トマト・マヨネーズ・ケチャップ・マスタード）
- **在庫ベース優先**: `fridge.md` にある食材から献立を組む（在庫外食材が必要な場合は「⚠️ 不足」表示）
- **タイムゾーン**: Notion 登録時は必ず `+09:00` を付ける（`notion-add.ts` が処理）
- **レシピは実在プラットフォームから**: 自作レシピ禁止
- **材料は完全網羅**: 調味料含め全材料を1人前換算で提示
- **イベント連動**: events DB にある食事時間帯は「外食」として献立から除外

## 既存コマンドとの連携

- `/meal` との重複を避ける: 登録済みの食事枠に `/kondate` で重複登録しない（`validate-entry.ts` で確認）
- `/cleanup` との整合: 過去日の未完了エントリは `/cleanup` で処理
- 買い出し不足食材が出た場合: `/event` や `/calendar` とは連携しないが、報告時に「買い出しリスト追加推奨」と表示する
