## 📊 Groundtruth — Recommendation Microservice（Go版）

### 状況

ユーザーがキャンペーン作成時にセグメントを選ぶ画面があったのですが、選択肢が肥大化して「何を選んでいいかわからない」という状態になっていました。Google Analytics で churn が確認されていたので、ユーザーがセグメントを迷わず選べる UX を提供することになりました。

まず「なぜ選べないのか」を分解しました。ユーザーには3つの状態があると判断しました。「自分のビジネスに合うセグメントがある程度わかっている人」「何があるか探したい人」「そもそも何を軸に選べばいいかわからない人」。それぞれに対応して、Personalized レコメンデーション・Elasticsearch 全文検索・Top 5 Popular の3モードを設計しました。1つの機能でカバーしようとするより、状態ごとに最適な入口を用意した方が churn の根本原因に刺さると判断しました。

### 設計判断

設計時の最初の問いは「既存の Flask モノリスに追加するか、新しいサービスとして切り出すか」でした。判断軸は2つです。まずデプロイ頻度。レコメンデーションは ML モデルの更新に合わせて頻繁にリリースが発生するので、毎回モノリス全体をデプロイすることになります。次に影響範囲。新機能のトラフィックがどれだけ伸びるか読めない状態でモノリスに同居させると、スパイク時に既存機能まで巻き込んでしまいます。切り出すことで、レコメンデーションのトラフィックに応じて独立してスケールアウトでき、モノリス側への影響をゼロにできます。トラフィックが読めないのに影響範囲がコントロールできない構成は怖いと判断して、サービスを切り出す決断をしました。

次に「切り出すとしてどの言語か」。当時社内で Go を全社的に採用していく方針があり、他チームにすでに運用知見が蓄積されていました。新規サービスを立ち上げるタイミングで Go に揃えるのは自然な判断でした。技術的にも、ML と ES の両方から候補を並列取得してマージする設計に対して、Go の goroutine と context でファンアウト・ファンインのパターンが自然に書けて、タイムアウト管理も明示的にできる点が合っていました。

フォールバック設計も重要でした。ML エンドポイントは AI チームが別管理で落ちることが予想された。リクエストのパターンは3つで、入力ありで ML 成功→Personalized、入力ありで ML 障害→Elasticsearch のみ、入力なし または ML/ES 両方障害→Redis にキャッシュした Top 5 Popular。Datadog で ML 障害による ES フォールバック率が 5% 以下であることを SLO として監視しました。

### スコープ管理

プロジェクト途中で PM から「セグメント選択時にリーチ数（何人にリーチできるか）を表示したい」というスコープ追加要求がありました。広告文脈では自然な要求ですが、別途集計パイプラインが必要で Go サービスの外側に新たな依存が増えるため、現スプリントには入らないと判断しました。リリース日を守る方が事業的な優先度が高いと PM に数字で示して合意してもらい、次フェーズに切り出しました。コアの3モード設計を守ったことで、当初スケジュールより2週間早くリリースできました。

### 結果

churn を 20% 減、広告インプレッションを 10% 向上。学んだのは、デプロイ頻度・影響範囲・全社の技術方針を軸に技術選択をすること。同じことが御社のモノリス採択にも作用していると思います。


**核トレードオフ**

- モノリス追加 vs マイクロサービス：デプロイ頻度・影響範囲の独立性の天秤
- Go 採用：全社方針への整合 + goroutine による並列処理の書きやすさ
- フォールバックの段階数：3段階がそのトラフィックで適切、過剰にしない
- スコープ調整：早期リリース vs 機能充実のトレードオフ

**想定ツッコミ**

- **Q: テスト戦略は？** → ML エンドポイントは AI チームから ONNX ファイルをもらってローカルで推論サーバーを立て、本物の推論結果でテストした。フォールバックのシナリオはステータスコード・タイムアウトをモックして検証。ES は LocalStack OpenSearch を使った API テストで実際のクエリ挙動まで確認した
- **Q: goroutine の漏れ対策は？** → context.WithTimeout で ML/ES 周辺にタイムアウト。runtime.NumGoroutine() を Datadog に送信し続けて、増加傾向が見えたらアラート
- **Q: マイクロサービスにしたことで増えた複雑さは？** → ネットワーク遅延が増えた。モノリスなら関数呼び出しで済む処理が HTTP 越しになるので、タイムアウト設計と分散トレーシング（Datadog APM）が必要になった。それでも独立デプロイ・独立スケールのメリットが上回ると判断した
- **Q: churn 20% の計測方法は？** → Google Analytics でキャンペーン作成フローの離脱率をリリース前後で比較。A/B テストはインフラが必要でこの規模感では過剰と判断した
- **Q: Top 5 Popular の Redis キャッシュはどう更新してた？** → バッチで1時間おきに集計して Redis に書き込む設計。Top 5 Popular はフォールバック用途なので多少古くても問題ない。リアルタイム集計（Redis Sorted Set）も検討したが、セグメント選択のたびに書き込みが発生して負荷が増える割に得られるものが小さいと判断した
- **Q: 業種別にしなかったのか？** → まずグローバル Top 5 からシンプルに始めて効果を測ることにした。業種別は改善余地として次フェーズに残した
- **Q: Redis による推論コスト最適化とは？** → ユーザーが何も入力していない状態で ML 推論を走らせないために Top 5 Popular を Redis にキャッシュ。ML 呼び出しゼロで返せるのでコスト削減になった
- **Q: リリース戦略は？** → Feature flag による段階的ロールアウト（10% → 50% → 100%）。新規マイクロサービスの初回本番デプロイは goroutine リーク・メモリ増加・レイテンシスパイクが負荷が増えないと見えないため、blast radius を最小化した。GA で churn の改善を確認しながら拡大。Shadow mode も検討したがフォールバック設計が堅牢なのでオーバーキルと判断した
- **Q: PM のスコープ追加をどう断ったか？** → 「断った」ではなく「次フェーズに移した」。リーチ数プレビューは別途集計パイプラインが必要で現スプリントに入らないことを工数で示し、リリース日を守る方が事業的優先度が高いと合意してもらった
- **Q: ML と ES が両方落ちたらどうなる？** → Redis の Top 5 Popular だけ返す。ユーザーには常に何かを返せるのでキャンペーン作成フローが完全に止まらない設計になっている。エラーにしない判断は、セグメント選択が必須ステップなので空白にするとフロー自体が詰まるため
- **Q: なぜ3モードと決めたのか？PM とどんな議論をした？** → PM が顧客会話から「何を選べばいいかわからない」という声を持ってきた。「なぜ選べないか」をエンジニア側で分解すると、セグメントを減らすと機能が失われるので選びやすくするしかない → どんなユーザーが選べないのかを GA と PM の知見で整理 → ユーザーの状態が3種類に分かれた → 状態ごとに入口を用意する3モード設計に落ち着いた。ユーザーインタビューはしていないが、リリース後に churn 20% 改善という結果が出たので仮説は正しかったと判断した
- **Q: 今やり直したら？** → 同じ構成。ただ REST の代わりに gRPC を検討する（御社も gRPC + SQLBoiler を使っているので話が合いやすい）
