# 勉強 - 2026-04-14

## 🎯 今日の目標・疑問

URL Shortener（Chapter 8）の容量見積もりと設計を理解する（17:10〜18:15）

## 📝 ノート

### URL のバイト数
- システム設計では URL は ASCII 前提 → 1文字 = 1バイト
- 日本語など非 ASCII はパーセントエンコーディングされるが、設計では考慮しない
- `https://example.com/page`（25文字）= 25バイト

### ビット vs バイト
- ストレージ・メモリの見積もり → **バイト**で計算
- ネットワーク帯域幅だけ **ビット**（慣習: 10 Gbps 表記）
- URL Shortener ではネットワーク帯域はボトルネックにならないのでビットはほぼ不要

### GB/s → Gbps 変換が出るシーン
- YouTube / Netflix など大量バイナリデータがネットワークを流れる設計
- URL Shortener では出てこない

### ストレージ見積もりをする理由
- インフラ設計（DBのシャーディング構成など）は後から変えるコストが高い
- 10年分を出すことで「今この設計にすべき理由」の根拠になる
- 面接では「なぜその設計か」を説明できることが重要

### API 設計
- 短縮URL生成: `POST /api/v1/data/shorten` + Body `{ "long_url": "..." }`
  - POST の理由①: DBへの書き込み（リソース作成）
  - POST の理由②: 長いURLがパスに収まりきらない（GET のURL長制限 ~2048文字）
- リダイレクト: `GET /{shortURL}` → 301 or 302 で Location ヘッダーに long URL をセット

### 301 vs 302 の tradeoff
- **301（恒久的）**: ブラウザがキャッシュ → 2回目以降サーバーに来ない → サーバー負荷↓ / クリック集計できない
- **302（一時的）**: キャッシュしない → 毎回サーバーを経由 → クリック数計測できる / サーバー負荷↑
- クリック分析が必要なら 302、負荷軽減優先なら 301

### Go での 301 リダイレクト実装（イメージ）
```go
http.Redirect(w, r, longURL, http.StatusMovedPermanently) // 301
// Location ヘッダーをセット → ブラウザが long URL へ飛ぶ
```

### RDBMSスタート vs 最初からNoSQL
- どちらが正解ではなく「なぜその選択か」を説明できるかが重要
- 段階的移行（RDBMS→シャーディング）: 現実的・コスト意識がある
- 最初からNoSQL: 将来を見越した設計力を示せる

### RDBMS vs NoSQL の選び方
- **ストレージ量より「アクセスパターンとデータ構造」で決める**
- RDBMS: JOIN が必要・トランザクションが必要・スキーマが固定
- NoSQL: キー引きだけ・スキーマが柔軟・最初から大規模確定
- 1〜3年・数GB規模ではどちらでも動く → チームの習熟度・後の移行コストで判断
- URL Shortener は `short → long` のキー引きのみ → NoSQL（DynamoDB等）が素直な選択

### ハッシュ関数の選択（ハッシュ＋衝突判定）
- CRC32: 出力32bit → 衝突しやすいので避ける
- MD5: 128bit → 速度重視でよく使われる（先頭7文字を使用）
- SHA-1: 160bit → やや遅い
- セキュリティ用途ではないので暗号的安全性は不要、速度重視でOK

### 衝突判定のコスト削減：ブルームフィルタ
- 「存在しないことを高速に判定」できるメモリ内データ構造
- 「絶対ない」→ DB問い合わせ不要
- 「あるかも」→ DB問い合わせ（衝突処理）
- 大半のケースでDB問い合わせをスキップできる

### BASE62 のデメリット：分散環境でのユニークID生成
- 複数サーバーで auto increment をそのまま使うと同じIDが衝突する
- 解決策:
  - **Twitter Snowflake**: タイムスタンプ + サーバーID + 連番を組み合わせ
  - **IDジェネレーターサーバー**: 発番専用サーバーで一元管理（単一障害点リスクあり）
  - **レンジ割り当て**: サーバーごとにID範囲を事前に分割
- 連番IDの予測可能性はセキュリティより分散ID生成の担保が本質的な課題
- **最も採用されているのは Twitter Snowflake**（Discord・Instagram・LINEなど）
  - 64bit = 41bit(タイムスタンプ) + 10bit(マシンID) + 12bit(連番)
  - 連番なのでDBインデックスに優しい・調整サーバー不要・1ms に4096個生成可能
  - 弱点: クロックスキュー（サーバー間の時刻ズレ）でID順序が狂う可能性
- UUID v4: 実装簡単だがランダムでDBインデックスが遅くなる・36文字で長い

### キャッシュ（補足知識・本には未記載）
- URL Shortener は読み取りが書き込みの10倍 → キャッシュ効果大
- Redis で short→long のマッピングをキャッシュ
- パレートの法則（80-20則）: アクセス上位20%をキャッシュするだけで負荷の80%を吸収

## 🔑 キーワード

- ASCII前提: システム設計での URL は 1文字 = 1バイト
- パーセントエンコーディング: 非 ASCII を `%XX` 形式に変換
- 帯域幅 (bandwidth): ビット/秒（bps）で表記するのが慣習
- シャーディング: DBを水平分割してスケールさせる手法
- アクセスパターン: どんな操作（キー引き・JOIN・集計）が多いかでDB選定が変わる

## 💡 まとめ

1. **システム設計の前提知識**: URLはASCII前提で1文字=1バイト。ビットはネットワーク帯域のみ使用
2. **容量見積もりの目的**: 10年分を出すことでDB設計（シャーディング要否・RDBMS vs NoSQL）の根拠になる。アクセスパターンとデータ構造でDB選定する
3. **URL短縮の2方式**: ハッシュ＋衝突判定（while loop・ブルームフィルタで最適化）vs BASE62変換（衝突なし・分散ID生成が課題）。BASE62の分散ID生成はTwitter Snowflakeが最も普及
