## 📝 ノート

### コンシステントハッシング

**基本の問題点（仮想ノードなし）**
- サーバー削除時、隣接サーバー1台が全負荷を引き継ぐ
- サーバーのハッシュ位置は運次第 → 偏りが生まれる

**仮想ノードで解決**
- 1台の物理サーバーをリング上に複数（100〜200個）配置
- サーバー削除時、影響がリング全体に分散される
- 実装: `"S0#0"`, `"S0#1"` ... という virtual key をハッシュして複数ポジションを生成

**実装の仕組み**
- `server: str` には IP アドレス（例: "192.168.1.1:8080"）が入る
- ハッシュ関数（MD5 や MurmurHash）が 0〜2^32 に擬似ランダム分布させる
- 等間隔ではない。統計的に均等（大数の法則）
- `sorted_keys`（昇順ソート済み配列）+ `bisect_left` でキールーティングを O(log N) に

**ハッシュ関数の選択**
- MD5: 教科書でよく使われるが実運用では衝突リスクあり
- MurmurHash: 速い・均等・衝突少ない → Cassandra 等で実際に使用
- SHA-256: 安全だが遅い、暗号用途向き

**virtual node vs virtual key**
- virtual node: リング上の「点」（概念）
- virtual key: その点の位置を計算するための入力文字列（実装詳細）

### 仮想ノード数の目安

| virtual nodes 数 | 負荷バラつき（標準偏差） |
|-----------------|-------------------------|
| 1 | ~50% |
| 10 | ~20% |
| 100 | ~10% |
| 150〜200 | ~7%（一般的な設定） |
| 500〜1000 | ~3% |

- Cassandra デフォルト: 256
- メモリコストは小さいので増やしやすい
- 面接では「100〜200 が一般的」と言えば OK
