# Day 1: 復習で言えなかったこと

## 2026-04-10

### システム設計: 可用性の数字（ダウンタイム）
- 99.9%（スリーナイン）: 年間約8.7時間、月43分 → 一般的なSaaS
- 99.99%（フォーナイン）: 年間約52分、月4.3分 → 決済・医療・インフラ系
- Resilireは災害対応システムなので「99.99%を目指す設計」と言うと刺さる

### システム設計: QPS計算でのピーク係数
- 計算した平均QPSに2〜3倍を掛けてピーク時を見積もる
- 例: 平均100 QPS → 「ピーク時300 QPSで設計します」と言う
- これを言うとシステム設計の深さが伝わる

### システム設計: 5ステップの4・5番
- 4. **深掘り**（自分が一番語れる部分から入る）
- 5. **ボトルネック**（弱点と改善案をセットで言う）
- 1〜3は言えた: 要件確認 → スケール感 → 全体設計

### DB: N+1
- **sqlcとGORMの違い**
  - GORM: GoコードでORM経由でクエリを書く。N+1が起きやすい
  - sqlc: SQLを先に書いてGoコードを自動生成。型安全でN+1に気づきやすい
  - Resilireはsqlcを使用（理由: SQLを意識して書くことでパフォーマンス問題を防ぐ）

### Go: Basics
- **mapのキーが存在するか確認する書き方**
  - `m["key"]` だけでは不十分（キーがないときゼロ値が返るため区別できない）
  - 正しくは2値形式: `v, ok := m["key"]`
  - `ok` が `true` なら存在、`false` なら存在しない
  - **注意:** 存在しないキーへのアクセスはコンパイルエラーにならない（ランタイムでゼロ値を返す）。Python/JS と違って例外も投げない
