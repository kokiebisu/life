# レートリミッター

## 🧒 一言で言うと（ファインマン式）
「誰が、どこで、何を守るか」によって設計がまったく変わる。まず要件を絞ることが最重要。

## サーバーサイド vs クライアントサイド

| | サーバーサイド | クライアントサイド |
|--|--|--|
| **守るもの** | サーバーリソース・API | 外部APIの利用制限を守る |
| **誰が設定** | API提供者 | API利用者 |
| **信頼性** | 高い（改ざん不可） | 低い（バイパス可能） |

**クライアントサイドがバイパス可能な理由:**
コードとして動くため、攻撃者はそのコードを無視して直接APIを叩ける。
- モバイルアプリ: リバースエンジニアリングしてエンドポイントを特定 → curl で直叩き
- フロントエンドJS: DevTools でコードを書き換えてスキップ

→ **悪意あるユーザーを止めるのは必ずサーバーサイドの役割。クライアントサイドはあくまで自衛手段。**

## 要件定義で確認すること

### サーバーサイド
- 制限の粒度（ユーザーID単位？IP単位？API key単位？）
- 分散環境か？（→ Redis等で共有カウンターが必要）
- 制限超過時の挙動（即拒否 or キュー？）

### クライアントサイド
- 外部APIのレート制限仕様（429が返ったら何秒待つ？）
- リトライ戦略（Exponential Backoff？）
- 複数サーバーから同じ外部APIを叩くか？（→ 共有カウンターが必要）

## 分散型レートリミッター

複数サーバーがそれぞれローカルでカウントすると合計が把握できない。

```
サーバー1 → 外部API（50req/s）
サーバー2 → 外部API（50req/s）
サーバー3 → 外部API（50req/s）
合計: 150req/s → 制限100req/sを超過
```

→ **Redis などの共有ストアでカウンターを一元管理する**のが定石。
競合状態（Race Condition）を防ぐため、Redis の INCR+TTL や Lua スクリプトによるアトミック操作が必要。

## 制限の粒度：IP vs ユーザーID

**IP単位の問題:**
- 企業・大学・VPNでは、数百〜数千人が同一IPを共有する
- 1人が超過すると他の全員も巻き添えを食う

**ユーザーIDだけでは足りない理由:**
- 未ログインユーザーにはユーザーIDが存在しない
- ログイン画面を叩きまくるブルートフォース攻撃がノーガードになる

→ **実務では「ログイン済み=ユーザーID、未ログイン=IP」の組み合わせが多い**

**IP単位の制限は緩めに設定する（1IPに複数人いる前提）:**
```
ユーザーID単位: 100req/分（厳しくOK）
IP単位:         1000req/分（緩め）
```
完璧な識別は諦めて「明らかに異常なトラフィックだけ弾く」運用が現実的。これは通常許容される。

## アルゴリズム5種

### バケット系（「残量」で管理）

**トークンバケット ★★★**
- トークンが溜まった分だけバーストOK
- 1トークン=1リクエストなら実質回数券と同じ
- 違いが出るのは重いリクエストに複数トークン消費するケース（例: 重い検索=5トークン）
- ユースケース: 一般的なAPI（GitHub・Stripe等）

**リーキーバケット ★**
- キューに入れて一定間隔で処理 → バーストを平滑化
- ユースケース: 決済・メール送信（均一な流量が必要な処理）

### ウィンドウ系（「時間枠内の回数」で管理）

**固定ウィンドウカウンタ ★★★**
- 枠が固定のため境界バースト問題がある
- 例: 59sに100回 + 61sに100回 → 2秒間で200回通ってしまう
- シンプルで実装しやすい
- ユースケース: 管理画面・社内ツール

**スライディングウィンドウログ ★**
- 全タイムスタンプを保存して「直近N秒」を常に正確にカウント
- 精度最高・メモリ大
- ユースケース: 認証・ログイン（セキュリティ重視）

**スライディングウィンドウカウンタ ★★**
- 固定ウィンドウの境界問題を近似で解決
- `前の枠のカウント × 重み + 現在の枠のカウント` で近似
- 精度とメモリ効率のバランスが良い
- ユースケース: 大規模な公開API

### 面接での推奨アプローチ
1. 固定ウィンドウで概要を説明
2. 「境界バースト問題があるのでトークンバケットに変える」と発展させる

## アーキテクチャ

### ミドルウェア vs アプリケーションロジック
**ミドルウェアが主流。**
- 全エンドポイントに1行で適用、漏れがない、関心の分離
- `express-rate-limit` 等のライブラリもミドルウェアとして提供
- 大規模では**APIゲートウェイ**（Kong、AWS API Gateway）でアプリサーバーの手前でブロック

### ルール管理
- ルール（誰を何回まで）は**S3等に設定ファイル（YAML/JSON）で保存**
- ワーカーが起動時・定期的にS3から読み込み → **メモリにキャッシュ**（毎リクエストでS3は遅すぎる）
- Redisはルールではなく**カウンター（何回叩いたか）の保存先**

```
ルール: S3（設定ファイル）→ メモリキャッシュ
カウンター: Redis（共有）
```

### Lua スクリプトによるアトミック操作
RedisはLuaインタープリターを内蔵。スクリプト全体をシングルスレッドで実行するため割り込み不可。
アプリは redis の EVAL コマンドでスクリプトを送るだけ。ネットワーク往復1回で完結。

## 同期問題の解決策

### 1. 集中型データストア（Redis）
全サーバーが1つのRedisを参照。シンプルだがRedisがボトルネック・SPOFになりうる。

### 2. Sticky Session
同じユーザーを常に同じサーバーに振り向けてローカルカウント。Redis不要だがサーバー障害時にカウントがリセットされる致命的な穴がある → 実務では避けられる。

### SPOF対策

| 手法 | 概要 |
|--|--|
| **Redisレプリケーション** | マスターのデータをスレーブにコピー。障害時にスレーブが昇格。読み取り分散は副次メリット |
| **Redis Cluster** | データを複数ノードにシャーディング。1ノード障害でも他は生きている |
| **マネージドサービス** | ElastiCache・GCP Memorystore等。フェイルオーバーをクラウド側が自動管理 → **実務では最多** |

面接では「Redis Clusterでシャーディング＋レプリケーション、実務ではElastiCacheに任せる」の順で答える。

### シャーディングキー
**ユーザーIDが最も一般的。** 均等分散しやすく、カウンターがユーザー単位でまとまる。
Redis Clusterはコンシステントハッシュでシャーディングしているのでノードの増減時の再配置コストが小さい（→ Ch.5）。

**セレブ問題は気にしなくてよい。** カウンターは整数1つ（数バイト）なので有名人でもデータ量は同じ。
セレブ問題が深刻なのはSNSタイムライン配信やキャッシュのホットスポット等の文脈。

## クライアントサイドのベストプラクティス

**キャッシュ ＋ バックオフはセットで使うのが理想。**

```
1. APIを叩く前にキャッシュを確認 → あれば返す（429自体を防ぐ）
2. 429が返ってきたら → staleキャッシュがあれば返す（UX維持）
3. キャッシュなければ → Exponential Backoffで待機後に再試行
```

相性が良い理由:
- キャッシュで429自体を減らせる
- 429になってもキャッシュでユーザー体験を維持
- バックオフ中の無駄な再試行を減らせる

## OSIモデルとレートリミットの関係

```
レイヤ7: アプリケーション層  ← ソフトウェアレベルのレートリミット（HTTP）
レイヤ6: プレゼンテーション層（暗号化・圧縮・文字コード変換）
レイヤ5: セッション層
レイヤ4: トランスポート層    ← ハードウェアレベルのレートリミット（TCP/UDP）
レイヤ3: ネットワーク層      ← ハードウェアレベルのレートリミット（IP）
レイヤ2: データリンク層
レイヤ1: 物理層
```

**ソフトウェアレベル（L7）:** HTTPの内容を見てリクエスト単位で制限。APIゲートウェイ・ミドルウェアが担う。ユーザーID・エンドポイント・ペイロードの内容まで見られる。

**ハードウェアレベル（L3/L4）:** IPアドレス・TCPコネクション数で制限。ファイアウォール・ロードバランサーが担う。L7より高速だがHTTPの中身は見えない。

### L4（トランスポート層）でのレートリミット
TCPコネクション数・UDPパケット数で制限する。

```
例:
- 同一IPからのTCPコネクション数を制限（SYN flood攻撃対策）
- 1秒あたりのUDPパケット数を制限（DDoS対策）
```

担うもの: ファイアウォール（AWS Security Group等）、ロードバランサー

### L3（ネットワーク層）でのレートリミット
IPアドレス単位でパケット数・帯域を制限する。

```
例:
- 特定IPからのパケットを丸ごとブロック
- 帯域制限（1IPあたり100Mbpsまで）
```

担うもの: ルーター、AWS WAF・Shield（DDoS防御）

> プレゼンテーション層（L6）はサーバーのControllerとは別物。データの形式変換・暗号化（TLS）・圧縮（gzip）を担う。
