
## Coinmiles CI/CD 70%短縮

### 面接中に見るカンペ

#### 30秒版

「Coinmiles では、40個程度の Lambda を毎回すべて順番にビルド・デプロイしていて、CI/CD が約2時間かかっていました。そこで GitHub Actions の matrix で Lambda ごとに並列実行し、git diff で変更があった Lambda だけを対象にしました。共通ライブラリ変更時だけは全 Lambda を再デプロイするルールにして、partial deploy の不整合を防ぎました。結果として CI/CD 時間を約70%短縮しました。」

#### 2分版

「Coinmiles では、開発生産性の改善として CI/CD の短縮に取り組みました。当時のバックエンドは 40 個程度の Lambda で構成されていて、main に push されるたびに全 Lambda を順番にビルド・デプロイしていました。そのため、CI が約2時間かかり、緊急修正や複数 PR をまとめてマージしたい時の開発体験が悪くなっていました。

ボトルネックは2つです。1つ目は、変更がない Lambda まで毎回ビルドしていたこと。2つ目は、処理が順次実行だったことです。そこで GitHub Actions の matrix を使い、Lambda ごとに並列実行できるようにしました。さらに git diff ベースで変更があった Lambda だけをビルド・デプロイする形にしました。

ただし、共通ライブラリが変わった場合は注意が必要です。一部の Lambda だけをデプロイすると、Lambda 間で共通ライブラリのバージョン不整合が起きる可能性があります。なので、共通ライブラリ変更時は全 Lambda を対象にするルールを入れました。Bazel や Turborepo のような追加ツールは入れず、3名チームでも保守できる GitHub Actions 標準機能を中心にしました。」

#### 深掘り版

- 判断軸: CI/CD時間、緊急修正のしやすさ、3名チームで保守できる運用コスト
- 検討した選択肢: 全 Lambda 順次実行の継続、matrix 並列化、差分検出、追加ビルドツール導入
- 採用理由: GitHub Actions 標準機能だけで大きな効果が見込め、運用負荷が低い
- トレードオフ: 差分検出ルールを間違えるとデプロイ漏れが起きる
- 一番のリスク: 共通ライブラリ変更時の partial deploy によるバージョン不整合
- 今なら改善: 依存グラフの明示化、対象 Lambda の dry-run 表示、CI 上での deploy 対象レビュー

「Coinmilesでは、CI/CDの改善にも取り組みました。当時のバックエンドは40個程度のLambdaで構成されていて、mainにpushされるたびに全Lambdaを順番にビルド・デプロイしていました。そのため、CIが約2時間かかり、緊急修正や複数PRをまとめてマージしたい時の開発体験がかなり悪くなっていました。

ボトルネックは大きく2つで、変更がないLambdaまで毎回ビルドしていたことと、処理が順次実行だったことです。そこで、GitHub Actionsのmatrixを使ってLambdaごとに並列実行できるようにし、さらにgit diffベースで変更があったLambdaだけをビルド・デプロイする形に変えました。

ただし、共通ライブラリが変わった場合だけは、一部のLambdaだけをデプロイするとバージョン不整合が起きる可能性があります。なので、共通ライブラリ変更時は全Lambdaを対象にするルールを入れました。追加ツールは入れず、GitHub Actionsの標準機能を中心にしたことで、少人数チームでもメンテしやすい構成にできました。結果として、CI/CD時間を70%短縮できました。」

なぜGitHub Actionsの標準機能にしたのかを聞かれたら、3名チームでは運用コストが大事で、BazelやTurborepoのような追加ツールを入れずとも十分効果が出る見込みだったと話す。partial deployのリスクについて聞かれたら、共通ライブラリ変更時は全Lambda再デプロイにすることで不整合を防いだと話す。この経験から、並列化そのものよりも、差分検出ルールの正確さが重要だと学んだ。

## Coinmiles 開発組織の引き継ぎ

### 面接中に見るカンペ

#### 30秒版

「Coinmiles では、参画後しばらくして CTO とリードエンジニアが続けて退職し、残ったエンジニアが3名程度になるタイミングがありました。自分はフロントエンドとバックエンドの両方を触れる立場だったので、バックエンドとインフラのリード役に志願しました。決済、キャッシュバック計算、Lambda のキュー連携など、止まると事業に影響する部分を優先して引き継ぎ、少人数でも運用できる粒度に整理しました。」

#### 2分版

「Coinmiles では、開発組織の引き継ぎも大きな経験でした。参画して6ヶ月ほど経ったタイミングで CTO が退職し、その1ヶ月後にリードエンジニアも退職することになりました。当時は PMF を模索しているフェーズで、残ったエンジニアも3名とインターンという小さい体制でした。

ここで、自分からバックエンドとインフラのリード役に志願しました。自分はフロントエンドとバックエンドの両方を触れる立場だったので、自分が動かないとエンジニアリング機能が止まると判断しました。退職するメンバーからは、決済、キャッシュバック計算、Lambda のキュー連携など、触らないと動かせないコア部分を優先して引き継ぎました。

その後は、過度に分かれていたマイクロサービスを、3名で安全に運用できる粒度へ整理しました。理想のアーキテクチャよりも、当時のチームサイズで止めずに運用できることを優先しました。CEO が事業判断、自分が技術判断を見る形に役割を整理し、既存プロダクトを止めずに新機能開発も続けました。」

#### 深掘り版

- 判断軸: 事業を止めないこと、引き継ぎ優先度、少人数で運用できる構造
- 検討した選択肢: 既存構成をそのまま維持、自分がコア領域を引き受ける、サービス粒度を整理する
- 採用理由: 決済やキャッシュバックなど、止まると事業影響が大きい領域を守る必要があった
- トレードオフ: 理想的な分散アーキテクチャより、チームサイズに合う運用性を優先した
- 一番のリスク: 属人化。自分が抱えすぎると次のボトルネックになる
- 今なら改善: 引き継ぎチェックリスト、運用 runbook、障害対応手順、オンコール観点の明文化

「Coinmilesでは、開発組織の引き継ぎも大きな経験でした。参画して6ヶ月ほど経ったタイミングでCTOが退職し、その1ヶ月後にリードエンジニアも退職することになりました。当時はPMFを模索しているフェーズで、残ったエンジニアも3名とインターンという小さい体制でした。

ここで、自分からバックエンドとインフラのリード役に志願しました。不安はありましたが、フロントエンドとバックエンドの両方を触れる立場だったので、自分が動かないとエンジニアリング機能が止まると判断しました。退職するメンバーからは、決済、キャッシュバック計算、Lambdaのキュー連携など、触らないと動かせないコア部分を優先して引き継ぎました。

その後は、過度に分かれていたマイクロサービスを、3名で安全に運用できる粒度へ整理しました。理想のアーキテクチャよりも、当時のチームサイズで止めずに運用できることを優先した判断です。既存プロダクトの運用を止めずに新機能開発も続け、最終的に1年以上、少人数体制でプロダクト開発を継続できました。」

なぜ志願したのかを聞かれたら、フロントエンドとバックエンドの両方を触れる立場で、自分が動かないとエンジニアリング機能が止まると判断した、と話す。不安について聞かれたら、不安はあったが、CEOが事業判断、自分が技術判断、という役割分担に変えたと話す。何を優先したかを聞かれたら、決済、キャッシュバック計算、キュー連携など、触らないと動かないコア部分を優先したと話す。
