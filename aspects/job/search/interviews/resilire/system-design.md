## 回答フレームワーク（5ステップ）

どんな設計問題にも使える型。これを体に染み込ませる。

```
① 要件確認     2〜3分
② スケール感   1〜2分
③ 全体設計     5〜7分
④ 深掘り       5〜10分
⑤ ボトルネック  2〜3分
```

### ① 要件確認（最重要・絶対飛ばさない）

> 「少し確認させてください」から入る。これをやらないと設計がずれる。

聞くべきこと：
- **機能要件**: 何ができないといけないか
- **非機能要件**: どのくらいのスケール？可用性は？レイテンシは？
- **制約**: 既存システムとの統合は？コストは？

例：「リアルタイム通知システムを設計してください」と言われたら
```
- 通知の種類は？（プッシュ、メール、SMS、Slack？）
- 対象ユーザー数は？（1000社 × 100人 = 10万人？）
- レイテンシ要件は？（災害なら即時、それ以外は数分以内？）
- 同じ通知を複数回送ってはいけない？（冪等性）
- 既存のメール送信基盤（SendGrid等）は使える？
```

### ② スケール感（数字を出す）

頭の中で計算する。正確でなくていい。「オーダー感」が大事。

```
ユーザー数: 10万人
通知頻度: 災害1回/日 × 10万人 = 10万通/日
ピーク: 震度6が発生した瞬間に集中 → 10万通/分も想定

→ 「バッチ処理では間に合わない、非同期キューが必要」という判断に使う
```

### ③ 全体設計（コンポーネントを並べる）

まず大きく書く。詳細は後。

```
Client → API Gateway → Service → Queue → Worker → 通知配信
                          ↓
                         DB
                          ↓
                        Cache
```

コンポーネント間の「なぜ」を言えることが大事：
- なぜキューを使うか → 通知配信はピーク時に集中するから非同期で捌く
- なぜキャッシュを使うか → ユーザー設定の読み取りが多いから

### ④ 深掘り（1〜2コンポーネントを詳しく）

面接官が「特に詳しく聞かせてください」と来たところ、または自分が一番語れるところ。

得意な部分から入る。Kenの場合：
- キャッシュ戦略・フォールバック（Story 1）
- 非同期処理・キュー（SQS経験あり）
- インフラ・デプロイ（Story 4）

### ⑤ ボトルネックと改善案

「この設計の弱点は〇〇です。対策として〇〇が考えられます」で締める。

## Resilire特化 設計問題 3本

### 問題1：災害アラート通知システム（最重要）

Resilireのコアプロダクトに直結。

**お題**：「自然災害発生時に、影響を受けるサプライチェーン施設を持つ企業に即時アラートを送るシステムを設計してください」

**① 要件確認**
```
機能:
- 気象庁のデータをリアルタイムで取得
- 施設の位置情報と被災エリアを照合
- メール・Slack・SMS等で通知

非機能:
- 震度6発生から5分以内に通知
- 同じアラートを二重送信しない（冪等性）
- Resilireのサービス自体が災害で落ちてはいけない（BCP）
```

**③ 全体設計**
```
[気象庁API/XMLフィード]
       ↓ Polling or Webhook
[Crawler Service] → Pub/Sub Topic「disaster-events」
       ↓
[Alert Processor]
  - PostGISで施設と被災エリアを地理的に照合
  - 影響企業・施設を特定
       ↓
[通知キュー] → [Worker] → [SendGrid/Twilio/Slack API]
       ↓
[DB: アラート送信履歴]  ← 冪等性チェックに使う
```

**④ 深掘り：冪等性（二重送信防止）**
```sql
-- alert_logs テーブルで送信済みチェック
INSERT INTO alert_logs (disaster_id, company_id, sent_at)
VALUES (:disaster_id, :company_id, NOW())
ON CONFLICT (disaster_id, company_id) DO NOTHING;
-- → 同じ (disaster_id, company_id) の組み合わせは1回しか送らない
```

**④ 深掘り：BCP（Resilireが最重視）**
```
- 東京・大阪の2リージョンで稼働
- 片方が落ちても通知が止まらない
- Pub/Subはマネージドサービスなので単一障害点にならない
```

**⑤ ボトルネック**
```
- 大規模災害で10万通が同時発生 → Workerをオートスケール
- 気象庁APIのレート制限 → Crawlerをシングルにして内部Pub/Sub経由で配信
```

### 問題2：大量データのインポート/エクスポート

Resilireが実際に設計を議論していた（PDF対応・バッチ事前生成）。

**お題**：「サプライチェーンデータを企業がCSV/Excelでインポートできる機能を設計してください。数万行のデータも想定」

**③ 全体設計**
```
[Client] → アップロードリクエスト
       ↓
[API Server]
  - バリデーション（ファイルサイズ・形式チェック）
  - GCS/S3にファイルを保存
  - ジョブIDを発行してクライアントに返す（非同期）
       ↓
[Pub/Sub / SQS] → [Import Worker]
  - CSVパース
  - バリデーション（行ごと）
  - DBにバルクインサート（チャンク単位）
  - 進捗をDBに保存
       ↓
[Client] ← ポーリング or WebSocketで進捗確認
```

**④ 深掘り：なぜ非同期か**
```
- 数万行のCSVをリクエスト内で処理するとタイムアウト
- 非同期にしてジョブIDで進捗確認させる
- Workerが落ちても再起動でジョブを再開できる（冪等なインポート）
```

**④ 深掘り：バルクインサートのチャンク**
```go
// 1000件ずつ処理してDBへの負荷を分散
const chunkSize = 1000
for i := 0; i < len(rows); i += chunkSize {
    end := min(i+chunkSize, len(rows))
    db.BulkInsert(rows[i:end])
    // 進捗更新
    job.UpdateProgress(i + chunkSize)
}
```

**⑤ ボトルネック**
```
- 同時に大量のインポートジョブが走るとDB負荷が高い
  → Workerの同時実行数を制限（semaphore）
- エラー行の扱い → 全失敗 or 成功行だけコミットかを要件で決める
```

### 問題3：キャッシュ戦略（Resilireが実際に議論）

Resilireは「quadkeyからGeoJSONへの変換コスト」でキャッシュ戦略を議論していた。

**お題**：「地図上に大量のサプライヤー施設をリアルタイムで表示する機能を設計してください」

**① 要件確認**
```
- 施設数: 1社あたり最大1万件、全体では100万件規模
- 更新頻度: 施設情報は月1回程度の更新
- レイテンシ: 地図操作時に1秒以内
```

**③ 全体設計**
```
[Client: 地図ズーム・移動]
       ↓ quadkey（地図タイル座標）を送る
[BFF]
  → Redisキャッシュに quadkey → GeoJSON の変換結果がある？
      Yes → そのまま返す
      No  → Backendに問い合わせ
               ↓
          [Backend: PostGIS]
            ST_Within() で quadkey の範囲内の施設を取得
            GeoJSON に変換
               ↓
          Redisにキャッシュ（TTL: 1時間）
          BFFに返す
```

**④ 深掘り：なぜサーバー側でキャッシュするか**
```
Resilireの記事での議論と同じ結論:
- クライアントで計算: ネットワーク転送量が増える
- リクエストごとに計算: PostGISのクエリが重い（10^3オーダーのJOIN）
- サーバーキャッシュ: 施設情報の更新頻度が低いのでヒット率が高い

判断基準: 「データ更新頻度 × 読み取り頻度 × キャッシュ失効コスト」
→ 更新が稀で読み取りが多い → キャッシュ効果大
```

**⑤ ボトルネック**
```
- 施設情報を更新したとき、古いキャッシュが残る
  → 施設更新時に関連quadkeyのキャッシュを明示的にINVALIDATE
- キャッシュが全部消えた直後（cold start）にDBに負荷集中
  → Cache Warm-up: デプロイ後に主要quadkeyを事前ロード
```

## キーワード集（最低限知っておく）

| キーワード | 一言説明 |
|-----------|---------|
| 非同期処理 | キューを使って処理を分離。タイムアウト防止・スケーラビリティ |
| 冪等性 | 同じ操作を何度やっても結果が変わらない設計 |
| 水平スケール | サーバーを増やして対応。ステートレスな設計が前提 |
| CDN | 静的ファイルをエッジで配信。レイテンシ削減 |
| Read Replica | DBの読み取りを別サーバーに分散 |
| Pub/Sub | 発行者と購読者を疎結合にする非同期メッセージング |
| Rate Limiting | APIの呼び出し頻度を制限 |
| Circuit Breaker | 外部サービスが落ちたとき自動でフォールバック |
| Cache Invalidation | キャッシュを適切なタイミングで無効化する難題 |
| Graceful Shutdown | 実行中のリクエストを完了させてからサーバーを停止 |
