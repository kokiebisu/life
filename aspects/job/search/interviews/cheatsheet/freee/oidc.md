## 📌 freee Eラーニング - 面接かんぺ(セリフ版)

### 🎤 状況設定(30〜40 秒)

「freee Eラーニングは、freee 人事労務上で OEM 提供元の Eラーニング基盤に OIDC SSO で遷移するための機能です。OEM 側に従業員アカウントを作成して、OIDC で SSO 認証することで、freee からシームレスに学習画面に遷移できる設計になっています。

2025 年 7 月から 7 ヶ月、3 名チームでゼロから構築し、2026 年 2 月にリリースしました。本番障害ゼロ、認証成功率 99.9% 以上で運用できています。

一番難しかったのは、freee と OEM の構造的な不一致を、**OEM 側のスキーマや API の大規模な変更も、freee 認証基盤の IdP 改修も使わずに**解いたことです。」

---

### 🎤 制約(45〜60 秒)

「制約からお話しします。

まず freee は **1 ユーザーが複数の事業所に所属できる構造**です。副業やグループ会社の親子関係で日常的に発生するケースで、ユーザーは『今どの事業所のセッションか』を切り替えながら使います。

一方、**OEM 側の OIDC は 1 ユーザー = 1 会社という前提**で設計されていて、ユーザーテーブルにも会社の外部キーが単一カラムで持たれています。freee の 1:N 構造を素直に表現する手段がない状態でした。

さらに、**OEM 自体がリアーキ中**で、スキーマや API 仕様の大きな変更には応じてもらえません。ただし、インデックス追加など軽微な変更や、OIDC 認証の実装合意の範囲内での追加実装は交渉の余地がありました。

freee 側にも制約があります。**認証基盤チームが IdP を所有している**ため、`sub` の値制御やカスタムクレームの追加といった IdP 挙動の変更は、すべて認証基盤チームの開発が必要です。彼らは freee 全体の認証 SLO と標準化方針を担っているので、**1 プロダクトの要件で例外的な処理を入れることは原則受け入れない方針**でした。

加えて、リリース日は固定、PSIRT のレビュー通過がリリース条件だったり、チームはも3 名だったのでメンテナンスがしやすい設計にする必要がありました」

---

### 🎤 設計判断(60 秒)

「採用した設計は、**OEM 側の既存のログイン ID フィールドに `{user_id}_{company_id}` という合成 ID を埋め込む方式**です。

理想は OEM 側に中間テーブルを置いて 1:N を素直に表現することですが、OEM のスキーマ変更ができないので不可能でした。代替案として、freee 側にマッピングテーブルを持ち `sub` に OEM の PK を逆流させる案、カスタムクレームで company_id を渡す案も検討しましたが、どちらも `sub` の値制御や IdP 挙動変更に認証基盤チームの開発が必要で却下されました。

そこで、OEM 既存のログイン ID という任意文字列フィールドに合成 ID を埋め込むことで、**OEM スキーマも freee IdP も触らない**設計に倒しました。

ログイン ID を選んだ理由は 3 つです。任意文字列で制約が緩いこと、認証主体を表す専用フィールドで意味が一致していること、ユニーク制約を担保できる前提だったこと。区切り文字の `_` は、user_id と company_id がどちらも整数なので数字に出てこない・URL セーフ・ログで目立つという基準で選びました。

OIDC の流れとしては、**`sub` には freee 内部の user PK が乗ります**。これは不変な値なので OIDC 標準と整合しています。**`company_id` は userinfo のクレームとして OEM に渡されます**。OEM 側は受け取った email と company_id を、ログイン ID から抽出した company_id と突き合わせて該当ユーザーを特定する仕組みです。」

---

### 🎤 タレマネ駆動の将来切替(30 秒、強い武器)

「現状の lookup は email + company_id ですが、将来的には **user_id + company_id への切替**を計画しています。

理由は、**人事労務には ID とパスワードで認証されている従業員アカウントが大量に存在する**からです。パートや派遣、非正規の方は会社支給のメールを持っていないケースが多くて、HR 側がログイン ID を発番して運用しています。タレマネ研修の本質はこういう層への展開にあるので、email 依存だとそもそも研修を届けたい対象を取りこぼします。

なので、認証基盤側に `user_id` を userinfo claim として追加してもらう交渉を進めています。OEM 側にとっても利用できる freee ユーザーが拡大するメリットがあり、切替修正自体も軽いので、双方メリットの観点で合意済みです。」

---

### 🎤 セキュリティ判断(60 秒)

「セキュリティ面では PSIRT レビューで重要な議論がありました。**ログイン ID は OEM 側で後から書き換え可能なフィールド**なので、攻撃者が他人の合成 ID に書き換えてなりすませるのではないか、という指摘です。

これに対して、なりすましの範囲を **2 段階に整理**しました。

まず **クロス事業所、つまり他企業の社員へのなりすましは構造的に不可能**です。なぜなら userinfo は認証主体(つまり攻撃者自身)の company_id を返すので、他事業所の company_id を詐称する手段がないからです。これは PSIRT の最重要懸念だったので、ここがゼロ容認できることは大きな防御線でした。

次に **同事業所内のなりすまし**ですが、これは OEM 側でログイン ID にユニーク制約を設けてもらうことで塞ぎました。インデックス追加 1 行という軽微な変更で、OIDC 認証の実装合意の範囲内に含めてもらえました。事前に既存データを調査して重複がゼロであることを確認した上で追加してもらいました。

仮にこの制約に塞ぎ漏れがあったとしても、なりすましで閲覧できるのは **従業員名と研修コンテンツ、それから同事業所内の LMS コンテンツに限定**されます。同事業所内に閉じる限り情報漏洩リスクにはならないラインとして PSIRT と合意しました。

~~設計の考え方としては、**OIDC 標準のトークン検証(署名・iss・aud・exp)を信頼の土台**にしつつ、**DB ユニーク制約を追加のセキュリティ境界として再解釈**した、という二段構成です。~~」

---

### 🎤 検知設計(45 秒)

「もう一つ意識したのは、`{user_id}_{company_id}` というフォーマットが**どこにも明文化されていない暗黙の契約**だったことです。これが壊れると認証ができずインシデントに繋がってしまうリスクがあったので多層の検知を入れました。

具体的には 4 層です。OEM ユーザー作成 API のバリデーション失敗を DLQ で受けてログから追える、フォーマット契約自体を rspec で保護する、Design Docにドキュメント化する。

~~加えて、退職や事業所離脱のタイミングで OEM 側の削除 API を同期実行し、夜間バッチで freee 側と OEM 側のユーザーを突合して生合成を毎日担保しています。~~

ただし、リリース時点では能動的な PagerDuty アラートまでは入れていなくて、**リリース後 1 ヶ月は手動レビューでカバーする運用にしました**。その後 PagerDuty 化する計画です。今やり直すならリリース時点から DLQ → PagerDuty まで繋ぎたいです。

結果として、認証成功率 99.9% 以上、本番障害ゼロでリリースできました。」

## 💬 想定 Q&A(セリフ版)

### Q1. なぜ中間テーブル(理想設計)にしなかったのか?

「ご指摘の通り理想は中間テーブルです。1:N の関係を素直に表現できますし、合成 ID を文字列に埋める必要もないクリーンな設計になります。ただ OEM 側のスキーマ変更が必要で、OEM がリアーキ中で大きな変更を受け付けられないという制約があったので、リリース日固定の中では選べませんでした。Design Doc には技術負債として明記していて、**OEM のリアーキ完了後に再評価する計画**です。」

### Q2. `sub` の immutability は満たしてる?

「**`sub` には freee 内部の user PK が入っています**。これは freee 側でユーザー作成時に発番される不変な値なので、OIDC の `sub` immutability 要件と整合しています。合成 ID を埋め込んだのは OEM 側のログイン ID フィールドであって、`sub` ではありません。」

### ~~Q3. freee 側マッピングテーブル + `sub`=OEM PK ではダメだった?~~

~~「それも検討しました。freee 側にマッピングテーブルを持って、OEM が発番した PK を `sub` に逆流させる案です。OEM スキーマには触らずに済むので筋は良かったのですが、**`sub` に何を入れるかを制御するには freee 認証基盤チームの開発が必要**で、これは標準化方針に反するということで却下されました。OEM 側ではなく **freee IdP 側のブロッカー**に当たった形です。」~~

### Q4. カスタムクレームで company_id を渡せばよかったのでは?

「カスタムクレームについては実は段階的な状況です。**`company_id` についてはすでに claim として userinfo に追加されていて、現在の設計でも使っています**。問題は `user_id` の方で、認証基盤側でデータ型をどうするか、freee の内部 user_id にすべきか、login_id を渡すべきか、あるいはダブルミーニング的に柔軟にするかという議論がまだ続いている状態です。

ただ、その議論とは別に、**`sub` を改造して合成 ID を埋め込むような『例外的な処理を IdP に持ち込む』提案については、認証基盤側の標準化方針として明確に却下**されています。」

### Q5. なぜ任意文字列フィールド(ログイン ID)への埋め込みか?

「ログイン ID は元々『認証主体を表す任意文字列』として設計されているフィールドで、**意味的にも合成 ID の用途と一致**していたからです。フォーマット制約も緩く、ユニーク制約も担保できる。表示名やメールアドレスなど、他機能と意味が衝突するフィールドは副作用が読めないので避けました。」

### Q6. 区切り文字に `_` を選んだ理由は?

「user_id と company_id がどちらも整数なので、**数字に出てこない・URL セーフ・ログで目立つ**という 3 点で `_` にしました。`-` も同条件ですが、OEM のログイン ID 制約への適合と、他フィールドでの先例があった点で `_` を選びました。」

### Q7. ユニーク制約で本当にセキュリティ要件を満たすか?

「アプリ層のチェックは抜け道がありえますが、**DB のユニーク制約は書き込み時点で物理的に弾かれる**ので、なりすましの実行手段がなくなります。攻撃者が DB に直接書ける状況はもはや OEM 全体の侵害なので認証フローのスコープ外、という整理で PSIRT に提出し、承認をもらえました。」

### Q8. OEM 側に変更不可と言っていたのにユニーク制約は依頼できたのか?

「これは **OEM とは OIDC 認証の実装自体について合意があって、その実装の中にログイン ID 抽出ロジックや関連する index 追加も含めてもらえた**、という形です。スキーマや API 仕様の大きな変更は受け付けられないという制約でしたが、**認証実装合意の範囲内なら追加実装は可能**でした。

さらに、将来 user_id への切替についても、OEM 側にとっても利用ユーザーが拡大するメリットがあり、切替修正自体も軽いので、**双方メリットの観点で合意**してもらえています。」

### Q9. email lookup でなりすまし可能では?

「**クロス事業所のなりすましは構造的にできません**。userinfo は認証主体自身の company_id を返すので、他事業所の company_id を詐称する手段がないからです。同事業所内のなりすましについては、ログイン ID のユニーク制約で塞いでいますが、PSIRT との議論で『仮に塞ぎ漏れても閲覧範囲は従業員名と研修コンテンツに限定されるので情報漏洩リスクにはならない』という許容ラインを合意済みです。」

### Q10. 既存データに重複はなかった?

「ユニーク制約を追加する前に、**OEM 側で既存データを調査して重複ゼロを確認**した上で index を追加しています。」

### Q11. なぜ email → user_id に切替えたいのか?

「人事労務には **ID とパスワードで認証されている従業員アカウントが大量に存在**します。パートや派遣、非正規の方は会社支給のメールを持っていないケースが多くて、HR 側がログイン ID を発番して運用しています。

**タレマネ研修の本質はこういう層への展開にある**ので、email 依存だとそもそも研修を届けたい対象を取りこぼします。なので、認証基盤側に `user_id` を userinfo claim として追加してもらえれば、メールに依存せずにマッチングできるようになる、という流れです。」

### Q12. `user_id` claim のデータ型、どっちが望ましいと思う?

「私としては **freee の内部 user PK を渡すべき**だと考えています。理由は 3 つです。

まず login_id は freee 側で運用上ユーザーが変更可能な値なので、**SSO の identifier として不適**です。OIDC の `sub` も内部 PK ベースなので、claim 側も同じ層に揃えた方が考え方が一貫します。

次に **ダブルミーニング形式は consumer 側の解釈ロジックが分散するのでバグ温床**になりやすい。

最後に内部 PK を使う方が長期的にシンプルです。認証基盤チームには この方向で提案しています。」

### Q13. ダブルミーニングを認証基盤が許容したい背景は?

「認証基盤側の動機は理解しています。彼らは **freee 全体の認証 SLO** を見ていて、Eラーニング以外にも会計や申告、新規プロダクトなど多数の consumer の要望を吸収する必要があります。固定すると他チームのユースケースを潰すリスクがあるので、柔軟形で逃げたいというのは横断責務の観点では筋が通っています。

ただ私としては、ダブルミーニングは consumer 側でバグ温床になるという懸念があるので、**schema versioning、たとえば claim を `user_id_v1` と `user_id_v2` で分けて、v1 は login_id、v2 は内部 PK、という形でクリーンに分ける**ことを提案しています。決定はまだ議論中です。」

### Q14. silent failure をどう検知する?

「3 層の技術的検知と、1 層の運用カバーで構成しています。

技術的には、OEM の create_user API バリデーション失敗を DLQ で受けてログから追える、OIDC コールバックの parse 失敗を同様に DLQ に逃がす、フォーマット契約自体を rspec の契約テストで保護する、加えて ADR に明文化する、という構成です。

運用面では、**リリース後 1 ヶ月は手動レビューで DLQ をモニタリング**し、その後 PagerDuty 化していく計画でした。今やり直すならリリース時点から PagerDuty まで繋いでおきたいです。」

### Q15. user_id 切替時に OEM 側の修正は発生しないのか?

「OEM 側のマッチング処理を email + extracted company_id から user_id + company_id に切り替える修正が必要になります。ただ切替修正自体は軽い実装なので、**これも OEM と合意済み**です。OEM にとっても利用できる freee ユーザーが email なしユーザーまで広がるメリットがあるので、双方メリットの観点で前向きに進めてもらえています。」

### Q16. ユーザー作成はいつ行うか? OIDC 到達時に OEM ユーザーが存在しないケースは?

「2 パスあります。1 つ目は **管理者が freee 上で OEM との認証情報連携を設定したタイミング**で管理者の OEM ユーザーが作成されます。2 つ目は **管理者が一般従業員を招待する操作の中で**、対象ユーザーの OEM アカウントが作成されます。

OIDC フローへの導線はどちらも OEM ユーザー作成後にしか現れないので、**未作成の状態で OIDC フローが来ることは設計上発生しません**。」

### Q17. 従業員退職時・事業所離脱時の扱いは?

「freee 側で従業員の退職や事業所離脱イベントが発生したタイミングで、**OEM の削除 API を同期実行**してアクセス権限を即時失効させています。加えて、**夜間バッチで freee 側と OEM 側のユーザー一覧を突合**して、孤立ユーザーを検知・削除することで生合成を毎日担保しています。」

### Q18. 1 ユーザーが 3 社・4 社所属の拡張時に破綻するか?

「破綻しません。OEM 側に `{user_id}_{company_id}` の組み合わせ単位で行ができる設計なので、**3 社所属なら 3 行、4 社所属なら 4 行できるだけ**です。アプリ側のロジックも組み合わせを引くだけなので変更不要です。」

### Q19. 技術負債としての扱いは? 将来巻き直す計画は?

「**明確に技術負債として ADR に記載**しています。ただロードマップ上、近い将来に複雑な拡張は計画されていないので、巻き直しコストの方が維持コストより大きい状態が続く見込みです。**OEM 側のリアーキ完了後にスキーマ拡張が解禁されたタイミングで再評価**する想定です。」

### Q20. 今やり直すなら何を変える?

「4 つあります。

1 つ目は **OEM のリアーキ完了後に中間テーブルへ巻き直す**こと。

2 つ目は **合成 ID にバージョン prefix、たとえば `v1_` を最初から入れておく**こと。今のフォーマットだと将来別形式を導入するときに分岐が必要になりますが、最初から `v1_` を付けておけば parser の分岐が自然に書けます。

3 つ目は **DLQ から PagerDuty への能動アラートを最初から入れておく**こと。

4 つ目は、**`user_id` claim のデータ型を最初から freee 内部 PK で固定するよう認証基盤に提案する**こと、これでダブルミーニング議論を回避できます。」
