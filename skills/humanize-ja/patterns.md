# NG パターン詳細

[gonta223/humanizer-ja](https://github.com/gonta223/humanizer-ja) の 20 パターンをベースに、就活ドメイン特化のパターンを追加。原典は MIT License。

## 凡例

- **NG**: AI が好む典型句
- **置き換え方針**: 削るか、何で埋めるか
- **就活向け補足**: 履歴書・職務経歴書・面接文脈での注意

---

## カテゴリ A: humanizer-ja 流用

### A1. 意義の過剰強調

「重要な」「極めて」「計り知れない」などで物事を盛る。

**NG 例:** この取り組みは、業界全体の DX 推進において極めて重要な役割を果たしており、その意義は計り知れない。
**置き換え:** 具体的な数字・事実に置き換える。
**就活向け補足:** 「〜することで〇〇に大きく貢献」のような形が頻出。「〇〇を 70% 短縮」のような数値で代替する。

### A2. 定型評価語

| NG | 置き換え方 |
|---|---|
| 浮き彫りにしており | 具体的に何が見えたか書く |
| 今後の展開が注目されます | 削除するか、自分の予測を書く |
| 多面的な / 包括的な / 画期的な | 何がどう多面的なのか具体化 |
| 注目に値する | なぜ自分が注目したか 1 文で |
| 〜と言えるでしょう | 断定するか、削除する |
| 〜ではないでしょうか | 言い切る |
| 重要な示唆を与えている | 何の示唆か具体的に書く |

### A3. カタカナ語の過剰使用

「ソリューションをレバレッジして、イノベーティブなアプローチで」のような形。**置き換え:** 和語で言えるなら和語。カタカナが 3 語以上連続したら疑う。

**就活向け補足:** 技術用語（`Kubernetes`, `Terraform`, `FastAPI` 等）は固有名詞として使ってよい。一般概念のカタカナ化（「ソリューション」「アプローチ」）は和語化する。

### A4. 曖昧な出典

「業界の専門家によると」「調査結果が示すように」など。**置き換え:** 出典を出すなら具体的に（「Ahrefs が 90 万ページを分析したデータでは」）。出せないなら自分の意見として書く。

### A5. 太字 + コロンの箇条書き

```
- **速度:** 処理速度が 3 倍に向上
- **安全性:** セキュリティ基準を満たす
```

**置き換え:** ラベルを消して内容だけにする。

```
- 処理速度が 3 倍になった
- セキュリティ基準もクリア
```

**就活向け補足:** 職務経歴書ではこの形式が頻出するため最重要警戒。「【見出し】」形式や「〇〇について」前置きでも代替可。

### A6. 三点セットの強制

メリット 3 つ・ステップ 3 つに何でもまとめる。**置き換え:** 本当に 3 つなら 3 つでいい。4 つあるなら 4 つ書く。1 つだけでもいい。

### A7. 全角ダッシュ（——）の乱用

**NG:** Claude Code——Anthropic が開発した CLI ツール——を使えば
**OK:** Claude Code（Anthropic の CLI ツール）を使えば

**就活向け補足:** カッコ or 読点に置き換える。voice 参照（`voice.md`）の `**bold** — 補足` 構造での `—`（em dash）は OK。`——`（ダブル em dash）が NG。

### A8. 見出しの過剰構造化

300 字以下のセクションに小見出しは不要。

### A9. 説教くさい前置き

「ここで重要なのは〜という点です」「〜について理解しておく必要があります」「注意すべき点として」**置き換え:** 削除して内容をそのまま書く。

### A10. 接続詞の多用

「一方で」「しかしながら」「加えて」「このように」「さらに」「とりわけ」が 1 文章に 3 回以上 → 半分削除。

### A11. 否定並列構文

「〜ではない。〜だ。」が 2 回以上 → 片方を別構文に。

**NG:** 単なるツールではない。パラダイムシフトだ。

**就活向け補足:** 「個人の生産性ツールではなくチームの開発基盤として」のような就活頻出形を含む。

### A12. -ing 的付け足し

データの後に AI が勝手に意義を付け足す。

**NG:** 売上は前年比 120% でした。これは同社の戦略が功を奏していることを示しており、今後の成長が期待されます。
**OK:** 売上は前年比 120% でした。

**就活向け補足:** 「〇〇を改善しました。これは〜を示しています」「〇〇を達成し、〜に貢献しました」が頻出。事実だけで止める。

### A13. 主語の過剰明示

**NG:** このツールは高速です。このツールはセキュリティも強固です。
**OK:** 高速で、セキュリティも強固。

### A14. 「〜することができます」

「設定を変更することができます」→「設定を変更できます」→「設定は変えられます」

### A15. 体温のない結論

「以上のことから〜と考えられます」→ 自分の体温を入れる。
**就活向け補足:** ただし履歴書では「思ってます」「感じてます」は NG。「考えています」「目指しています」程度の温度感に留める（媒体別注意 → SKILL.md）。

## カテゴリ B: 不採用 / 調整

humanizer-ja の以下のパターンは履歴書文脈では採用しない:

- **追従的トーン**（「素晴らしいご質問」等） — 履歴書では発生しない
- **敬語の均一化** — 履歴書はむしろ均一な方が読みやすい
- **「正直〜思ってます」系の voice** — 履歴書媒体に不適合（カバーレターでは可）

## カテゴリ C: 就活ドメイン特化（追加）

### C1. 空疎動詞の連発

以下が **同段落で 2 回以上** 出現したら警戒:

「担当」「実現」「貢献」「主導」「推進」「最大化」「横断的に」「一貫して」「組み込み」「取り組み」「整備」

**置き換え:** 具体動作に書き換える。

| NG | OK |
|---|---|
| 〇〇を担当 | 〇〇を設計・実装・QA まで自分が握った |
| 〇〇を実現 | 〇〇を達成（数字を添える） |
| 〇〇に貢献 | 〇〇を XX% 改善 |
| 〇〇を主導 | 〇〇の方向性を決め、チームに渡した |
| 横断的に推進 | 設計・実装・QA・リリース判断まで自分が責任を持った |
| AI を組み込み | Claude Code を設計→実装→レビュー→QA で回した |

### C2. 盛り構文（抽象名詞でドラマ化）

**NG パターン:**
- 「〜という二重の技術的不確実性」
- 「〜という前例のない」
- 「複雑なステークホルダー構造」
- 「〇〇という困難」「〇〇という制約」

**置き換え:** 具体的状況描写に分解する。

**例:**
- ❌ 二重の技術的不確実性
- ✅ コードベースが新規・OEM 連携も初

### C3. チーム規模の盛り

「3 名のチームを率いて」を毎セクションで繰り返さない。**1 つの職歴内で 1 回までが目安**。

### C4. AI 活用節の決まり文句

特に警戒すべき頻出句:

- 「AI を組み込み」
- 「アウトプット最大化」
- 「個人の生産性ツールではなくチームの開発基盤として」
- 「設計・実装・レビュー・QA まで開発プロセス全体に」
- 「少人数チームでのアウトプット量を最大化することにも取り組んできました」

**置き換え:** 具体的に何をしたか（何のフローで・何を回したか・どれくらいの効果が出たか）。

### C5. 数字の単独提示原則

「〇〇を改善」より「〇〇を 70% 短縮」を優先。

- ❌ レスポンスタイムを向上
- ✅ レスポンスタイムを 1 秒以内に短縮（Datadog 計測）

数字の根拠（計測ツール）が出せるとさらに信頼度が上がる。

### C6. 文末「ました」連発

職務経歴書では「〜しました」が並びやすい。3 文連続したら 1 つは体言止め or「〜した」「〜だった」に変える。

**NG（文末連発）:**
- 設計を担当しました。
- 実装を担当しました。
- QA を実施しました。

**OK:**
- 設計から実装、QA までを自分が握る形だった。
- リリース判断も同じく自分が責任を持った。
