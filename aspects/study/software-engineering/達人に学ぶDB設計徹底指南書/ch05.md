# 勉強 - 2026-04-21

## 🎯 今日の目標・疑問

- なぜ正規化をするのか？そのトレードオフは？

## 📝 ノート

### 正規化のメリット・デメリット

- **メリット**: データの整合性の保持
- **デメリット**: 正規化されたテーブル群に対する SQL が非常に遅い（JOIN のコスト）

### 正規化 vs 非正規化の例

正規化済み（JOIN が必要）:
```sql
SELECT 会社.会社名, 社員.社員名
FROM 社員
INNER JOIN 会社 ON 社員.会社コード = 会社.会社コード
```

非正規化であれば結合を使わなくても済む。

### 非正規化のスタンス

**基本的に非正規化は許さない。あくまで最後の手段。**

### どうしても非正規化する場合の例

#### 例1: サマリデータの冗長保持

```sql
-- 正規化のまま（遅い）
SELECT 受注.受注日, COUNT(*) AS 商品数
FROM 受注
INNER JOIN 受注明細 ON 受注.受注ID = 受注明細.受注ID
GROUP BY 受注.受注日
```

→ 部分的にサマリデータ（商品数）をカラムとして事前に持たせる。正規形には違反するが検索は高速化。

#### 例2: 検索条件カラムの冗長保持

```sql
SELECT *
FROM 受注
INNER JOIN 受注明細 ON 受注.受注ID = 受注明細.受注ID
WHERE 受注.受注日 BETWEEN '2024-01-06' AND '2024-01-07'
```

→ 受注日カラムを受注明細テーブルにも持たせる → 受注テーブルとの JOIN が不要に。

### あえての非正規化によるリスク

1. **検索パフォーマンスUP ↔ 更新パフォーマンスDOWN**
2. **データのリアルタイム性の低下**（冗長データの同期遅延）
3. **手戻りが大きい**（後から正規化に戻すのが困難）

### 重要な原則

> リレーショナルDBの**読み込み処理のチューニング方法はたくさんある**（インデックス、キャッシュ、パーティション等）。**書き込み処理のチューニングはあまりない。**
>
> → だから安易に非正規化（=書き込みを複雑にする）に飛びつかず、まず読み込み側のチューニングを検討すべき。

## 🔑 キーワード

- 正規化 / 非正規化

- 読み込みチューニング vs 書き込みチューニング

## 💡 まとめ

1. 正規化の目的はデータ整合性の保持。代償として JOIN によるパフォーマンス低下がある
2. 非正規化は最後の手段。サマリデータや検索条件カラムの冗長保持が典型例
3. 読み込みチューニング手段は豊富なので、非正規化に飛びつく前にそちらを先に検討すべき

## 🔁 復習で詰まったところ

### 2026-04-27 - 🔺 あいまい
- **Q: 非正規化が「最後の手段」とされる理由は？先に検討すべきことは何か？**
  - 詰まった内容: 「先に検討すべきこと」（インデックス・レプリケーション）は答えたが、「なぜ最後の手段か」の3理由が抜けた
  - 正解ポイント: 非正規化が最後の手段である3理由 = ① 検索パフォーマンスUP ↔ **更新パフォーマンスDOWN**（冗長カラム書く分コスト増）② **データのリアルタイム性低下**（同期遅延で不整合リスク）③ **手戻りが大きい**（後から正規化に戻すコストが高い）。先に検討すべきは読み込みチューニング（インデックス・キャッシュ・パーティション・レプリケーション・マテリアライズドビュー）
- **Q: 非正規化の2つの典型例（サマリデータの冗長保持・検索条件カラムの冗長保持）を具体的に説明せよ**
  - 詰まった内容: 検索条件カラムの冗長保持を「わからん」と即答
  - 正解ポイント: ① **サマリデータ冗長保持** = COUNT/SUM などの集計値をカラムに事前格納、GROUP BY 計算を回避 ② **検索条件カラム冗長保持** = 別テーブルの WHERE 用カラム（例: 受注日）を自テーブル（例: 受注明細）にコピー、JOIN を回避。共通する代償は「元の値更新時に冗長コピーも全更新が必要」
