# 面接対応: 広告セグメントレコメンデーション（Groundtruth）

**期間:** 〜2024年6月リリース  
**役割:** バックエンドエンジニア（5名チーム）

---

## 30秒イントロ

広告プラットフォームのセグメント選択ステップで離脱率が全体の 38%を占め、SQL LIKE 検索の P99 が 3,200ms だった。キャンペーン CTR も業界平均（0.71%）を下回る 0.48%。

既存の Flask モノリスへの追加ではなく、FastAPI の独立マイクロサービスとして新規構築した。ML エンドポイントと Elasticsearch の並列実行で P99 1,000ms 以内の目標を達成し、CTR 向上と離脱率削減を達成した。

---

## 核心となる設計判断

### 1. Flask モノリスへの追加ではなく独立マイクロサービスを選んだ

**判断の根拠は定量比較:**

| 観点 | Flask への追加 | 独立マイクロサービス |
|------|--------------|-------------------|
| 並列 I/O | WSGI 同期。Elasticsearch(100ms) + ML(500ms) を逐次 = 600ms+ | asyncio で並列 = max(100ms, 500ms) ≈ 500ms |
| スケーリングコスト | EC2 インスタンス追加で月 $150〜300 | Fargate タスク追加で月 $20〜50 |
| 障害隔離 | ML 障害が Ads Manager 全体に波及 | Ads Manager への影響ゼロ |

並列化だけで P99 を ~400ms 改善できる。レイテンシ目標（1,000ms）の 40% を単一の設計変更で解決できるなら、投資対効果は明確。

**反論への回答:**
「マイクロサービスは運用コストが増える」は正しいが、Fargate により OS パッチ・ノード管理が不要。チームに ECS の運用経験があり、実質的な増加は「デプロイ設定の追加」程度だった。

---

### 2. レイテンシバジェットを事前に分解した

P99 1,000ms の目標をコンポーネントに分解して上限を定義した。

```
経験ユーザー（最も遅いケース）:
  ネットワーク:            30ms
  Redis 確認:              5ms
  RDS Read Replica:       20ms
  asyncio.gather():
    Elasticsearch:        150ms
    ML エンドポイント:     800ms  ← ボトルネック
  レスポンス整形:          20ms
  合計:                1,025ms ← 目標わずかに超過
```

この分解が示したこと: ML タイムアウトを 2,000ms に設定していたが、P99 800ms に変更しないと目標を達成できない。この数値を ML チームとの SLA 交渉の根拠にした。

---

### 3. ML フォールバック戦略（可用性を ML チームの SLA に依存させない）

ML エンドポイントが P99 800ms 超過または障害の場合、Elasticsearch の結果のみを返す（フォールバック）。ML が完全に落ちても Ads Manager の機能に影響しない設計。

フォールバック率 5% を SLO として定義し、Datadog で監視。ML チームのデプロイが影響するたびに検知できるようにした。これにより「ML チームのデプロイに自分たちのサービスが引きずられる」リスクを定量的に管理できた。

---

### 4. Top 5 Popular のスコアリングに CTR を加重した

単純な「使用頻度順」ではなく、「使用頻度 60% + 平均 CTR 正規化値 40%」の複合スコアを採用。

理由: 使用頻度だけだと「昔よく使われたが効果の低いセグメント」が上位に残り続ける。CTR を加重することで「よく使われ、かつ効果的な」セグメントを推薦できる。

**限界と承認:** CTR は広告カテゴリ（CPG vs 不動産）で 5〜10倍の差がある。z-score 正規化でバラつきを補正しているが完全には解決できない。この限界を PM にドキュメントとして共有した上で、V1 では許容することを合意した。

## よく聞かれる質問と回答

**Q: ML チームとどう連携したか？**

OpenAPI spec を事前に合意し、実装の独立性を確保した。ML エンドポイントが「どういうリクエストを受けてどういうレスポンスを返すか」を文書化してから実装を開始したので、ML チームのモデル実装と並行して進められた。また P99 800ms の SLA をレイテンシバジェット分解の結果として提示し、根拠ある交渉ができた。

**Q: 新規ユーザーと経験ユーザーの分岐をなぜ設けたか？**

新規ユーザー（履歴 0〜2件）に ML でパーソナライズを出しても意味がない（学習データがない）。この場合は Redis から Top 5 Popular を返すだけで十分で、ML 呼び出しのコストを払わなくていい。ユーザーの状態によって必要な処理を変えることで、コストとレイテンシを最適化した。

**Q: P99 1,000ms を実際に達成できたか？**

リリース後の実測値は手元に残っていないが、P99 1,000ms 以内という目標は達成できたと認識している。Datadog でモニタリングしていて、SLO アラートが継続的に鳴らなかったのが根拠。ML フォールバック率も 5% 以内で推移していた。

設計上は、経験ユーザーの最悪ケースが 1,025ms で目標をわずかに超えるため、ML タイムアウトを P99 800ms に設定し超過時はフォールバックする構造にした。「P99 1,000ms 以内かつ ML フォールバック率 5% 以内」を両立することで実質達成とした。

**Q: Redis キャッシュの設計で気をつけたことは？**

Redis がダウンした場合のフォールバックとして、月次 cron が Top 5 の結果を S3 にバックアップしている。Redis MISS 時は S3 から読み込む（200ms 程度）。キャッシュ障害がサービス全体の障害にならない設計にした。

## 深掘り質問への準備

### リリース後、想定外のことはありましたか？

> ※ 実際に起きたことに差し替えること。以下はたたき台。

ML エンドポイントのレイテンシが本番で想定より高い時間帯があった。設計時に「P99 800ms」で ML チームと SLA を合意したが、〔実際の状況を記入。例: 特定の時間帯にスパイクがあった、cold start の影響があった等〕。フォールバック率が一時的に SLO の 5% を超えたが、Datadog のアラートで検知して ML チームに連絡した。この経験から「ML フォールバック率の監視を入れておいてよかった」と感じた。

Top 5 Popular のスコアリングについては、リリース後に特定の広告カテゴリのセグメントが上位を占有する問題が出た（CTR の業界差を z-score で補正しきれていなかった）。〔実際に起きたことを記入〕。これは設計時に「将来は業界別 Top 5 への分割を検討」とドキュメントに書いていた問題が早期に顕在化した形。

---

### 今やり直すなら何を変えますか？

1つ目は **Top 5 Popular を業界別にすること**。V1 では全業界共通のスコアを出したが、CPG と不動産では CTR の絶対値が 5〜10倍違う。z-score で補正してもバイアスが残る。最初から業界タグを持ってセグメントを管理し、「同じ業界内での Top 5」を出す設計にすれば精度が上がった。当時は「まず動くものを出す」を優先したが、このバイアスはリリース前に想定できた問題だった。

2つ目は **ML チームとの SLA 合意を先にすること**。レイテンシバジェット分解をして「P99 800ms が必要」と ML チームに交渉したが、この交渉が設計の後半になった。ML チームの実装状況によっては P99 800ms が達成できないケースもあり得た。アーキテクチャ決定の前に ML チームの現状 P99 を確認して、フォールバック戦略を先に固めるべきだった。

---

### 答えられないときの動き

> 「正直なところ、そこまでは検討していませんでした。ただ今考えると、〜という観点が重要になりそうですね。当時は P99 目標の達成を最優先にしていたので、〔何を犠牲にしたか〕については後回しになっていたと思います。」
