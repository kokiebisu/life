## Groundtruth レコメンド基盤

### 30秒版

「Groundtruth では、広告キャンペーン作成時のセグメント選択で離脱が出ていました。選択肢が数千件あり、ユーザーが選び切れない状態です。そこで、Go のマイクロサービスで ML と Elasticsearch を並列に呼び、失敗時は Redis の人気 Top 5 に落とすレコメンド基盤を作りました。狙いは、画面操作の体感を重くせずに『効果的な候補を必ず返す』ことです。」

### 2分版

「Groundtruth の Ads Manager で、広告キャンペーン作成時のセグメント選択がボトルネックになっていました。セグメント数が数千件まで増えていて、Google Analytics で離脱が見えていました。

設計で一番重かった制約は、キャンペーン作成画面の体感を重くしないことです。ML エンドポイントと Elasticsearch はどちらも外部依存で、直列に待つと平均値だけでなく遅延のばらつきや timeout も足し合わされます。既存の Ads Manager は Flask モノリスでしたが、そのまま足すと timeout / fallback の扱いが複雑になります。なので Go の独立サービスにして、ML と ES を並列に呼ぶ設計にしました。

障害前提の設計にもしています。ML と ES が両方成功したらパーソナライズ結果、ML が落ちたら ES のみ、両方落ちたら Redis に持っている業種別 Top 5 を返すようにしました。レコメンドは『完全に最新であること』より『何か返ってユーザーが進めること』の方が重要です。

運用ではレイテンシだけでなくフォールバック率も見るようにしました。速く返っていても ML が落ちて ES だけで返っているなら品質は落ちています。結果として、キャンペーン作成時の churn が約 20% 減、広告インプレッションが約 10% 増になりました。」

### 技術判断

- 判断軸: 画面の体感速度、外部依存、ゼロ件を避ける UX
- 選択肢: Flask モノリスに追加 / FastAPI 独立サービス / Flask + gevent / Go 独立サービス
- 採用: Go 独立サービス
- 理由: 小さな集約サービスに限定すれば、goroutine と context で deadline / fallback を一貫して扱いやすい
- トレードオフ: Go の学習コストはある。FastAPI でも成立しうるため、Go が唯一解とは言わない
- 今なら改善: shadow traffic、cold start 対策、フォールバック率の監視設計

### Q&A

**Q. なぜ並列化が必要だった？**  
A. 100ms を削るためだけではない。ML と ES は互いに依存しない外部I/Oなので、直列にすると遅延のばらつきや timeout まで足し合わされる。並列化すると、片方が遅い/落ちた時にもう片方の結果で返せる。

**Q. なぜ Go？**  
A. Go が唯一解というより、今回の本質はサービス分離、deadline、fallback。Go は goroutine と context timeout が標準で、小さな集約APIでは実装と運用が読みやすいと判断した。

**Q. FastAPI じゃダメだった？**
A. ダメではない。FastAPI + async client でも成立する。ただ、async 対応 client を end-to-end で揃える必要があり、sync client が混ざると event loop を塞ぐ。当時は FastAPI の本番運用知見も薄かったので、小さな独立サービスに限定して Go を選んだ。今なら FastAPI も真面目に比較する。

**Q. ML が落ちたら品質が下がらない？**  
A. 下がる。ただ、ゼロ件でユーザーを止めるより、ES や人気 Top 5 で進められる方が UX 上よい。だからフォールバック率を品質指標として見る。

詳細: [groundtruth/groundtruth-recommendation-deep-dive.md](groundtruth/groundtruth-recommendation-deep-dive.md)

## freee 従業員マスタ統合

### 30秒版

「freee福利厚生では、人事労務の共通従業員マスタと、福利厚生独自DBの統合を担当しました。課題は、人事労務で従業員を登録済みなのに、福利厚生導入時にもう一度 CSV で登録する必要があったことです。ただし福利厚生DBには社宅情報や OEM ログイン ID など固有データがあるため、DB 自体は消せません。そこで、既存ユーザー影響ゼロを優先して段階移行にし、基本情報だけ共通マスタを正にし、福利厚生固有データは既存DBに残す設計にしました。」

### 2分版

「freee福利厚生では、人事労務の共通従業員マスタと、福利厚生の独自DBをどう統合するかを設計しました。事業的には、人事労務から福利厚生へアップセルしたいのに、導入時に従業員をもう一度 CSV 登録する必要があり、体験上の大きな摩擦になっていました。

一方で、福利厚生DBを丸ごと廃止することはできませんでした。社宅情報、OEM ログイン ID、利用履歴のような福利厚生固有データは、共通従業員マスタには乗っていないからです。そこで、統合対象を『基本情報』に絞り、福利厚生固有データは既存DBに残す段階移行にしました。

設計で一番大事にしたのは、書き込み先を一意に決めることです。既存の CSV 起源ユーザーと、新しく共通マスタ起源で作るユーザーを曖昧に混ぜると、どちらのDBが正なのか分からなくなります。なので、入口の UI とデータフローで経路を分け、呼び出し側には master/CSV の分岐を漏らさず、リポジトリ層で取得元を切り替える設計にしました。

既存 CSV ユーザーを master 起源へ切り替える場合も、レコードを作り直しません。既存の福利厚生ユーザーレコードに `master_employee_id` を付けて、基本情報の参照元だけを切り替えます。これにより、福利厚生ユーザーIDや固有データを守ったまま移行できます。」

### 技術判断

- 判断軸: 二度入力の解消、既存ユーザー影響ゼロ、福利厚生固有データの保護
- 選択肢: 一括移行 / 段階移行 / dual-write / 福利厚生DB廃止
- 採用: 段階移行
- 理由: 基本情報だけ共通マスタへ寄せ、固有データは福利厚生DBに残せる
- トレードオフ: CSV 起源と master 起源が共存するため、取得元の分岐を閉じ込める必要がある
- 今なら改善: 移行候補の確認 UX、移行監査ログ、重複検知、出口戦略

### Q&A

**Q. なぜ一括移行にしなかった？**  
A. 既存ユーザーへの影響と戻し作業が重いため。福利厚生固有データもあるので、基本情報だけを段階的に寄せる方が安全だった。

**Q. 整合性はどう守った？**  
A. 書き込み先を入口で一意に決めた。後段でDBをまたいで整合性を取るのではなく、そもそも跨り書き込みが起きにくい構造にした。

**Q. 既存CSVユーザーを master 起源に切り替える時、固有データは？**  
A. レコードを作り直さない。既存レコードに `master_employee_id` を付け、基本情報の参照元だけを切り替える。

詳細: [freee/master-integration-deep-dive.md](freee/master-integration-deep-dive.md)

## freee OIDC 認証フロー統合

### 30秒版

「freee Eラーニングでは、OEM 側との OIDC 認証フロー統合を担当しました。一番難しかったのは、freee は 1 ユーザーが複数事業所に所属できる一方で、OEM 側は 1 ユーザー 1 会社に近い前提だったことです。OEM 側はリアーキ中でスキーマ変更もできなかったので、既存のログイン ID フィールドに `freee_user_id` と `company_id` の組み合わせを入れ、OIDC トークンと照合する設計にしました。PSIRT からのなりすまし懸念には、OEM 側のユニーク制約とトークン照合で説明し、リリースまで持っていきました。」

### 2分版

「freee Eラーニングでは、freee 人事労務から OEM 提供の Eラーニング基盤へ SSO するために、OIDC 認証フローを設計しました。難しかったのは、freee と OEM のユーザーモデルが合っていなかったことです。freee では 1 人のユーザーが複数事業所に所属できますが、OEM 側は 1 ユーザー 1 会社に近い構造でした。

理想だけで言えば、OEM 側に `(freee_user_id, company_id)` を表す中間テーブルや external_id を追加するのが自然です。ただ、OEM 側はリアーキ中で、スキーマ変更や API 変更を依頼できない状況でした。そこで、OEM 側に既に存在していたログイン ID フィールドに、`freee_user_id` と `company_id` の組み合わせを埋め込む設計にしました。

セキュリティ面では、PSIRT から『ログイン ID を書き換えられるなら、他人になりすませるのでは』という懸念が出ました。ここは、OEM 側のユニーク制約で他人が使っているログイン ID には変更できないこと、OIDC トークン内の値とログイン ID から取り出した値を照合することを説明しました。つまり、既存カラムを使いながらも、DB 制約とトークン検証を組み合わせて本人性を担保しました。」

### 技術判断

- 判断軸: freee の 1 user : N companies と OEM の 1 user : 1 company の不一致
- 選択肢: 中間テーブル / external_id / OEM 主キー指定 / 既存ログインIDへの合成ID埋め込み
- 採用: 既存ログインIDへの合成ID埋め込み
- 理由: OEM 側のスキーマ/API を変えずに、事業所コンテキストを OIDC に載せられる
- トレードオフ: 既存ログインIDに意味を持たせるため、検証ロジックと説明責任が増える
- 今なら改善: 外部IDカラムや中間テーブルを OEM 側に正式に持たせる

### Q&A

**Q. 既存ログインIDに意味を持たせるのは危なくない？**  
A. 理想ではない。ただ、OEM 側を変更できない制約があり、既存ユニーク制約と OIDC トークン照合を使うことで安全性を説明した。

**Q. なりすましはどう防いだ？**  
A. 他人が使っているログインIDにはユニーク制約で変更できない。さらに OIDC トークンの値とログインIDから取り出した値を照合する。

**Q. 理想設計なら？**  
A. OEM 側に external_id または中間テーブルを持たせる。既存ログインIDに複合キーの意味を持たせない。

詳細: [freee/oidc-integration-deep-dive.md](freee/oidc-integration-deep-dive.md)

## freee rate limit 対策

### 30秒版

「freee Eラーニングでは、OEM API に 1 IP あたり 300 req/sec の rate limit がありました。freee 側は全顧客のリクエストが同じ出口 IP から出るため、複数社が同時に一括招待すると上限に当たりやすい構造でした。そこで Sidekiq で非同期化し、N+1 解消、worker 分散、Redis rate limiter、同一企業の組織同期 mutex、retry と冪等性を入れました。UX は最低限だったので、今なら進捗表示や既存ジョブへの合流 UX を追加します。」

### 2分版

「freee Eラーニングでは、OEM の Eラーニング基盤と API 連携していました。従業員アカウント作成、招待、削除、ロール変更、組織同期などが対象です。OEM 側には 1 IP あたり 300 req/sec の rate limit があり、freee 側は全顧客のリクエストが同じ出口 IP から出る構造でした。つまり、複数社が同時に一括招待すると、全社で 300 req/sec を奪い合う状態になります。

対応として、重い処理は Sidekiq で非同期化し、N+1 クエリを解消しました。さらに worker を 2 つに分け、負荷試験で一括招待上限を把握しました。API 呼び出し側には Redis rate limiter を入れ、同一企業の組織同期が並列で走らないよう mutex も入れました。リトライは単純に叩き続けるのではなく、冪等性を保った上で失敗時に再実行できるようにしました。

当時の割り切りとして、UX は最低限でした。管理者が待っている操作と、一括招待のようなバッチ処理を完全には分け切れていません。今なら urgent / bulk queue を分け、管理者操作を優先し、一括招待は soft cap をかけます。さらに進捗表示や、後から押した同期ボタンを既存ジョブに合流させる UX を作ります。」

### 技術判断

- 判断軸: OEM API の上限、全顧客で出口IP共有、worker の詰まり、管理者UX
- 選択肢: 同期処理のまま待つ / 非同期化だけする / rate limiter と mutex まで入れる
- 採用: 非同期化 + Redis rate limiter + mutex + retry / 冪等性
- 理由: OEM 側を守りつつ、freee 側の worker も詰まらせない必要があった
- トレードオフ: UX は最低限。進捗表示やジョブ合流 UX は後回し
- 今なら改善: urgent / bulk queue 分離、bulk soft cap、進捗表示、失敗分だけ再試行

### Q&A

**Q. rate limit なら retry だけでよくない？**  
A. retry だけだと相手をさらに叩き続ける危険がある。手前で rate limiter を入れて、送る量自体を制御する必要がある。

**Q. worker が sleep で詰まらない？**  
A. 詰まりやすい。だから今なら urgent / bulk queue を分け、管理者操作を一括処理に巻き込まない構成にする。

**Q. どこが実装済みで、どこが改善案？**  
A. Sidekiq 非同期化、N+1解消、worker分散、Redis rate limiter、mutex、retry/冪等性は実装済み。進捗表示や合流UXは改善案。

詳細: [freee/rate-limit-design-deep-dive.md](freee/rate-limit-design-deep-dive.md)

## freee EKS クラスタ統合

### 30秒版

「freee福利厚生では、独自 EKS クラスタを人事労務側の EKS クラスタへ統合しました。独自クラスタ運用には、EKS バージョンアップ、権限管理、ログ/監視、監査対応、コストの負荷がありました。福利厚生用 namespace をもらい、Terraform、Kubernetes manifest、IAM role、RBAC、ArgoCD、Cron を integration、staging、production の順に段階移行しました。一番注意したのは Cron の二重起動で、メール二重送信などを避けるため、起動時間を集約し、その時間帯を避けて移行しました。」

### 2分版

「freee福利厚生では、独自に持っていた EKS クラスタを、人事労務側の EKS クラスタに統合する移行を担当しました。背景として、福利厚生チーム単独で EKS を持つと、バージョンアップ、権限管理、ログ/監視、監査対応などの運用負荷が継続的に発生していました。プロダクト開発に集中するには、人事労務側の運用基盤に寄せる方が合理的でした。

自分は、福利厚生用の namespace を提供してもらう調整をした上で、Terraform、Kubernetes manifest、IAM role、RBAC、ArgoCD、Cron などを integration、staging、production の順に段階移行しました。一気に本番だけを切り替えるのではなく、環境ごとに差分を確認しながら進めたのがポイントです。

一番注意したのは Cron の二重起動です。旧クラスタと新クラスタの両方で同じ Cron が動くと、メール二重送信や外部連携の重複実行につながります。そこで Cron の起動時間を集約し、その時間帯を避けて切り替えました。必要に応じて CS 経由で事前通知も行い、技術対応と運用対応の両方でリスクを下げました。」

### 技術判断

- 判断軸: 独自クラスタ運用コスト、監査/権限管理の統一、チーム規模に合う運用責任
- 選択肢: 独自EKS継続 / 人事労務クラスタへの namespace 統合 / 大きな基盤再設計
- 採用: 人事労務クラスタへの namespace 統合
- 理由: 小さなチームで運用責任を持ち続けるより、既存基盤に寄せる方が集中できる
- トレードオフ: 統合先クラスタの影響範囲は広がるため、namespace、RBAC、移行手順で境界を作る
- 今なら改善: CronJob suspend、DB lock、idempotency key、rollback runbook

### Q&A

**Q. 統合すると blast radius が広がらない？**  
A. 広がる。だから namespace、RBAC、権限、移行手順で境界を作る。独自クラスタ運用を続けるコストとの比較で判断した。

**Q. 一番危なかった点は？**  
A. Cron 二重起動。メール二重送信や外部連携の重複実行につながるため、起動時間を集約し、その時間帯を避けた。

**Q. 今ならどう改善する？**  
A. CronJob suspend、DB lock、idempotency key、rollback runbook をより明示的に用意する。

## Coinmiles CI/CD 70%短縮

### 30秒版

「Coinmiles では、40個程度の Lambda を毎回すべて順番にビルド・デプロイしていて、CI/CD が約2時間かかっていました。そこで GitHub Actions の matrix で Lambda ごとに並列実行し、git diff で変更があった Lambda だけを対象にしました。共通ライブラリ変更時だけは全 Lambda を再デプロイするルールにして、partial deploy の不整合を防ぎました。結果として CI/CD 時間を約70%短縮しました。」

### 2分版

「Coinmiles では、開発生産性の改善として CI/CD の短縮に取り組みました。当時のバックエンドは 40 個程度の Lambda で構成されていて、main に push されるたびに全 Lambda を順番にビルド・デプロイしていました。そのため、CI が約2時間かかり、緊急修正や複数 PR をまとめてマージしたい時の開発体験が悪くなっていました。

ボトルネックは2つです。1つ目は、変更がない Lambda まで毎回ビルドしていたこと。2つ目は、処理が順次実行だったことです。そこで GitHub Actions の matrix を使い、Lambda ごとに並列実行できるようにしました。さらに git diff ベースで変更があった Lambda だけをビルド・デプロイする形にしました。

ただし、共通ライブラリが変わった場合は注意が必要です。一部の Lambda だけをデプロイすると、Lambda 間で共通ライブラリのバージョン不整合が起きる可能性があります。なので、共通ライブラリ変更時は全 Lambda を対象にするルールを入れました。Bazel や Turborepo のような追加ツールは入れず、3名チームでも保守できる GitHub Actions 標準機能を中心にしました。」

### 技術判断

- 判断軸: CI/CD時間、緊急修正のしやすさ、3名チームで保守できる運用コスト
- 選択肢: 全 Lambda 順次実行 / matrix 並列化 / 差分検出 / 追加ビルドツール導入
- 採用: matrix 並列化 + git diff による対象絞り込み
- 理由: GitHub Actions 標準機能だけで大きな効果が見込め、運用負荷が低い
- トレードオフ: 差分検出ルールを間違えるとデプロイ漏れが起きる
- 今なら改善: 依存グラフの明示化、対象 Lambda の dry-run 表示

### Q&A

**Q. なぜ Bazel や Turborepo を入れなかった？**  
A. 3名チームでは運用コストが重い。GitHub Actions の標準機能で十分な効果が見込めたため。

**Q. partial deploy のリスクは？**  
A. 共通ライブラリ変更時に一部 Lambda だけを出すと不整合が起きる。だから共通ライブラリ変更時は全 Lambda を対象にした。

**Q. 一番重要だった判断は？**  
A. 並列化そのものより、差分検出ルールの正確さ。どれを出すべきかを間違えないこと。

- 理由: 決済やキャッシュバックなど、止まると事業影響が大きい領域を守る必要があった
- トレードオフ: 理想的な分散アーキテクチャより、チームサイズに合う運用性を優先した
