### ☆ Freee Eラーニング プロダクトリード立ち上げ @ Freee（0→1 包括）

**Resilire 向け冷頭フック**

> 「カジュアル面談で『PRD が書けるエンジニアに来てほしい』と伴っていました。私が freee で Eラーニングをゼロイチで立ち上げた経験を、プロダクトリードとしての判断を中心にお話しします」
> 

**話すストーリー本文（3-4分）**

「freee 人事労務上で Eラーニング SaaS をゼロから立ち上げるプロジェクトで、プロダクトリードを担当しました。24卒・25卒を含むエンジニア 3名、自分は中途入社のリード、期間は 7 ヶ月、OEM 提供元が上場企業でリリース日が契約固定、という制約の中で2026年 2 月に 1,000コース・6,000本動画の受講管理 SaaS をリリースしました。

4つの層で動きました。

1つ目は **技術選定**。人事労務のモジュールとして統合するか、standalone として展開するかを比較検討しました。セキュリティ要件・開発速度・運用コストで総合評価してモジュール統合を選び、Design Doc で基盤チームと合意を取りました。OIDC 認証では freee 1:n と OEM 1:1 の構造的不一致を、OEM のユーザー作成 API に `{user_id}_{company_id}` を login_id として渡し、認証時は userinfo の email + company_id で突合する設計で解決し、PSIRT 承認を得ました。

2つ目は **プロダクト設計**。PdM と協働で PRD を作成し、特に組織同期機能では従業員データの状態遷移（新規・更新・削除・権限変更）を網羅的に洗い出して過渡期の UX を設計しました。

3つ目は **チーム運営**。初期は「大きい機能を分散して手分け」で動きましたが、インクリメントが届かない状態になり、「小さい単位を一緒に届ける」へ方針転換。ペアプロでジュニアのドメイン理解を加速。中盤で PdM↔Eng、freee↔OEM、自分↔メンバーの 3 層で信頼が崩れた局面があり、最初は「個人の努力」で解決しようとして失敗。意思決定権限の明文化・PdM 定例へのエンジニア代表枠・OEM 制約の可視化・メンバーへ設計レビューのオーナーシップを任せる、という構造的な打ち手で立て直しました。

4つ目は **ステークホルダー調整**。OEM 先とのコミュニケーション、freee PSIRT の 3 者調整、契約上の機密情報は「なぜ」だけ要約してエンジニアに翻訳する運用にしました。

結果、3名で7ヶ月のスコープを完遂。ジュニア 2名も自走するチームに成長しました。一番の学びは、**「問題が人に見えるときはまず構造を疑う」** ことと、**「リーダーの仕事は自分がいなくても回る仕組みを作ること」**。御社の『仕組みで守る』文化と同じ結論に、自分も失敗から辿り着いた経験でした」

**核となる判断ポイント**

- **モジュール統合 vs standalone** → モジュール統合（セキュリティ + 開発速度）
- **分散並行 vs ペアプロ小単位インクリメント** → 後者（総合速度で勝つ）
- **個人の努力 vs 構造的打ち手** → 後者（スケールするリーダーシップ）
- **全部拾う vs 捨てる判断** → 中盤で「何を捨てるか」をステークホルダーと再合意

**想定ツッコミ**

- **Q: プロダクトリードと EM の違いは？** → EM はチーム運営にフォーカス。プロダクトリードは What/Why から How まで横断。PdM と協働で PRD を書き、技術選定を主導し、リリース判断までコミット
- **Q: 3名で完遂できた要因は？** → ①技術選定（モジュール統合）で複雑性を削った、②スクラム再設計で「小さいインクリメント」に転換、③構造的打ち手で信頼関係を立て直した
- **Q: 一番の失敗は？** → 前半、自分が個人の努力で全てを解決しようとしてチーム停滞を招いた。構造の問題に気づいて打ち手を切り替えた
- **Q: 次に同じ状況なら何を変える？** → 「引き受ける」判断と「何を捨てるか」の合意をセットで初日にやる。構造的打ち手を試す前に上長エスカレーションに行かない

### ☆ Story 1′: Recommendation Microservice（Go版）@ Groundtruth

**なぜ Resilire に刺さるか**

Go + goroutine 並列処理・マイクロサービス vs モノリスの判断軸・障害波及防止の3段階フォールバック、Datadog SLO 監視。御社の SCRM 3.0 で「マイクロサービス vs モジュラモノリスを議論してモノリス採択」という同系の判断をしているので、**判断軸で会話ができる**。

**Resilire 向け冷頭フック**

> 「御社が SCRM 3.0 でマイクロサービス vs モジュラモノリスを議論して、チーム規模と試行錯誤フェーズを理由にモノリス採択された記事を読みました。私も Groundtruth でレコメンデーション機能を作るとき、逆の判断をしたことがあります。そののときの判断軸をお話しさせてください」
> 

**話すストーリー本文（3-4分）**

「ユーザーがキャンペーン作成時にセグメントを選ぶ画面があったのですが、選択肢が肥大化して「何を選んでいいかわからない」という状態になっていました。Google Analytics で churn が確認されていたので、レコメンデーション機能を追加することになりました。

設計時に選択肢が 3つありました。パターン A: 既存の Ads Manager モノリスに追加。パターン B: 既存言語（Flask/Python）で独立サービス、C: 別言語（Go）で独立サービス。

判断軸は 3 つでした。最上位は P99 1,000ms。ノンブロッキングでは ML エンドポイント（最大 800ms）+ Elasticsearch 検索が直列で走るので、SLO 超過する。並列化が必須だと分かった時点でパターン A は消えました。次に B と C で迷いました。Flask は WSGI 同期で asyncio.gather() が使えない。gevent で無理矢理退けることもできたが、モンキーパッチの副作用・社内に運用実績がないことを懸念して Go にしました。goroutine で ML と ES を並行呼び出しして、レスポンスを asyncio.gather() 相当で集約する設計です。

もう 1 つ重要な判断が、フォールバック設計でした。MLエンドポイントは AI チームが別管理で、デプロイ影響で落ちることが予想されました。3 段階のフォールバックを組みました。ML 成功→Personalized 、ML 障害→Elasticsearch のみ、ユーザー未入力→Redis にキャッシュした Top 5 Popular。Datadog でフォールバック率 5% を SLO として監視し、AI チームのデプロイ影響を定量的に追えるようにしました。

結果、キャンペーン作成時の churn を 20% 減、広告インプレッションを 10% 向上させられました。実踐を通して学んだのは、経体的 SLO とチームの運用能力を軸に技術選択をすること。同じことが御社のモノリス採択にも作用していると思います」

**核トレードオフ**

- モノリス追加 vs マイクロサービス：SLO 達成と独立性の天秤
- Flask/gevent vs Go：チームの運用経験の有無で判断
- フォールバックの段階数：3 段階がそのユーザー数・トラフィックで適切、過剰にしない

**想定ツッコミ**

- **Q: goroutine の漏れ対策は？** → context.WithTimeout で ML/ES 周辺にタイムアウト。起動後も runtime.NumGoroutine() を Datadog に送信し続けて、増加傾向が見えたらアラート
- **Q: P99 1,000ms の予算配分は？** → ML 800ms、ES 400ms と仮定し、並列で最大エッジの ML 800ms + 並列集約コスト 50ms + 余裕 150ms の内訳
- **Q: 今やり直したら？** → 同じ構成。ただ Pub/Sub 代わりに gRPC を検討する（御社も gRPC + SQLBoiler を使っているので、そういう話にもなりそう）

### ☆ Story 5: AI 駆動開発の導入 @ Freee Eラーニング（補助エピソード）

**メインストーリーの補足位置**。プロダクトリード立ち上げの総ぶりストーリーの中で、「少人数でのアウトプット最大化の仕組み」の一部として AI 駆動開発を触れる。御社の Claude Code + MCP (Figma/Asana/Playwright/O3) と同じツールセットなので、尚さらに話が扱いやすい。詳細が問われたら以下へ展開。

**話すストーリー本文（3分）**

「Eラーニングは 3 名チーム（24卒・25卒含む）で 7 ヶ月でSaaS をゼロから立ち上げる必要がありました。人員増員は間に合わない、AI を使うしかない。ただし「個人の工夫」ではジュニアのコードがバラバラになる、なので**チーム標準化**が必要でした。

ツール選定で Claude Code + MCP を選んだ理由は 3 つあります。Copilot だけだと設計壁打ちやシェル操作、並列エージェント運用ができない。Cursor 統一は IDE 変更のコストが重い。Claude Code なら CLI で CI 統合もでき、MCP で Figma/Slack/GitHub などを経由したコンテキスト収集が自動化できる。これが決め手でした。

標準化の仕組みとして、`.claude` skills/hooks をリポ共通設定で配った。「PR を single responsibility で作る」 command を共有して、ジュニアの PR 粒度が上がるようにした。もう一つ大事にしたのは **AI 過依存防止**です。テストでは「ロジック・骨格は人間が書く、肉付けは AI に壁打ち」スタイルをリコメンドし、根拠を持った実装が绰めないようにしました。CI には Claude GitHub Actions + CodeRabbit を入れて、AI 生成コードも必ずレビューが入る構造にしました。

結果、少人数・短期間で SaaS をリリースできました。もう一つ大事な学びは、**AI を「個人の工夫」でやるのではなく、仕組みに入れてチームで使うこと**。御社も「仕組みで守る」文化だと読んだので、そこに貢献できると思っています」

**核トレードオフ**

- 人員増員 vs AI 活用：期限と採用リードタイムが決め手
- 個人裁量 vs チーム標準化：再現性・監査性が必要なら後者
- Claude Code vs Copilot vs Cursor：設計壁打ち・並列運用・CI 統合の 3 軸で評価

**想定ツッコミ**

- **Q: AI 過依存の具体例は？** → ジュニアが AI 生成コードを「読まずにマージ」しかけたケースがあり、PR 粒度を小さくする command を作って解決。個人指導ではなく仕組みで防く
- **Q: MCP でよく使うサーバーは？** → Storybook/Figma/GitHub/Slack/Confluence/Jira/社内脆弱性診断ツール。御社の Playwright MCP / O3 MCP も付けたい
- **Q: コストは？** → Claude 利用料 + Claude GitHub Actions 実行時間の合計。ユーザーあたり月 [要確認] 程度、人材採用より桁違いに安い

### ☆ Story 7: 「人を変えるのではなく構造を変える」を学んだ失敗 @ Freee Eラーニング

**なぜ Resilire に刺さるか**

御社の「仕組みで守る」文化に**直結**するストーリー。「個人の努力ではなく構造で解決する」という決断は、シニア・リード候補に必要なリーダーシップの粒度。失敗談として語ることで成熟度も示せる。

**Resilire 向け冷頭フック**

> 「御社の zenn を読んでいて『個人の努力ではなく仕組みで守る』という表現に強く共感しました。私も同じ結論に至った失敗があり、その経験をお話しさせてください」
> 

**話すストーリー本文（4-5分）**

「Eラーニングは技術的にも組織的にもナイトメアモードでした。二重の技術的未知領域、上場企業 OEM の厳格な口調・固定スケジュール、そして 24卒・25卒のチーム。自分は中途入社。

3 層の信頼が同時に崩れました。PdM ↔ エンジニアの信頼、freee ↔ OEM の信頼、そして自分 ↔ メンバーの権限認識。メンバーが「上長が指示したのであれば問題ないです」と言ったときは、正直ショックでした。

最初は自分が「もっと頽張れば」で解決しようとしました。議論で折れない、PdM の意図をエンジニアに翻訳する、タスクを巻き取る、上長に 1on1on1 を設定する。あらゆる「個人の努力」で。

しかし悪循環に陥りました。余裕がない→熟考できない→説明が弱い→メンバーが納得しない→議論が長引く→さらに余裕がない。「自分の努力量」と「チームの状態」が逆相関していることに気づいたとき、**これは構造の問題だ**と認識が変わりました。

打ち手を 5 つ変えました。第一に、意思決定権限の明文化。技術選定は Eng が決める、スコープは PdM が決める、両方に関わるものは weekly sync 。口頭合意を排除しました。第二に PdM 定例へのエンジニア代表枠を作り、「なぜこの仕様なのか」の一次情報が底にまで流れる構造にしました。第三に OEM 制約の可視化。第四に、メンバーの不満を聞くのではなく「何を担いたいか」を聞いて設計レビューのオーナーシップを任せ、**権限で応えた**。第五に 1on1on1 を「助けてください」から「これを試した、次はこうする、見落としがないか」の壁打ちにシフトしました。

結果、チームの動き方が変わり、2026 年 2 月のリリースを達成しました。学んだことは 2 つあります。一つは「問題が人に見えるとき、まず構造を疑う」。もう一つは「リーダーの仕事は、自分がいなくても回る仕組みを作ること」。これが御社の「仕組みで守る」文化と同じ結論になった点だと認識しています」

**核トレードオフ**

- 個人の努力 vs 仕組み設計：後者だけがスケールするリーダーシップ
- 上長に頃合いでエスカレーション vs 構造的打ち手をやってから壁打ちに使う：順序が大事
- メンバーの信頼を「得る」 vs 「参加の仕組みで返す」：後者の方が直接的

**想定ツッコミ**

- **Q: もし従来の個人の努力路線で残り泳ぎほどの期間だったら？** → それでも構造に切り替える。一時的に遅くなっても、個人依存の仕組みは属人性が富み、将来の交代応勝で負は確実に大きい
- **Q: 他部署との対立でこの透明化・権限明文化アプローチは効かなかったケースは？** → あった。意思決定の外側に政治的な力学が強く履く組織だと構造明文化だけでは弱い。その場合は上長や VP を巻き込んで仕組みを上位に包み込む必要があった
- **Q: 御社で同じことをやるなら何から始める？** → ADR 文化があるなら、まず「今の権限分担」を ADR に起こして、曖昧な部分を 1 つずつ決めていく。ズレが見えた点をここから修正する

## ❓ 典型 Q&A（フルスクリプト）

### Q: 自己紹介を（2分版）

「沖胡献一と申します。およそ 5 年間、フルスタックエンジニアとしてスタートアップから中規模 SaaS まで経験してきました。直近 1 年半は freee で **Ruby (Rails) をメインに**、福利厚生の開発と freee Eラーニングのプロダクトリードを担ってきました。

### E-Learning — OIDC 認証設計

**Q: `{user_id}_{company_id}` 設計で将来 user_id が変わったら？**
→ freee の user_id は内部連番（変わらない）。OEM 側のログインIDフィールドは varchar(256) で余裕があり、正規表現バリデーションを PSIRT と合意済み。変わる場合は migration スクリプト + OEM 側ログイン URL リダイレクトで対応する設計を Design Doc に明記した。

**Q: freee 1:n（1ユーザーが複数会社）と OEM 1:1（1ユーザー1会社）の不一致は具体的にどう解いたか？**
→ OEM のユーザー作成 API に `{user_id}_{company_id}` を login_id として渡して provisioning。認証時は OIDC 後に userinfo の email + company_id で突合する設計。sub は freee 側で暗号化済みで復号アルゴリズムを OEM に渡せないため使えない。OEM 側の DB スキーマは触らない。

振り返ると突合キーに email を使ったのは改善余地がある。2点理由がある。①email はドメイン移行・改姓で変わりうる（org sync で追従しているが依存が生まれる）。②ターゲット層の派遣社員・バイトは freee に email を登録していないケースが多く、email がない従業員はそもそも突合できない。`freee_user_id` を userinfo のカスタムクレームに追加すれば両方解決できた。ただし認証基盤チームへの依頼が必要な内容だったため、プロジェクト序盤で優先タスクとして早めに打診すべきだったというのが反省点。

**Q: 組織同期の状態遷移、どのパターンを洗い出したか？**
→ 新規追加 / 属性更新（名前・役職・メール）/ 会社間異動 / 休職 / 退職（論理削除）/ 権限変更（管理者昇格/降格）の 6 パターン。過渡期の UX（同期中に別タブでアクセスしたら？）は楽観的ロック + リトライで設計。

**Q: PSIRT 承認で何が懸念されたか？**
→ パース失敗時の挙動（不正アクセスにならないか）→ エラー時は 401 で弾く設計を明示。

**Q: 意思決定権限の「明文化」はどんな形式で？**

### Rate Limiting（一括招待・組織同期）

**Q: 一括招待のrate limiting設計は？**
→ OEMのユーザー作成APIにrate limitがある。workerをurgentとbulkに分け、それぞれに独立したRedis rate limiterを用意した。

- **bulk worker**: 一括招待と組織同期の両方を担当。大量リクエストを保守的な閾値でスロットリング
- **urgent worker**: ロール変更・従業員削除など即時性が求められる操作を担当。bulkがquotaを使い切ってもurgentのquotaは独立して確保

Luaスクリプトで「残り確認→インクリメント」をアトミックに実行。2コマンドに分けると複数ジョブが同時に「残り1」を読むrace conditionが起きる。OEMのレスポンスヘッダー（`X-RateLimit-Remaining`系）は制御には使わず、アラート指標として使用。自前カウンタとOEM側のドリフトリスクは各workerの閾値を保守的に設定して吸収した。exponential backoffをリトライに組んだが本番では一度も発火しなかった。

**Q: なぜOEMヘッダーを制御に使わなかったか？**
→ OEMヘッダーはリクエストを送った後にしか読めない。つまり「送る前の判断（制御）」には使えない。事前にスロットリングするには自前カウンタが必要で、ヘッダーは送った後の確認（事後モニタリング）にしか使えない。

**Q: 自前カウンタとOEMヘッダーでズレは生じないか？**
→ 窓の実装が違う場合（自前 fixed window、OEM rolling window）は境界付近でズレが起きうる。自前カウンタがリセットされてもOEM側には前の窓のリクエストが残っているため、OEM視点では超過して見える。対策は2層。①閾値をOEMのlimitより保守的に設定して安全マージンを確保。②万が一429が返ってきてもexponential backoffで復帰できる設計にした。OEMヘッダーで実際の残量を事後モニタリングして、ズレが常態化していないか監視する運用も入れた。

### テーブルロック（一括招待）

**Q: 一括招待で行ロックがurgent操作と競合するリスクはどう対処したか？**
→ UIレベルで制約を設けた。「全員を一気に招待」ボタンは作らず、100件ずつのchunkでしか招待できないページネーションにした。100件という数字は人事労務側の既存仕様に合わせたもので、freeeプロダクト全体のUX統一が目的。結果として1チャンクのトランザクション保持時間が短くなり、urgentとのロック競合の確率と影響範囲を下げた。ゼロリスクではないが、アドミンがページをめくる操作の間隔がurgentの入れる窓になる。

**Q: ジョブ失敗時の再実行でOEM APIを二重呼び出しするリスクは？**
→ `provisioning_status`カラムで冪等性を担保した。`pending` / `success` / `failed` の3ステータスを持ち、OEM API呼び出し前に`success`でないことを確認してから実行。リトライ時に二重作成が起きない設計。

### セッション無効化

**Q: ユーザー削除後のセッション無効化はどうしたか？**
→ Webhook が打てない（基盤側実装待ち）という制約があった。対応は3層。①TTL: OEM 元々30分、freee のセッションと同じ長さなので変更不要と判断。②即時対応: 管理者が組織同期ボタンを押すことで即時反映できる仕組みを用意。③フォールバック: 「ログインできない場合は組織同期してください」のヘルプページを用意。ベストではないが webhook 実装待ちの間の許容できるトレードオフと判断した。

### テスト哲学（Resilire は testcontainers / mock 禁止派）

**Q: テストでモックを使っていたか？**
→ freee E-Learning では統合テストに Docker Compose で DB を直接立てるアプローチを取っていた。Go マイクロサービスでは外部 API（ML・ES）は interface でモックしていた。testcontainers の哲学（「DB はモックするな、本物を立てろ」）には共感できる。本番環境との乖離がテスト失敗の原因になった経験があるため。御社方針に合わせてすぐに切り替えられる。

**Q: Testing Trophy（統合テスト重視）に対してどう思うか？**
→ 賛成。ユニットテストだけだと、統合境界（DB・API）のバグを見逃す。一方 E2E だけだと遅くてフィードバックループが長い。統合テストを厚くして、ユニットは純粋な計算ロジックに絞る方針が再現性と速度のバランスで一番良いと実感している。
