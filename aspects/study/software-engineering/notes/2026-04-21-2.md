# 勉強 - 2026-04-21

## 🎯 今日の目標・疑問

- パフォーマンスチューニングの2大要素（インデックスと統計情報）を理解する

## 📝 ノート

### パフォーマンスで大事な2要素

1. **インデックス** — DBMS内にテーブルとは独立に保持されるオブジェクト。`(キー値, レコードへのポインタ)` の配列
2. **統計情報** — SQLのアクセスパス（実行計画）を決める情報

> SQLには「このデータが欲しい」とは伝えられても「こうやってデータを取りに行って」とは言えない。

---

### インデックス設計

**特徴:**
- DBMSが自動で使うか判断する（開発者が意識しなくてよい）
- **アプリケーション透過性**: アプリケーションコードに影響しない
- **データ透過性**: テーブルのデータ自体に影響しない
- 非正規化よりもハードルが低い

**注意:** 全列にインデックスを貼るのは NG → **インデックスショットガン**

---

### B-Tree インデックス（最も有名）

**強み: オール4の秀才型** — どんな値を指定しても同じくらいの速度で結果を得られる

- **平衡木（Balanced Tree）** の特性: 毎回探索量が同じ
- テーブルへの挿入・更新・削除で木が崩れる → 自動修正機能あり

**計算量:**

| 操作 | フルスキャン | B-Tree |
|------|------------|--------|
| 検索 | O(N) | O(log N) |
| 挿入 | - | O(log N) |
| 更新 | - | O(log N) |
| 削除 | - | O(log N) |

**対応する演算子:** `=`, `>`, `<`, `<=`, `>=` → 高速
**効果なし:** `!=`（否定）

**ソートの最適化:**
- DBMSにとってソートは重い演算
- B-Treeはデータをソート済みで保持 → ORDER BY 指定時にソート処理をスキップできる

---

### インデックスを貼る列の選定基準

1. **大規模テーブル**（レコード数10万件超）
2. **カーディナリティの高い列**（値の種類が多い）
   - 例: 性別 = 2（低い）、ユーザーID = ユニーク（高い）
3. **WHERE句・結合条件に使用されている列**

**複合インデックスのカーディナリティ:**
- a=2種, b=10種, c=5種 → `(a,b,c)` = 2×10×5 = 100

**注意:** カーディナリティが高くても分布に偏りがあれば実質低い
- 例: 値100が99%、値1〜99が1% → 実質カーディナリティは1 → B-Tree非推奨

---

### インデックスが適用されないパターン

1. **インデックス列に演算** — `WHERE col_1 * 1.1 < 100`
2. **インデックス列に関数** — `WHERE SUBSTR(col_1, 1, 1) = 'a'`（解決: 関数インデックス）
3. **IS NULL** — `WHERE col_1 IS NULL`
4. **否定** — `WHERE col_1 <> 100`
5. **後方一致・中間一致の LIKE**
   - NG: `LIKE '%a'` / `LIKE '%a%'`
   - OK: `LIKE 'a%'`（前方一致のみ有効）

**その他の注意:**
- 主キー・一意制約の列 → インデックス作成済みなので不要
- B-Treeインデックスは**更新性能を低下させる**（独立オブジェクトとして保持されるため）
- 定期的な**インデックスの再構築**（メンテナンス）が必要

---

### 統計情報

**SQL実行の流れ:**
ユーザー → パーサー（文法チェック）→ オプティマイザー（最適パス模索）↔ カタログマネージャー → テーブル

- オプティマイザーは開発者から見たら**ブラックボックス**

**統計情報の収集タイミング:**
- データが大きく更新されたタイミング（夜間推奨）
- 重い処理なので必要テーブルにのみ実施

**統計情報の凍結:**
- 実行計画が最適だと分かっている場合、変更されないようにロックする

---

### その他のチューニング方法

#### パーティション

物理的な読み込みを減らし、**ストレージI/O（最も重い処理）を削減**。

**例:** `order_month` でパーティショニング → 同じ月のレコードが物理的にまとまる → 特定月の検索時に不要パーティションを読み飛ばす（**パーティションプルーニング**）

**パーティションの種類:**

| 種類 | 用途 | 例 |
|------|------|-----|
| レンジ | 時系列データ（連続値） | 注文月、年度 |
| リスト | カテゴリなど離散的な値 | 商品コード、地域 |
| ハッシュ | 均等分散（キーに規則性がない場合） | ユーザーID等、分布が予測しにくい列 |

**コンポジットパーティションの例:**
リストで商品コード（A,B,C,D）に分けたが、商品Aだけレコードが偏っている → 商品Aのパーティション内をさらにハッシュで均等分散させる（リスト×ハッシュ）

**注意:**
- WHERE句でパーティションキーを指定しないと意味がない
- パーティションキーは大きな偏りがない方がよい
- 複数パーティション組み合わせ可能（コンポジットパーティション）
- パーティション数に上限あり

**使い分け:**
- カーディナリティが**高い** → インデックス
- カーディナリティが**低い** → パーティション

#### ヒント句

実行計画を手動で変更する。ハマれば絶大だが**最終手段**。標準機能ではなく、薬にも毒にもなる。

#### パラレルクエリ

CPU/ストレージのリソースが潤沢に余っているときのみ。不足すると他のクエリまで巻き添えに。

#### オンメモリ

メモリが潤沢に余っている場合のみ。みだりに使用しない。利用できるDBMSも限定的。

## 💡 まとめ

1. パフォーマンスチューニングの2大要素は**インデックス**と**統計情報**。インデックスはアプリケーション/データ透過性があり非正規化より導入しやすい
2. B-Treeインデックスは O(log N) のオール4型。ただし貼る列の選定（カーディナリティ・分布の偏り）と適用されないパターン（演算・関数・NULL・否定・後方一致LIKE）を理解する必要がある
3. パーティションはI/O削減が目的。レンジ（時系列）・リスト（カテゴリ）・ハッシュ（均等分散）の使い分けと、コンポジットで組み合わせ可能
