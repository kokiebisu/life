## 📌 Groundtruth レコメンド基盤 - 面接かんぺ(セリフ版)

### 🎤 状況設定(30〜40 秒)

「Groundtruth は B2B 広告 SaaS で、私はその Ads Manager というプロダクトでセグメントレコメンド機能を設計・実装しました。

当時、広告主がキャンペーン作成時にターゲットセグメントをドロップダウンから選ぶ UI がありましたが、選択肢が**数千件**まで膨れていて、Google Analytics で離脱が観測されていました。効果の薄いセグメントを選んでしまうケースも多く、広告効果も伸びていない。プロダクト側からレコメンド追加の意思決定が出ました。

方針は、**未経験ユーザーには業種別人気 Top 5**、**経験ユーザーには ML によるパーソナライズ候補**を出して、両者をマージして返すという形です。リリース後 Q 単位の集計で、キャンペーン作成時の churn が約 20% 減、広告インプレッションが約 10% 増、締切より 2 週間前倒しで完了しました。」

---

### 🎤 制約(45〜60 秒)

「制約は大きく 3 つありました。

1 つ目は **P99 1,000ms の SLO**。これを超えると元のドロップダウン選択より UX が悪化するという、プロダクト側で引いたラインです。

2 つ目は**外部依存のレイテンシ特性**。AI チームが管理する ML エンドポイントが最大 800ms、Elasticsearch が約 300ms。**直列に呼ぶと 1,100ms で SLO を超える**ことが制約段階で確定していました。

3 つ目は **ML エンドポイントの可用性**。AI チームが別管理で、彼らのデプロイ影響で ML が一時的に落ちることが現実的に予想されました。『ML 必須』設計だと、AI チームのデプロイ毎にレコメンドが死ぬ構造になります。

さらに既存 Ads Manager は **Flask モノリス、WSGI 同期処理**なので、`asyncio.gather()` のような並列化が素直に使えませんでした。観測スタックは Datadog と Sentry で、フォールバックが起きたことを定量的に把握できる必要もありました。」

---

### 🎤 設計判断(60 秒)

「採用した設計は、**Go で独立マイクロサービスに切り出して、goroutine で ML と ES を並列に呼ぶ**形です。

選択肢は 3 つ検討しました。

A: 既存 Flask モノリスに Python で追加。WSGI 同期処理で並列化できず、直列 1,100ms で SLO 超過確定なので消滅。

B: Flask + gevent で独立サービス化。並列はできますが、**monkey patch が stdlib 全体を書き換える副作用**があり、チーム内に gevent の運用実績も薄い。本番デバッグが詰むリスクがありました。

C: Go で独立マイクロサービス化。**goroutine と `context.WithTimeout` が言語標準**、副作用なし、deadline 伝播が明確、単一バイナリでデプロイがシンプル。

採用理由は 3 つです。**並列化が言語レベルで素直**、**副作用リスクなし**、**ML が落ちてもモノリス全体に波及しない障害分離**。学習コストは認めましたが、比較対象は『Go の学習コスト』ではなく『Python のまま SLO や運用リスクを抱えるコスト』で、後者の方が長期で重いと判断しました。」

---

### 🎤 並列実行とフォールバック(60 秒、強い武器)

「並列実行は **`sync.WaitGroup` + `context.WithTimeout`** で組みました。サービス全体の deadline を 950ms に置き、ML と ES を 2 本の goroutine で独立に呼んで、結果を構造体に書き込んで待ち合わせる形です。

**`errgroup` を使わなかった理由**は明確で、errgroup は最初のエラーで他の goroutine を cancel しますが、今回は『**ML が失敗しても ES の結果は使う**』graceful degradation が要件でした。errgroup の挙動と要件が合わないので、結果とエラーを独立に保持する WaitGroup を選びました。

フォールバックは 3 段階です。

- **Tier 1**: ML + ES 両方成功 → パーソナライズ + ES サジェストの混合(想定 95%)
- **Tier 2**: ML 失敗 → ES のみ(想定 3%)
- **Tier 3**: 両方失敗 or 新規ユーザー → Redis にキャッシュした業種別 Top 5(想定 2%)

Redis Top 5 はオフラインバッチで日次更新、TTL 7 日。メインパスでは使わず、**最終フォールバック専用**です。レコメンドは決済と違って強整合性より可用性が大事なので、**ゼロ件で UI を壊さないこと**を優先しました。」

---

### 🎤 「並列処理は本当に必要だったか」への答え(45 秒)

「必要でした。理由は数字と依存関係の 2 つです。

**数字**: ML 最大 800ms + ES 約 300ms = 直列 1,100ms。P99 1 秒の SLO を超える。

**依存関係**: ML と ES は互いに独立した I/O です。ML の結果を使って ES を呼ぶわけではないので、直列に待つ理由がありません。

それから並列化はフォールバックにも効きます。直列だと先に呼んだ ML が timeout した時点で ES を呼ぶ時間も失いやすい。並列なら ML が遅延・失敗しても、ES の結果だけで Tier 2 として返せます。

複雑さは認めます。ただ今回は **1 リクエストあたり goroutine 2 本に固定**で、数が増え続ける fan-out ではない。timeout とメトリクスを入れれば運用できる複雑さだと判断しました。」

---

### 🎤 「ML が 800ms なら並列化してもそこまで縮まらないのでは」への答え(30 秒、強い武器)

「おっしゃる通りで、並列化しても 800ms が消えるわけではありません。**ML が律速なのは認めます**。

並列化の価値は『800ms を縮める』ではなく『**800ms + 300ms にしない**』ことです。直列だと 1,100ms ですが、並列なら `max(ML, ES)` ≒ 800ms + 集約処理 50ms に収まる。

ただ SLO 1 秒に対して余裕は大きくありません。なので **ML を無制限に待つ設計にはしない**。サービス全体に 950ms の deadline を置いて、deadline 内に ML が返れば使う、返らなければ ES / Redis に落とす。つまり SLO は**並列化・deadline・fallback のセット**で守る設計です。」

---

### 🎤 SLO 監視と可用性判断(45 秒)

「観測は Datadog と Sentry を分けました。

- **Datadog**: フォールバック率、レイテンシ p50/p95/p99、`runtime.NumGoroutine()`
- **Sentry**: アプリケーション例外
- **PagerDuty**: フォールバック率 5% 超過で AI チーム / SRE に通知

特に大事だったのは**フォールバック率 5% を SLO**にしたことです。レイテンシだけ見ていると、ML が落ちて ES だけで『速くパーソナライズなし』で返っている状態を見逃します。だから**品質指標として独立に持つ**。

Datadog ダッシュボードには **AI チームのデプロイタイムスタンプを overlay** していて、デプロイ起因の障害が一目で分かるようにしました。3 つを並べて見ることで『ML の遅延なのか、ES の障害なのか、自前コードのバグなのか』が切り分けられます。」

---

### 🎤 P99 予算配分とリリース後の学び(45 秒)

「P99 1,000ms の予算配分はこうです。

- 並列実行(`max(ML, ES)`): 800ms(ML の P95 に律速)
- 結果集約・マージ: 50ms(O(n)、n 最大 30 件)
- ネットワーク・シリアライゼーション: 50ms
- 余裕(GC pause / jitter / TLS handshake): 100ms

リリース後の学びとして、**ECS の Cold start で初回 ML レイテンシが跳ね上がる問題**がありました。深夜のスケールイン後、最初のリクエストで稀に SLO 超過する。対処として **minimum instance = 1** にして、常に温まったコンテナを 1 つ保つ運用にしました。

~~`runtime.NumGoroutine()` を 10 秒ごとに Datadog 送信して、ベースライン + 100 を超えたらアラート、というリーク対策も入れています。~~

今やり直すなら、**リリース前に shadow traffic で事前計測**します。本番リリース後の実測だけだと、Cold start のような問題は遅れて気づく形になります。」

---

## 💬 想定 Q&A(セリフ版)

### Q1. なぜモノリスではなくマイクロサービスに?

「直列呼び出しになると 1,100ms で SLO 超過するからです。Flask モノリスは WSGI 同期前提で並列化が素直に書けない。独立サービスに切り出すことで、**並列実行・独立スケール・障害分離**の 3 つを同時に取れました。

特に障害分離は大きくて、ML エンドポイントが AI チームのデプロイで落ちた時に、Ads Manager 本体まで巻き込まれない構造にしたかった。」

### Q2. なぜ Flask + gevent ではなく Go?

「gevent は monkey patch で stdlib の I/O を greenlet 化する仕組みで、副作用が広範です。requests、ssl、threading など stdlib 全体が書き換わるので、**スタックトレースが読みにくくなる**、C 拡張モジュールでパッチが効かないケースがある、といった運用リスクがありました。

加えてチーム内に gevent の運用実績が薄い。本番でデバッグが必要になった時に詰む可能性が高かったので、**goroutine と `context.WithTimeout` が言語標準**の Go の方が長期運用コストで勝つと判断しました。」

### Q3. これは並列処理で行う必要があったんですか?

「はい、必要でした。ML が最大 800ms、Elasticsearch が約 300ms で、直列だと 1,100ms。P99 1 秒の SLO を超えます。加えて、ML と ES は互いに依存していないので、直列に待つ理由がありません。

もう 1 つ、**フォールバックの観点でも並列が効きます**。ML が timeout しても ES が返っていれば、ES only として Tier 2 で返せる。直列だと先に呼んだ依存先が遅いだけで後段の結果を取りに行く時間も失いやすい。並列化は高速化だけでなく、障害時にも何か返すための設計でした。」

### Q4. ML が 800ms なら、並列化してもそこまで縮まらないのでは?

「おっしゃる通りで、並列化しても ML の 800ms が消えるわけではありません。ML が律速なのは変わりません。

**並列化の目的は、ES の 300ms をその後ろに足さないこと**です。直列だと 1,100ms ですが、並列なら遅い方の 800ms に集約処理を足す形になります。それでも SLO 1 秒に対して余裕は大きくないので、ML を無制限に待つのではなく deadline で打ち切って ES / Redis に落とす。SLO は**並列化、timeout、fallback のセット**で守る設計です。」

### Q5. 学習コストがかかるのに Go にする必要があったんですか?

「学習コストはありました。ただ、比較したのは『Go の学習コスト』と『Python のまま SLO や運用リスクを抱えるコスト』です。既存 Flask に足すと同期処理で直列になりやすい。Flask + gevent なら並列化できますが monkey patch の副作用と運用経験の薄さがある。Go は goroutine と timeout が標準機能なので、この要件ではリスクが小さいと判断しました。

それから、**全体を Go にしたわけではありません**。HTTP で受けて ML、ES、Redis を呼んで返す、境界が明確な小さい集約サービスに限定しました。学習コストを払う範囲を絞った上で、SLO と障害分離を優先しています。」

### Q6. なぜ `errgroup` ではなく `sync.WaitGroup` を選んだ?

「**graceful degradation の要件と errgroup の挙動が合わなかった**からです。errgroup は最初のエラーで他の goroutine を cancel するので、ML が失敗した瞬間に ES の呼び出しも止まってしまう。

今回は『ML が落ちても ES の結果は使う』『両方落ちたら Redis Top 5 に落とす』という設計なので、**各 goroutine の結果とエラーを独立に保持して、後段の fallback ロジックで判定する**形が自然でした。WaitGroup なら結果構造体に独立に書き込めるので、要件に素直にマッチします。」

### Q7. goroutine リーク対策は?

「3 つです。

1. **全外部呼び出しに `context.WithTimeout` を必須化**、`defer cancel()` で関数終了時に解放
2. **`runtime.NumGoroutine()` を 10 秒ごとに Datadog 送信**してベースライン + 100 を超えたらアラート
3. **リリース前の負荷試験**で 1,000 同時リクエスト後にベースラインに戻ることを確認

`context.WithTimeout` は linter で強制する形にして、開発時点で忘れにくくしました。」

### Q8. P99 予算配分の根拠は?

「律速箇所が ML の 800ms なので、これに合わせて全体予算を組みました。集約 50ms、ネットワーク 50ms、GC pause と jitter のための余裕 100ms を積み上げて 1,000ms です。

数字の根拠は、ML 800ms は AI チームから提示された P95 の値、ES 300ms は過去のクエリログから自分で計測、集約 50ms は O(n)、n 最大 30 件で見積り、余裕 100ms は Go の GC・ネットワーク往復・TLS handshake のばらつきから取りました。

本番リリース後は Datadog の production traffic で実測し SLO 達成を確認しています。今やり直すならリリース前に shadow traffic で事前計測したいです。」

### Q9. フォールバックの段階数の妥当性は?

「3 段階に絞りました。**細かく分けすぎると保守と監視が難しくなる**ので、運用で実際に起きるシナリオに合わせています。

- ML + ES 両方成功(正常パス)
- ML 障害(AI チームのデプロイ起因で最も起きやすい)
- 両方失敗 or 新規ユーザー(最終セーフティネット)

各 Tier への落ち込み率を Datadog でメトリクス化しているので、**想定外の偏りに気づける構造**にもなっています。例えば Tier 3 が増えてきたら ES 側の障害を疑う、というふうに切り分けに使えます。」

### Q10. 結果整合性 vs 強整合性の判断は?

「**AP 寄りに倒しました**。レコメンドは決済とは違って『最新の完璧な結果』より『何か返る』が UX 上大事です。

Redis キャッシュの Top 5 はバッチで日次更新なので鮮度が落ちますが、フォールバック時のパーソナライズ品質低下と、何も返らないことによる離脱を比べると、**後者の方がダメージが大きい**。だから可用性を取りました。

決済のような『正確に 1 回・最新の状態』が必要な機能だったら逆の判断になりますが、レコメンドは性質的に AP で良い領域だと整理しています。」

### Q11. Redis Top 5 の鮮度が落ちる問題は?

「日次更新なので**当日の急変は反映できない**のは事実です。ただ Redis Top 5 はメインパスではなく、ML / ES が両方落ちた時の**最終フォールバック専用**なので、想定発生頻度は 2% 程度。

TTL は 7 日にしていて、**バッチが万一落ちても直近のキャッシュで運用継続できる**ようにしました。鮮度が要件になる機能ではないので、保守の単純さを優先しています。」

### Q12. ECS Cold start 対策の判断軸は?

「**コスト vs SLO のトレードオフ**です。常時 1 インスタンス分のコストは払いますが、Cold start による P99 悪化を防ぐ価値が勝つと判断しました。

加えて **ML エンドポイント側もウォームアップが必要**だったので、AI チームと連携して min instance を合わせました。ヘルスチェックエンドポイントで定期的にモデルを叩いて warm を維持する運用も入れています。

Provisioned Concurrency のような追加機構もありますが、ECS の min instance で十分対処できたので、最小構成に留めました。」

### Q13. 観測・モニタリングは?

「Datadog でフォールバック率・レイテンシ p50/p95/p99・goroutine 数を見て、Sentry でアプリケーション例外を拾う構成です。AI チームのデプロイタイムスタンプを Datadog ダッシュボードに overlay しているので、**デプロイ起因の障害が一目で分かる**ようになっています。

フォールバック率 5% を SLO にして、超えたら PagerDuty で AI チーム / SRE に通知。レイテンシだけだと『速いがパーソナライズなし』を見逃すので、**品質指標として独立に持つ**のがポイントでした。」

### Q14. 1 リクエスト 2 goroutine 固定だが、スケールアウトはどうした?

「**1 リクエスト内の goroutine 数は 2 で固定**ですが、リクエスト数自体は ECS の auto scaling で水平に増やしました。min=1、max は当時のピークから逆算して設定しています。

なので fan-out が爆発する設計ではなく、**1 リクエストあたりの並列度は読める、リクエスト数はインスタンス数で吸収する**という素直な構成です。goroutine 数の上限が予測できるので、メモリと CPU の見積もりも安定していました。」

### Q15. 今やり直すなら何を変える?

「3 つあります。

1 つ目は **リリース前に shadow traffic で事前計測**を入れること。本番リリース後の実測だけだと、Cold start のような問題に遅れて気づきます。

2 つ目は **Cold start 対策を最初から入れる**こと。リリース後に min instance = 1 を後付けしましたが、設計段階で予見できた問題でした。

3 つ目は **goroutine 数監視と context.WithTimeout の linter 強制を最初から組み込む**こと。後から入れると既存コードの修正が増えるので、リリース時点で運用ガードレールを揃えておきたいです。」
