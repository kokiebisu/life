# 勉強 - 2026-04-15

## 🎯 今日の目標・疑問

システム設計の面接本 Chapter 10〜 通知システムの設計を理解する

## 📝 ノート

### 通知システムの要件確認ポイント

- リアルタイム性（soft real-time か厳密か）
- 通知種別（Email / SMS / モバイルプッシュ）
- 1日あたりのユーザーあたりの通知量
- DAU（スループット試算に必要）
- 対応端末（iOS / Android / Web）
- オプトアウトの可否

### 通知チャネルごとの仕組み

**iOS プッシュ通知**
```
Provider → APNs → iOS端末
```
- APNs に渡すもの: Device Token + Payload（JSON）
- Device Token は「アプリ × 端末」の組み合わせを一意に識別（同一端末でもアプリが違えば別Token）
- 端末登録フロー: アプリ起動 → 通知許可 → APNs に登録 → Token発行 → サーバーに送信・保存
- Token はアプリ再インストール / OS更新で変わる可能性あり → 無効Token検知して DB から削除する仕組みが必要

**Android プッシュ通知**
- FCM（Firebase Cloud Messaging）を使用。仕組みは APNs とほぼ同じ
- FCM は iOS にも送れる（内部で APNs に転送） → FCM に統一して両OS をさばく設計も可能

**Email**
- SendGrid（汎用）、Mailchimp（マーケティング特化）などの Third-party を利用

**SMS**
- Twilio、Nexmo などの Third-party を利用。国際対応が必要な場合も多い

### 通知許可のタイミング

- サインアップ**後**が一般的（ユーザーが価値を感じた後に許可率が上がる）
- iOS は一度「許可しない」を選ぶと再度ダイアログを出せない → タイミングが重要

### 冪等性（Idempotency）

| 状況 | 二重送信 | 二重処理 |
|------|---------|---------|
| 冪等性なし | 危険（禁止・防止が必要） | 発生しうる |
| 冪等性あり | 許容（安全にリトライ可） | 防止される |

**仕組み:**
```
POST /charges
Idempotency-Key: abc-123
```
- 同じキーで2回送っても課金処理は1回だけ
- 2回目は「最初の処理結果をそのまま返す」（再実行しない）

**タイムアウト時のリトライが安全な理由:**
```
クライアント → リクエスト送信 → タイムアウト（成否不明）
            → 同じキーで再送（safe!）
サーバー    → 「このキーは処理済み」と判断 → 最初の結果を返すだけ
```

冪等性 = 「何回実行しても結果が同じ」。二重送信を許容する代わりに、二重処理を防ぐ設計。

### 非機能要件の観点（毎回のシステム設計で必ず確認）

| 観点 | 英語 | 一言定義 |
|------|------|---------|
| 信頼性 | Reliability | 処理が確実に完了するか |
| 可用性 | Availability | システムが常に稼働しているか |
| 一貫性 | Consistency | どのノードから読んでも同じデータが見えるか |
| スケーラビリティ | Scalability | 負荷増加に対応できるか |
| 耐久性 | Durability | データが消えないか |
| レイテンシ | Latency | 応答速度 |

面接では最初に「どの非機能要件が重要か」を宣言してから設計に入る。

### コンポーネント設計

```
[サービス 1,2,3...]
       ↓
[通知サービス]  ← User Info Service からデバイストークン・連絡先取得 / オプトアウト確認
       ↓
  [メッセージキュー]  ← チャネルごとに分離（iOS / Android / SMS / Email）
  /      |      \
[iOS]  [Android] [SMS/Email]
Worker  Worker    Worker
  ↓       ↓         ↓
APNs    FCM    Twilio/SendGrid
```

**通知サービス**
- 他サービスからの通知リクエストを受け付ける
- User Info Service からデバイストークン・連絡先を取得
- オプトアウト確認（ユーザー設定 DB を参照）
- メッセージキューに投げる

**メッセージキュー（チャネルごとに分離）**
- チャネルによって処理速度・障害を切り分けたい
- 例: SMS が落ちてもプッシュ通知は落ちない

**Worker**
- キューからメッセージを取り出して Third-party に送信
- 失敗時はリトライ（Idempotency-Key で冪等性を担保）

**DB**
- ユーザー設定: デバイストークン・連絡先・オプトアウト状態
- 通知ログ: 送信履歴・ステータス

**なぜキューを積むか？**
- 送信処理が時間かかるため、キューなしだとメインスレッドをブロックする
- Worker が落ちてもキューが残るのでリトライできる（通常のアプリサーバーでは難しい）
- ピーク時のバッファ（突然100万件来てもキューが吸収）

**スロットリング**
- 1ユーザーへの通知量を時間単位で制限
- Worker 側で実装（チャネル間で共通の rate limit 制約があるわけではないため）

## 💡 まとめ

- 通知システムは APNs / FCM / SMS / Email の各チャネルに Third-party を使い、Worker + キューで非同期処理する
- キューはバースト吸収・デカップリング・障害時リトライの三役を担う
- 冪等性は `event_id + channel` をキーにした notification_log で担保し、レースコンディションは unique 制約で防ぐ
- 非機能要件（信頼性・可用性・一貫性・スケーラビリティ・耐久性・レイテンシ）は毎回の設計で最初に宣言する習慣をつける
