## 📝 ノート

### POA vs DOA

- **POA（Process Oriented Approach）**: かつての主流。プロセス（機能・処理）を中心に設計し、データは後回し
  - 問題点: データ構造が後付けになるため、要件変更のたびにモデルの変更が頻発 → 非効率
- **DOA（Data Oriented Approach）**: 現在の主流。データ構造を先に設計し、プロセスはそこに乗せる
  - メリット: データは業務の本質を反映するため変化しにくい → 安定したシステム設計が可能

### 3層スキーマ

| 層           | 別名     | 視点    | 内容                                 |
| ------------ | -------- | ------- | ------------------------------------ |
| 外部スキーマ | —        | ユーザー | ビュー・アプリから見えるデータの姿   |
| 概念スキーマ | 論理設計 | 開発者  | ER図・正規化。業務の本質的なデータ構造 |
| 内部スキーマ | 物理設計 | DBMS    | テーブル・インデックス・ストレージ   |

- **2層スキーマ**（外部 + 内部のみ）: 小規模では機能するが、外部スキーマの変更が直接内部に波及するため脆い
- **概念スキーマの役割**: 外部と内部の緩衝材。ユーザー要件の変化をここで吸収し、物理層を守る

## 🔑 キーワード

- POA（Process Oriented Approach）
- DOA（Data Oriented Approach）
- 外部スキーマ / 概念スキーマ（論理設計）/ 内部スキーマ（物理設計）
- 正規化
- ER図

## 💡 まとめ

1. **POA → DOA**: 業務フロー（画面・バッチ）は変わりやすいが、データは本質的に変わりにくい。データを土台（基礎）に設計する DOA が現在の主流
2. **3層スキーマ**: 外部（ユーザー視点、ビューなど）・概念（開発者視点・論理設計）・内部（DBMS視点・物理設計）の3層構造。概念スキーマが外部と内部の緩衝材になる
3. **2層 vs 3層**: 小規模なら2層でも動くが、外部変更が内部に直撃するリスクがある
