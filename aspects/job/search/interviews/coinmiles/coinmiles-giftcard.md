## 🎁 Coinmiles — ギフトカード購入・払い戻しフロー

### 状況

モバイルアプリでユーザーがギフトカードを購入し、仮想通貨でキャッシュバックを得られる機能を実装した。決済は Stripe の Payment Intent を使用。ギフトカードプロバイダーとの連携部分で失敗が発生した場合の払い戻しフローを設計した。

### アーキテクチャ概要（サーバーレス構成）

Lambda・SQS・SNS・Amplify・Cognito・GraphQL・MongoDB Cloud の構成。Lambda はデフォルトで IP が固定されないため、NAT Gateway + Elastic IP で固定して MongoDB Cloud への接続を安定させた。後からギフトカードプロバイダー側の IP ホワイトリスト対応も同じ構成で対応した。

### アーキテクチャ

**購入フロー：**
Stripe Payment Intent で決済確定 → ギフトカードプロバイダー API 呼び出し → 成功時はギフトカード発行

**失敗時フロー：**
ギフトカード API 失敗 → SQS に払い戻しイベントを積む → 非同期でリファンド処理

同期で払い戻しを叩くとタイムアウトやリトライで API レスポンスが詰まる。SQS に積んで非同期にすることで、購入フローを即座にユーザーに返しつつ、バックグラウンドで確実に払い戻しを処理できる設計にした。

### Stripe Payment Intent を使った理由

2つの理由。①冪等性の担保。②2フェーズコミットに近い設計を実現したかった。

Payment Intent を作成した時点では決済を仮確保するだけで引き落としは発生しない。ギフトカードプロバイダー API を叩いて 200 が返ってきた時点で初めて Payment Intent を確定（キャプチャ）する。API が失敗した場合は Payment Intent をキャンセルするだけで払い戻しは不要。「両方成功を確認してから確定する」設計にすることで、大半の失敗ケースを払い戻しなしで処理できた。

### DLQ（Dead Letter Queue）

SQS コンシューマーのエラーを2種類に分類した。

- **リトライ対象**：ネットワークエラー・5xx。一時的な障害なので時間を置けば回復する
- **DLQ 行き**：4xx（プロバイダーの API 変更・こちら側のロジックの不備）。リトライしても同じエラーが返るだけなので SQS のスループットを圧迫しないよう即 DLQ に分離した

DLQ にイベントが入ったら CloudWatch → Slack にアラートを飛ばして開発者が Datadog で手動調査。「自動回復するもの」と「人間が見るもの」を明確に分けた。

### 想定ツッコミ

- **Q: なぜ同期で払い戻しを叩かなかったか？** → 決済フローをユーザーに即返す必要があった。同期でリトライすると購入レスポンスが詰まる。SQS で非同期にして購入体験を壊さず、バックグラウンドで確実に処理する設計にした
- **Q: Payment Intent を使った理由は？** → 冪等性の担保と2フェーズコミット的な設計のため。API 成功を確認してから決済を確定することで、大半の失敗ケースを払い戻しなしで処理できた
- **Q: DLQ に入ったイベントはどう処理した？** → 4xx はリトライしても無意味なので即 DLQ に分離。CloudWatch → Slack アラート → Datadog で手動調査する運用にした
- **Q: Payment Intent 確定後にコード取得レスポンスがロストした場合は？** → このケースは Payment Intent を確定済みでユーザーに請求が発生している。コードの再取得ができなければ CS 経由で払い戻しフローに流す対応になる
