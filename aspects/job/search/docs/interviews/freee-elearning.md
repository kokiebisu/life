# 面接対応: freee Eラーニング

**期間:** 〜2026年2月リリース（7ヶ月）  
**役割:** プロダクトリード（エンジニア 3名チーム、うち 2名はジュニア）

---

## 30秒イントロ

freee 人事労務上に Eラーニングプロダクトを新規追加した。OEM（LMS プロバイダー）と OIDC 連携して、人事担当者が従業員に研修を受けさせる管理機能を作った。3名のチームが 7ヶ月でゼロからリリース。

制約は 3つ: チームが小さい（ジュニア 2名）、OEM がリアーキ中で仕様変更に応じられない、PSIRT 要件（なりすまし防止）は絶対条件。この制約の中で「3名が保守できる最もシンプルな設計」を選び続けた。

---

## 核心となる設計判断

### 1. 人事労務のモノリスにモジュールとして追加する判断を下した

**選択肢:**
- A: スタンドアローンアプリ（独自リポジトリ・独自 ECS/EKS）
- B: 人事労務 Rails アプリに `Elearning::` 名前空間のモジュールとして追加 → **採用**

**スタンドアローンを却下した理由:**
- 扱うデータは従業員の氏名・所属のみ（給与・銀行情報ではない）。独立したセキュリティ境界を設ける必然性がない
- OEM がコア機能（コース管理・動画配信・受講記録）をすべて担う。freee 側が作るのは「どの従業員に利用を許可するか」の権限管理と組織同期だけ。この規模に独自インフラのコストを払うのは投資対効果が合わない
- 3名チームが新しいリポジトリ・インフラ・CI/CD・監視を並行して構築するのは現実的ではない

**受け入れたトレードオフ:**
人事労務本体の障害が Eラーニングにも波及するリスクを許容した。逆方向（Eラーニング障害 → 人事労務本体）については、同一プロセス・同一 DB 接続プールであるため**完全な隔離はできない**。ただし「Eラーニングのルートがエラーになっても人事労務のルートに直接影響しない」というコントローラーレベルの分離は担保した。DB コネクション枯渇やメモリリークは波及し得るリスクとして受け入れた——その規模のリスクではないと判断したから。将来の切り出しを見越し、`Elearning::` 名前空間を厳守して人事労務コアへの依存を最小化した。

---

### 2. OIDC の「1:n 不一致」を OEM 側のリアーキを待たずに解決した

**問題の本質:**
freee は 1ユーザーが複数事業所に属せる（副業・グループ会社）。OEM は 1ユーザー = 1会社の前提で設計されており、OEM がリアーキ中のため仕様変更に応じられなかった。

**前提:** freee の事業所ごとに別々の OEM アカウントを作成する必要があった。これは避けられない制約。

**論点:** OEM 側が「このアカウントは freee のどの事業所と紐づいているか」をどう把握するか。

**却下した案（OEM 側に事業所情報を持たせる方法）:**
- OEM 側に freee の事業所情報を持ったテーブルを追加 → OEM に断られた
- OEM 側の企業作成時に freee の `external_cid` を使ってもらう → OEM に断られた

**採用した解決策: ログインID フィールドへの埋め込み**

OEM のアカウント作成時に、ログインID フィールド（任意文字列）へ `"{user_id}_{company_id}"` を詰める。ログインID 自体に事業所の情報を含めることで、OEM 側に追加のテーブルや仕様変更を求めずに紐づけを実現した。このフィールドに OEM 側でユニーク制約を設けることで、ログインID と freee の `(user_id, company_id)` の組み合わせが一意に対応する。OIDC 認証時、freee 側はトークンリクエストからそのユーザーの `external_cid` と `external_uid` を取得し、ログインID（`{user_id}_{company_id}`）から抽出した値と照合して本人確認を行う。

**受け入れたトレードオフ:**
受講履歴が事業所ごとに分断される。複数事業所に属するユーザーの履歴を横断的に見ることはできない。ただし、Eラーニングの管理単位は「事業所ごと」であることが多く、実運用上の問題は限定的と判断した。

**PSIRT との調整点:**
「ログインID は後から変更できる。他人の値に書き換えてなりすませないか」という懸念が出た。OEM 側でログインID に**ユニーク制約**を設けてもらうことで対処した。既に別アカウントが使っている値への書き換えは OEM 側で拒否されるため、なりすましが物理的に不可能になる。PSIRT 承認。

---

### 3. Rate Limit を「技術で解決しない部分」を最初から特定した

**制約:** OEM の Rate Limit は 1IP あたり。freee は全顧客のリクエストを同一 IP から送るため、通常想定の何十倍ものリクエストが集中する。OEM との交渉で 300 req/sec まで上げてもらったが、これ以上は不可。

さらに招待時は毎回組織同期（部門ツリー更新）も必要で、1従業員あたりの実質 API コールが「アカウント作成 1件」より大幅に多い。

**設計:**
- Sidekiq で 100件ずつチャンク分割、1分間隔で投入（`perform_later(wait: index.minutes)`）
- チャンクレベルのリトライ（Rate Limit 超過時は 1分後にリトライ）
- 進捗は `OemSyncBatch` テーブルで可視化（タイムアウトで状態不明になる問題を排除）

**リリース後の継続チューニング設計:**
「最初から完璧には調整できない」という前提で、429 率を company_id タグ付きで Datadog に記録し、チャンクサイズ・インターバルの最適値を実測ベースで調整する仕組みを最初から組み込んだ。

## よく聞かれる質問と回答

**Q: なぜ Eラーニングを人事労務のサイドバーに置いたのか？**

OEM がコア機能を持つため freee 側の実装規模が小さく、スタンドアローンに独立するコストが合わなかった。プロダクトとして「人事担当者が研修管理も人事労務の画面内で完結する」ことが freee の統合体験ミッションに合致していた。福利厚生（従業員向け）ではなく人事労務（管理者向け）に置いた理由は、Eラーニングのコアニーズが「管理者が従業員に研修を受けさせる」機能だから。

**Q: PSIRT との調整で一番難しかったことは？**

なりすまし防止の実装方法。OEM の公開 API に `external_user_id` の専用フィールドがなかったため、ログインID フィールドに値を埋め込む方式を提案した。その値が後から書き換えられるリスクを PSIRT に指摘され、OEM 側での重複禁止設定で対処した。「freee 側のコードを変えない解決策」を OEM との交渉で実現した点が評価された。

**Q: ジュニア 2名のチームでどう品質を担保したか？**

`Elearning::` 名前空間による分離を徹底し、「このモジュールに触れば影響範囲が明確」な設計にした。レビューの工数を減らすため、人事労務本体のモデルへの依存を最小限にする方針を最初に決め、ドキュメントに明記した。複雑な実装（OIDC・非同期バッチ）は自分が担当し、ジュニアには影響範囲が明確なモジュール内の実装を任せた。

**Q: 同じアプリサーバーなのに「Eラーニング障害が本体に波及しない」と言えるのか？**

厳密には言えない。同一プロセス・同一 DB 接続プールなので、DB コネクション枯渇やメモリリークは人事労務本体にも波及し得る。

担保したのは「コントローラーレベルの分離」。Eラーニングのルートがエラーになっても、人事労務の給与計算や申請フローのルートには直接影響しない。逆方向（人事労務障害 → Eラーニング）は許容した。

この限界を受け入れた理由は、Eラーニングが扱うのは氏名・所属データのみで、DB 負荷やメモリ使用量が人事労務のコア機能を圧迫するほどの規模にならないと判断したから。将来的に規模が大きくなった場合に切り出せるよう、`Elearning::` 名前空間を厳守して依存を最小化した。

**Q: SLO をどう定義したか？**

5つ（OIDC 認証成功率・従業員同期成功率・インポート完了時間・Rate Limit エラー率・本体への影響ゼロ）を設定し、それぞれに Datadog のアラート閾値まで落とした。SLO を定義して終わりではなく「どのメトリクスで計測するか・どこでアラートを上げるか」まで設計した。

## 深掘り質問への準備

### リリース後、想定外のことはありましたか？

Sidekiq ワーカーのメモリが想定以上に不足した。リリース後に OOM（Out of Memory）が頻発し、ワーカーの割り当てメモリを 1GB → 3GB に引き上げる対応が必要になった。

ただ、OOM でワーカーが落ちても問題にならなかった。Sidekiq のリトライ機構と `OemSyncBatch` テーブルによる冪等性が担保されていたため、再起動後に途中から処理を再開できた。DLQ（デッドレターキュー相当）に積まれたジョブも二重実行なく正常に完了した。

「最初から完璧に調整できない」前提で設計していた部分が実際に機能した場面だった。

---

### 今やり直すなら何を変えますか？

1つ目は **OEM との仕様合意を先にすること**。OIDC の 1:n 問題は OEM がリアーキ中であることを最初から知っていたが、「どこまで変更できてどこからできないか」の境界を早期に文書化すべきだった。途中で「あれもできない、これもできない」が出てくるたびに設計を修正した。最初にできないことを全部洗い出しておけば設計のやり直しが減った。

2つ目は **モジュール境界をもっと厳しく守ること**。途中でジュニアメンバーが人事労務コアのモデルに直接アクセスするコードを書いてしまい、レビューで差し戻しが発生した。「なぜ `Elearning::` 名前空間の外に出てはいけないか」をドキュメントだけでなく、CI の lint ルールで自動検知できる仕組みを最初から入れるべきだった。
