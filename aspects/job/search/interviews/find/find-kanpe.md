## ① 技術選定の意思決定力

**使うエピソード:** Groundtruth レコメンデーションマイクロサービス（詳細: `resilire-groundtruth.md`）

**一言要約:**
「モノリスに追加するか切り出すか」という判断を「デプロイ頻度の独立性」「トラフィック影響範囲のコントロール」の2軸で決めた。

**比較軸の説明（面接で言う構造）:**
1. **デプロイ頻度が違う** → MLスキーマ変更・ESチューニングのたびにモノリス全体を巻き込まない
2. **影響範囲が読めない** → 新機能のトラフィックがモノリスのスレッドプールを圧迫するリスクを排除したかった

**合意形成のポイント:**
- PMのスコープ追加（リーチ数プレビュー）を「次フェーズへ移動」として工数で示し納得させた
- 結果: 当初スケジュールより2週間早くリリース、churn -20%、インプレッション +10%

**想定ツッコミ:** `resilire-groundtruth.md` の「想定ツッコミ」セクション参照

## ② AIと共創する開発文化づくり

**使うエピソード:** freee Eラーニングチームへの展開（詳細: `resilire-freee.md` の「開発プロセスの標準化」）

**一言要約:**
個人の工夫で終わらせず、チーム全体に再現性のある仕組みとして組み込んだ。

### レビューの多層設計（CI パイプライン）

| レイヤー | ツール | 役割 |
|---|---|---|
| Layer 1 | CodeRabbit | パターン的な指摘の自動化（命名・フォーマット・既知の anti-pattern） |
| Layer 2 | Claude Code GitHub Actions | コードロジック・実装意図のレビュー |
| Layer 3 | PSIRT チームの MCP エージェント | 脆弱性診断（社内セキュリティチームが提供） |

- 人間のレビューに入る前に3層で機械的に潰せるものを潰す設計
- PSIRT の MCP エージェントは「セキュリティ専門家の知識をエージェント化してチーム全体に配布した」形。自前で書くより精度が高く、維持コストもゼロ

### AI 統合開発パイプライン（最重要エピソード）

チケット起票から PR まで、文脈が一切抜け落ちない一本のパイプラインを設計した。

```
① Jira チケットで AI と壁打ち → チケットに決断の背景・理由を蓄積
        ↓
② チームでチケットレビュー（コードより前に意図を合わせる）
        ↓
③ チケットをベースに TDD → AI がテストケースを生成
        ↓
④ PR skill → チケットの内容（壁打ちの経緯・決断背景）をそのまま PR に添付
```

**なぜこれが強いか:**
- 普通のチームは ticket → code → PR で文脈が抜けていく
- このパイプラインは各ステップで文脈を蓄積して次に渡す設計
- 「なぜこの実装にしたか」がコードレビュー時に PR を見れば全部わかる
- ジュニアが実装中に迷わない（チケット段階で壁打ちが終わっているから）
- TDD のテストケースがチケットから自然に出てくる（要件が既に具体化されているから）

**効果:**
- 後戻り減少・リードタイム改善
- PR レビューが「意図の確認」より「実装の検証」に集中できるようになった
- 将来オンボードするエンジニアが PR を見るだけで設計の背景を学べる

### 開発フローのコード化（`.claude/skills`）

チーム向けに作成した skills:
- **Jira 壁打ち skill** — チケット単位で AI と実装方針を議論し、決断の背景をチケットに記録
- **Design Doc 壁打ち skill** — GitHub に置いた Design Doc を読み込んで実装との整合性をチェック
- **PR 作成 skill** — Jira チケット内容を自動添付。single responsibility チェック + サブチケット自動生成

### クロスリポジトリのコンテキスト読み込み

VS Code ワークスペースに**全基盤リポジトリを読み込んだ状態**で Claude Code を起動する運用を確立した。

- ビジネスロジックが複数リポジトリにまたがる場合でも、「この変更は隣のサービスの仕様と矛盾しないか」を壁打ちで確認できた
- Design Doc が GitHub 上にあるため、Claude Code が直接参照してコードと設計判断の一貫性をチェックできる

### その他の仕組み

- **git worktrees での並列開発** — 複数フィーチャーブランチを同時に進め、コンテキストスイッチのコストを削減
- **`PostToolUse` hooks でテスト自動実行** — Claude がコード変更するたびに対象 RSpec が走り、テスト壊しを即検知
- **`CLAUDE.md` で開発ルール共有** — 口頭説明ゼロ、参照型の規約ガイドが生まれた

### 数値で示す

- PR サイズ: 500行→150行
- レビュー TTA: 2日→4時間
- 意思決定会議: 週3回→週1回

## ③ コンテナ基盤への移行・大規模化への対応

**使うエピソード:** freee EKS クラスタ統合（詳細: `eks-migration-deep-dive.md`）+ Groundtruth ECS 運用

**技術スタック（実経験）:**

| 技術 | 経験 |
|---|---|
| EKS / Kubernetes | ✅ freee でクラスタ統合移行をリード |
| Terraform | ✅ IAM / namespace / ArgoCD 設定の移行 |
| ECS | ✅ Groundtruth でレコメンドサービスをコンテナ運用 |
| ArgoCD | ✅ app 定義・sync 管理・環境ごとの設定移行 |

---

### EKS クラスタ統合（freee）— 移行設計とダウンタイムゼロの実行

**背景:**
freee 福利厚生は独自 EKS クラスタを持っていたが、チーム規模に対して運用負荷が重かった（定期バージョンアップ・権限管理・監査統制）。人事労務側のクラスタに統合することで、運用コストと SOC1 対応の両方を解消した。

**移行したもの（一通りを自分で担当）:**
- Kubernetes manifest（Deployment / Service / ConfigMap / CronJob / Ingress）
- Terraform（IAM ロール / namespace 周辺 / ArgoCD 設定）
- RBAC（service account / role / role binding）
- ArgoCD（app 定義 / sync 設定 / 環境ごとの設定）

**段階移行の設計:**
integration → staging → production の順。production 前に staging で手順をリハーサルし、rollback ポイントを決めてから実施。

**一番危なかった点: Cron 二重起動**
- 旧新クラスタが並走する期間に同じ Cron が両方で動くと、メール二重送信・外部 API 二重呼び出しが起きる
- Cron の起動時間を一枠に集約し、その時間帯を避けて移行した
- CronJob suspend で旧クラスタ側を止め、新クラスタだけ unsuspend して確認
- 今ならさらに DB lock（advisory lock）と idempotency key を入れる

**結果:** ダウンタイムなしで production 移行完了。

---

### ECS 運用（Groundtruth）— コンテナ設計と cold start 対処

- Go のレコメンデーションマイクロサービスを ECS の Docker 1 コンテナで運用
- **cold start 問題を本番で踏んだ**: minimum instance = 0 のままだと初回リクエストの ML レイテンシが跳ね上がる → minimum instance = 1 に変更して常時温めた
- Datadog APM でコンテナの goroutine 数・レイテンシ・エラー率を監視

---

### 面接で使う一言要約

> 「freee では独自 EKS クラスタを共有基盤に統合する移行をリードしました。Terraform・Kubernetes manifest・IAM・RBAC・ArgoCD を全部含めた移行で、一番気をつけたのは Cron の二重起動です。segment 単位で起動時間を集約して危険な時間帯を避け、CronJob suspend を使ってダウンタイムなしで本番移行できました。」

**想定深掘り:** `eks-migration-deep-dive.md` の Q&A セクション（Q1〜Q9）参照

## ④ テックリードとしての牽引姿勢

**使うエピソード:** freee Eラーニング 全体（詳細: `resilire-freee.md`）

**一言要約:**
技術方針の策定・レビュー文化・後進育成、この3つを「仕組みで守る」一本軸でつないだ。

**語るべき3つの打ち手:**

### 技術方針の策定
- 意思決定マトリクスを作成（Confluenceに1ページ）
  - 「技術選定・実装方式 → Eng単独可」「スコープ変更 → PdM+リード必須」「リリース判断 → 双方サイン」
  - 「先週決めた議論が今週また始まる」繰り返しを止めた
  - ADRがある御社なら、このマトリクスをADRに昇格させるのが自然（と繋げる）

### レビュー文化の醸成
- AI + 人間のレイヤー設計:
  - Layer 1: CodeRabbit（パターン的指摘の自動化）
  - Layer 2: プロジェクト固有AIエージェント（OEM制約・Design Docとの整合）
  - Layer 3: 人間レビュー（本質的判断のみ）
- 結果: TTA 2日→4時間、人間のレビューが「本当に必要なところ」だけになった

### 後進育成の実績

**① Jira チケット粒度の AI skill 化（②と共通）**

- **問題の構造**: ジュニア × 新コードベース = チケット粒度が荒い → 後戻り・PR 肥大化・ベロシティが読めないという三重苦
- **解決**: AI skill が Jira チケットを single responsibility で切れているか検査し、満たない場合はサブチケットを自動生成
- **効果**:
  - 後戻りが減少
  - リードタイム改善
  - PR の説明が Jira チケット内容をそのまま添付できる粒度に
  - チームでベロシティの判断ができるようになった（属人化していた見積もりが共有可能に）
  - 将来オンボードするエンジニアの学習コストも下がった

**② デプロイ手順書（インシデント後の再発防止と仕組み化）**

- **背景**: 人事労務の定期デプロイと研修基盤のデプロイに順序依存があり、間違えると新規プロダクトの研修権限がリリース前に公開されてしまう
- **インシデント発生** → ただ謝って終わりにせず、「同じことが起きない構造」を作ることにした
- **アクション**: デプロイ手順書を作成し、新規プロダクト開発者のオンボードに組み込んだ
- **ポイント**: 「知っている人がいれば防げた」属人知識を「誰でも防げる手順」に変えた。テックリードの仕事は個人の注意力に頼る運用をなくすこと

**③ コードベース走査による CPU/RAM 影響分析 skill（②と共通）**

- **課題**: 新しいコードを書くとき「どのメトリクスを監視すべきか」が属人的で、見落としが起きやすかった
- **解決**: AI skill がコードベースを走査し、CPU/RAM に影響が出そうな処理を検出 → 対応するメトリクス指標をドキュメントに記録
  - 大量ループ・N+1 的なパターン → `db.query_duration_p99`
  - goroutine 生成が多い箇所 → `runtime.goroutines`
  - 大きなデータロード → メモリ使用量系メトリクス
- **効果**: オブザーバビリティ設計が「気づいた人がやる」から「コード変更時に自動でリストが更新される」に変わった

**④ TDD 導入（実装前の認識合わせ）**
- 実装前にテストケースのタイトルだけを列挙させてレビューする手法
- 「仕様をどう理解しているか」が透けて見える → 実装後の「思ってたのと違う」手戻りが激減

**弱みセット（牽引姿勢の文脈で）:**
過度に巻き取りがちだった → 仕組みで歯止め（詳細: `resilire-freee.md` の「弱みを教えてください」）
