# 面接対応: ギフトカード BTC キャッシュバック（Coinmiles）

**期間:** 〜2022年6月リリース  
**役割:** バックエンドエンジニア（バックエンド・インフラをほぼ 1名で担当）

---

## 30秒イントロ

ユーザーがアプリでギフトカードを購入すると BTC キャッシュバックが付与される機能を設計・実装した。「バックエンド・インフラをほぼ 1名で担当」という制約の中で 4つの問題を同時に解決する必要があった。

1. クレジットカード情報が自社サーバーを通ると PCI DSS スコープに入る
2. 決済とキャッシュバック付与を同一トランザクションで処理すると状態が曖昧になる
3. ギフトカードプロバイダーの API キーが 30日で期限切れになるため、うまく立ち回る仕組みが必要だった
4. 複雑な設計は自分しか保守できなくなる

---

## 核心となる設計判断

### 1. Stripe Checkout ではなく Payment Intent を使った

**問題の発端:** 最初のプロトタイプは Stripe Checkout を使っていた。実装が 3時間で終わり PCI 準拠も自動クリアできた。しかしレビューで 2つの致命的な問題が発覚した。

1つ目は **React Native モバイルアプリとの相性**。Stripe Checkout はウェブブラウザへのリダイレクト方式のため、アプリから外部ブラウザに飛んだ後に戻れなくなる。モバイルアプリでは使えない。

2つ目は **webhook の信頼性**。決済完了後に Stripe が webhook で通知する方式のため、webhook が届かなかった場合（ネットワーク障害・Stripe 側の障害）に「決済は完了したがシステムが知らない」状態になる。ギフトカードプロバイダーへの購入リクエストが送られないまま、ユーザーの画面には成功と表示される可能性があった。

**Payment Intent を選んだ理由:**
Stripe SDK がアプリ内で決済を完結させるためリダイレクト不要（React Native フレンドリー）。クライアントは SDK でカードをトークン化し、`paymentMethodId` のみを自社サーバーに送る（PCI スコープ外）。サーバー側は Payment Intent を作成し、状態を DynamoDB に記録してから SQS に投入する。決済完了と下流処理の分離が設計上保証される。

---

### 2. 決済とキャッシュバック付与を SQS で非同期分離した

**問題:** 決済とキャッシュバックを同一トランザクションで処理すると、どちらかが失敗した際に状態が曖昧になる（二重請求・付与漏れ）。

**設計:**
- Lambda（決済）: Payment Intent 作成 → DynamoDB に pending 書き込み → SQS 投入 → クライアントに 200 を返す
- Lambda（キャッシュバックワーカー）: SQS から取得 → DynamoDB で冪等チェック → ギフトカード API 呼び出し → BTC 付与 → completed に更新

SQS FIFO キューの `MessageDeduplicationId` に `payment_intent_id` を使い、重複投入を防いだ。ワーカー側も DynamoDB の ConditionalUpdate で `pending → processing` の状態遷移を原子的に行い、二重処理を防いだ。

5回リトライ後も失敗した場合は DLQ に移動し、CloudWatch Alarm → PagerDuty で即座に通知。DLQ 深度ゼロを SLO として定義した。

---

### 3. API キー失効問題を EventBridge で根本解決した

**経緯:** ギフトカードプロバイダーの API キーは 30日で期限切れになる。「毎月手動でローテーションする」では属人的な運用になり、見落としのリスクが残る。構造的に解決する必要があった。

**解決策:** EventBridge で毎月 25日に自動ローテーションを実行。Lambda がプロバイダーの API を呼んで新しいキーを取得し、Secrets Manager を更新する。Lambda 本体は起動時に Secrets Manager から最新のキーを取得する（コールドスタート時のみ）。人間が期限を追う必要を設計で排除した。

---

### 4. Strategy パターンでプロバイダー差し替えを可能にした

CEO から「より安いギフトカードプロバイダーが現れたらすぐ対応できるようにしてほしい」という要望があった。`GiftCardProvider` インターフェースを定義し、プロバイダーごとに実装クラスを持つ Strategy パターンを採用した。コア決済ロジックはインターフェースにのみ依存するため、新プロバイダーへの差し替えは実装クラスを追加するだけで済む。

依存関係を明確にすることで、コードを読んだ人がプロバイダー実装の場所をすぐ特定できる構造にした。

## よく聞かれる質問と回答

**Q: バックエンド・インフラをほぼ 1名で担当する状況でどう判断を下したか？**

設計ドキュメントを書いて CTO（当時は別の役割で在籍）に非同期でレビューしてもらった。「なぜ Stripe Checkout ではないのか」「SQS を使う理由は何か」を文書に書くことで、自分の判断の抜けを発見できた。1名でも「自分が未来の自分に説明する」ドキュメントを書く習慣で品質を担保した。

**Q: 冪等性の設計で一番気をつけたことは？**

二重付与のリスク。SQS の `MessageDeduplicationId` と DynamoDB の ConditionalUpdate の 2層で防いだ。SQS 側の重複排除だけに頼らず、ワーカー側でも状態遷移を原子的に行うことで「SQS が重複メッセージを配信した場合」にも対応した。

**Q: DLQ の運用方針は？**

DLQ 深度ゼロを SLO として設定し、1件でも入ったら PagerDuty に飛ぶ設定にした。DLQ のメッセージは自動では再処理しない方針にした。理由は「なぜ 5回リトライが全て失敗したか」を人間が確認してから再処理すべき問題だから。自動再処理は問題を隠蔽するリスクがある。

**Q: PCI DSS にどう対処したか？**

カードデータ（番号・CVV）を自社サーバーに送らないことで、PCI DSS のスコープ対象から外れる設計にした。クライアントの Stripe SDK がデバイス上でカードをトークン化し、`paymentMethodId` のみを送る。自社サーバーは `paymentMethodId` を受け取って Payment Intent を作成するだけ。

## 深掘り質問への準備

### リリース後、想定外のことはありましたか？

ギフトカードプロバイダー側の一時障害で API が 503 を返し続け、5回リトライ後に DLQ に移動した。PagerDuty で即座に検知し、プロバイダーの復旧を確認してから手動で再処理した。二重付与せずに復旧できた。「DLQ に入ったら自動再処理しない」という設計判断が正しかったと確信した瞬間だった。

---

### 今やり直すなら何を変えますか？

1つ目は **Strategy パターンのインターフェースをテストで明文化すること**。CEO の要望でプロバイダー差し替えを前提に `GiftCardProvider` インターフェースを設計したが、そのインターフェースの契約をテストで担保していなかった。新プロバイダーの実装クラスが正しくインターフェースを満たしているかはコードレビューだけに頼っていた。契約テストがあれば差し替えの安全性を機械的に検証できた。

2つ目は **SLO の監視をリリースと同時に入れること**。「決済成功率 ≥ 98%」という SLO を定義したが、監視の Datadog ダッシュボードは後から整備した。SLO を定義するなら、リリース当日から計測できる状態にすべきだった。最初の数週間はアラートが正しく設定されているか自信が持てない状態だった。
