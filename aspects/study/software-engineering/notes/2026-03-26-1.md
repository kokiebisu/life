# 勉強 - 2026-03-26

## 🎯 今日の目標・疑問

- 単一KVと分散KVの違いを理解する
- パーティション決定・レプリケーション・フェイルオーバーの仕組み
- CAP定理の適用範囲
- クォーラム（W+R>N）の理解
- 具体例で単一 vs 分散を判断できるようになる

## 📝 ノート

### 単一 vs 分散KV Store

- 単一: 1台。低レイテンシ・SPOF。スケールアップのみ
- 分散: 複数ノードにシャーディング。水平スケール可能。CAP定理が絡む

### パーティション決定 — Coordinatorノードが担う

- クライアントはどのノードに接続してもよい → そのノードがCoordinatorになる
- CoordinatorがConsistent Hashingでパーティションを決定して転送

### Master/Slaveの概念

| モデル | 代表 | 特徴 |
|--------|------|------|
| Leaderless | Cassandra, DynamoDB | どのノードにも書ける。クォーラムで整合性 |
| Leader-based | Redis Cluster | シャードごとにLeader1台。Leaderにだけ書く |

### Replication & Failover

- Leaderless: フェイルオーバー不要。残りノードで継続。復帰時はヒントハンドオフで追いつく
- Leader-based: LeaderがダウンするとFollowerが選挙で昇格（RDBと同じ流れ）

### CAP定理

- **分散システム全般**に当てはまる（KV Storeだけでなく分散RDBも対象）
- スタンドアロンRDBは分散でないので対象外
- P（Partition tolerance）は実質必須 → CP か AP の選択
- CP: HBase, ZooKeeper / AP: Cassandra, DynamoDB

### クォーラム（W + R > N）

- N=レプリカ数, W=書き込み成功必要数, R=読み込み必要数
- 例: N=3, W=2, R=2 → W+R=4 > 3 → 強整合
- 例: N=3, W=1, R=1 → W+R=2 ≤ 3 → 結果整合（速いが古い値返す可能性あり）
- 直感: 「古いノード数 = N-W」。R > N-W なら古いノードだけ引き当てることが不可能
  → W=2なら最新が2台。残り1台が古い。R=2なら必ず最新の1台を読む
- R>=2で複数ノードから読んだとき: バージョン番号（タイムスタンプ）で比較し、大きい方を採用
- W=3（全台合意）: ネットワーク分断中は書き込み不可。タイムアウトでエラー → これがCPの動き
- コンセンサス = 「W台が成功を返したら書き込み完了とみなす」合意の仕組み
- W+R ≤ N: 古いノードだけを引き当てる可能性がある（整合性保証なし）
- 設定の使い分け: W=3,R=1（書き遅・読速）/ W=1,R=3（書き速・読遅）/ W=2,R=2（バランス）/ W=1,R=1（キャッシュ用・整合性なし）

### CP / APの正しい理解

- CP = 「古いデータは絶対返さないが、全ノードに伝わらない場合はエラー」
- AP = 「常に何かを返す（古いデータかもしれない）。分断解消後に整合性を合わせる」
- ※「CPが古いレスポンス」は逆。CPは古いデータを返さない代わりに応答できないことがある

### Coordinatorの補足

- 両モデルに存在するが役割が違う
  - Leaderless: CoordinatorがW台全てに並行書き込みを行う（ボトルネックなし）
  - Leader-based: Coordinatorはキーに対応するLeaderを特定して転送するだけ
- 接続はラウンドロビン or 最寄り優先。リクエストごとに変わることもある

### 単一 vs 分散の判断基準

| ケース | 判断 | 理由 |
|--------|------|------|
| セッション管理（100万ユーザー、10万QPS） | 分散 | QPS・冗長性が必要 |
| 設定値キャッシュ（数MB、低QPS） | 単一 | データ量・QPSが小さい |
| 在庫カウンター（強整合必須） | CP系分散 or RDB | APは不可。過剰販売が致命的 |

### Read/Write/Memory トレードオフ設計

| ケース | 要件 | 設計方針 |
|--------|------|---------|
| Read Heavy（SNSプロフィール） | 読100万/s, 書1000/s | N増やす, R=1で高速読み込み, 結果整合OK |
| Write Heavy（IoTセンサー） | 書10万/s, 読少ない | W=1, LSM-Tree（Cassandra/RocksDB）選ぶ |
| Memory Heavy（動画メタデータ1TB） | データ大 | 20%ホットデータをメモリキャッシュ, TTL設定 |

- **B-Tree**: 読み取り速い・書き込み遅い（RDB, Redis）
- **LSM-Tree**: 書き込み速い・読み取りやや遅い（Cassandra, RocksDB）
- ホットデータはパレート則（20%のデータがリクエストの80%を担う）
- Redis の INCR はアトミックで競合なし → カウンター系に最適

### 独立KV Store vs アプリ内ハッシュテーブル

| 観点 | ハッシュテーブル | KV Store（Redis） |
|------|----------------|-----------------|
| スコープ | プロセス内のみ | 全サーバーから共有 |
| 永続性 | 再起動で消える | RDB/AOFで永続化可 |
| メモリ | アプリのヒープを圧迫 | 独立プロセスで影響なし |
| 競合 | lock実装が必要 | シングルスレッドでアトミック |
| TTL | 自前実装 | EXPIRE コマンドで標準対応 |

- 水平スケール時はハッシュテーブルは使えない（サーバー間で共有不可）
- 2層キャッシュ: アプリ内HashMap(L1, 0.1μs) → Redis(L2, 1ms) → DB
- 別プロセス（同居）vs 別サーバー: どちらもKV Store。レイテンシと可用性が違うだけ

### メモリの種類

- Stack: 関数ローカル変数。関数終了で自動解放。容量小（数MB）
- Heap: newで確保するオブジェクト。HashMap等はここ。GCが管理
- GCプレッシャー: ヒープに大量データ → GCがStop-the-Worldで一瞬アプリが止まる → Redisに外出しで回避

### KV Storeの用途整理

- セッション → 水平スケール対応のため（race conditionとは無関係）
- カウンター → INCR がシングルスレッドでアトミック → race condition回避
- ロック → SETNX（Set if Not eXists）でdistributed lock → race condition回避
- 判断基準: 「リクエストが終わっても残すべきデータ」→ KV Store

### 不整合の解消：バージョン管理

- **書き込み遅延**: バージョン番号を比較して大きい方を採用（Last Write Wins / LWW）
- **LWWの問題**: サーバーの時計がズレると間違った方が勝つ
- **同時書き込み（Conflict）**: バージョン番号では解決不可。Vector Clockが必要

#### Vector Clock
- 形式: `{NodeA: 回数, NodeB: 回数}`
- 「どのノードの書き込みを知った上で書いたか」を追跡
- タイムスタンプが信用できない理由: サーバーの時計がズレると因果関係が壊れる
- Vector Clockの記録: 「自分が書いた時点で各ノードの何回目まで知っていたか」
- NodeAが書く: `{A:1}` → NodeBがAを読んでから書く: `{A:1, B:1}`（Aを知った上で書いた）
- 包含判定: YのすべてがX以上なら「YはXを知った上で書いた」→ Yが後 → Xを捨ててOK
- `{A:1}` vs `{B:1}` → 互いを知らない = CONFLICT（競合）
- `{A:1}` vs `{A:1, B:1}` → 後者がAの情報を含む → 後者が「後」とわかる → 前者を捨ててOK

#### 競合解決の3戦略
| 戦略 | 説明 | 向いている用途 |
|------|------|--------------|
| LWW（Cassandraデフォルト） | タイムスタンプが新しい方が勝つ | ログ・センサー |
| クライアント解決（DynamoDB） | 両方の値をアプリに返してビジネスロジックで解決 | 在庫・金融 |
| マージ | ユニオンなどで両方を合成。削除はTombstoneで表現 | ショッピングカート |

## 🔑 キーワード

- Coordinator（コーディネーター）: パーティション決定ノード
- マルチキャスト: 1人が全員に送信。速いがスケールしない（送信元に負荷集中）
- ゴシッププロトコル: ランダムな隣人にうわさを伝播。O(log N)で全ノードへ届く。スケーラブル・耐障害性高い
- ハートビート: 各ノードが定期的に「生きてる」カウンターを送信。止まったら障害と判定
- Consistent Hashing（一貫性ハッシュ）: ノード追加時の再配置最小化
- Leaderless / Leader-based: レプリケーションモデルの2種類
- ヒントハンドオフ: 障害ノード復帰後のデータ追いつき
- CAP定理: Consistency / Availability / Partition tolerance
- クォーラム: W + R > N で強整合性を保証
- 結果整合性（Eventual Consistency）: 遅延はあるがいずれ一致

## 💡 まとめ（絶対知っておくべき内容）

### 1. 単一 vs 分散の判断

- データ量・QPS・可用性要件が小さければ単一でOK
- 水平スケール・複数サーバー間共有が必要になったら分散
- **セッションは水平スケールのためにKV Storeへ（race conditionとは無関係）**

### 2. CAP定理

- 分散システム全般に当てはまる（KV Storeだけでなく分散RDBも対象）
- P（Partition tolerance）は実質必須 → **CP か AP の選択**
- CP = 古いデータを返さない代わりに、分断中はエラーを返す
- AP = 常に何かを返す。古いデータかもしれない。後で整合性を合わせる

### 3. クォーラム（W + R > N）

- W=書き込み成功必要数、R=読み込み台数、N=レプリカ数
- **古いノード数 = N-W。R > N-W なら古いノードだけ引き当てることが不可能**
- W=2,R=2,N=3 がバランス型。W=1,R=1 はキャッシュ用（整合性なし）
- 複数ノードから読むときはバージョン番号で最新を判断

### 4. Leaderless vs Leader-based

- Leaderless（Cassandra）: どのノードにも書ける。Coordinatorが並行書き込み
- Leader-based（Redis Cluster）: シャードごとにLeader。フェイルオーバーはFollowerが昇格
- Coordinator は両モデルに存在。ランダムではなく接続したノードがなる

### 5. 障害検知：ゴシッププロトコル

- 毎秒全ノードがランダムな2〜3台にハートビートテーブルを送り合う
- カウンターが一定時間止まったノードを障害と判定
- 全ノードが独立して検知。中央管理者不要。O(log N)で伝播

### 6. 不整合の解消

- **LWW**: タイムスタンプが新しい方が勝つ。時計ズレが問題
- **Vector Clock**: 「誰の書き込みを知った上で書いたか」を追跡。競合検出に使う
- 競合解決: LWW（センサー）/ クライアント解決（在庫）/ マージ（カート）
